## 项目一：量化算法对比实验

### 目标
对同一个模型，实现并对比至少 3 种量化方法，撰写量化报告。

### 场景设定
你是某 AI 公司的推理优化工程师。CTO 要求你在下周技术评审会上给出明确的量化方案推荐。需要对同一模型跑通多种量化方法，用数据说话。

### 任务分解

#### Step 1：模型准备（30 min）
- 选择一个预训练语言模型（推荐：Qwen2.5-0.5B 或 GPT-2 small）
- 加载模型并记录基线性能（参数量、模型大小、推理速度、Perplexity）
- 准备 128-256 条校准数据

#### Step 2：实现 3 种量化方法（2-3h）
必须包含：
1. **对称逐张量 INT8 量化**（最基础，作为下界）
2. **逐组 INT4 量化**（group_size=128，中等水平）
3. **AWQ 风格 INT4 量化**（激活感知，参考课程 9.1 的实现）

可选加分：
4. SmoothQuant 风格 W8A8
5. 使用 GPTQ 官方库

#### Step 3：系统化评估（1-2h）
对每种量化方法，测量并记录：

| 指标 | 测量方法 |
|------|---------|
| 模型大小 | 理论压缩比 + 实际文件大小（如导出 GGUF） |
| 逐层权重余弦相似度 | 各层的量化前后余弦相似度，找最敏感的层 |
| Perplexity（WikiText-2 或自定义测试集） | 量化前后的 PPL 变化 |
| 输出 Token 分布一致性 | 同一输入下，量化前后 Top-5/10 token 的重叠率 |
| 推理速度 | Prefill（256/512 tokens）和 Decode（单 token）的延迟 |
| 内存峰值 | 推理时的最大内存占用 |

#### Step 4：撰写量化选型报告（1h）

报告结构：
```
1. 实验环境（硬件、PyTorch版本、模型信息）
2. 方法概述（每种量化方法的原理一句话 + 关键参数）
3. 精度对比（表格 + 逐层余弦相似度折线图）
4. 性能对比（延迟、内存、吞吐的柱状图）
5. 综合分析（精度-性能的帕累托前沿，找最优 trade-off）
6. 推荐方案（针对不同场景的量化推荐）
7. 附录：复现步骤
```

### 交付物
- 一个 Jupyter Notebook（包含完整实验代码和分析）
- 一份量化选型报告（Markdown/PDF）

### 自评检查清单
- [ ] 至少实现了 3 种不同的量化方法
- [ ] 所有量化方法的逐层余弦相似度都 > 0.95
- [ ] 包含精度、速度、内存三个维度的对比
- [ ] 推荐方案有明确的适用场景说明
- [ ] 代码可以完整复现（从头跑到尾不出错）
- [ ] 图表清晰，有轴标签和图例

### 提示
- 如果模型太小（<50M 参数），量化带来的性能差异可能不明显——可以用更大的模型或更多层来放大效果
- 逐层余弦相似度要找最敏感的层：通常第一层和最后一层对量化最敏感
- PPL 不是唯一精度指标——输出 token 分布一致性有时更能反映用户体感

## 项目二：设计一个端侧友好的轻量级架构

### 目标
从零设计一个小型语言模型架构，专门为端侧推理优化，并对比标准架构的性能。

### 场景设定
你的团队在开发一款离线翻译 App，目标硬件是 2 年前的安卓中端机（骁龙 7 Gen1，4GB 可用内存）。你需要设计一个模型架构，在 500M-1B 参数量下，最大化推理速度和翻译质量。

### 任务分解

#### Step 1：基线架构搭建（1-2h）
- 实现一个标准 GPT-2 风格的小型 Transformer（约 500M 参数）
- 包含标准 MHA（8-12 头）、GELU 激活、LayerNorm
- 在 CPU 上测量基线性能

#### Step 2：逐个应用优化策略（3-4h）
按以下顺序逐步优化，每步都记录变化：

**优化 A：GQA 替换 MHA**
- 从 MHA（12 头）改为 GQA（4 组 KV，12 个 Q 头）
- 测量 KV Cache 大小变化和推理速度变化

**优化 B：ReLU² 替换 GELU**
- 标准 GELU 包含 erf 计算（需要查表或近似），端侧 NPU 可能不支持
- ReLU² = ReLU(x)² 同样有非线性，但仅需乘法和比较
- 对比两者的推理速度和精度差异

**优化 C：RMSNorm 替换 LayerNorm**
- LayerNorm 需要计算均值和方差（两次遍历）
- RMSNorm 只需要均方根（一次遍历），计算量减半
- 对比精度损失

**优化 D：Weight Tying**
- 输入 Embedding 和输出 LM Head 共享权重
- 测量参数量节省

**优化 E：（加分项）尝试更深更窄的设计**
- 参考 MobileLLM 的思路，在相同参数量下尝试不同的深度×宽度比例
- 如：16层×256维 vs 8层×512维

#### Step 3：最终架构对比（1-2h）
将优化后的架构与基线进行全维度对比：

| 维度 | 基线架构 | 优化架构 | 变化 |
|------|---------|---------|------|
| 参数量 | | | |
| KV Cache 大小（seq_len=1024） | | | |
| Prefill 延迟（512 tokens） | | | |
| Decode 延迟（单 token） | | | |
| Perplexity（测试集） | | | |
| 端侧内存占用估算 | | | |

#### Step 4：架构设计文档（1-2h）
撰写设计文档，解释每个优化决策的权衡和理由。

### 交付物
- 一个 Jupyter Notebook（包含基线架构和优化架构的完整实现 + 对比实验）
- 一份架构设计文档（说明每个选择的理由）

### 自评检查清单
- [ ] 包含了至少 4 项架构优化（GQA、激活函数、归一化层、权重共享等）
- [ ] 每项优化都有量化的前后对比数据
- [ ] 最终架构的 KV Cache 比基线减少 50% 以上
- [ ] 推理速度有可测量的提升（>15%）
- [ ] 精度损失 < 5%（PPL 或下游任务）
- [ ] 文档中解释了每项优化的"为什么"而不是"做了什么"

## 项目三：硬件选型决策模拟

### 目标
基于课程第 5 章的硬件知识和 benchmark 数据，为一个具体的业务场景做完整的硬件选型分析。

### 场景设定
你是一家 AI 创业公司的技术负责人，正在做一款"智能会议录音笔"产品。产品需求：
- 实时录音转文字 + 要点摘要（运行 1.5B-3B 参数的语言模型）
- 电池续航 ≥ 8 小时（持续使用）
- 设备成本（芯片部分）< $50
- 首 token 延迟 < 500ms
- 离线运行（不依赖网络）

### 任务分解

#### Step 1：候选硬件调研（1-2h）
基于课程第 5 章和第 10 章的知识，列出至少 4 个候选硬件平台：

建议覆盖：
- 高通骁龙系列（如 7 Gen3、8 Gen2）
- 联发科天玑系列
- 瑞芯微 RK3588
- 地平线/华为昇腾边缘芯片
- Apple Silicon（如果产品支持 iOS）

对每个候选硬件，收集：
- 标称 NPU 算力（TOPS）
- 内存带宽
- 可用内存（芯片支持的最大 RAM）
- TDP（热设计功耗）
- 芯片批量采购价格
- SDK 成熟度和 LLM 算子支持度
- 实际 LLM 推理性能（如果能找到公开 benchmark）

#### Step 2：Roofline 分析（1-2h）
利用课程 5.3/5.4 的 Roofline 分析工具：
- 为每个候选硬件绘制 Roofline 图
- 标注目标模型（1.5B 和 3B）关键算子的位置
- 计算每个硬件上目标模型的估算性能

#### Step 3：五维评估（1-2h）
使用课程 5.4 的五维框架做量化评分：

| 维度 | 权重 | 候选1 | 候选2 | 候选3 | 候选4 |

| 性能（TTFT + 吞吐） | 30% | | | | |
| 内存（是否够跑模型） | 25% | | | | |
| 功耗（续航要求） | 25% | | | | |
| 生态（SDK+算子支持） | 10% | | | | |
| 成本（芯片价格） | 10% | | | | |
| **加权总分** | | | | | |

#### Step 4：敏感性分析（30 min）
- 如果续航要求从 8h 放宽到 4h，推荐会变吗？
- 如果模型从 1.5B 升级到 3B，推荐会变吗？
- 如果成本上限从 $50 提到 $80，推荐会变吗？

#### Step 5：选型报告（1h）
撰写正式的技术选型报告，包含：
1. 业务需求概述
2. 候选方案对比
3. 五维评估打分
4. 最终推荐及理由
5. 风险与备选方案
6. 建议的下一步（原型验证计划）

### 交付物
- 一份硬件选型分析报告（Markdown/PDF）
- 可选：Roofline 分析代码和图表

### 自评检查清单
- [ ] 至少评估了 4 个硬件候选方案
- [ ] 每个候选都有可量化的数据支撑（不是"感觉"）

## 项目四：端到端部署流水线实战

### 目标
从 HuggingFace 模型出发，完成量化 → 导出 → 推理 → 基准测试 → 故障排查的完整链路。

### 场景设定
你需要将一个开源模型部署到端侧设备。模型已经选定，量化方案需要你设计，部署后需要证明精度和性能达标。

### 任务分解

#### Step 1：模型选型与基线建立（1h）
- 从 HuggingFace 选择一个 1B-3B 参数的模型（推荐：Qwen2.5-1.5B 或 SmolLM2-1.7B）
- 加载模型，跑通基线推理
- 记录基线 Perplexity 和推理速度

#### Step 2：量化方案设计（2-3h）
你需要自己决定量化策略，而不是照搬课程代码。至少回答以下问题：
- 用什么量化算法？为什么？（考虑目标硬件的精度支持）
- W 多少 bit？A 多少 bit？为什么？
- group_size 选多少？为什么？
- 需要校准数据吗？多少条？
- 不同层需要不同精度吗？（混合精度）

实现你选择的量化方案并验证精度。

#### Step 3：模型导出（1-2h）
选择一条导出路径并实现：
- **GPU 路径**：导出 GGUF（使用 llama.cpp 的 convert 脚本或自己实现 GGUF 格式写入）
- **ONNX 路径**：导出 ONNX → 用 ONNX Runtime 推理验证
- 如果硬件条件允许，可以两条路径都做

#### Step 4：基准测试（1-2h）
设计并执行完整的基准测试方案：
- Prefill 阶段：测试不同输入长度（128, 256, 512, 1024）的 TTFT
- Decode 阶段：测试 ITL 均值和 P95/P99
- 内存：测试不同 seq_len 下的峰值内存
- 画 TTFT-seq_len 曲线和内存-seq_len 曲线

#### Step 5：故障排查模拟（1-2h）
人为制造 2 个故障并演示排查过程。故障可以从以下选择：
1. **量化精度异常**：故意对某一层用过大的 group_size，导致精度异常，然后演示如何通过逐层余弦相似度定位问题层
2. **内存模拟泄漏**：模拟 KV Cache 没有被正确释放，演示内存增长曲线和排查方法
3. **算子不兼容**：模拟 ONNX 导出时某个自定义算子导致导出失败，演示如何分解算子

#### Step 6：部署报告（2h）
撰写正式的部署报告：

```
1. 项目概述
   - 目标模型、目标硬件、使用场景
2. 量化方案设计
   - 方案选择的逻辑链
   - 量化参数配置及理由
3. 精度验证
   - 逐层余弦相似度
   - PPL 对比
   - 输出一致性
4. 性能基准
   - TTFT 曲线
   - ITL 统计
   - 内存占用
5. 故障排查记录（模拟）
6. 部署检查清单验证
7. 结论与后续优化建议
```

### 交付物
- 一个 Jupyter Notebook（完整流水线代码）
- 部署报告（Markdown/PDF）
- 量化后的模型文件（或模型导出结果截图）

### 自评检查清单
- [ ] 量化方案有明确的选择理由（不是照抄课程）
- [ ] 逐层余弦相似度 > 0.99（至少 90% 的层）
- [ ] PPL 变化 < 5%
- [ ] 包含了 TTFT 和 ITL 的多场景基准测试
- [ ] 模拟了至少 2 个故障并演示了排查过程
- [ ] 报告专业、可复现、结构清晰

## 项目五：多平台部署方案的完整设计与对比

### 目标
针对同一个模型，设计 3 条不同的部署路径（CPU/GPU/NPU），对比它们的优劣势，给出最终推荐。

### 场景设定
你的产品需要在以下 3 类设备上运行同一个 AI 功能：

| 设备类型 | 代表硬件 | 约束 |
|---------|---------|------|
| 旗舰手机 | 骁龙 8 Elite（Hexagon NPU + Adreno GPU） | 可用内存 5GB，需利用 NPU |
| 中端手机 | 骁龙 7 Gen3（仅有 CPU + 小 GPU） | 可用内存 3GB，NPU 不够强 |
| 桌面/笔记本 | Intel i7 + RTX 4060 / Apple M 系列 | 可用内存 8GB+，按需选择 |

你需要为每种设备设计最优部署方案，并给出跨平台的统一推荐。

### 任务分解

#### Step 1：统一模型选择（1h）
选择一个 3B 左右参数的模型（如 Qwen2.5-3B），确保它能在这 3 类设备上都跑起来。

#### Step 2：路径 A — CPU 部署方案（llama.cpp）（2-3h）
- 选择 GGUF 量化格式（Q4_K_M / Q5_K_M / Q8_0 等）
- 使用 llama.cpp 或 llama-cpp-python 做推理
- 测量不同量化格式的精度-速度 trade-off
- 分析：哪些 K-Quant 变体（Q4_K_S vs Q4_K_M）适合什么场景？

#### Step 3：路径 B — GPU 部署方案（2-3h）
选择以下之一：
- **NVIDIA**：TensorRT-LLM 或 vLLM + AWQ 量化
- **Apple Silicon**：MLC-LLM 或 Core ML 转换
- **通用 GPU**：ONNX Runtime + CUDA/ROCm EP

设计并测试 GPU 推理方案的性能。

#### Step 4：路径 C — NPU 部署方案（模拟）（2-3h）
由于大多数人没有 NPU 开发板，本项目用模拟方式完成：
- 调研目标 NPU（如高通 QNN）的算子兼容性列表
- 对模型做算子兼容性检查（哪些算子不支持？需要分解？）
- 设计算子分解方案（不能直接跑的算子怎么拆？）
- 撰写 NPU 部署方案文档（如果真实硬件可用，加分）
- 估算 NPU 推理性能（基于课程 5.1 的参考数据）

#### Step 5：跨平台对比分析（2-3h）

| 维度 | CPU (llama.cpp) | GPU (TensorRT/MLC) | NPU (QNN模拟) |
|------|:---:|:---:|:---:|
| 模型体积 | | | |
| TTFT (512 tokens) | | | |
| ITL | | | |
| 峰值内存 | | | |
| 功耗估算 | | | |
| 开发复杂度 | | | |
| 可维护性 | | | |
| 跨平台兼容性 | | | |

#### Step 6：最终推荐与实施路线图（1-2h）
给产品经理写一份推荐方案：
- 如果只选一条路径，推荐哪条？为什么？
- 如果预算允许做两条路径，优先级怎么排？
- 3 个月的实施路线图（里程碑、风险、依赖）

### 交付物
- 3 个独立的部署方案文档（CPU/GPU/NPU 各一份）
- 一份跨平台对比分析报告
- 一份最终推荐方案（面向非技术决策者）

### 自评检查清单
- [ ] 3 条路径都有可运行或可验证的方案
- [ ] 每个方案都有 quantifiable 的性能数据
- [ ] 跨平台对比表格完整
- [ ] 最终推荐有清晰的优先级和路线图
- [ ] NPU 方案（即使模拟的）有严肃的算子兼容性分析
- [ ] 至少包含一个"遇到问题→排查→解决"的完整记录

3. **真实硬件实战**：参考硬件实操路线图，在真实设备上跑通项目四和项目五
4. **追踪前沿**：关注 ICML/NeurIPS/MLSys 的端侧部署相关论文，尝试复现
