## 第1章 模型压缩

### 思考题 1：为什么 SmoothQuant 将激活的量化难度"迁移"到权重？这种迁移的代价是什么？

**核心原因：激活值的离群通道（outlier channels）是量化的主要障碍。**

在 LLM 中，激活值存在一个普遍现象：少数几个通道（通常不到 1%）的数值范围远大于其他通道（可达 10-100 倍）。这些离群通道导致：
- 如果按全局 max 确定 scale，99% 的通道会被压缩到极少量化级别中，精度大幅损失
- 如果逐通道量化激活，虽能解决离群值问题，但硬件不支持——因为激活量化后需要与量化后的权重做矩阵乘法，逐通道量化的激活无法直接与量化权重相乘

**SmoothQuant 的解决方案：**

引入平滑因子 s_j = max(|X_j|)^α / max(|W_j|)^(1-α)，其中 α 通常取 0.5：
- 激活除以 s_j：将离群通道的激活值"压平"，使激活值分布更均匀，易于量化
- 权重乘以 s_j：将权重的对应通道放大，但权重本身没有离群值问题，放大后仍可均匀量化

等价性保证：XW = (X · diag(s)^{-1}) · (diag(s) · W)，数学上完全等价。

**代价：**

1. **权重通道的数值范围增大**：被放大的权重通道可能有更大的 scale，导致该通道的量化步长变大，引入额外的舍入误差。好在这种误差通常远小于直接量化离群激活的误差。

2. **需要校准数据来确定 α**：不同模型的最优 α 不同，需要在小规模校准数据上搜索。

3. **迁移不完美**：对于激活离群值和权重离群值重叠的通道，s_j 在两边都会产生误差，这是内在的精度上限。

4. **仅适用于 W8A8**：SmoothQuant 目标是实现权重和激活都量化为 8bit。如果要进一步到 W4A8 或 W4A4，离群值问题会重新出现，需要 AWQ 等激活感知方法补充。

### 思考题 2：AWQ 和 GPTQ 都利用校准数据，它们的根本区别是什么？为什么 AWQ 不需要 Hessian 逆而 GPTQ 需要？

**根本区别：两者解决问题的思路完全不同。**

| 维度 | GPTQ | AWQ |
|------|------|-----|
| **核心思路** | 误差补偿：逐列量化后立即修正其余列的权重，补偿已造成的量化误差 | 误差预防：在量化前通过缩放保护重要通道，减少量化时产生的误差 |
| **理论基础** | OBQ (Optimal Brain Quantization)：基于二阶泰勒展开，用 Hessian 逆计算最优补偿 | 激活感知：基于激活值的统计分布，识别对输出影响最大的权重通道 |
| **为什么需要/不需要 Hessian** | 需要 Hessian 逆来计算每量化一列后对剩余列的最优修正量。ΔW = -(w_q - w)/[H^{-1}]_{ii} · H^{-1}_{:,i} | 不需要。AWQ 只关心"哪些通道更重要"，通过对激活值取绝对值的最大值来识别 salient channels，这是统计量而非优化量 |

**GPTQ 的 Hessian 依赖——细节：**

GPTQ 将量化问题建模为：在已量化部分权重固定的约束下，通过调整未量化权重最小化输出误差。这等价于求解一个带约束的二阶优化问题。Hessian 矩阵（或 Fisher 信息矩阵的近似）表征了每个权重对损失函数曲率的影响。关键公式：

对于要量化的第 i 列权重 w_i，量化误差为 δ_i = w_i^q - w_i。为补偿此误差，需对剩余列 j > i 做修正：

δ_j = -δ_i · [H^{-1}]_{i,j} / [H^{-1}]_{i,i}

这直接使用了 Hessian 逆的元素。GPTQ 不需要完整的 Hessian 逆矩阵，而是利用 Cholesky 分解逐步计算——每量化一列就更新一次，复杂度为 O(d_col³)。

**AWQ 不需要 Hessian 的原因：**

AWQ 的哲学是"重要的权重不要量化得太狠"，而非"量化错了再补偿"。它通过分析校准数据的激活值分布，找到 salient channels（对输出影响大的通道），在量化前对这些通道乘以缩放因子（放大权重使其量化步长相对变小），量化后再除回来。整个过程是前向的统计操作，不需要二阶梯度信息。

**哪个更好？**
- GPTQ 理论上精度上限更高（有最优性保证），但依赖校准数据的 Hessian 估计质量
- AWQ 更简单、更快（不需要 Hessian 计算和 Cholesky 分解），且保护 salient channels 的策略在实践中非常有效
- 两者可以互补：AWQ 先做缩放保护，GPTQ 再做误差补偿

## 第2章 高效推理架构

### 思考题 1：H2O 淘汰策略保留"最近窗口"和"历史高注意力 token"，为什么单独任何一种策略都不够？

**仅保留最近窗口（Sliding Window Attention）的问题：**

最近窗口保留了局部上下文，这对语法连贯性和最近依赖关系建模很重要。但关键问题是：

1. **长程依赖丢失**：如果用户问题是"请总结上面三段话的核心观点"，回答时需要回顾远距离的上下文。仅保留最近窗口意味着模型只能看到最后 W 个 token 的内容，之前的重要信息全部丢失。

2. **注意力汇聚效应（Attention Sink）**：研究发现，某些早期 token（尤其是 BOS token 和初始的几个 token）会持续获得高注意力分数，即使它们在语义上已经"过时"。仅保留最近窗口会让这些被大量 token 关注的"注意力汇聚点"丢失，破坏注意力分布结构。

3. **信息检索任务失效**：当模型需要在长文档中检索某个特定信息时，该信息可能位于窗口之外，却恰好是回答问题所必需的。

**仅保留历史高注意力 token（Heavy-Hitter）的问题：**

Heavy-Hitter 策略保留历史上累积注意力分数最高的 token。但：

1. **局部语法结构断裂**：生成的最近几个 token 对下一个 token 的预测有最强的直接依赖（语法搭配、代词指代等）。如果这些 token 因为注意力分数还不够高而被淘汰，会导致语法混乱和连贯性下降。

2. **注意力分数的时间滞后**：一个 token 的注意力分数需要经过多步推理才能累积到较高水平。刚生成的 token 几乎不可能立刻成为 heavy-hitter，但它们却是当前推理最关键的上下文。

3. **注意力汇聚 token 过度保留**：BOS token 通常是最强的 heavy-hitter，但它对大部分生成步骤的语义贡献有限。仅按注意力分数保留会导致大量 KV Cache 被语义价值低的 token 占据。

**为什么 H2O 同时保留两者才有效：**

H2O 的设计是：**最近 W 个 token（保证局部连贯性）+ 最近窗口之外历史上累积注意力最高的 token（保留长程关键信息）**。

这形成了一个互补结构：
- 最近窗口保证了对"当下"的精确建模（语法、代词、最新语义）
- Heavy-Hitter 保证了跨越长距离的关键信息不丢失（如问题的核心约束、前文的命名实体等）

视觉上，H2O 的 KV Cache 结构像一个"Φ"形：底部是最近的密集窗口，上方是稀疏但高价值的历史 token。这种结构同时保留了短程密集依赖和长程稀疏关键信息。

### 思考题 2：在计算资源极度受限的端侧，Flash Attention 是否仍然有效？为什么？

**简短回答：仍然有效，但原因和 GPU 上不同。**

**Flash Attention 在 GPU 上有效的核心原因：**

GPU 上 Flash Attention 通过分块计算（tiling）避免将 O(n²) 的注意力矩阵写入 HBM。GPU 的关键瓶颈是 HBM 带宽——SRAM 很快但很小（~20MB），HBM 很大但带宽是瓶颈。

**端侧设备的不同之处：**

端侧设备（尤其是手机 NPU 和低端 CPU）的内存层次结构完全不同：

| 硬件 | 瓶颈类型 | Flash Attention 的收益 |
|------|---------|---------------------|
| 手机 NPU（如 Hexagon） | DRAM 带宽极低（~60GB/s），SRAM 极小（~2-8MB） | **仍然有效**：O(n²) 中间结果无法放入 SRAM，必须分块。分块减少 DRAM 访问的收益在低带宽设备上**更显著**，因为每节省一次 DRAM 读写的代价更高 |
| 手机 CPU（ARM） | 内存带宽非常有限（~40GB/s），无专用 SRAM | **部分有效**：tiling 减少 cache miss 和内存访问，但由于 CPU 没有 GPU 那样的 SRAM 层次，收益来自更好的 cache 局部性而非专门的 SRAM 利用 |
| IoT 设备（Cortex-M） | 内存极小（<1MB），根本无法存储完整注意力矩阵 | **几乎必需**：不用 tiling 根本跑不动长序列。分块计算不是优化而是前提条件 |

**端侧 Flash Attention 的额外考量：**

1. **分块大小需要重新调优**：GPU 上分块大小通常以 SRAM 容量为准（如 16KB/块）。端侧 NPU 的 SRAM 可能只有 2-8MB，需要更小的块，这意味着更多次外层循环迭代，可能有额外开销。

2. **在线 softmax 的开销**：维护 m、ℓ、O 三个运行统计量并做修正因子计算，在计算能力弱的端侧 CPU 上可能成为新的瓶颈。需要权衡分块带来的 DRAM 节省 vs. 额外计算开销。

3. **NPU 算子支持度**：Flash Attention 的 tiling + online softmax 是复杂的自定义 kernel，NPU SDK（QNN、ANE）可能不直接支持。需要用底层 API 手写或等待厂商支持。

4. **与量化的联合优化**：端侧通常会对 KV Cache 做量化（如 INT8 KV Cache），此时 Flash Attention 的 tiling 需要与反量化操作融合，否则 tiling 带来的带宽节省会被反量化的额外读写抵消。

**总结：Flash Attention 在端侧不仅有效，而且其核心思想（分块计算避免 O(n²) 中间结果）在端侧比在 GPU 上更加重要——因为端侧的 O(n²) 绝对数字虽然小（n 通常 < 4096），但相对可用内存的比例可能更大。只是实现形式需要针对端侧硬件重新设计。**

## 第3章 高效模型架构

### 思考题 1：SSM（如 Mamba）声称 O(n) 复杂度，为什么在实际端侧部署中可能不比优化的 Transformer 快？

**理论上的 O(n) vs. O(n²) 是渐近复杂度的比较。在实际端侧部署中，常数因子和硬件效率往往比渐近复杂度更重要。**

**原因 1：硬件对矩阵乘法的极致优化 vs. 对 SSM 扫描操作的弱优化**

Transformer 的核心计算是矩阵乘法（QKV 投影、Attention Score、FFN），这是过去几十年硬件优化的绝对重点：
- GPU 的 Tensor Core、NPU 的 MAC 阵列都是为矩阵乘法设计的
- 矩阵乘法在现有硬件上可以达到峰值算力的 80-90%
- cuBLAS、MKL 等库对矩阵乘法的优化经过了数十年积累

而 SSM 的核心计算是**选择性扫描（selective scan）**：每个时间步根据当前输入动态调整状态转移矩阵，然后做一次小的矩阵-向量乘法。这个操作：
- 不是标准的矩阵乘法，无法直接利用 Tensor Core / MAC 阵列的全部算力
- 硬件利用率通常只有 10-30%
- 缺乏经过数十年优化的底层库支持

结果：即使理论 FLOPS 更少，实际 wall-clock 时间可能更长。

**原因 2：序列长度不够长时，O(n²) 不是瓶颈**

端侧部署的典型序列长度：
- 对话机器人：输入 512-2048 tokens
- 文档摘要：输入 2048-4096 tokens

在这种长度下：
- Transformer 的 O(n²) 注意力矩阵大小：n=2048 时约 16MB（FP32），对于 4-6GB 的端侧内存来说可以承受
- Prefill 阶段的 O(n²) 计算可以通过 Flash Attention 大幅优化
- Decode 阶段是自回归的，每步 O(n)，且已经通过 KV Cache 减少了重复计算

SSM 的 O(n) 优势要等到 n > 8000 甚至 16000 才能明显体现，而这超出了大多数端侧应用的需求。

**原因 3：SSM 的隐状态维度与内存**

SSM（如 Mamba）需要在推理时维护隐状态（类似 RNN 的 hidden state）。对于 Mamba：
- 隐状态大小 = d_model × d_state（d_state 通常为 16）
- 对于 d_model=4096 的模型，隐状态约 256KB
- 看起来不大，但这意味着 SSM 在 decode 阶段每步都需要读取和更新这 256KB 的状态

对比：Transformer 的 decode 阶段是读取 KV Cache（随序列增长），但 GQA 架构下 KV Cache 的每步增量（1个新 token 的 KV）可能只有几十 KB。

**原因 4：Transformer 的端侧优化生态极其成熟**

- 量化方案：GPTQ/AWQ/SmoothQuant 都针对 Transformer 架构优化
- 推理框架：llama.cpp、MLC-LLM、ExecuTorch 都默认支持 Transformer
- NPU 适配：QNN/Core ML 对 Transformer 算子的支持远好于 SSM

**结论：SSM 在理论上的 O(n) 优势是真实的，但在当前硬件和端侧典型序列长度下，其硬件利用率的劣势抵消了复杂度优势。这一局面可能随着专用 SSM 硬件加速器和更长序列端侧应用的出现而改变。**

### 思考题 2：MoE 模型的总参数量可以达到大模型水平，但端侧推理时只有部分专家活跃，为什么依然难以在手机上部署 Mixtral 8x7B？

**核心矛盾：稀疏激活节省的是计算量，不是内存占用。**

虽然每次推理只激活 Top-2 专家（约 13B 参数的计算量），但：

**问题 1：所有专家权重必须驻留在内存中**

MoE 的路由决策发生在推理时——路由器看到输入 token 后，才决定激活哪些专家。这意味着：
- 所有 8 个专家（每个是标准 FFN，约 7B × 6/8 ≈ 5.25B 参数/专家）的权重都必须加载到内存中
- 即使压缩到 W4A16，8 个专家的 INT4 权重仍需约 46B × 0.5 bytes ≈ 23GB
- 旗舰手机可用内存约 6-8GB，其中能给模型的通常不超过 4-5GB
- 即使只加载专家的 FFN 部分（约占总参数的 60%），也需要约 14GB

**问题 2：专家权重的按需加载面临延迟挑战**

理论上可以将不活跃的专家放在慢存储（Flash），推理时按需加载。但这引入巨大延迟：
- 手机 UFS 4.0 存储读取速度约 4GB/s
- 加载一个 W4A16 专家（约 0.7GB）需要 ~175ms
- 每个 token 需要 2 个专家 = ~350ms 加载时间
- 加上推理计算时间，单个 token 的总延迟远超用户体验阈值（<50ms）

而且路由结果无法提前预知——你不知道下一个 token 需要哪个专家，所以无法提前预取。

**问题 3：路由器本身的随机性**

训练良好的 MoE 路由应该让不同专家处理不同类型的 token。但在实际推理中：
- 路由器的 softmax 输出是概率分布，即使 Top-2 专家占 90% 的概率，仍有 10% 概率选中其他专家
- 这意味着理论上任何一个专家都可能被选中，你不能安全地卸载任何专家
- Expert Choice routing（让专家选 token）可以强制负载均衡，但牺牲了精度

**问题 4：MoE 的辅助损失与端侧推理的矛盾**

为防止路由器坍塌（总是选同几个专家），MoE 训练时添加了负载均衡损失。这会促使专家之间有一定冗余（多个专家能处理相似的 token 类型）。冗余对于训练稳定性是好消息，但对于端侧部署：
- 如果不能安全卸载某些专家，冗余反而变成了纯存储开销
- 如果能安全卸载，说明专家确实有分工，那路由器的错误分类代价很大

**可行的端侧 MoE 方案：**

1. **小规模 MoE**：2-4 个专家，每个较小（如 1B 模型 + 4 个专家），总参数量可控制在 3-4GB INT4
2. **专家蒸馏/合并**：将 8 个相似专家蒸馏为 2-3 个，牺牲一些专业能力换取可部署性
3. **MoE + LoRA 混合**：基座模型用稠密小模型，专家用极小的 LoRA 适配器表示，按路由选择激活不同 LoRA
4. **投机预取**：用小型预测网络提前猜测下一个 token 可能需要哪些专家，在 GPU/NPU 计算当前层时异步预取下一层的专家权重

## 第4章 编译与运行时优化

### 思考题 1：为什么 NPU 上 PagedAttention 难以直接映射？有什么可能的解决方案？

**为什么难以映射：**

PagedAttention 的核心设计依赖于 GPU 的三个特性，这些在 NPU 上都不成立：

**1. NPU 不支持动态内存分配**

PagedAttention 的核心是动态管理 KV Cache 的物理块（blocks）。当新请求到来时动态分配 block，请求完成后释放 block 回 free pool。这需要：
- 运行时动态内存分配（malloc/free）
- 非连续物理地址空间的管理（page table）
- GPU 的 CUDA 编程模型原生支持这些操作（cudaMalloc、指针运算等）

而 NPU 通常是预编译的静态计算图：
- 编译期确定所有内存大小和布局
- 没有运行时的动态内存分配 API
- 所有张量必须在编译时声明固定大小

**2. NPU 不支持非连续内存的 gather/scatter**

PagedAttention 的注意力计算需要对非连续存储的 KV blocks 进行访问。在 GPU 上，这些 block 通过 page table 中的指针找到，GPU 的指针运算和灵活的 load/store 指令可以高效处理非连续访问。

NPU 的 DMA 引擎和 MAC 阵列是为连续内存访问优化的：
- DMA 引擎擅长连续的大块数据传输，非连续的小块随机访问效率极低
- 缺乏 GPU 那样的 warp-level 编程能力来做细粒度的非连续 gather
- SRAM 的地址空间是高度结构化的，不适合任意指针跳转

**3. NPU 的计算图是编译期确定的**

PagedAttention 需要根据 page table 在运行时决定读取哪些 KV block、跳过哪些。这本质上是动态控制流，而 NPU 的计算图：
- 所有算子的执行顺序在编译时确定
- 不支持运行时根据数据内容改变执行路径
- 条件分支、while 循环等动态结构受限或根本不存在

**可能的解决方案：**

1. **虚拟连续映射 + 物理碎片整理**：在 CPU 侧维护 page table（逻辑 → 物理映射），但给 NPU 提供一个连续的虚拟 KV 空间。当有新请求需要连续空间时，CPU 在后台做碎片整理（compaction），将分散的 block 拼成连续的物理区域。整理期间用双缓冲保证推理不中断。代价是碎片整理的计算开销和延迟。

2. **预留最大 + 掩码利用**：不为每个请求动态分配，而是为每个请求预留一个较大的固定 KV 区域（如最大支持 4096 tokens），用 attention mask 控制可见范围。这回到传统方案，但简单可靠。代价是内存利用率降低（回到 60% 左右）。

3. **编译期固定块分配**：在编译时就确定 N 个固定大小的 KV block 区域（如 8 个 block，每个 512 tokens），运行时在 CPU 上做调度分配。这是 PagedAttention 和静态分配的折中方案，适合请求数量可预测的场景。

4. **厂商原生支持**：推动 NPU 厂商在 SDK 层面提供类似 PagedAttention 的原语。例如高通的 QNN 和苹果的 Core ML 已经在为 LLM 推理增加 KV Cache 管理 API。这是长期方向。

### 思考题 2：激活重计算用计算换内存，端侧设备上计算和内存哪个更稀缺？如何据此决定重计算策略？

**端侧设备上，内存通常比计算更稀缺。**

这不是说端侧设备计算能力强——恰恰相反，端侧 NPU 的算力（几十 TOPS）远低于 GPU（几百 TFLOPS）。但：

**端侧内存的绝对瓶颈：**
- 手机可用内存：4-8GB（其中系统占用 2-3GB，可用仅 2-5GB）
- 7B W4A16 模型：约 3.5-4GB（不含 KV Cache 和激活值）
- KV Cache（2048 tokens, GQA）：约 0.5-1GB
- 剩余给激活值的空间：几乎没有

而一个 7B 模型的推理峰值激活值（prefill 阶段，batch=1, seq_len=2048）可以达到 1-2GB。如果不做重计算，7B 模型在手机上根本跑不起来。

**端侧计算的相对灵活性：**
- 计算可以用时间来换（用户等待）
- 内存不够就是 OOM crash，没有退路
- 用户对 200ms vs 300ms 的延迟差异不敏感，但对 crash 零容忍

**因此，端侧重计算策略应该比 GPU 端更激进：**

| 策略 | GPU 服务器 | 端侧设备 |
|------|-----------|---------|
| 重计算粒度 | 通常每 1-2 层做一次检查点 | 可以每层都做检查点，甚至不保存任何激活值 |
| 重计算范围 | 通常只对 FFN 层重计算 | 可以对注意力层也做重计算 |
| FLOP 开销预算 | 控制在 20-30% | 可以接受 50-80% |

**具体决策框架：**

1. **先算内存天花板**：模型权重 + KV Cache 最小占用 = M_base。设备可用内存 - M_base = M_budget（激活值预算）。

2. **估算无重计算时的激活值峰值**：A_peak = f(模型架构, batch_size, seq_len)。这是理论最大值。

3. **计算必须通过重计算释放的内存量**：A_peak - M_budget。如果为负，不需要重计算；如果为正，必须重计算。

4. **确定检查点策略**：每 k 层保存一次激活值。k 越大 → 内存节省越多 → 额外计算越多。在满足 M_budget 约束下取最小的 k。

5. **对 Prefill 和 Decode 分别考虑**：Prefill 阶段激活值大（与序列长度成正比），计算量大，重计算代价高。Decode 阶段每步只处理 1 个 token，激活值小，重计算代价可以忽略。因此可以考虑只在 Prefill 阶段做重计算，Decode 阶段不做。

**关键公式：**

重计算后激活值内存 = A_peak / k（检查点保存）+ A_peak × (k-1)/k × α（重计算期间的临时激活）

其中 α < 1 是因为重计算时不需要保存所有中间结果（只为了计算梯度才需要；推理时重计算只需前向激活值）。

**总结：端侧部署中，内存是硬约束，计算是软约束。重计算策略应该"激进到刚好不 OOM"，而不是"适度以平衡性能"。**

## 第5章 硬件部署

### 思考题 1：端侧 NPU 的 Ridge Point（1000+ FLOP/B）远高于 GPU（~150 FLOP/B），这对部署策略有何影响？

**核心影响：量化在端侧的价值远超 GPU。**

Ridge Point（拐点）= Peak FLOPS / Memory Bandwidth，它标记了 memory-bound 和 compute-bound 的分界线：
- 算术强度 < Ridge Point → memory-bound（性能 = 算术强度 × 带宽）
- 算术强度 > Ridge Point → compute-bound（性能 = 峰值算力）

**为什么 Ridge Point 差异如此之大？**

| 硬件 | Peak FLOPS | BW (GB/s) | Ridge Point |
|------|-----------|-----------|-------------|
| NVIDIA A100 | 312 TFLOPS | 2032 | ~154 FLOP/B |
| 高通 Hexagon NPU | 75 TOPS | 60 | ~1250 FLOP/B |

原因：GPU 的算力和带宽大体同步增长（都有大量晶体管预算），而 NPU 的算力主要通过堆 MAC 阵列获得（面积效率高），但带宽受限于 DRAM 物理接口（引脚数限制）。所以 NPU 的算力/BW 比值天然更高。

**部署策略影响：**

**1. 更多算子落在 memory-bound 区域**

LLM Decode 阶段的算术强度约 1-10 FLOP/B（因为每步只处理 1 个 token，权重搬运占主导）。在 A100（Ridge=154）上，这显然是 memory-bound；但在 Hexagon NPU（Ridge=1250）上，同样是 memory-bound——但因为带宽更低（60 vs 2032 GB/s），实际性能差距比算力差距更大。

即使 Prefill 阶段（算术强度约 100-500 FLOP/B），在 A100 上可能 compute-bound，在 Hexagon NPU 上仍是 memory-bound 或接近边界。

**2. 量化对端侧加速效果远大于 GPU**

量化本质上是减少数据搬运量（压缩权重从 FP16 到 INT4 = 4x 减少）。对于 memory-bound 算子：
- 性能 ≈ 算术强度 × 带宽
- 量化后算术强度提升约 4x（同样 FLOPS，搬运字节数减少 75%）
- 在 bandwidth-limited 区域，性能线性增长：提升接近 4x

对于 compute-bound 算子（GPU Prefill 场景），量化仅仅减少了内存占用，不直接提升速度——除非硬件有原生 INT4 Tensor Core。

**结论：端侧量化从 W16A16 → W4A16 的 decode 加速可达 3-4x，而 GPU 上这个数字可能只有 1.2-1.5x。**

**3. 需要更激进的内存优化**

因为端侧内存带宽如此之低，任何减少内存访问的技术都价值巨大：
- KV Cache 量化（INT8 → 2x 带宽节省，INT4 → 4x）
- 权重共享（tied embeddings）在端侧的价值比 GPU 大
- 算子融合（减少中间结果写回 DRAM）的收益更显著
- 双缓冲、预取等 latency hiding 技术更重要

**4. 设计目标从"最大化 FLOPS 利用率"转向"最小化内存访问"**

GPU 优化常关注提高 Tensor Core 利用率（occupancy、wave scheduling 等）。端侧 NPU 优化应首先关注 DRAM 访问次数，其次才是 MAC 利用率。这影响了设计优先级：

| 优先级 | GPU 部署 | NPU 端侧部署 |
|--------|---------|------------|
| 1 | 最大化 batch size | 量化权重（减少搬运） |
| 2 | 混合精度（FP16 Tensor Core） | KV Cache 压缩 |
| 3 | Kernel fusion（减少 kernel launch） | Kernel fusion（减少 DRAM 往返） |
| 4 | 量化（适度收益） | 双缓冲/预取 |

### 思考题 2：芯片的标称 TOPS（峰值算力）和实际 LLM 推理性能之间为什么存在巨大差距？

**差距的量级：**

以高通 Hexagon NPU 为例：
- 标称算力：75 TOPS (INT8)
- 实际 LLM Decode（Qwen2.5-1.5B W4A16）：约 25-35 tok/s

25 tok/s × 1.5B 参数 × 2 FLOP/参数 ≈ 75 GFLOP/s = 0.075 TOPS 有效算力。利用率仅 0.1%。

**原因 1：Decode 阶段是 memory-bound，算力不是瓶颈**

标称 TOPS 是在一切数据都在寄存器/SRAM 中的理想条件下测得的。但 LLM Decode 阶段：
- 每处理 1 个 token，需要读取全部权重
- 1.5B W4A16 模型权重约 0.75GB
- NPU DRAM 带宽 60 GB/s → 仅读取权重就需要 0.75/60 = 12.5ms
- 加上激活值读写、KV Cache 访问：总 DRAM 时间远超计算时间

这就像一辆最高时速 300km/h 的跑车在市区开——理论最高速度与红绿灯（带宽）限速无关。

**原因 2：标称 TOPS 的测量条件与实际推理条件不同**

TOPS 测量通常基于：
- 最大的矩阵乘法（如 4096×4096×4096 GEMM），FLOPS/Byte 比极高
- 连续内存访问模式
- 无中断、无同步开销
- 完全利用所有 MAC 单元

实际 LLM 推理：
- Decode 的矩阵乘是 GEMV（矩阵-向量），M=1，FLOPS/Byte ≈ 1
- 注意力计算、LayerNorm、激活函数等非 GEMM 算子
- NPU 与 CPU 之间的同步和数据搬运
- 多个小算子的 launch overhead

**原因 3：NPU 的 MAC 阵列利用率在 GEMV 场景极低**

NPU 的 MAC 阵列通常设计为处理大矩阵乘（如 256×256 tile）。当 M=1（Decode），MAC 阵列中只有 1/256 的单元在有效工作，其余闲置。这是一个硬件架构级的效率损失——不是软件能优化的。

部分 NPU 针对 GEMV 有专门的快速路径（减少 launch overhead），但核心问题是 MAC 阵列本身是为 M×N 大矩阵乘设计的。

**原因 4：不可忽视的非计算开销**

- NPU 子图编译的启动延迟
- CPU ↔ NPU 数据同步（Top-K sampling、token embedding 在 CPU 上）
- 多个小算子的 kernel launch overhead
- 动态 shape 导致的 re-compilation

**衡量真实性能的正确指标：**

1. **tok/s（端到端吞吐）**：最直观的用户体验指标
2. **TTFT（首 token 延迟）**：Prefill 阶段效率
3. **ITL（token 间延迟）**：Decode 阶段效率
4. **有效算力利用率**：实际 FLOPS / 标称 TOPS，反映硬件架构与工作负载的匹配度
5. **能效比（tok/J）**：对电池设备更重要的指标

**选择硬件的建议：看 benchmark 不看标称 TOPS。** 不同 NPU 的 TOPS 定义方式不同（有算 INT4 的、有算 INT8 的、有算 FP16 的），且架构差异导致同样 TOPS 的 NPU 在 LLM 推理上可能差 2-3 倍。

## 第7章 端云协同

### 思考题 1：端云协同推理中，中间特征传输替代原始数据上传，是否真正保护了用户隐私？攻击者能从中间特征重建原始输入吗？

**简短回答：能提供一定保护，但不是绝对隐私。攻击者确实可以从中间特征重建原始输入，尤其在弱攻击模型下效果惊人。**

**为什么不是绝对隐私？——特征反演攻击**

中间特征是原始输入经过神经网络浅层处理后的表示。从信息论角度，这个过程不是无损加密——保留了输入的语义内容，只是改变了表示形式。攻击方法包括：

**1. 模型反演（Model Inversion）**

给定中间特征 z = f_shallow(x)，攻击者训练一个解码器 g 使得 g(z) ≈ x。方法：
- 如果攻击者拥有浅层模型的副本（白盒攻击），可以训练端到端的反演网络
- 使用 GAN 或扩散模型作为解码器，以特征 z 为条件生成重建图像/文本
- 对于图像：已有研究（Dosovitskiy & Brox, 2016; He et al., 2019）证明从 CNN 中间特征可以重建出视觉上可辨认的原始图像
- 对于文本：中间特征（如 BERT 的 hidden states）可以通过 probing classifiers 恢复词性、实体、甚至原始词汇

**2. 梯度泄漏**

如果端侧需要接收云端的反向梯度来做训练（如 split learning），梯度本身可以泄漏信息。Zhu et al. (2019, "Deep Leakage from Gradients") 证明从梯度可以在像素级别重建训练图像和文本。

**3. 属性推断攻击**

即使无法完整重建原始输入，攻击者也可以从中间特征推断出敏感属性：
- 文本中的敏感关键词（疾病、财务、身份信息）
- 图像中的人脸、位置、文本内容

**在什么情况下特征传输确实保护了隐私？**

1. **信息瓶颈**：如果浅层网络有足够强的降维（信息瓶颈），丢弃了细粒度信息，重建会变得困难。但这与任务性能矛盾——信息瓶颈太强会损害云端模型的推理质量。

2. **差分隐私加持**：在传输前对中间特征添加校准噪声（如高斯噪声），使得从带噪特征重建原始输入的信息论上界可控。这是唯一有理论保证的方案。

3. **对比原始数据上传**：虽然不完美，中间特征确实比原始数据更难被第三方（如传输层攻击者）直接理解。原始文本可以被直接阅读，中间特征是高维浮点向量——对不具备 ML 能力的攻击者有意义。

4. **去标识化**：对于特定类型的隐私信息（如人脸），浅层特征会压缩掉身份相关的细粒度信息，保留任务相关的语义信息（如表情、姿态）。

**实践建议：**

1. 不要声称"绝对隐私保护"，对用户坦诚"特征传输提供部分保护但不是完全匿名"
2. 需要隐私保护的场景，叠加差分隐私（ε=1-8 的噪声校准）
3. 定期做红队测试——尝试从中间特征重建输入，评估实际风险
4. 对特别敏感的输入类型（医疗、金融），直接禁止上传任何形式的数据

### 思考题 2：浏览器端推理（WebGPU vs WASM）与原生 App 推理相比，在哪些场景下是更优选择？

**浏览器端推理更优的场景：**

**1. 零安装 / 即时使用场景**
- 用户不想为试用一个功能下载 500MB 的 App
- 分享链接即可体验（如"试试我们的 AI 翻译功能"）
- Demo、试用、一次性任务
- *例子*：HuggingFace Space、ChatGPT 网页版、各类 AI Demo 页面

**2. 隐私敏感的临时场景**
- 用户需要处理敏感数据但不想安装 App（担心 App 有更多权限）
- 用完即走，本地处理，数据不留存
- *例子*：在线加密工具、临时文件处理、医疗报告解读

**3. 跨平台快速覆盖**
- 一套代码覆盖所有平台（Windows/Mac/Linux/iOS/Android）
- 无需为每个平台单独开发和维护 App
- 迭代速度快（网页部署 vs App Store 审核）
- *例子*：企业内部工具、教育平台、快速原型验证

**4. 搜索引擎 / 内容平台集成**
- 在搜索框旁边直接提供 AI 功能，用户无需离开页面
- 网页内在线推理（如实时翻译、内容摘要）
- *例子*：Chrome 内置 Gemini Nano

**5. 低频使用场景**
- 用户每个月只用 1-2 次的功能，不值得装 App
- Web 端随时可用，无需常驻设备存储

**原生 App 更优的场景：**

1. **高频使用**（每天多次）→ App 的启动速度和离线能力更有优势
2. **需要高级硬件权限**（相机实时处理、后台运行、通知推送）
3. **极致性能要求**（原生 NPU 加速 → WASM 有 1.5-2x 性能损失）
4. **大型模型**（>2GB → 浏览器内存限制 1-2GB 不允许）
5. **需要离线始终可用**（Service Worker 不如原生可靠）

**技术对比：**

| 维度 | 浏览器端 | 原生 App |
|------|---------|---------|
| 性能 | WASM 慢 1.5-2x, WebGPU 接近但调度受限 | 最快，直接访问硬件 |
| 模型大小限制 | WASM 内存 <4GB, Tab 限制 1-2GB | 受设备内存限制（通常 4-6GB） |
| 首次加载 | 需下载模型（CDN） | 预装或 App Store 下载 |
| 硬件访问 | 有限（WebGPU, WebNN 草案） | 完整（NPU, GPU, DSP, 传感器） |
| 分发 | 即时，零审核 | 有审核流程 |
| 用户留存 | 低 | 高 |

## 第8章 端侧训练

### 思考题 1：端侧微调带来个性化能力的同时，为什么可能导致灾难性遗忘？如何检测和防止？

**为什么会灾难性遗忘？**

端侧微调通常在用户私有数据上进行，这些数据有两个特点：
1. **分布偏斜**：用户数据往往集中在少数话题/风格上（如某个用户的短信都是工作邮件风格）
2. **数据量小**：单个用户的数据量远小于预训练语料

当模型在这样的小规模偏斜数据上做梯度更新时：
- **参数覆盖**：与用户数据相关的参数被强梯度更新，覆盖了预训练时学到的通用知识
- **表示崩塌**：模型的隐藏表示逐渐偏离预训练的通用表示空间，向着狭窄的用户数据分布坍缩
- **输出分布偏移**：模型开始对所有输入（包括与用户数据无关的）产生偏向用户风格的输出

数学上，灾难性遗忘的本质是：在损失函数空间 L(θ) 中，用户数据的局部最小值离预训练模型的全局最小值很远。端侧的小学习率梯度下降沿着一条狭窄的峡谷走，最终到达一个仅在用户数据上低损失、但在通用数据上高损失的局部极小点。

**如何检测？**

1. **预留评测集**：端侧保留少量通用评测样本（如标准问答对、常识推理题），定期测试模型在这些样本上的表现
2. **输出分布监控**：统计模型输出 token 的分布熵。灾难性遗忘通常表现为输出分布熵降低（模型变得"过于确定"，只输出用户数据中的常见模式）
3. **梯度范数监控**：如果微调时某些层的梯度范数远大于其他层 → 这些层可能正在被过度重写
4. **对比原始模型输出**：在相同输入上比较微调后模型与原始模型输出的 KL 散度，如果持续增大 → 正在偏离原始能力

**如何防止？**

1. **EWC（Elastic Weight Consolidation）**：对预训练中的重要参数施加 L2 正则约束。重要性通过 Fisher 信息矩阵估计——Fisher 信息大的参数在预训练任务上重要，应被强约束：

   L(θ) = L_task(θ) + λ/2 Σ F_i (θ_i - θ_i*)^2

   其中 F_i 是第 i 个参数的 Fisher 信息，θ_i* 是预训练值。

2. **经验回放（Experience Replay）**：在端侧保留少量通用数据样本（如 100-500 条），微调时与用户数据混合训练。

3. **极低学习率 + 极小适配器**：使用 LoRA 且 rank 很小（r=4-8），学习率极低（1e-5 级别）。适配器参数少意味着"可破坏"的参数少。

4. **正则化微调**：在微调损失中加入与原始模型输出分布的 KL 散度作为正则项，防止输出分布偏离太远。

5. **用户数据筛选**：不是所有用户数据都值得微调。筛选与预训练分布差异适中（既不重复也不离群）的数据用于训练。

### 思考题 2：联邦学习中"梯度"传输替代"数据"传输，差分隐私如何防止梯度泄漏用户信息？

**梯度为什么能泄漏信息？**

联邦学习中，客户端上传的是模型梯度（或参数更新量），而非原始数据。直觉上这更安全，但：

1. **梯度是数据在模型参数空间中的投影**。给定梯度 ∂L(f(x;θ), y)/∂θ，攻击者可以通过求解优化问题来重建 x：
   
   x* = argmin ||∂L(f(x';θ), y)/∂θ - 真实梯度||²

2. **对于语言模型**，嵌入层的梯度只对输入中实际出现的 token 位置非零 → 直接泄漏词表信息。Zhu et al. (2019) 和 Deng et al. (2021) 已证明可以从梯度重建 NLP 训练数据。

3. **梯度大小与输入特征成正比**：某个特征的梯度大 → 该特征在输入中数值大或出现频繁。

**差分隐私如何解决？**

差分隐私的核心操作：在梯度上传前添加校准噪声。

**具体做法（DP-SGD / DP-FedAvg）：**

1. **梯度裁剪**：对每个样本的梯度做范数裁剪：ḡ = g / max(1, ||g||/C)。这限制了单个样本对梯度的最大影响。

2. **添加高斯噪声**：对裁剪后的梯度加噪声：g̃ = ḡ + N(0, σ²C²I)。噪声的标准差与裁剪阈值 C 和隐私参数 ε 相关。

3. **隐私会计（Privacy Accounting）**：通过 Moments Accountant 跟踪多次迭代累积的隐私损失 (ε, δ)。每轮训练消耗一定量的隐私预算。

**差分隐私的数学保证：**

(ε, δ)-差分隐私意味着：对于任意两个仅在一条数据上不同的训练集 D 和 D'，以及任意输出集合 S：

P[M(D) ∈ S] ≤ e^ε · P[M(D') ∈ S] + δ

ε 越小，隐私保护越强。典型联邦学习设置 ε = 2-8。

**为什么加噪声能防止梯度泄漏？**

1. **信息论限制**：噪声使得梯度中的每个分量都附带不确定性。从含噪梯度重建原始输入的精度有理论下界——噪声越大，重建越不准。

2. **掩盖个体贡献**：即使攻击者能看到加噪后的梯度，也无法确定某个特定样本是否存在于训练集中。这是差分隐私的定义性保证。

3. **组合保护**：多轮训练累积的隐私损失可以通过 Moments Accountant 精确追踪，确保总 ε 在预设范围内。

**代价：**

- 模型精度下降（噪声干扰了梯度信号），ε 越小精度损失越大
- 需要更多训练轮次来达到相同精度
- 梯度裁剪阈值 C 的选择对效果影响大（太大则噪声也大，太小则信号丢失）
- 对 LoRA 微调影响相对较小（因为只有少量参数有梯度，噪声可集中在这些参数上）
