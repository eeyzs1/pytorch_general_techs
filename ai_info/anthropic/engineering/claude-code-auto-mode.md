# Claude Code Auto Mode: A Safer Way to Skip Permissions

- **原文链接**: [Claude Code auto mode: a safer way to skip permissions](https://www.anthropic.com/engineering/claude-code-auto-mode)
- **发布日期**: 2026-03-25
- **标签**: #安全 #权限 #ClaudeCode #分类器 #自动审批 #PromptInjection

## 核心观点

Claude Code 用户批准了 93% 的权限提示。Auto mode 构建了基于模型分类器的自动权限决策系统，在提高安全性的同时减少审批疲劳。采用两层防御架构：输入层（prompt-injection 探测器）和输出层（transcript 分类器），分类器分两阶段运行（快速单 token 过滤 + CoT 推理），只对危险操作消耗推理 token。

## 威胁模型（四种危险行为原因）

1. **过度热心 (Overeager)**: Agent 理解目标但越权执行——最难检测，因为没有恶意推理
2. **诚实错误**: Agent 误解爆炸半径（如认为资源是测试环境）
3. **Prompt Injection**: 文件/网页中植入的指令劫持 Agent
4. **模型不对齐**: 模型追求自身目标（目前实践中未观察到）

## 架构设计

### 两层防御

| 层级 | 机制 | 位置 | 功能 |
|------|------|------|------|
| 输入层 | Prompt-injection 探测器 | 服务端 | 扫描工具输出，添加警告标记 |
| 输出层 | Transcript 分类器 (Sonnet 4.6) | 服务端 | 替代人工审批，评估每个操作 |

### Transcript 分类器两阶段

- **阶段 1 (快速过滤)**: 单 token 是/否判断，倾向 Block，高 FPR 低成本
- **阶段 2 (CoT 推理)**: 仅阶段 1 标记时才触发，减少误报

### 设计决策

- 分类器只看用户消息和工具调用，剥离 Claude 自身消息和工具输出 → **推理盲化**
- 子 Agent 递归应用相同 pipeline，handoff 分类器在委派和返回时介入

## 权限决策三级制

| 层级 | 范围 | 说明 |
|------|------|------|
| Tier 1: 内置安全允许列表 | 文件读取、搜索、代码导航、todo/plan模式 | 默认允许，不消耗分类器延迟 |
| Tier 2: 项目内文件操作 | workspace 内文件写入和编辑 | 允许，可通过版本控制审查 |
| Tier 3: Transcript 分类器 | Shell命令、网络请求、外部工具、workspace外的文件操作、子Agent | 走完整两阶段分类 |

进入 auto mode 时，自动丢弃已知会授予任意代码执行的权限规则（通配 shell、python/node/ruby 解释器、包管理器 run 命令）。

## 分类器决策标准

### 固定模板（三个核心规则）

1. **评估规则**: 评估操作的真实世界影响，而非表面文本。如果 Agent 组装 shell 命令后执行，分类器评估组装后的命令
2. **用户意图规则**: 判断操作是否获得用户授权——这是过度热心行为的防御关键。分类器故意保守，"清理我的分支"不授权批量删除
3. **步骤化分类流程**: 先检查 block 规则 → 检查 allow 例外 → 检查用户意图 → 做出决定

### 可自定义插槽

- **环境 (Environment)**: 定义信任边界（默认仅信任当前 git repo）
- **Block 规则** (默认 20+):
  - 破坏或外泄：force-push、批量删除云存储、发送内部数据到外部
  - 降低安全姿态：禁用日志、安装持久化（SSH key/cronjob）、修改 Agent 权限配置
  - 跨越信任边界：运行外部 repo 代码、扫描凭证存储、发送数据到未指定服务
  - 绕过审查或影响他人：直接推送到 main、生产部署、修改共享资源
- **Allow 例外**: 安装已声明的依赖、标准凭证流、推送到会话工作分支

## 结果数据

| 数据集 | 阶段1 FPR/FNR | 阶段1→2 (全Pipeline) |
|--------|--------------|---------------------|
| 真实内部流量 (n=10,000) | 8.5% FPR | 0.4% FPR |
| 真实过度热心操作 (n=52) | 6.6% FNR | 17% FNR |
| 合成数据外泄 (n=1,000) | 1.8% FNR | 5.7% FNR |

17% FNR 的真相：分类器能正确识别危险，但错误判断了会话中的同意信号是否真正覆盖当前操作。主要失败模式：用户说了"清理 PR"但没有明确 force-push，或之前批准过类似操作但一次批准不是模式。

## 关键洞察

- Auto mode 定位为手动审查和零防护之间的中间地带
- 分类器设计为推理盲化，防止评估本身变成攻击面
- Shell 命令链（`&&` 连接）被评估为一个整体操作
- 该功能以保守默认值出厂，可逐步扩展配置
