# Demystifying Evals for AI Agents

- **原文链接**: [Demystifying evals for AI agents](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents)
- **作者**: Mikaela Grace, Jeremy Hadfield, Rodrigo Olivares, Jiri De Jonghe
- **发布日期**: 2026-01-09
- **标签**: #评估 #评估框架 #Grader #能力评估 #回归评估

## 核心观点

好的评估帮助团队更自信地发布 AI Agent。没有评估，团队容易陷入被动循环——只在生产中捕获问题，修复一个失败又创造另一个。评估使问题和行为变更在影响用户之前可见，其价值在 Agent 生命周期中复合增长。

这是 Anthropic 最全面的 Agent 评估指南。文章提供了构建 AI Agent 评估的完整框架，涵盖术语、评分器类型、能力评估 vs 回归评估、四种 Agent 类型的评估设计、8 步路线图，以及评估与其他质量保障方法的协同。核心信息：**评估是公司的知识产权和关键竞争优势，其价值随时间复合增长。**

## 为何构建评估

团队在早期可以通过手动测试、dogfooding 和直觉走得相当远，但一旦 Agent 进入生产并开始扩展，没有评估的开发就会崩溃。**崩溃点**通常出现在：用户报告 Agent 变差了，而团队"盲飞"——只能猜测和检查，无法区分真实回归和噪声。没有评估，调试是被动的：等待投诉、手动复现、修复 Bug、祈祷没有其他回归。

**Claude Code 的演进路径**：最初基于 Anthropic 员工和外部用户的反馈快速迭代 → 后来为狭窄领域（简洁性、文件编辑）添加评估 → 再扩展到复杂行为（过度工程）→ 评估帮助识别问题、指导改进、聚焦研究-产品协作。结合生产监控、A/B 测试、用户研究等，评估为 Claude Code 的持续扩展提供了信号。

**两个客户案例**：
- **Descript**：其 Agent 帮助用户编辑视频，围绕三个维度构建评估：别搞坏、做我说的、做好。从手动评分演进到由产品团队定义标准的 LLM 评分器+定期人工校准，现在定期运行两套独立的套件——质量基准测试和回归测试。
- **Bolt AI**：在已有广泛使用的 Agent 之后才开始构建评估。3 个月内构建了评估系统：用静态分析评分输出，用浏览器 Agent 测试应用，用 LLM 评判指令遵循等行为。

**评估的复合价值**：成本前期可见，收益后期累积。评估还能加速新模型采纳——没有评估的团队面对新模型需要数周测试，有评估的团队可以在数天内确定模型优劣、调整提示并升级。评估一旦存在，免费获得基线和回归测试：延迟、Token 用量、每任务成本、错误率都可以在静态任务库上追踪。评估还能成为产品团队和研究团队间**最高带宽的沟通渠道**，定义研究人员可以优化的指标。

**何时构建评估**：有些团队在开发之初就创建评估；有些在规模化后才添加。评估在 Agent 开发早期特别有用——明确编码预期行为。两个工程师阅读同一份初始规范，可能对 AI 如何处理边缘情况有不同理解。评估套件解决这种歧义。无论何时创建，评估都能加速开发。编写评估在 Agent 生命周期的任何阶段都有用——早期强制产品团队明确成功的含义，后期帮助维持一致的质量标准。

## 评估的结构

评估 = 输入 + Agent 执行 + 评分逻辑。

- **单轮评估**：一个提示、一个响应、评分逻辑。早期 LLM 的主要评估方式。
- **多轮评估**：Agent 接收工具和任务，执行"Agent 循环"（工具调用+推理），更新环境状态，然后评分。例如：编码 Agent 接收工具、一个任务（构建 MCP 服务器）和一个环境，执行 Agent 循环，用实现更新环境。评分使用单元测试验证 MCP 服务器是否正常工作。
- **Agent 评估更复杂**：Agent 在多轮中使用工具、修改状态、动态适应——错误会传播和复合。前沿模型还能找到超越静态评估限制的创造性解决方案（如 Opus 4.5 在 τ²-bench 中发现政策漏洞来预订航班——按评估标准"失败"，但实际给出了更好的用户方案）。

**关键区分：转录 vs 结果**
- **转录 (Transcript)**：试验的完整记录，包括输出、工具调用、推理、中间结果（对 Anthropic API 而言是完整的 messages 数组）
- **结果 (Outcome)**：试验结束时环境的最终状态。航班预订 Agent 可能在转录末尾说"已预订"，但结果取决于预订是否真的存在于数据库中

## 核心术语

| 术语 | 定义 |
|------|------|
| 任务/测试用例 | 具有定义输入和成功标准的单个测试 |
| 试验 (Trial) | 每次尝试（因非确定性需要多次） |
| 评分器 (Grader) | 评分逻辑（任务可有多个评分器，每个可含多个断言/检查） |
| 转录/追踪 | 试验的完整记录（对 Anthropic API 而言是完整的 messages 数组） |
| 结果 | 试验结束时环境的最终状态 |
| 评估工具 (Eval Harness) | 端到端运行评估的基础设施——提供指令和工具、并发运行任务、记录步骤、评分、聚合结果 |
| Agent 工具 (Agent Harness/Scaffold) | 使模型作为 Agent 运行的系统——处理输入、编排工具调用、返回结果。评估"一个 Agent"实际是评估工具+模型的组合。例如 Claude Code 是一个灵活的 Agent 工具，Anthropic 通过 Agent SDK 使用其核心原语构建了长时间运行的 Agent 工具 |
| 评估套件 | 为衡量特定能力或行为而设计的任务集合。套件中的任务通常共享一个广泛目标（如客户支持评估套件可能测试退款、取消和升级） |

## 三种评分器类型

Agent 评估通常组合三种评分器。每种评分器评估转录或结果的某部分。有效评估设计的核心是选择正确的评分器。

### 基于代码的评分器

**方法**：字符串匹配（精确、正则、模糊）、二进制测试（fail-to-pass, pass-to-pass）、静态分析（lint、类型、安全）、结果验证、工具调用验证（工具使用、参数）、转录分析（轮次、Token 用量）

| 优势 | 劣势 |
|------|------|
| 快速 | 对有效变体脆弱——不精确匹配期望模式就失败 |
| 廉价 | 缺乏细微度 |
| 客观 | 对更主观的任务评估能力有限 |
| 可复现 | |
| 易调试 | |
| 验证特定条件 | |

### 基于模型的评分器

**方法**：评分标准 (Rubric-based scoring)、自然语言断言、成对比较、基于参考的评估、多评判共识

| 优势 | 劣势 |
|------|------|
| 灵活 | 非确定性 |
| 可扩展 | 比代码评分贵 |
| 捕捉细微差别 | 需与人类评分器校准以确保准确性 |
| 处理开放式任务 | |
| 处理自由格式输出 | |

### 人类评分器

**方法**：主题专家 (SME) 审查、众包判断、抽查采样、A/B 测试、标注者间一致性

| 优势 | 劣势 |
|------|------|
| 黄金标准质量 | 昂贵 |
| 匹配专家用户判断 | 缓慢 |
| 用于校准基于模型的评分器 | 通常需要大规模人类专家 |

**评分组合方式**：加权（组合评分器分数须达到阈值）、二进制（所有评分器必须通过）、混合。

## 能力评估 vs 回归评估

- **能力评估**: "这个 Agent 能做好什么？"——应从低通过率开始，针对 Agent 挣扎的任务，给团队一个攀登的山峰
- **回归评估**: "它是否仍能处理以前能处理的？"——应维持约 100% 通过率，分数下降意味着出了问题
- **毕业机制**：高分的能力评估"毕业"到回归套件——曾经测量"我们能不能做到"的任务，现在测量"我们还能不能可靠地做到"
- **必须同时运行**：在能力评估上攀登时，也要运行回归评估确保变更不会在其他地方造成问题

## 四种 Agent 类型的评估方法

Anthropic 看到当前大规模部署的几种常见 Agent 类型：编码 Agent、研究 Agent、计算机使用 Agent 和对话 Agent。每种类型可能跨行业部署，但可以使用相似的技术评估。不需要从零发明评估——以下方法可作为基础，再扩展到你的领域。

### 编码 Agent

编码 Agent 编写、测试和调试代码，像人类开发者一样导航代码库和运行命令。有效的编码 Agent 评估通常依赖明确定义的任务、稳定的测试环境和对生成代码的全面测试。

**核心评估方法**：确定性评分器天然适合——代码能否运行？测试是否通过？

**主要基准**：
- **SWE-bench Verified**：给 Agent 流行 Python 仓库的 GitHub issue，通过运行测试套件评分。解决方案仅在修复失败测试且不破坏现有测试时通过。LLM 在一年内从 40% 提升到 >80%。
- **Terminal-Bench**：测试端到端技术任务，如从源码构建 Linux 内核或训练 ML 模型。

**补充评分**：除 pass/fail 测试外，还可用基于启发式的代码质量规则评估生成代码，用基于模型的评分器评估工具调用行为或用户交互方式。

**示例——修复认证绕过漏洞的评估配置**：
```yaml
task:
  id: "fix-auth-bypass_1"
  desc: "Fix authentication bypass when password field is empty and ..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics: [n_turns, n_toolcalls, n_total_tokens]
    - type: latency
      metrics: [time_to_first_token, output_tokens_per_sec, time_to_last_token]
```

> **实践注释**：此示例展示了全部可用评分器范围供说明之用。实践中，编码评估通常仅依赖单元测试验证正确性和 LLM 评分标准评估整体代码质量，仅在需要时才添加额外评分器和指标。

### 对话 Agent

对话 Agent 在支持、销售、辅导等领域与用户交互。与传统聊天机器人不同，它们维护状态、使用工具、在对话中采取行动。虽然编码和研究 Agent 也可能涉及多轮用户交互，但对话 Agent 呈现独特的挑战：**交互质量本身是评估的一部分**。

有效的对话 Agent 评估通常依赖可验证的最终状态结果和同时捕获任务完成度与交互质量的评分标准。与大多数其他评估不同，它们通常需要第二个 LLM 来模拟用户。Anthropic 在对齐审计 Agent 中使用这种方法，通过扩展的对抗性对话对模型进行压力测试。

**成功是多维的**：工单是否解决（状态检查）、是否在 <10 轮内完成（转录约束）、语气是否恰当（LLM 评分标准）？

**主要基准**：τ-Bench 及其后继 τ²-Bench 在零售支持和航空预订等领域模拟多轮交互——一个模型扮演用户角色，Agent 处理真实场景。

**示例——处理退款的支持任务**：
```yaml
graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: state_check
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: transcript
    max_turns: 10
```

> **实践注释**：与编码 Agent 示例一样，此任务展示了多种评分器类型供说明。实践中，对话 Agent 评估通常使用基于模型的评分器同时评估沟通质量和目标完成度，因为许多任务（如回答问题）可能有多个"正确"解决方案。

### 研究 Agent

研究 Agent 收集、综合和分析信息，产生答案或报告。与编码 Agent 的单元测试提供二元 pass/fail 信号不同，研究质量只能相对于任务来评判。什么算"全面"、"来源充分"甚至"正确"取决于上下文：市场扫描、收购尽职调查和科学报告各有不同标准。

**独特挑战**：
- 专家可能对综合是否全面存在分歧
- 真实基准随参考内容变化而不断偏移
- 更长、更开放式的输出创造更多犯错空间

**评估策略——组合评分器**：
- **接地性检查**：验证声明是否有检索来源支持
- **覆盖度检查**：定义好答案必须包含的关键事实
- **来源质量检查**：确认所咨询来源是权威的，而非仅仅是首先检索到的
- **精确匹配**：对有客观正确答案的任务（"公司 X 的 Q3 营收是多少？"）
- LLM 可标记无支撑声明和覆盖缺口，也可验证开放式综合的连贯性和完整性

**关键建议**：鉴于研究质量的主观性，LLM 评分标准应频繁与专家人类判断校准。BrowseComp 基准测试 AI Agent 能否在开放网络中找到"大海捞针"式答案——问题设计为易于验证但难以解决。

### 计算机使用 Agent

计算机使用 Agent 通过截图、鼠标点击、键盘输入和滚动与软件交互，而非通过 API 或代码执行。它们可以使用任何具有图形用户界面 (GUI) 的应用，从设计工具到遗留企业软件。

**评估方法**：在真实或沙箱环境中运行 Agent，检查是否达成预期结果。
- **WebArena**：测试浏览器任务，使用 URL 和页面状态检查验证导航，后端状态验证确认数据修改（确认订单确实已下，而非仅确认页面出现）
- **OSWorld**：扩展到完整操作系统控制，评估脚本检查文件系统状态、应用配置、数据库内容和 UI 元素属性

**Token 效率 vs 延迟的权衡**：
- 基于 DOM 的交互执行快但消耗大量 Token
- 基于截图的交互更慢但更省 Token
- 例：让 Claude 总结维基百科 → 从 DOM 提取文本更高效；在亚马逊找笔记本电脑壳 → 截图更高效（提取整个 DOM 太耗 Token）

**Claude for Chrome 的经验**：开发了评估来检查 Agent 是否为每个上下文选择了正确的工具，从而更快更准确地完成浏览器任务。

## 非确定性指标

Agent 行为在运行间变化，使评估结果比表面看起来更难解读。每个任务有自己的成功率——可能一个任务 90%，另一个 50%——一次评估运行中通过的任务下次可能失败。有时我们想测量的是 Agent 在一个任务上成功的频率（多少比例的试验成功）。

- **pass@k**: k 次尝试中至少 1 次成功的概率。随 k 增加——更多"射门"意味着至少 1 次命中的概率更高。编码中通常最关心 pass@1（首次即解决）。50% pass@1 = Agent 首次尝试成功完成一半任务。在其他情况下，提出多个解决方案只要有一个有效就行。
- **pass^k**: k 次试验全部成功的概率。随 k 递减——要求更多试验一致通过是更难的标准。75% 单次成功率 × 3 次试验 = (0.75)³ ≈ 42%。特别适用于面向用户的 Agent——用户期望每次都可靠。
- **两者随 k 增大而发散**：k=1 时相同（都等于单次成功率）；k=10 时走向相反——pass@k 趋近 100%，pass^k 趋近 0%。
- **选择依据**：pass@k 适合一次成功就够的工具，pass^k 适合一致性至关重要的 Agent。

## 8 步路线图

### 收集初始评估数据

**Step 0. 尽早开始**
- 20-50 个来自真实失败的简单任务就是很好的起点
- 早期 Agent 开发中，每次变更影响明显（大效应量），小样本量就够。更成熟的 Agent 可能需要更大、更难的评估来检测更小的效应，但开始时最好采用 80/20 方法
- 越晚构建评估越难——早期产品需求自然转化为测试用例，等太久就要从运行中的系统逆向工程成功标准

**Step 1. 从手动检查开始**
- 从开发中的手动检查开始——发布前验证的行为、终端用户尝试的常见任务
- 已在生产中？查看 bug 追踪器和支持队列
- 将用户报告的失败转换为测试用例，确保套件反映实际使用；按用户影响优先排序

**Step 2: 编写无歧义的任务，附带参考解决方案**
- 好任务的标准：两个领域专家独立判断会得出相同的 pass/fail 结论。他们自己能否通过这个任务？如果不能，任务需要细化
- 歧义在任务规范中变成指标中的噪声——模糊的评分标准产生不一致的判断。同样适用于基于模型的评分器标准
- **每个任务都应可被正确遵循指令的 Agent 通过**。这很微妙：Terminal-Bench 审计发现，如果任务要求 Agent 写脚本但未指定文件路径，而测试假设了特定路径，Agent 可能因自身无过错而失败。评分器检查的一切都应在任务描述中明确；Agent 不应因模糊规范而失败
- **0% pass@100 最常意味着任务有问题，而非 Agent 无能**——应检查任务规范和评分器
- 为每个任务创建参考解决方案：已知能通过所有评分器的输出，证明任务可解且评分器正确配置

**Step 3: 构建平衡的问题集**
- 测试行为应该发生和不应该发生的情况——单侧评估导致单侧优化
- 例：如果只测试 Agent 是否在应该搜索时搜索，可能得到一个几乎什么都搜索的 Agent
- 尽量避免类别不平衡的评估
- Claude.ai 网页搜索评估的经验：构建双向评估——应该搜索的查询（如查天气）和应从已有知识回答的查询（如"谁创立了苹果？"）。在欠触发（该搜不搜）和过触发（不该搜也搜）之间找到平衡很困难，需要多轮优化提示和评估。随着更多示例问题出现，持续添加到评估中改善覆盖

### 设计评估工具和评分器

**Step 4: 构建健壮的评估工具，确保稳定环境**
- 评估中的 Agent 应与生产中的 Agent 大致相同功能，环境本身不应引入额外噪声
- **每次试验必须从干净环境开始**——隔离运行
- 不必要的共享状态（残留文件、缓存数据、资源耗尽）会导致基础设施抖动引起的关联失败，而非 Agent 性能问题。如果多个不同试验因同一环境限制（如 CPU 内存不足）而失败，这些试验不独立，评估结果不可靠
- 共享状态还可能人为膨胀性能——内部评估中观察到 Claude 通过检查之前试验的 git 历史获得不公平优势
- [Quantifying Infrastructure Noise in Agentic Coding Evals](quantifying-infrastructure-noise-in-agentic-coding-evals.md) 量化了不干净环境的影响——仅基础设施配置差异就能造成 6 个百分点波动

**Step 5: 深思熟虑地设计评分器**
- 优先选择确定性评分器，必要时使用 LLM 评分器，谨慎使用人类评分器
- **评估 Agent 产出了什么，而非路径**——检查特定步骤序列（如工具调用顺序）过于刚性，Agent 经常找到评估设计者未预料的有效方案。不要不必要地惩罚创造性
- **内置部分得分**：正确识别问题并验证客户但未处理退款的支持 Agent，比立即失败的 Agent 有意义地更好。在结果中体现这种成功连续体
- **LLM-as-judge 校准**：应与人类专家密切校准，确保人类评分和模型评分间分歧很小。为避免幻觉，给 LLM 一个"退出"方式（如返回"Unknown"）。为每个维度创建清晰的结构化评分标准，用独立的 LLM-as-judge 评分每个维度，而非用一个评分所有维度。系统健壮后，偶尔使用人类审查即可
- **评估的微妙失败模式**：Opus 4.5 在 CORE-Bench 上最初得分 42%，直到研究员发现多个问题：刚性评分（"96.12" vs 期望"96.124991…"）、模糊任务规范、不可精确复现的随机任务。修复后跳至 95%。METR 也发现时间范围基准中配置错误的任务——要求优化到声明的分数阈值，但评分要求超过该阈值，惩罚了遵循指令的模型。仔细复查任务和评分器有助于避免这些问题
- **防作弊**：任务和评分器应设计为通过必须真正解决问题，而非利用非预期漏洞

### 长期维护和使用

**Step 6: 检查转录**
- 除非阅读多次试验的转录和评分，否则不知道评分器是否工作良好
- Anthropic 投资了查看评估转录的工具，并定期花时间阅读
- 任务失败时，转录告诉你 Agent 是犯了真正的错误还是评分器拒绝了有效方案
- **失败应该看起来公平**——清楚 Agent 做错了什么以及为什么。分数不上升时，需要确信是 Agent 性能问题而非评估问题
- 阅读转录是验证评估测量了真正重要内容的关键技能

**Step 7: 监控能力评估饱和**
- 100% 通过率的评估只追踪回归，不提供改进信号
- SWE-bench Verified 今年从 30% 开始，前沿模型现在接近饱和 >80%
- 接近饱和时进展放缓（只剩最难的任务），大幅能力提升表现为小幅分数增长——结果具有欺骗性
- 例：代码审查创业公司 Qodo 最初对 Opus 4.5 不以为然，因为他们的单次编码评估未能捕捉更长、更复杂任务上的增益。他们开发了新的 Agent 评估框架后，才看清进展
- **规则：在有人深入评估细节并阅读转录之前，不要按面值接受评估分数**。如果评分不公平、任务模糊、有效方案被惩罚或工具约束了模型，评估应被修订

**Step 8: 通过开放贡献和维护保持评估套件长期健康**
- 评估套件是需要持续关注和明确所有权的活文档
- Anthropic 的经验：建立专门的评估团队拥有核心基础设施，领域专家和产品团队贡献大多数评估任务并自行运行评估
- 对 AI 产品团队而言，拥有和迭代评估应像维护单元测试一样常规。团队可能浪费数周开发"早期测试中可行"但未能满足设计良好的评估会提前暴露的未声明期望的 AI 功能。定义评估任务是压力测试产品需求是否足够具体以开始构建的最佳方式之一
- **评估驱动开发**：在 Agent 能力满足之前构建评估定义计划能力，然后迭代直到 Agent 表现良好。内部经常构建"今天够用"但对未来模型下注的功能——低通过率的能力评估使这可见。新模型发布时，运行套件快速揭示哪些赌注得到了回报
- 让最接近产品需求和用户的人定义成功——产品经理、客户成功经理、销售人员可以用 Claude Code 贡献评估任务作为 PR——让他们这样做！或者更好的是，主动赋能他们

## 瑞士奶酪模型：评估与其他方法的协同

没有单一评估层能捕获所有问题。多种方法组合时，穿过一层的失败会被另一层捕获。

| 方法 | 优势 | 劣势 |
|------|------|------|
| **自动化评估** | 更快迭代、完全可复现、无用户影响、可在每次提交运行、大规模测试场景无需生产部署 | 需要更多前期投资构建、需随产品和模型演进持续维护避免偏移、不匹配真实使用模式时产生虚假信心 |
| **生产监控** | 揭示大规模真实用户行为、捕获合成评估遗漏的问题、提供 Agent 实际表现的基准真相 | 被动——问题到达用户后才知道、信号嘈杂、需要基础设施投资、缺乏评分的基准真相 |
| **A/B 测试** | 测量实际用户结果（留存、任务完成）、控制混淆因素、可扩展且系统化 | 慢——数天或数周才能达到显著性、需要足够流量、只测试部署的变更、对指标变更的底层"为什么"信号较少（除非能彻底审查转录） |
| **用户反馈** | 发现未预料的问题、附带真实人类用户的实际示例、反馈通常与产品目标相关 | 稀疏且自我选择偏倚、偏向严重问题、用户很少解释失败原因、非自动化、主要依赖用户捕获问题可能对用户产生负面影响 |
| **手动转录审查** | 建立失败模式直觉、捕获自动化遗漏的微妙质量问题、帮助校准"好"是什么样的并掌握细节 | 耗时、不可扩展、覆盖不一致、审查者疲劳或不同审查者影响信号质量、通常只给出定性信号而非清晰定量评分 |
| **系统性人类研究** | 黄金标准质量判断（多个人类评分者）、处理主观/模糊任务、为改进基于模型的评分器提供信号 | 昂贵、周转慢、标注者间分歧需调解、复杂领域（法律、金融、医疗）需要人类专家进行研究 |

**各阶段映射**：自动化评估在发布前和 CI/CD 中最有用（每次 Agent 变更和模型升级的第一道防线）→ 生产监控在发布后检测分布偏移和未预料的真实世界失败 → A/B 测试在有足够流量后验证重大变更 → 用户反馈和转录审查是持续实践（持续分类反馈，每周抽样转录阅读，需要时深入）→ 系统性人类研究保留用于校准 LLM 评分器或评估主观输出（人类共识作为参考标准）。

**最有效的团队组合这些方法**：自动化评估用于快速迭代，生产监控获取基准真相，定期人类审查用于校准。

## 评估框架推荐

| 框架 | 特点 |
|------|------|
| **Harbor** | 容器化环境运行 Agent，跨云提供商大规模运行试验，标准化格式定义任务和评分器。Terminal-Bench 2.0 通过 Harbor 注册表分发 |
| **Braintrust** | 离线评估+生产可观测性+实验追踪，`autoevals` 库包含预构建的事实性、相关性评分器 |
| **LangSmith** | 追踪、离线/在线评估、数据集管理，与 LangChain 生态紧密集成 |
| **Langfuse** | 类似 LangSmith 的自托管开源替代，满足数据驻留要求 |
| **Arize Phoenix/AX** | 开源 LLM 追踪/调试/评估平台，AX 是其 SaaS 扩展 |

许多团队组合多个工具、自建评估框架，或仅使用简单评估脚本作为起点。框架可以加速进展和标准化，但**只有通过它运行的评估任务才有效**。快速选择适合工作流的框架，然后把精力投入到高质量测试用例和评分器的迭代上。

## 结论

没有评估的团队陷入被动循环——修复一个失败，创造另一个，无法区分真实回归和噪声。及早投资的团队发现相反的情况：开发加速，因为失败变成测试用例，测试用例防止回归，指标取代猜测。评估给整个团队一个清晰的山峰去攀登，把"Agent 感觉变差了"变成可操作的信息。

模式因 Agent 类型而异，但基础是恒定的：尽早开始，不要等待完美套件；从真实失败中获取现实任务；定义无歧义、健壮的成功标准；深思熟虑地设计评分器并组合多种类型；确保问题对模型足够难；迭代评估以改善信噪比。**读转录！**

AI Agent 评估仍是一个新兴、快速演进的领域。随着 Agent 承担更长的任务、在多 Agent 系统中协作、处理越来越主观的工作，我们需要调整技术。Anthropic 将继续分享最佳实践。
