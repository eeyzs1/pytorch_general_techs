## 1. L2 回头看 L0 项目1：Web日志分析系统（2h）

**原始方案：** Python 脚本处理日志

### 新视角一：用 Spark 重写日志分析

对比 Python 单机处理 vs Spark 分布式处理的性能和代码量。

**旧方案（Python 单机）：**

```python
from collections import Counter
import re

log_pattern = re.compile(r'(\d+\.\d+\.\d+\.\d+).*?"(\w+) (\S+).*?" (\d{3})')

ip_counter = Counter()
status_counter = Counter()
path_counter = Counter()

with open('access.log', 'r') as f:
    for line in f:
        m = log_pattern.match(line)
        if m:
            ip_counter[m.group(1)] += 1
            status_counter[m.group(4)] += 1
            path_counter[m.group(3)] += 1

print("Top 10 IPs:", ip_counter.most_common(10))
print("Status distribution:", dict(status_counter))
print("Top 10 paths:", path_counter.most_common(10))
```

**新方案（Spark）：**

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, count, desc

spark = SparkSession.builder.appName("LogAnalysis").getOrCreate()

df = spark.read.text("hdfs:///logs/access.log")

parsed = df.select(
    regexp_extract("value", r'^(\d+\.\d+\.\d+\.\d+)', 1).alias("ip"),
    regexp_extract("value", r'"(\w+) (\S+)', 2).alias("path"),
    regexp_extract("value", r'" (\d{3})', 1).alias("status")
).filter(col("ip") != "")

parsed.groupBy("ip").count().orderBy(desc("count")).show(10)
parsed.groupBy("status").count().show()
parsed.groupBy("path").count().orderBy(desc("count")).show(10)
```

**对比维度：**

| 维度 | Python 单机 | Spark |
|------|------------|-------|
| 数据规模 | < 1GB | TB 级 |
| 代码量 | ~20行 | ~15行 |
| 执行时间（10GB日志） | ~120s | ~30s（4节点） |
| 扩展性 | 无法水平扩展 | 线性扩展 |
| 容错性 | 无 | 自动重试 |

### 新视角二：用 Flink 实现实时日志分析

从 T+1 批处理到实时流处理。

```python
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment

env = StreamExecutionEnvironment.get_execution_environment()
t_env = StreamTableEnvironment.create(env)

t_env.execute_sql("""
    CREATE TABLE access_log (
        ip STRING,
        method STRING,
        path STRING,
        status STRING,
        event_time TIMESTAMP(3),
        WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
    ) WITH (
        'connector' = 'kafka',
        'topic' = 'access-logs',
        'properties.bootstrap.servers' = 'kafka:9092',
        'format' = 'json'
    )
""")

t_env.execute_sql("""
    CREATE TABLE ip_stats (
        window_start TIMESTAMP(3),
        window_end TIMESTAMP(3),
        ip STRING,
        cnt BIGINT
    ) WITH (
        'connector' = 'jdbc',
        'url' = 'jdbc:clickhouse://ck:8123',
        'table-name' = 'ip_stats',
        'username' = 'default',
        'password' = ''
    )
""")

t_env.execute_sql("""
    INSERT INTO ip_stats
    SELECT
        TUMBLE_START(event_time, INTERVAL '1' MINUTE) AS window_start,
        TUMBLE_END(event_time, INTERVAL '1' MINUTE) AS window_end,
        ip,
        COUNT(*) AS cnt
    FROM access_log
    GROUP BY
        TUMBLE(event_time, INTERVAL '1' MINUTE),
        ip
""")
```

### 新视角三：用 ClickHouse 存储分析结果

从 CSV 文件到 OLAP 列式数据库。

```python
import csv

with open('result.csv', 'w', newline='') as f:
```

**新方案（ClickHouse）：**

```sql
CREATE TABLE log_analysis.ip_stats (
    date Date,
    ip String,
    request_count UInt64,
    avg_response_time Float64,
    error_rate Float32
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (ip, date);

-- 查询 Top IP：毫秒级返回
SELECT ip, sum(request_count) AS total
FROM log_analysis.ip_stats
WHERE date >= today() - 7
GROUP BY ip
ORDER BY total DESC
LIMIT 10;
```

## 2. L2 回头看 L0 项目2：电商数据库设计与SQL分析（2h）

**原始方案：** MySQL 单机 + 30条SQL

### 新视角一：用 Hive 分区表重做分析

从 GB 到 TB 的扩展性。

**旧方案（MySQL 单表）：**

```sql
SELECT
    category_id,
    DATE(order_time) AS order_date,
    COUNT(*) AS order_count,
    SUM(amount) AS total_amount
FROM orders
WHERE order_time >= '2024-01-01'
GROUP BY category_id, DATE(order_time);
```

**新方案（Hive 分区表）：**

```sql
CREATE TABLE dwd_orders (
    order_id STRING,
    user_id STRING,
    category_id STRING,
    amount DECIMAL(12,2)
)
PARTITIONED BY (dt STRING)
STORED AS ORC;

-- 分区裁剪：只扫描目标分区
SELECT
    category_id,
    dt AS order_date,
    COUNT(*) AS order_count,
    SUM(amount) AS total_amount
FROM dwd_orders
WHERE dt >= '2024-01-01' AND dt <= '2024-12-31'
GROUP BY category_id, dt;
```

**关键差异：** Hive 分区裁剪避免全表扫描，ORC 列存减少 I/O，数据量从 GB 到 TB 无需改 SQL。

### 新视角二：用 Spark SQL 替代 MySQL 分析

```python
spark = SparkSession.builder.appName("ECommerceAnalysis").getOrCreate()

df = spark.read.parquet("hdfs:///warehouse/dwd_orders/")

df.createOrReplaceTempView("orders")

result = spark.sql("""
    SELECT
        category_id,
        COUNT(*) AS order_count,
        SUM(amount) AS total_amount,
        PERCENTILE(amount, 0.5) AS median_amount
    FROM orders
    WHERE dt >= '2024-01-01'
    GROUP BY category_id
    ORDER BY total_amount DESC
""")

result.show()
```

**对比：** Spark SQL 支持分布式执行，MySQL 单机在大数据量下 PERCENTILE 等聚合函数性能急剧下降。

### 新视角三：用窗口函数重写复杂 SQL

**旧方案（子查询）：**

```sql
SELECT o.user_id, o.order_id, o.amount
FROM orders o
INNER JOIN (
    SELECT user_id, MAX(amount) AS max_amount
    FROM orders
    GROUP BY user_id
) m ON o.user_id = m.user_id AND o.amount = m.max_amount;
```

**新方案（窗口函数）：**

```sql
SELECT user_id, order_id, amount
FROM (
    SELECT
        user_id,
        order_id,
        amount,
        ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY amount DESC) AS rn
    FROM orders
) t
WHERE rn = 1;
```

**对比维度：**

| 维度 | 子查询方案 | 窗口函数方案 |
|------|-----------|-------------|
| 可读性 | 需理解 JOIN 逻辑 | 语义清晰 |
| 执行计划 | 两次扫描 + JOIN | 一次扫描 + 排序 |
| 扩展性 | 取 Top N 需改写 | 改 `rn <= N` 即可 |
| 维护性 | 多层嵌套难维护 | 结构扁平易维护 |

对 30 条 SQL 逐一优化，记录每条 SQL 的：
- 优化前执行计划与耗时
- 优化后执行计划与耗时
- 优化手段（分区裁剪/窗口函数/列裁剪/谓词下推）
- 性能提升倍数

## 3. L3 回头看 L1 项目4：离线数仓构建（3h）

**原始方案：** Hive + Spark + Sqoop

### 新视角一：用 Flink CDC 替代 Sqoop

**旧方案（Sqoop 增量导入）：**

```bash
sqoop import \
  --connect jdbc:mysql://mysql:3306/ecommerce \
  --table orders \
  --incremental append \
  --check-column id \
  --last-value 1000000 \
  --target-dir /warehouse/ods/orders \
  --fields-terminated-by '\t'
```

**新方案（Flink CDC）：**

```sql
-- Flink SQL CDC 表定义
CREATE TABLE orders_cdc (
    id BIGINT,
    user_id BIGINT,
    amount DECIMAL(12,2),
    order_time TIMESTAMP(3),
    PRIMARY KEY (id) NOT ENFORCED
) WITH (
    'connector' = 'mysql-cdc',
    'hostname' = 'mysql',
    'port' = '3306',
    'database-name' = 'ecommerce',
    'table-name' = 'orders',
    'server-time-zone' = 'Asia/Shanghai'
);

CREATE TABLE orders_ods (
    id BIGINT,
    user_id BIGINT,
    amount DECIMAL(12,2),
    order_time TIMESTAMP(3),
    dt STRING,
    PRIMARY KEY (id, dt) NOT ENFORCED
) WITH (
    'connector' = 'jdbc',
    'url' = 'jdbc:mysql://mysql:3306/dw',
    'table-name' = 'ods_orders'
);

INSERT INTO orders_ods
SELECT
    id, user_id, amount, order_time,
    DATE_FORMAT(order_time, 'yyyy-MM-dd') AS dt
FROM orders_cdc;
```

**对比：**

| 维度 | Sqoop | Flink CDC |
|------|-------|-----------|
| 延迟 | T+1（定时调度） | 秒级（实时捕获） |
| 数据一致性 | 需手动处理 | Snapshot + Binlog 保证 |
| 运维 | 需 Cron 调度 | 常驻进程 |
| 全量+增量 | 分两步操作 | 一体化 |

### 新视角二：用 Iceberg 替代 Hive 表格式

**旧方案（Hive 表）：**

```sql
CREATE TABLE dwd_orders (
    order_id STRING,
    user_id STRING,
    amount DECIMAL(12,2)
)
PARTITIONED BY (dt STRING)
STORED AS ORC;
```

**新方案（Iceberg 表）：**

```sql
CREATE TABLE dwd_orders (
    order_id STRING,
    user_id STRING,
    amount DECIMAL(12,2),
    dt STRING
)
USING iceberg
PARTITIONED BY (dt);

-- ACID：并发写入不冲突
-- Schema 演化：无痛加列
ALTER TABLE dwd_orders ADD COLUMN payment_method STRING;

-- 时间旅行：查询历史版本
SELECT * FROM dwd_orders VERSION AS OF 20240601000000;

-- 回滚：修复错误数据
CALL catalog.system.rollback_to_snapshot('dw.dwd_orders', 20240531000000);
```

**Iceberg 核心优势：**

| 特性 | Hive | Iceberg |
|------|------|---------|
| ACID | 不支持 | 支持 |
| Schema 演化 | 需重建分区 | 原生支持 |
| 时间旅行 | 不支持 | 支持 |
| 隐式分区 | 需手动指定 | 支持 |
| 小文件合并 | 需外部脚本 | 自带 Compaction |

### 新视角三：用 dbt 重构 ETL 逻辑

**旧方案（Spark 硬编码 ETL）：**


```sql
-- models/dwd/dwd_orders.sql
{{ config(
    materialized='incremental',
    partition_by={'field': 'dt', 'data_type': 'date'},
    cluster_by=['user_id']
) }}

SELECT
    o.order_id,
    o.user_id,
    u.user_name,
    u.city,
    o.amount,
    o.dt
FROM {{ ref('stg_orders') }} o

    ON o.user_id = u.user_id
```

## 4. L3 回头看 L2 项目7：实时交易监控大屏（3h）

**原始方案：** Kafka + Flink + ClickHouse + Grafana

### 新视角一：添加 Flink CEP 做复杂事件检测

**旧方案（简单聚合）：**

```java
DataStream<Transaction> txStream = ...;

txStream
    .keyBy(Transaction::getUserId)
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new AvgAmountAggregator())
    .addSink(new ClickHouseSink());
```

**新方案（Flink CEP 模式匹配）：**

```java
DataStream<Transaction> txStream = ...;

Pattern<Transaction, ?> fraudPattern = Pattern
    .<Transaction>begin("start")
        .where(tx -> tx.getAmount() > 10000)
    .followedBy("rapid")
        .where(tx -> tx.getAmount() > 10000)
        .within(Time.minutes(5));

PatternStream<Transaction> patternStream = CEP.pattern(
    txStream.keyBy(Transaction::getUserId),
    fraudPattern
);

DataStream<Alert> alerts = patternStream.select(map -> {
    Transaction first = map.get("start").get(0);
    Transaction second = map.get("rapid").get(0);
    return new Alert(
        first.getUserId(),
        "RAPID_HIGH_VALUE",
        first.getAmount() + " -> " + second.getAmount(),
        first.getTimestamp()
    );
});

alerts.addSink(new AlertSink());
```

**对比：** 从"5分钟窗口聚合"升级为"5分钟内连续两笔大额交易"的精确模式匹配。

## 5. L4 回头看全部项目：全局重构（4h）

从架构师视角审视 L0-L3 所有项目。

### 5.1 识别技术债务

| 项目 | 当时合理的设计 | 现在需要重构的原因 | 重构方案 |
|------|--------------|------------------|---------|
| L0 日志分析 | Python 单机脚本 | 无法处理 TB 级数据 | Spark/Flink 分布式 |
| L0 电商SQL | MySQL 单机 | 无分区、无分布式 | Hive/Spark SQL |
| L1 离线数仓 | Sqoop T+1 导入 | 延迟太高 | Flink CDC 实时入仓 |

| L2 推荐系统 | 规则引擎 | 精度不足 | ML 模型 + A/B 测试 |

| L0 电商SQL | MySQL root 无密码 | 高 | 最小权限 + 密码管理 |
| L1 数仓 | Hive 无认证 | 高 | Ranger 权限控制 |
| L2 大屏 | Kafka 无 ACL | 高 | Kafka ACL + SASL |
| L2 推荐 | 用户行为数据未脱敏 | 高 | 动态脱敏 + 审计日志 |
| L3 湖仓 | S3 Bucket 公开 | 严重 | IAM 策略 + Bucket 加密 |

**安全加固示例（Kafka ACL）：**

```bash
kafka-acls.sh --authorizer-properties zookeeper=zk:2181 \
  --add --allow-principal User:flink-producer \
  --producer --topic transactions

kafka-acls.sh --authorizer-properties zookeeper=zk:2181 \
  --add --allow-principal User:flink-consumer \
  --consumer --topic transactions \
  --group flink-consumer-group
```

## 6. 回头看方法论

### 为什么要回头看

- **避免"学完就忘"**：知识不用就会遗忘，回头看是主动回忆和巩固
- **建立知识连接**：初学时知识是碎片化的，回头看能将新旧知识串联成体系
- **培养批判思维**：用更高视角审视旧方案，理解"为什么当时这样设计"和"现在为什么需要改"

### 什么时候回头看

| 时机 | 目的 |
|------|------|
| 每升级一个 Level 后 | 用新能力重新评估旧方案 |
| 面试前 | 系统梳理项目经验，准备深度追问 |
| 做新项目前 | 避免重复踩坑，复用已有经验 |

### 怎么回头看

```
Step 1：运行旧代码
  → 确保旧项目能跑起来
  → 记录当前性能基线

Step 2：识别痛点
  → 性能瓶颈在哪里？
  → 哪些操作需要人工干预？
  → 哪些地方容易出错？

Step 3：用新方案重写
  → 选择 1-2 个痛点用新技术解决
  → 不要试图一次重写所有内容

Step 4：对比评估
  → 量化性能差异
  → 评估代码可维护性
  → 考虑学习成本和迁移风险

Step 5：写复盘笔记
  → 用下方模板记录
  → 归档到项目文档
```
