# 大数据面试题库与刷题指南

## 一、面试准备方法论

### 1.1 大数据岗位分类

| 岗位方向 | 核心职责 | 技术栈侧重 | 典型公司 |
|---------|---------|-----------|---------|
| 数据开发 | ETL开发、数据清洗、报表开发 | Hive/Spark SQL/Airflow | 传统互联网、金融 |
| 数据工程 | 数据管道、流处理、平台建设 | Kafka/Flink/Spark | 大厂、数据驱动型公司 |
| 大数据平台 | 集群运维、平台开发、组件调优 | Hadoop生态/K8s/监控 | 大厂基础架构团队 |
| 数据架构 | 架构设计、技术选型、规范制定 | 全栈+业务理解 | 中大型公司架构组 |

### 1.2 各级别面试侧重点

- **L1（初级工程师）**：基础概念 + 编码能力 + SQL实战 + 简单场景分析
- **L2（中级工程师）**：原理深入 + 调优经验 + 系统设计 + 故障排查
- **L3（高级工程师）**：源码理解 + 架构设计 + 技术决策 + 团队影响力

### 1.3 面试准备时间线

**提前3个月**：
- 系统梳理技术知识体系，查漏补缺
- 每天刷2-3道面试题，建立答题框架
- 开始整理项目经验，用STAR法则重构叙述
- 针对薄弱环节深入学习（如源码、调优）

**提前1个月**：
- 高频题集中突破，每道题限时3分钟作答
- 模拟面试：找朋友或同事进行mock interview
- 准备3-5个项目的深度讲述，覆盖不同技术栈
- 整理薪资数据和目标公司信息

**提前1周**：
- 回顾高频题和易错题
- 准备自我介绍（1分钟/3分钟/5分钟版本）
- 准备反问环节问题清单
- 调整作息，保持良好状态

### 1.4 简历优化

**项目经验包装原则**：
- 用STAR法则描述：Situation（背景）→ Task（任务）→ Action（行动）→ Result（结果）
- 量化产出：处理数据量、性能提升百分比、成本节省金额、SLA指标
- 突出技术深度：不只是"做了什么"，更要说"为什么这样做"

**示例**：
- ❌ "负责数据仓库ETL开发"
- ✅ "设计并实现ODS→DWD→DWS三层数仓架构，日均处理2TB数据，ETL耗时从4h优化至1.5h，支撑20+业务方实时查询，P99延迟<500ms"

## 二、L1级别面试题（初级工程师，100题）

### 2.1 Hadoop生态（25题）

**Q1：HDFS读写流程**
- 核心考点：HDFS数据流转机制与副本策略
- 参考答案要点：
  - 写流程：Client→NameNode（获取DataNode列表）→Pipeline方式写入→ACK确认→NameNode提交
  - 读流程：Client→NameNode（获取Block位置）→选择最近DataNode读取→校验checksum
  - 副本放置策略：第一个副本同机架、第二个副本不同机架、第三个副本同第二机架不同节点
  - Pipeline写入：数据包逐个传递，ACK逐个返回，保证所有副本写入完成
- 常见追问：NameNode挂了还能读吗？DataNode挂了写流程怎么处理？

**Q2：NameNode HA方案**
- 核心考点：HDFS高可用架构与脑裂防护
- 参考答案要点：
  - Active/Standby双NameNode架构，通过ZKFC切换
  - EditLog共享存储：QJM（Quorum Journal Manager）或NFS
  - ZKFC通过ZooKeeper实现自动故障转移，包含健康检查和选举逻辑
  - 防脑裂：SSH fence或Shell fence强制杀死旧Active
- 常见追问：QJM的半数写入机制怎么保证一致性？Standby如何保持元数据同步？

**Q3：MapReduce Shuffle过程**
- 核心考点：Map端到Reduce端数据流转全流程
- 参考答案要点：
  - Map端：分区→排序→Spill到磁盘→Merge合并
  - Reduce端：Copy拉取→Merge合并→排序→交给Reduce处理
  - Spill触发条件：缓冲区达到阈值（默认80%）开始Spill
  - Combiner可在Spill和Merge阶段执行，减少传输数据量
- 常见追问：Shuffle阶段可以做哪些优化？Combiner和Reducer的区别？

**Q4：Hive内部表vs外部表**
- 核心考点：表类型对数据生命周期管理的影响
- 参考答案要点：
  - 内部表：Hive管理数据和元数据，DROP时数据一起删除
  - 外部表：Hive只管理元数据，DROP时只删除元数据，数据保留
  - 生产建议：原始数据用外部表（防误删），加工后数据用内部表
  - 通过`EXTERNAL`关键字区分，`DESC FORMATTED table`可查看表类型
- 常见追问：内部表转外部表怎么做？外部表的数据路径变了怎么办？

**Q5：Hive分区vs分桶**
- 核心考点：数据组织方式与查询优化策略
- 参考答案要点：
  - 分区：按列值划分目录，粗粒度，避免全表扫描，对应HDFS目录
  - 分桶：按hash划分文件，细粒度，优化JOIN和采样，对应HDFS文件
  - 分区适合低基数列（如日期、地区），分桶适合高基数列（如用户ID）
  - 分桶数建议为2的幂次，分桶列选择JOIN键可优化MapJoin
- 常见追问：分区过多会有什么问题？分桶JOIN的原理是什么？

**Q6：小文件问题及解决方案**
- 核心考点：HDFS小文件对NameNode和计算引擎的影响
- 参考答案要点：
  - 影响：NameNode内存压力（每个文件~150B元数据）、Map任务过多、调度开销大
  - 产生原因：低频写入、分区过细、Spark小批次输出
  - 解决方案：`concat`/`merge`命令合并、`distribute by`控制Reducer数、定期小文件合并任务
  - 预防：控制分区粒度、使用CombineHiveInputFormat、Spark输出coalesce/repartition
- 常见追问：NameNode能存多少文件？小文件对Spark有什么影响？

**Q7：YARN资源调度器**
- 核心考点：三种调度器的特点与适用场景
- 参考答案要点：
  - FIFO：先进先出，单队列，不适合多租户
  - Capacity：多队列，保证最低资源，弹性借用，适合多租户
  - Fair：公平共享，队列间按权重分配，适合多用户共享
  - 生产多用Capacity Scheduler，支持队列层级和ACL控制
- 常见追问：Capacity Scheduler的弹性分配怎么工作？队列资源借还不还怎么办？

**Q8：Hive SQL执行顺序**
- 核心考点：SQL逻辑执行顺序与优化理解
- 参考答案要点：
  - 逻辑顺序：FROM→JOIN→WHERE→GROUP BY→HAVING→SELECT→DISTINCT→ORDER BY→LIMIT
  - 理解执行顺序有助于写高效SQL：先过滤再聚合，减少数据量
  - WHERE在GROUP BY前过滤行，HAVING在GROUP BY后过滤组
  - 子查询优化：谓词下推将过滤条件尽早执行
- 常见追问：LEFT JOIN时WHERE和ON的区别？Hive如何做谓词下推？

**Q9：MapReduce中的Combiner**
- 核心考点：Combiner的作用与使用限制
- 参考答案要点：
  - 作用：Map端本地聚合，减少Shuffle传输数据量
  - 限制：输入输出KV类型必须一致，操作必须满足交换律和结合律
  - 适用场景：SUM/COUNT/MAX/MIN，不适用AVG（需自定义实现）
  - 执行时机：Spill前和Merge后各可能执行一次
- 常见追问：为什么Combiner不能替代Reducer？求平均数用Combiner会怎样？

**Q10：HDFS数据一致性**
- 核心考点：HDFS如何保证数据正确性与一致性
- 参考答案要点：
  - Checksum校验：每个Block对应一个校验文件，读取时自动校验
  - 副本一致性：Pipeline写入保证所有副本数据一致
  - 租约机制：同一文件同一时刻只有一个写入者
  - 损坏恢复：Client检测到校验失败，从其他副本恢复
- 常见追问：HDFS支持修改操作吗？Append写入的一致性保证？

**Q11：Hive数据类型与文件格式**
- 核心考点：存储格式选择对性能的影响
- 参考答案要点：
  - TextFile：默认格式，行存储，压缩率低，查询慢
  - ORC：列存储，高压缩比，支持谓词下推和索引，Hive首选
  - Parquet：列存储，跨平台兼容好，Spark生态首选
  - 选择原则：Hive为主选ORC，Spark为主选Parquet，交互查询选ORC+Snappy
- 常见追问：列存储为什么比行存储查询快？ORC的索引机制是什么？

**Q12：Hive UDF开发**
- 核心考点：自定义函数扩展Hive能力
- 参考答案要点：
  - UDF：一进一出，继承`org.apache.hadoop.hive.ql.exec.UDF`
  - UDAF：多进一出（聚合），如自定义SUM
  - UDTF：一进多出，如explode的变体
  - 注册方式：`CREATE TEMPORARY FUNCTION`临时注册，打JAR包永久注册
- 常见追问：UDF的性能问题怎么优化？Python UDF和Java UDF的区别？

**Q13：Hadoop完全分布式搭建步骤**
- 核心考点：集群部署流程与核心配置
- 参考答案要点：
  - 核心配置文件：core-site.xml（NameNode地址）、hdfs-site.xml（副本数/目录）、yarn-site.xml（调度器/资源）
  - 关键步骤：SSH免密→配置文件→格式化NameNode→启动HDFS/YARN
  - NameNode格式化只做一次，重复格式化会导致ClusterID不一致
  - workers文件配置所有DataNode主机名
- 常见追问：格式化NameNode报ClusterID不一致怎么修复？如何动态添加DataNode？

**Q14：SecondaryNameNode的作用**
- 核心考点：检查点机制与元数据管理
- 参考答案要点：
  - 不是NameNode的热备，是辅助检查点节点
  - 定期合并FsImage和EditLog，减少NameNode重启恢复时间
  - 工作流程：下载FsImage→回放EditLog→生成新FsImage→上传给NameNode
  - 触发条件：按时间（默认1h）或按EditLog条数（默认100万条）
- 常见追问：有了HA还需要SecondaryNameNode吗？Checkpoint间隔怎么调？

**Q15：Hive JOIN优化**
- 核心考点：不同JOIN策略的适用场景
- 参考答案要点：
  - Map Join：小表放入内存广播，避免Shuffle，适合小表JOIN大表
  - Reduce Join：通用方式，Shuffle后Reduce端JOIN，数据倾斜风险大
  - Bucket Map Join：分桶表JOIN，按桶匹配，减少内存压力
  - SMB Join（Sort Merge Bucket Join）：分桶+排序，流式JOIN，适合大表JOIN大表
- 常见追问：Map Join的内存阈值怎么设？SMB Join的前置条件是什么？

**Q16：HDFS Block大小设置**
- 核心考点：Block大小对性能的影响与调优
- 参考答案要点：
  - 默认128MB（Hadoop 2.x+），可配置
  - 过小：NameNode内存压力大，Map任务数过多
  - 过大：并行度不足，数据本地性差
  - 调优依据：数据量、文件大小分布、集群规模
- 常见追问：为什么不是64MB或256MB？SSD对Block大小选择有影响吗？

**Q17：MapReduce数据倾斜**
- 核心考点：倾斜的识别与处理方法
- 参考答案要点：
  - 表现：某个Reduce任务运行时间远超其他
  - 原因：Key分布不均匀、空值过多、关联键类型不匹配
  - 解决：加盐打散（加随机前缀）、两阶段聚合、Map Join替代Reduce Join
  - 空值处理：过滤空值或给空值加随机数分散
- 常见追问：两阶段聚合的具体流程？加盐后结果怎么还原？

**Q18：Hive窗口函数**
- 核心考点：窗口函数语法与常见应用场景
- 参考答案要点：
  - 语法：`函数() OVER (PARTITION BY ... ORDER BY ... ROWS/RANGE ...)`
  - ROWS：按物理行偏移，RANGE：按逻辑值偏移
  - 常用函数：ROW_NUMBER/RANK/DENSE_RANK、LEAD/LAG、SUM/AVG累计
  - 经典场景：排名、连续登录、留存计算、TopN
- 常见追问：ROW_NUMBER和RANK的区别？累计求和的窗口怎么写？

**Q19：Hadoop机架感知**
- 核心考点：机架感知对数据本地性和容错的作用
- 参考答案要点：
  - NameNode维护机架拓扑，DataNode启动时上报机架信息
  - 副本放置策略依赖机架感知：同机架→跨机架→同机架不同节点
  - 读数据时优先选择同机架DataNode，减少跨交换机流量
  - 自定义机架感知脚本：实现`DNSToSwitchMapping`接口
- 常见追问：没有配置机架感知会怎样？机架感知脚本的返回格式？

**Q20：Hive动态分区**
- 核心考点：动态分区的配置与使用
- 参考答案要点：
  - 设置：`hive.exec.dynamic.partition=true`，`hive.exec.dynamic.partition.mode=nonstrict`
  - 用法：INSERT时只指定分区列名，值由SELECT字段自动确定
  - 分区列必须是SELECT最后几个字段
  - 注意：控制最大分区数`hive.exec.max.dynamic.partitions`，防止分区爆炸
- 常见追问：动态分区和静态分区可以混用吗？分区数过多怎么办？

**Q21：MapReduce InputFormat**
- 核心考点：不同InputFormat对数据读取的影响
- 参考答案要点：
  - TextInputFormat：默认，按行读取，key是偏移量，value是行内容
  - KeyValueTextInputFormat：按分隔符切分key和value
  - CombineFileInputFormat：合并小文件，减少Map任务数
  - NLineInputFormat：按行数切分，每个Map处理固定行数
  - 自定义InputFormat：继承FileInputFormat，重写getSplits和createRecordReader
- 常见追问：CombineFileInputFormat的合并策略？如何自定义InputFormat？

**Q22：Hive元数据管理**
- 核心考点：Metastore架构与配置
- 参考答案要点：
  - 三种模式：内嵌Derby（测试）、本地MySQL（单用户）、远程Metastore（生产）
  - Metastore存储：表定义、分区信息、列类型、存储位置、序列化信息
  - 生产建议：独立Metastore服务+MySQL后端，支持多HiveServer2共享
  - 备份策略：定期导出MySQL元数据，防止元数据丢失
- 常见追问：Metastore挂了Hive还能查询吗？如何迁移Metastore后端数据库？

**Q23：HDFS权限控制**
- 核心考点：HDFS安全机制与访问控制
- 参考答案要点：
  - 类POSIX权限模型：owner/group/other，rwx权限位
  - ACL：更细粒度的权限控制，支持给特定用户/组授权
  - Kerberos认证：生产环境必须开启，防止伪造身份
  - Ranger/Sentry：统一的权限管理平台，支持列级权限
- 常见追问：HDFS权限和Linux权限的区别？没有Kerberos会有什么风险？

**Q24：MapReduce Counter**
- 核心考点：计数器的使用与自定义
- 参考答案要点：
  - 内置计数器：Map输入记录数、Spill次数、Shuffle字节数等
  - 自定义计数器：`context.getCounter()`统计业务指标
  - 用途：数据质量校验、异常数据统计、性能分析
  - 架构级别：Task→ApplicationMaster→HistoryServer
- 常见追问：Counter的值在哪里查看？自定义Counter的性能影响？

**Q25：Hadoop联邦（Federation）**
- 核心考点：Federation解决NameNode单点扩展问题
- 参考答案要点：
  - 多个NameNode各自管理一部分命名空间（Block Pool）
  - 每个NameNode独立，互不干扰，共享DataNode存储
  - 解决单NameNode内存瓶颈，支持水平扩展
  - ViewFS提供统一命名空间视图给客户端
- 常见追问：Federation和HA可以同时使用吗？Block Pool的概念？

### 2.2 Spark核心（25题）

**Q1：RDD vs DataFrame vs Dataset**
- 核心考点：Spark三种数据抽象的区别与选择
- 参考答案要点：
  - RDD：底层抽象，无Schema，无Catalyst优化，Java序列化，灵活但性能差
  - DataFrame：行式Schema，Catalyst优化+Tungsten执行，运行时类型安全
  - Dataset：编译时类型安全+Catalyst优化，Scala/Java可用，Python通过DataFrame
  - 选择：优先DataFrame/Dataset，只在需要底层控制时用RDD
- 常见追问：为什么DataFrame比RDD快？Python能用Dataset吗？

**Q2：Spark提交模式（YARN client vs cluster）**
- 核心考点：两种部署模式的区别与适用场景
- 参考答案要点：
  - Client模式：Driver运行在提交任务的客户端，本地可见日志，适合交互调试
  - Cluster模式：Driver运行在YARN的ApplicationMaster中，客户端可退出，适合生产
  - 区别：Driver位置不同→日志位置不同→资源分配方式不同
  - 生产建议：生产用Cluster模式，开发调试用Client模式
- 常见追问：Client模式Driver挂了会怎样？Cluster模式怎么看Driver日志？

**Q3：数据倾斜诊断与解决**
- 核心考点：Spark数据倾斜的识别与处理
- 参考答案要点：
  - 诊断：Spark UI看Task运行时间分布，某个Task数据量远超其他
  - 原因：Key分布不均、空值、默认Hash分区
  - 解决方案：增加并行度、盐化（salting）、两阶段聚合、广播JOIN
  - 两阶段聚合：先加随机前缀局部聚合→去掉前缀全局聚合
- 常见追问：`repartition`和`coalesce`在处理倾斜时的区别？如何判断是数据倾斜还是内存不足？

**Q4：Spark内存模型**
- 核心考点：Executor内存划分与管理
- 参考答案要点：
  - 堆内内存：Execution（shuffle/buffer）+ Storage（cache）+ User（用户数据）+ Reserved
  - 堆外内存：Off-heap，直接操作系统内存，减少GC开销
  - `spark.memory.fraction`（默认0.6）：统一内存占堆内比例
  - `spark.memory.storageFraction`（默认0.5）：Storage占统一内存比例
  - Execution和Storage可互相借用，Storage不够可驱逐缓存
- 常见追问：Storage被驱逐的条件？OOM通常是什么内存不够？

**Q5：广播变量与累加器**
- 核心考点：Spark共享变量的原理与使用
- 参考答案要点：
  - 广播变量：只读共享变量，Driver分发到每个Executor一份，避免每Task复制
  - 使用场景：小表JOIN、配置参数分发、维度表广播
  - 累加器：只写共享变量，用于计数和求和，只有Driver能读取值
  - 注意：在Transformation中累加器可能重复计算，应在Action中读取
- 常见追问：广播变量的大小限制？累加器在RDD中为什么不准确？

**Q6：AQE自适应查询执行**
- 核心考点：Spark 3.0 AQE三大优化机制
- 参考答案要点：
  - 动态合并Shuffle分区：运行时合并小分区，减少Task数
  - 动态切换JOIN策略：运行时根据数据量切换SortMergeJoin→BroadcastJoin
  - 动态优化数据倾斜：检测倾斜分区并拆分处理
  - 开启：`spark.sql.adaptive.enabled=true`
- 常见追问：AQE的倾斜检测阈值怎么设？AQE和CBO的关系？

**Q7：Spark Shuffle机制**
- 核心考点：HashShuffle vs SortShuffle
- 参考答案要点：
  - HashShuffle（已废弃）：每个Map Task生成R个文件，文件数M×R
  - SortShuffle：Map端排序后写入一个数据文件+索引文件，文件数2M
  - BypassMergeSortShuffle：无排序无预聚合，直接写后合并，适合无需排序的场景
  - UnsafeShuffle（Tungsten）：序列化数据直接排序，零拷贝，性能最优
- 常见追问：Shuffle文件数过多怎么办？Bypass触发的条件？

**Q8：Spark Stage划分**
- 核心考点：DAG调度与Stage切分逻辑
- 参考答案要点：
  - 宽依赖（Shuffle Dependency）切分Stage边界
  - 窄依赖（Narrow Dependency）在同一个Stage内流水线执行
  - Stage内部Task并行执行，Stage之间串行依赖
  - DAGScheduler从后往前遍历RDD血缘，遇到宽依赖就切分
- 常见追问：map和filter是宽依赖还是窄依赖？union呢？

**Q9：Spark缓存策略**
- 核心考点：persist/cache的存储级别与选择
- 参考答案要点：
  - `MEMORY_ONLY`：纯内存，不序列化，速度快但占内存大
  - `MEMORY_AND_DISK`：内存放不下溢写到磁盘，安全但慢
  - `MEMORY_ONLY_SER`：序列化存储，省内存但需序列化开销
  - `DISK_ONLY`：纯磁盘，适合大数据量低频访问
  - 选择原则：内存够用MEMORY_ONLY，不够用MEMORY_ONLY_SER，要容错MEMORY_AND_DISK
- 常见追问：cache()和persist()的区别？缓存什么时候该清除？

**Q10：Spark任务调度**
- 核心考点：从提交到执行的调度全流程
- 参考答案要点：
  - 流程：Application→Job→Stage→Task
  - Action触发Job，每个Job对应一个DAG
  - DAGScheduler划分Stage，TaskScheduler分配Task
  - 数据本地性：PROCESS_LOCAL→NODE_LOCAL→RACK_LOCAL→ANY
  - 推测执行：慢Task在其他节点重试，`spark.speculation=true`
- 常见追问：一个Job可以有多个Action吗？推测执行有什么风险？

**Q11：Spark Checkpoint**
- 核心考点：检查点机制与容错
- 参考答案要点：
  - 作用：截断RDD血缘，将中间结果持久化到可靠存储（如HDFS）
  - 场景：血缘链过长、依赖过深、计算代价大
  - 与Cache区别：Checkpoint切断血缘，Cache保留血缘
  - 使用：`sc.setCheckpointDir()`→`rdd.checkpoint()`，需先Action触发
- 常见追问：Checkpoint和Persist的区别？为什么Checkpoint前要先Action？

**Q12：Spark SQL执行流程**
- 核心考点：SQL到物理执行的完整流程
- 参考答案要点：
  - SQL→Parser（ANTLR解析为AST）→Analyzer（绑定元数据）→Optimizer（Catalyst优化）→Planner（生成物理计划）→Execution（执行）
  - Catalyst优化：谓词下推、列裁剪、常量折叠、JOIN重排序
  - Tungsten优化：全阶段代码生成、Off-heap内存管理
  - 物理计划选择：基于代价（CBO）或基于规则（RBO）
- 常见追问：CBO和RBO的区别？全阶段代码生成的原理？

**Q13：Spark Join策略**
- 核心考点：不同JOIN实现方式的选择
- 参考答案要点：
  - Broadcast Hash Join：小表广播，适合小表<广播阈值（默认10MB）
  - Sort Merge Join：两边排序后归并，适合大表JOIN大表，默认策略
  - Shuffle Hash Join：小表分桶后构建Hash表，适合中等大小表
  - 选择优先级：Broadcast > Shuffle Hash > Sort Merge
- 常见追问：广播阈值怎么调？Sort Merge Join为什么要排序？

**Q14：Spark OOM排查**
- 核心考点：内存溢出的常见原因与解决
- 参考答案要点：
  - Map端OOM：数据膨胀、大对象、缓冲区不足
  - Shuffle/Reduce端OOM：数据倾斜、内存配置不足
  - 排查：Spark UI看Task数据量、GC时间、Shuffle读写量
  - 解决：增加Executor内存、调整并行度、处理倾斜、优化代码避免collect
- 常见追问：Driver OOM和Executor OOM分别怎么排查？`collect()`为什么容易OOM？

**Q15：Spark与MapReduce对比**
- 核心考点：Spark的优势与适用场景
- 参考答案要点：
  - 内存计算：Spark中间结果放内存，MR每次写磁盘
  - DAG模型：Spark Stage内流水线执行，MR只有Map和Reduce两阶段
  - 启动开销：Spark Task启动快（线程级），MR启动慢（进程级）
  - 迭代计算：Spark缓存中间结果，MR每次重新读取
  - MR优势：超大规模数据稳定性更好、资源隔离更强
- 常见追问：Spark什么时候不如MR？Spark的稳定性问题有哪些？

**Q16：Spark Streaming微批处理**
- 核心考点：DStream模型与微批处理机制
- 参考答案要点：
  - DStream：离散化流，将流数据按时间间隔切分为微批RDD
  - 处理模型：接收数据→按batch interval切分→每个batch生成一个RDD→处理
  - 批间隔设置：根据数据处理延迟确定，保证处理时间<批间隔
  - 背压机制：`spark.streaming.backpressure.enabled=true`自动调节接收速率
- 常见追问：DStream和Structured Streaming的区别？微批的延迟问题？

**Q17：Spark序列化**
- 核心考点：序列化对性能的影响与选择
- 参考答案要点：
  - Java序列化：默认，慢、体积大，兼容性好
  - Kryo序列化：快10倍、体积小，需注册类，部分类型不支持
  - 配置：`spark.serializer=org.apache.spark.serializer.KryoSerializer`
  - 注册：`spark.kryo.classesToRegister`注册自定义类提升性能
- 常见追问：Kryo为什么比Java快？哪些场景不能用Kryo？

**Q18：Spark资源调优**
- 核心考点：Executor和Driver资源配置
- 参考答案要点：
  - Executor数量：`--num-executors`，通常为集群总核数的2-3倍
  - Executor核数：`--executor-cores`，建议2-5核，避免HDFS客户端竞争
  - Executor内存：`--executor-memory`，根据数据量和缓存需求设置
  - 动态分配：`spark.dynamicAllocation.enabled=true`，按需增减Executor
  - 经验公式：总核数 = 并行度，每个Executor 5核 8GB 是常见起点
- 常见追问：动态分配的策略是什么？Executor太多有什么问题？

**Q19：Spark UDF与性能**
- 核心考点：UDF对Catalyst优化的影响
- 参考答案要点：
  - UDF是黑盒，Catalyst无法优化UDF内部逻辑
  - 影响列裁剪、谓词下推、常量折叠等优化失效
  - Pandas UDF（Spark 2.3+）：使用Arrow批量处理，性能更好
  - 建议：优先用内置函数，必须用UDF时考虑Pandas UDF
- 常见追问：Pandas UDF有哪几种类型？为什么内置函数比UDF快？

**Q20：Spark Partition并行度**
- 核心考点：并行度设置对性能的影响
- 参考答案要点：
  - `spark.default.parallelism`：RDD默认并行度
  - `spark.sql.shuffle.partitions`：SQL Shuffle默认分区数（默认200）
  - 原则：每个分区128MB-200MB，总分区数≥总核数×2-3
  - 分区过少：并行度低、数据倾斜、OOM
  - 分区过多：调度开销大、小任务多
- 常见追问：如何判断当前并行度是否合理？`repartition`和`coalesce`的区别？

**Q21：Spark容错机制**
- 核心考点：RDD血缘与容错恢复
- 参考答案要点：
  - RDD血缘（Lineage）：记录从源头到当前RDD的转换链
  - 分区丢失时，根据血缘重新计算该分区
  - 窄依赖：只需重算丢失分区的父分区
  - 宽依赖：可能需要重算所有父分区（因为Shuffle）
  - Checkpoint可截断血缘，加速恢复
- 常见追问：宽依赖恢复为什么代价大？如何减少恢复时间？

**Q22：Spark Whole-Stage Code Generation**
- 核心考点：Tungsten全阶段代码生成原理
- 参考答案要点：
  - 将多个物理算子融合为一个Java函数，消除虚函数调用
  - 避免中间数据在内存中的物化，减少内存分配和GC
  - 类似于手写循环的性能，消除框架开销
  - Spark UI中可以看到"WholeStageCodegen"标记
- 常见追问：哪些算子支持全阶段代码生成？为什么不全部支持？

**Q23：Spark Streaming Exactly-Once**
- 核心考点：流处理端到端精确一次语义
- 参考答案要点：
  - 挑战：输出可能重复（重启后重放）
  - 幂等输出：重复写入不影响结果（如Redis SET、HBase PUT）
  - 事务输出：将offset和结果在同一个事务中提交
  - WAL方案：Spark Streaming内置Write-Ahead Log
- 常见追问：幂等和事务的区别？Structured Streaming如何实现Exactly-Once？

**Q24：Spark Structured Streaming**
- 核心考点：新一代流处理模型
- 参考答案要点：
  - 无界表模型：流数据不断追加到表中，查询产生结果表
  - 统一API：DataFrame/SQL同时处理批和流
  - 输出模式：Append/Complete/Update
  - Watermark：处理迟到数据，`withWatermark()`设置延迟阈值
  - 支持Event Time处理和状态管理
- 常见追问：Structured Streaming和Flink的区别？状态存储用什么？

**Q25：Spark性能调优总结**
- 核心考点：系统化的调优方法论
- 参考答案要点：
  - 数据层：压缩（Snappy）、列存（Parquet）、分区裁剪、小文件合并
  - 代码层：避免collect、用内置函数替代UDF、合理缓存、广播小表
  - 配置层：并行度、序列化、内存分配、Shuffle参数
  - 架构层：数据倾斜处理、Stage优化、AQE开启
  - 调优流程：看Spark UI→定位瓶颈→针对性优化→验证效果
- 常见追问：如何用Spark UI定位性能瓶颈？调优的优先级顺序？

### 2.3 SQL与数据仓库（25题）

**Q1：窗口函数实战题**
- 核心考点：窗口函数的灵活运用
- 参考答案要点：
  - 排名：`ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC)`
  - 累计：`SUM(amount) OVER (PARTITION BY user_id ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)`
  - 偏移：`LAG(date, 1) OVER (PARTITION BY user_id ORDER BY date)`取上一行
  - 占比：`salary / SUM(salary) OVER (PARTITION BY dept)`算部门占比
- 常见追问：ROWS和RANGE的区别？如何用窗口函数求移动平均？

**Q2：数仓分层设计（ODS/DWD/DWS/ADS）**
- 核心考点：数仓分层的意义与各层职责
- 参考答案要点：
  - ODS：原始数据层，保持与源系统一致，不清洗
  - DWD：明细数据层，清洗、标准化、维度退化，保持粒度不变
  - DWS：汇总数据层，按主题轻度汇总，宽表
  - ADS：应用数据层，面向具体业务场景的指标表
  - 分层意义：复用性、可维护性、数据一致性、问题追溯
- 常见追问：DWD和DWS的粒度区别？为什么不能跳过DWD直接到DWS？

**Q3：维度建模（星型/雪花）**
- 核心考点：两种维度建模方式的对比
- 参考答案要点：
  - 星型：事实表居中，维度表直接关联，维度表不规范化，查询快
  - 雪花：维度表进一步规范化拆分，节省存储但JOIN多、查询慢
  - 星型更常用：查询性能优先，存储成本相对低
  - 选择依据：查询性能 vs 存储空间 vs ETL复杂度
- 常见追问：星座模型是什么？什么时候用雪花模型更合适？

**Q4：缓慢变化维度（SCD）**
- 核心考点：维度数据变化的处理策略
- 参考答案要点：
  - SCD1：直接覆盖，不保留历史，简单但丢失变化轨迹
  - SCD2：新增行，用起止时间和当前标志记录历史，最常用
  - SCD3：新增列，只保留上一次值，空间有限
  - SCD2实现：`start_date`、`end_date`、`is_current`三个字段
- 常见追问：SCD2如何查询当前有效记录？SCD2的ETL怎么写？

**Q5：数据质量保障方法**
- 核心考点：数据质量维度与保障体系
- 参考答案要点：
  - 六大维度：完整性、准确性、一致性、及时性、唯一性、有效性
  - 事前：数据规范制定、Schema校验
  - 事中：ETL过程校验、异常检测、阈值告警
  - 事后：数据对账、质量报告、问题追溯
  - 工具：Great Expectations、dbt test、自研质量平台
- 常见追问：如何量化数据质量？数据质量问题怎么追溯？

**Q6：SQL连续登录问题**
- 核心考点：经典窗口函数应用
- 参考答案要点：
  - 方法一：`ROW_NUMBER() - 登录日期`得到分组标识，相同标识为连续
  - 步骤：去重→ROW_NUMBER按日期排序→日期减序号→分组计数
  - 方法二：`LAG`取上次登录日期，差值=1则连续
  - 扩展：连续N天登录、最大连续登录天数、断点续登
- 常见追问：如何求最大连续登录天数？允许间隔1天算连续怎么改？

**Q7：留存率计算**
- 核心考点：用户留存分析的SQL实现
- 参考答案要点：
  - 定义：第N日留存率 = 第N日回访用户数 / 首日新增用户数 × 100%
  - 实现：首日表JOIN第N日表，`DATEDIFF(login_date, first_date) = N`
  - 一次查询多日留存：用CASE WHEN + GROUP BY
  - 注意：分母是首日新增用户数，不是首日登录用户数
- 常见追问：如何计算周留存？留存率的业务意义是什么？

**Q8：TopN问题**
- 核心考点：分组取前N名的SQL写法
- 参考答案要点：
  - `ROW_NUMBER() OVER (PARTITION BY group_col ORDER BY value DESC) AS rn` → `WHERE rn <= N`
  - RANK vs ROW_NUMBER：并列排名用RANK，严格TopN用ROW_NUMBER
  - 大数据量优化：先过滤再排名，减少排序数据量
  - Hive优化：设置`hive.limit.row.max.size`控制Reduce端数据量
- 常见追问：如何取每个部门薪资前三名？并列第三名怎么处理？

**Q9：数据仓库与数据库的区别**
- 核心考点：OLAP与OLTP的本质区别
- 参考答案要点：
  - 数据库（OLTP）：事务处理、行存储、实时增删改、规范化建模
  - 数仓（OLAP）：分析查询、列存储、批量读写、反规范化建模
  - 数仓面向主题，数据库面向应用
  - 数仓保留历史，数据库通常只保留当前状态
- 常见追问：HTAP是什么？为什么数仓用列存储？

**Q10：SQL执行计划分析**
- 核心考点：EXPLAIN的使用与执行计划解读
- 参考答案要点：
  - `EXPLAIN`查看逻辑计划，`EXPLAIN EXTENDED`查看物理计划
  - 关注：JOIN类型、扫描行数、分区裁剪、过滤条件位置
  - 优化信号：全表扫描、大表Shuffle JOIN、未下推的过滤
  - Hive：`EXPLAIN`看Stage依赖和Operator链
- 常见追问：如何判断JOIN顺序是否合理？分区裁剪怎么验证？

**Q11：数据仓库建模方法论**
- 核心考点：Inmon vs Kimball
- 参考答案要点：
  - Inmon（自顶向下）：先建企业级数据仓库（3NF），再建数据集市
  - Kimball（自底向上）：先建维度模型数据集市，再合并为数仓
  - Inmon：数据一致性好，建设周期长，成本高
  - Kimball：见效快，迭代快，可能数据不一致
  - 实践：大公司倾向Inmon+Kimball混合，小公司用Kimball
- 常见追问：Data Vault建模是什么？适合什么场景？

**Q12：事实表类型**
- 核心考点：不同事实表的特点与选择
- 参考答案要点：
  - 事务事实表：记录每次事件，粒度最细，如订单明细
  - 周期快照事实表：固定时间间隔快照，如每日余额
  - 累积快照事实表：记录业务全生命周期，如订单从创建到完成
  - 无事实事实表：只记录关系，如学生选课
- 常见追问：订单表用哪种事实表？累积快照的关键日期字段有哪些？

**Q13：SQL去重方法**
- 核心考点：不同去重方式的性能与语义
- 参考答案要点：
  - `DISTINCT`：全字段去重，全局Shuffle
  - `GROUP BY`：按字段分组去重，可配合聚合函数
  - `ROW_NUMBER()`：灵活去重，可保留特定行（如最新一条）
  - 性能：DISTINCT通常优化为GROUP BY，大数据量用ROW_NUMBER更灵活
- 常见追问：DISTINCT和GROUP BY哪个快？如何去重并保留最新记录？

**Q14：数据增量同步方案**
- 核心考点：增量数据的捕获与同步
- 参考答案要点：
  - 时间戳：源表有`update_time`字段，按时间增量抽取
  - CDC（Change Data Capture）：Binlog实时捕获变更
  - 全量比对：MD5哈希比对，无侵入但性能差
  - 触发器：数据库层面捕获变更，影响源库性能
  - 生产建议：优先CDC（Canal/Debezium），次选时间戳增量
- 常见追问：CDC的延迟怎么控制？增量同步如何保证不丢不重？

**Q15：SQL NULL处理**
- 核心考点：NULL值的特殊行为与处理
- 参考答案要点：
  - `NULL = NULL`结果为NULL（不是TRUE），`IS NULL`判断
  - 聚合函数忽略NULL：`COUNT(*)`包含NULL行，`COUNT(col)`忽略NULL
  - `COALESCE(col, default)`替换NULL，`IFNULL`/`NVL`类似
  - JOIN中NULL不匹配，需特殊处理
- 常见追问：`COUNT(1)`和`COUNT(*)`的区别？`NULL`在排序中怎么处理？

**Q16：数据仓库粒度设计**
- 核心考点：粒度选择对数仓灵活性和性能的影响
- 参考答案要点：
  - 粒度越细：灵活性越高，存储成本越大
  - 粒度越粗：查询越快，但无法下钻到细节
  - DWD保持最细粒度（事务级），DWS按需聚合

**Q1：零拷贝原理**
- 核心考点：Kafka高性能IO的底层机制
- 参考答案要点：
  - 传统IO：磁盘→内核缓冲→用户缓冲→Socket缓冲→网卡（4次拷贝）
  - 零拷贝（sendfile）：磁盘→内核缓冲→网卡（2次拷贝，DMA直接传输）
  - Kafka使用`java.nio.channels.FileChannel.transferTo()`实现零拷贝
  - 前提：Consumer从Broker拉取消息时走零拷贝，Producer写不走
- 常见追问：mmap和sendfile的区别？Java NIO的零拷贝实现？

**Q2：ISR机制**
- 核心考点：Kafka副本同步与一致性保证
- 参考答案要点：
  - ISR（In-Sync Replicas）：与Leader保持同步的副本集合
  - 判断标准：`replica.lag.time.max.ms`（默认10s），超过未同步则移出ISR
  - Leader选举：优先从ISR中选举，保证不丢数据
  - `unclean.leader.election.enable=false`（默认）禁止非ISR副本成为Leader
- 常见追问：ISR为空时Leader挂了怎么办？ISR频繁伸缩怎么处理？

**Q3：Exactly-Once语义实现**

  - 幂等Producer：`enable.idempotence=true`，PID+SequenceNumber去重
  - 事务：`transactional.id`，保证跨分区原子写入
  - Consumer端：`isolation.level=read_committed`只读已提交消息

**Q4：Consumer Rebalance策略**
- 核心考点：消费者重平衡的触发与策略
- 参考答案要点：
  - 触发条件：Consumer加入/退出、订阅Topic分区变化、Consumer超时
  - RangeAssignor：按分区范围连续分配，可能不均匀
  - RoundRobinAssignor：轮询分配，更均匀
  - StickyAssignor：尽量保持原有分配，减少迁移
  - CooperativeStickyAssignor：增量重平衡，不全量停止
- 常见追问：Rebalance期间Consumer能消费吗？如何避免频繁Rebalance？

**Q5：分区数如何确定**
- 核心考点：分区规划的方法论
- 参考答案要点：
  - 吞吐量公式：分区数 = 目标吞吐量 / 单分区吞吐量
  - 单分区生产吞吐约10MB/s，消费吞吐约20-30MB/s
  - 分区数上限：影响文件句柄数、Leader选举时间、内存开销
  - 建议：按未来1-2年增长预估，适当冗余，不宜过多
- 常见追问：分区数可以增加但不能减少，为什么？分区过多有什么问题？

**Q6：Kafka消息丢失场景与防护**
- 核心考点：消息可靠性保障
- 参考答案要点：
  - Producer端：`acks=all` + 重试 + 幂等
  - Broker端：`min.insync.replicas=2` + `unclean.leader.election.enable=false`
  - Consumer端：手动提交offset，处理完再提交
  - 端到端：事务或幂等写入下游
- 常见追问：acks=1会丢数据吗？Consumer自动提交有什么风险？

**Q7：Kafka消息顺序性**
- 核心考点：消息顺序保证的范围与限制
- 参考答案要点：
  - 单分区有序：消息按写入顺序消费
  - 多分区无全局顺序：不同分区之间无顺序保证
  - 保证顺序：相同Key的消息发往同一分区（Key Hash分区）
  - 失序场景：重试、Rebalance、多Producer并发
- 常见追问：如何保证多分区的全局有序？有序性和吞吐量怎么平衡？

**Q8：Kafka存储架构**
- 核心考点：日志存储与索引机制
- 参考答案要点：
  - 存储结构：Topic→Partition→Segment（.log + .index + .timeindex）
  - .log：消息数据文件，顺序追加写
  - .index：稀疏索引，offset→物理位置映射，二分查找
  - .timeindex：时间戳索引，支持按时间查找
  - Segment滚动：`log.segment.bytes`（默认1GB）或`log.roll.hours`
- 常见追问：稀疏索引的间隔怎么设？如何根据offset快速定位消息？

**Q9：Kafka Controller**
- 核心考点：Controller的职责与选举
- 参考答案要点：
  - 第一个注册`/controller`节点的Broker成为Controller
  - 职责：分区Leader选举、副本重分配、Topic管理、Broker上下线处理
  - 故障转移：Controller节点挂了，其他Broker竞争创建临时节点
  - KRaft模式（Kafka 3.x+）：移除ZooKeeper依赖，内置Raft共识
- 常见追问：Controller单点问题怎么解决？KRaft模式的优势？

**Q10：Kafka延迟队列实现**
- 核心考点：延迟消息的设计方案
- 参考答案要点：
  - 方案一：多级延迟Topic，到期转入目标Topic
  - 方案二：时间轮算法，Kafka内部用层级时间轮管理延迟操作
  - 方案三：外部存储（Redis）记录到期时间，定时扫描
  - 生产建议：简单场景用多级Topic，复杂场景用时间轮
- 常见追问：时间轮的原理？Kafka内部哪些操作用了时间轮？

**Q11：Kafka消息压缩**
- 核心考点：压缩算法选择与批量压缩机制
- 参考答案要点：
  - Producer端压缩，Broker端保持压缩传输，Consumer端解压
  - 算法对比：gzip（高压缩比慢）、snappy（中等快）、lz4（低极快）、zstd（最优平衡）
  - 批量压缩：整个batch一起压缩，压缩比更高
  - `compression.type=producer`让Producer决定压缩算法
- 常见追问：压缩在哪个阶段生效？Broker端会重新压缩吗？

**Q12：Kafka高水位与Leader Epoch**
- 核心考点：消息可见性与一致性机制
- 参考答案要点：
  - HW（High Watermark）：所有ISR副本都已复制的offset，Consumer只能读到HW之前
  - LEO（Log End Offset）：每个副本的最新offset
  - Leader Epoch：单调递增的Leader版本号，解决HW截断导致的数据不一致
  - Leader Epoch方案：新Leader不依赖HW截断，而是根据Epoch恢复
- 常见追问：HW截断导致数据丢失的场景？Leader Epoch如何解决？

**Q13：Kafka消费者组**
- 核心考点：消费者组的协调与管理
- 参考答案要点：
  - Group Coordinator：每个消费者组分配一个Coordinator Broker
  - 分配策略：`partition.assignment.strategy`配置
  - Offset管理：`__consumer_offsets`内部Topic，默认自动提交
  - 消费者数量>分区数：多余消费者空闲
- 常见追问：Coordinator如何选择？消费者数和分区数的关系？

**Q14：Kafka生产者发送模式**
- 核心考点：同步/异步发送与可靠性配置
- 参考答案要点：
  - 同步：`future.get()`等待确认，低吞吐高可靠
  - 异步：回调函数处理结果，高吞吐需处理异常
  - `acks=0`：不等确认，最快但可能丢
  - `acks=1`：Leader确认，折中
  - `acks=all`：所有ISR确认，最可靠最慢
- 常见追问：异步发送失败怎么处理？`retries`和`retry.backoff.ms`怎么配？

**Q15：Kafka集群部署规划**
- 核心考点：集群规模与资源配置
- 参考答案要点：
  - Broker数量：基于吞吐量和可用性，至少3个
  - 磁盘：SSD优先，容量 = 日数据量 × 副本数 × 保留天数 × 1.2
  - 内存：Broker堆内存6-8GB，页面缓存尽量大
  - 网络：万兆网卡，避免跨机房
  - OS优化：文件描述符、Swap关闭、文件系统XFS
- 常见追问：如何估算磁盘容量？JVM参数怎么调？

**Q16：Kafka消息积压处理**
- 核心考点：Consumer Lag的诊断与解决
- 参考答案要点：
  - 监控：`kafka-consumer-groups.sh --describe`查看Lag
  - 原因：Consumer处理慢、Consumer数不足、分区不均、GC停顿
  - 临时方案：增加Consumer实例、临时Consumer组只消费不处理
  - 长期方案：优化消费逻辑、增加分区数、异步处理
- 常见追问：分区数固定怎么增加Consumer？临时方案的具体实现？

**Q17：Kafka日志清理策略**
- 核心考点：日志保留与清理机制
- 参考答案要点：
  - 基于时间：`log.retention.hours`（默认168h=7天）
  - 基于大小：`log.retention.bytes`（默认-1不限）
  - 基于Key压缩：`cleanup.policy=compact`，保留每个Key最新值
  - 删除 vs 压缩：删除是默认策略，压缩适合changelog场景
- 常见追问：Compact清理的时机？什么场景用Compact？

**Q18：Kafka安全机制**
- 核心考点：认证与授权
- 参考答案要点：
  - SASL认证：PLAIN/SCRAM-SHA-256/GSSAPI(Kerberos)/OAUTHBEARER
  - SSL加密：Broker-Client和Broker-Broker间TLS
  - ACL授权：`kafka-acls.sh`配置Topic/Group级别权限
  - 生产建议：SASL_SSL组合，SCRAM支持动态增删用户
- 常见追问：SCRAM和PLAIN的区别？如何实现多租户隔离？

**Q19：Kafka幂等性原理**
- 核心考点：Producer幂等性的实现细节
- 参考答案要点：
  - PID（Producer ID）：每个Producer实例唯一标识
  - Sequence Number：每个<TopicPartition>单调递增的序列号
  - Broker端：保存每个PID+TopicPartition的最新序列号，重复序列号拒绝
  - 限制：只能保证单分区单会话幂等，跨分区需事务
- 常见追问：PID怎么生成？Producer重启后PID会变吗？

**Q20：Kafka与RabbitMQ对比**
- 核心考点：消息中间件选型
- 参考答案要点：
  - Kafka：高吞吐、持久化、分布式、拉模式、适合大数据流
  - RabbitMQ：低延迟、丰富路由、推模式、适合业务消息
  - Kafka：百万级TPS、顺序IO、批量处理
  - RabbitMQ：万级TPS、灵活路由、即时消费
  - 选型：大数据/日志/事件流→Kafka，业务消息/任务队列→RabbitMQ
- 常见追问：Kafka能替代RabbitMQ吗？Pulsar和Kafka的区别？

### 3.2 Flink核心（20题）

**Q1：Watermark机制详解**
- 核心考点：事件时间处理与迟到数据处理
- 参考答案要点：
  - Watermark = 最大事件时间 - 允许延迟时间
  - 作用：触发窗口计算，Watermark超过窗口结束时间则触发
  - 生成方式：周期性（`WatermarkGenerator`）或Punctuated（按标记）
  - 多分区取最小Watermark，保证全局一致性
  - 迟到数据：`allowedLateness()`设置允许延迟，`sideOutputLateData()`收集超迟数据
- 常见追问：Watermark传播机制？多Source的Watermark怎么对齐？

**Q2：Checkpoint vs Savepoint**
- 核心考点：两种状态持久化机制的区别
- 参考答案要点：
  - Checkpoint：自动定期触发，轻量级，用于故障恢复，可增删
  - Savepoint：手动触发，完整快照，用于版本升级/迁移，兼容性好
  - Checkpoint格式可能随版本变化，Savepoint保证跨版本兼容
  - Savepoint需要指定路径，Checkpoint由状态后端管理
- 常见追问：Checkpoint如何触发？从Savepoint恢复时算子可以修改吗？

**Q3：端到端Exactly-Once**
- 核心考点：Flink端到端精确一次语义实现
- 参考答案要点：
  - 前提：Source可重放（Kafka offset）、Sink支持事务或幂等
  - 两阶段提交（2PC）：Checkpoint开始→预提交→确认提交
  - Kafka Sink：事务写入，Checkpoint完成时提交事务
  - 幂等Sink：HBase/Redis的PUT操作天然幂等
  - `exactly-once`语义需要Source+框架+Sink三方配合
- 常见追问：2PC的Coordinator挂了怎么办？HDFS Sink如何实现Exactly-Once？

**Q4：Flink vs Spark Streaming**
- 核心考点：流处理引擎的架构对比
- 参考答案要点：
  - Flink：事件驱动、真正的流处理、毫秒级延迟、状态管理强
  - Spark Streaming：微批处理、秒级延迟、与Spark生态集成
  - Flink优势：事件时间、状态管理、窗口丰富、Exactly-Once
  - Spark优势：生态成熟、SQL强、批流统一API、社区大
  - 趋势：Flink主导实时，Spark主导离线，Structured Streaming追赶
- 常见追问：Structured Streaming和Flink的差距？什么场景选Spark Streaming？

**Q5：状态后端选择**
- 核心考点：不同状态后端的特点与适用场景
- 参考答案要点：
  - HashMapStateBackend：内存存储，JVM堆，适合小状态，速度快
  - EmbeddedRocksDBStateBackend：RocksDB存储，支持超大状态，延迟略高
  - 选择：状态<内存可用量用HashMap，状态大用RocksDB
  - RocksDB优化：Block Cache、Write Buffer、Compaction策略
- 常见追问：RocksDB的性能瓶颈在哪？如何监控RocksDB状态？

**Q6：Flink窗口类型**
- 核心考点：四种窗口的特点与使用场景
- 参考答案要点：
  - Tumbling Window：固定大小，不重叠，如每5分钟
  - Sliding Window：固定大小，可重叠，如每5分钟滑动1分钟
  - Session Window：按活跃间隔划分，不固定大小，如30分钟无活动
  - Global Window：全局一个窗口，需自定义Trigger触发
- 常见追问：Session Window的Gap怎么设？窗口的Trigger和Evictor？

**Q7：Flink反压机制**
- 核心考点：反压的原理与处理
- 参考答案要点：
  - 反压：下游处理慢，向上游传递压力，自动限流
  - Credit-based（1.5+）：下游告知可用Buffer数，上游按需发送
  - 影响：数据延迟增加、Checkpoint超时、吞吐下降
  - 排查：Web UI看BackPressure指标、Task指标、Sink写入速率
  - 解决：增加并行度、优化算子逻辑、异步IO、外部缓存
- 常见追问：Credit-based和TCP反压的区别？反压导致Checkpoint超时怎么办？

**Q8：Flink时间语义**
- 核心考点：三种时间的区别与应用
- 参考答案要点：
  - Event Time：事件发生时间，最准确，需Watermark配合
  - Processing Time：处理时间，延迟最低但不稳定
  - Ingestion Time：进入Flink的时间，介于两者之间
  - 生产建议：实时报表用Event Time，监控告警可用Processing Time
  - Event Time需要指定Timestamp Assigner和Watermark Generator
- 常见追问：Event Time和Processing Time的延迟差异？乱序数据怎么处理？

**Q9：Flink CEP复杂事件处理**
- 核心考点：模式匹配与实时规则引擎
- 参考答案要点：
  - 定义模式：`Pattern.begin("start").where(...).followedBy("middle").where(...).times(3)`
  - 连续性：`next`严格连续、`followedBy`宽松连续、`followedByAny`非确定性宽松
  - 时间约束：`within(Time.seconds(10))`限制模式匹配时间
  - 应用：风控规则、用户行为分析、设备异常检测
- 常见追问：CEP的状态开销怎么控制？动态规则怎么实现？

**Q10：Flink SQL与Table API**
- 核心考点：关系型API在流处理中的应用
- 参考答案要点：
  - 统一API：同一套SQL同时处理批和流
  - 动态表：流→表→流的转换，Append/Retract机制
  - 窗口：`TUMBLE`/`HOP`/`SESSION`/`CUMULATE`窗口函数
  - Join：Regular Join（状态大）、Interval Join（时间范围）、Temporal Join（维度表）
  - Catalog：管理元数据，支持Hive Catalog集成
- 常见追问：Retract机制是什么？Interval Join如何控制状态？

**Q11：Flink CDC**
- 核心考点：变更数据捕获在Flink中的实现
- 参考答案要点：
  - Flink CDC Connector：基于Debezium，无锁读取Binlog
  - 支持MySQL/PostgreSQL/Oracle等
  - 全量+增量：先全量快照读取，再切换到Binlog增量
  - 无锁算法：Snapshot Phase用SELECT一致性读，不锁表
  - 应用：实时同步、数仓更新、数据湖写入
- 常见追问：无锁读取如何保证一致性？CDC的延迟怎么优化？

**Q12：Flink并行度设置**
- 核心考点：并行度对性能的影响与调优
- 参考答案要点：
  - 四级并行度：算子级 > Execution Environment级 > Client级 > 配置文件默认
  - 原则：并行度≥Kafka分区数，避免数据倾斜
  - Source并行度=分区数，Sink并行度=下游系统并发能力
  - 过高：调度开销、状态开销、网络开销
  - 过低：资源利用不足、反压
- 常见追问：如何判断并行度是否合理？算子级并行度什么时候需要单独设？

**Q13：Flink内存模型**
- 核心考点：TaskManager内存划分
- 参考答案要点：
  - JVM Heap：Framework Heap + Task Heap（用户代码）
  - Off-heap：Managed Memory（RocksDB/排序）、Network Memory（Shuffle缓冲）
  - Network Buffer：`taskmanager.network.memory.min/max`，影响Shuffle效率
  - 调优：状态大增加Managed Memory，Shuffle多增加Network Memory
- 常见追问：Network Buffer不够会怎样？RocksDB状态大怎么调内存？

**Q14：Flink侧输出（Side Output）**
- 核心考点：分流处理与迟到数据收集
- 参考答案要点：
  - 用途：将数据分流到不同输出，避免多次过滤
  - 定义：`OutputTag<T>`标记侧输出流
  - 发送：`ctx.output(outputTag, value)`
  - 获取：`mainStream.getSideOutput(outputTag)`
  - 典型场景：迟到数据收集、异常数据分流、多目标输出
- 常见追问：侧输出和Filter的区别？侧输出的状态管理？

**Q15：Flink异步IO**
- 核心考点：异步查询外部系统的优化
- 参考答案要点：
  - 问题：同步IO查询外部系统导致反压和低吞吐
  - 解决：`AsyncDataStream.unorderedWait/orderedWait`异步请求
  - 有序：结果按请求顺序输出，延迟高
  - 无序：结果谁先回来谁先输出，吞吐高
  - 配置：超时时间、最大并发请求数
- 常见追问：异步IO的线程模型？超时怎么处理？

**Q16：Flink状态TTL**
- 核心考点：状态过期清理机制
- 参考答案要点：
  - 配置：`StateTtlConfig.newBuilder(Time.hours(24))`
  - 清理策略：`cleanupInRocksdbCompactFilter`（RocksDB压缩时清理）
  - `cleanupIncrementally`（每次访问清理部分）
  - 过期时间：从状态创建/更新开始计算
  - 注意：TTL只对Keyed State有效，ListState等需注意语义
- 常见追问：TTL和窗口清理的区别？TTL清理的时机？

**Q17：Flink重启策略**
- 核心考点：故障恢复策略选择
- 参考答案要点：
  - Fixed Delay：固定间隔重启N次，`restart-strategy.fixed-delay.attempts=3`
  - Failure Rate：失败率重启，时间窗口内不超过N次
  - No Restart：不重启，直接失败
  - Checkpoint + 重启策略配合：Checkpoint保证状态恢复，重启策略控制重试
- 常见追问：重启策略和Checkpoint的关系？Flink on YARN的恢复机制？

**Q18：Flink广播状态**
- 核心考点：广播流的模式匹配与规则分发
- 参考答案要点：
  - 用途：将规则/配置广播到所有并行实例，与主流关联
  - API：`mainStream.connect(broadcastStream).process(new BroadcastProcessFunction)`
  - 广播状态：`MapStateDescriptor`定义，所有TaskManager持有完整副本
  - 典型场景：动态规则引擎、实时风控、配置热更新
- 常见追问：广播状态的大小限制？广播状态更新时如何保证一致性？

**Q19：Flink Sink幂等写入**
- 核心考点：输出端Exactly-Once的实现方式
- 参考答案要点：
  - 幂等写入：重复写入结果相同，如HBase PUT、Redis SET
  - 事务写入：两阶段提交，如Kafka事务Sink
  - Upsert写入：主键去重，如MySQL INSERT ON DUPLICATE KEY UPDATE
  - 文件写入：分区+文件名唯一（包含Checkpoint ID）
- 常见追问：MySQL Sink如何实现Exactly-Once？文件Sink的命名策略？

**Q20：Flink作业提交模式**
- 核心考点：不同部署模式的区别

### 3.4 场景题（20题）

**Q1：Kafka Consumer Lag持续增长怎么排查**
- 核心考点：消费延迟问题的系统化排查
- 参考答案要点：
  - 监控：`kafka-consumer-groups.sh`查看Lag，Burdock/Prometheus告警
  - 排查路径：Consumer处理慢？→ Consumer数不足？→ GC问题？→ 下游瓶颈？
  - Consumer处理慢：优化消费逻辑、异步处理、批量写入
  - Consumer数不足：增加实例（不超过分区数）
  - GC问题：调整JVM参数，减少Full GC
  - 下游瓶颈：数据库慢查询、外部API超时
- 常见追问：分区数不够怎么紧急扩容？Consumer频繁Rebalance怎么处理？

**Q2：Flink反压怎么处理**
- 核心考点：反压问题的定位与解决
- 参考答案要点：
  - 定位：Web UI BackPressure标签，找到第一个高反压算子
  - 原因：数据倾斜、算子逻辑慢、下游写入慢、GC
  - 数据倾斜：重分区（rebalance/rescale）、加盐打散
  - 算子慢：异步IO、增加并行度、优化逻辑
  - 下游慢：批量写入、增加连接池、缓存
  - GC：调整内存、减少对象创建、使用RocksDB
- 常见追问：反压和CheckPoint超时的关系？如何区分反压和数据倾斜？

**Q3：数据倾斜怎么发现和解决**
- 核心考点：全链路数据倾斜治理
- 参考答案要点：
  - 发现：Spark UI/Flink UI看Task数据分布、监控Shuffle量
  - Map端倾斜：输入文件大小不均→repartition
  - Shuffle倾斜：Key分布不均→加盐/两阶段聚合/广播JOIN
  - Reduce端倾斜：聚合Key集中→局部聚合+全局聚合
  - 预防：合理分区、预聚合、均匀Key设计
- 常见追问：两阶段聚合的具体实现？如何判断倾斜的Key？

**Q4：实时和离线数据不一致怎么办**
- 核心考点：Lambda/Kappa架构的数据一致性
- 参考答案要点：
  - 原因：计算逻辑不同、数据延迟、时间窗口差异、重算未同步
  - 检测：定期对账任务，实时和离线指标比对
  - 解决：统一计算引擎（Flink批流一体）、统一SQL逻辑
  - Kappa架构：只用实时链路，离线重算走同一套逻辑
  - 短期方案：以离线为准，实时做修正
- 常见追问：Kappa架构的局限性？如何保证对账的准确性？

**Q5：Spark任务OOM怎么排查**
- 核心考点：内存溢出的定位与解决
- 参考答案要点：
  - 定位：看Executor日志、Spark UI的Task指标
  - Driver OOM：collect()拉大数据、广播大变量、SQL结果集大
  - Executor OOM：数据倾斜、内存配置不足、Shuffle溢出
  - 解决：增加内存、处理倾斜、调整并行度、避免collect
  - 预防：合理资源配置、代码审查、监控告警
- 常见追问：如何区分是数据倾斜导致的OOM还是内存配置不足？Heap Dump怎么分析？

**Q6：Hive查询突然变慢怎么排查**
- 核心考点：性能突降的排查思路
- 参考答案要点：
  - 确认范围：单个查询慢还是整体慢？
  - 单查询慢：数据量变化、统计信息过期、执行计划变化
  - 整体慢：集群资源不足、YARN队列满、DataNode故障
  - 排查步骤：EXPLAIN看执行计划→数据量对比→资源使用→集群状态
  - 常见原因：小文件增多、统计信息未更新、分区裁剪失效
- 常见追问：如何自动更新统计信息？分区裁剪失效怎么发现？

**Q7：Kafka集群磁盘报警怎么处理**
- 核心考点：存储容量问题的应急与根治

  - 应急：删除过期Topic、缩短保留时间、手动删除旧日志段

**Q8：Flink Checkpoint超时怎么处理**
- 核心考点：Checkpoint失败的排查与优化
- 参考答案要点：
  - 原因：状态过大、反压、Barrier对齐慢、HDFS写入慢
  - 排查：看Checkpoint详情、Barrier耗时、状态大小
  - 状态过大：增量Checkpoint（RocksDB）、TTL清理、状态拆分
  - 反压：先解决反压问题

**Q1：Spark Catalyst优化流程**

**Q2：Flink Checkpoint源码流程**
- 核心考点：Checkpoint的完整源码链路
- 参考答案要点：
  - 触发：JobManager的CheckpointCoordinator定时触发
  - Barrier注入：Source算子收到CheckpointBarrier，向下游传递
  - Barrier对齐：算子等待所有输入的Barrier到达
  - 状态快照：调用StateBackend.snapshot()持久化状态
  - 汇报：所有算子完成后向JobManager汇报，JobManager确认完成
  - Unaligned Checkpoint：Barrier不对齐，先存Buffer数据
- 常见追问：Aligned vs Unaligned Checkpoint的源码差异？增量Checkpoint的实现？

**Q3：Kafka Controller选举过程**
- 核心考点：Controller选举的源码实现
- 参考答案要点：
  - 选举：Broker启动时尝试创建`/controller`临时节点，成功则成为Controller
  - 已有Controller：监听`/controller`节点变化，Controller下线时重新选举
  - 初始化：Controller选举成功后初始化分区状态、副本状态、Leader选举

**Q4：Raft协议与ZAB协议对比**
- 核心考点：分布式共识协议的异同
- 参考答案要点：
  - 相同：Leader选举、日志复制、多数派确认、安全性保证
  - Raft：强Leader模型、Term概念、日志连续性保证、更易理解
  - ZAB：为ZooKeeper设计、ZXID（epoch+counter）、支持快速Leader选举
  - ZAB有恢复模式（选主）和广播模式（消息广播）两阶段
  - Raft更通用，ZAB更针对ZooKeeper的读写场景
- 常见追问：Raft的日志压缩怎么实现？ZAB的过半机制和Raft的区别？

**Q5：Spark Shuffle Read源码分析**

**Q6：Flink Watermark传播源码**

**Q7：Kafka日志段读写源码**

**Q17：设计数据湖事务机制**

**Q18：设计数据平台可观测性**

  - 对齐：公司→部门→团队→个人逐级对齐
  - 周期：季度OKR，周进度跟踪，月度Review

  - 止损：停任务、切流量、回滚数据
  - 修复：数据重跑、增量修正、业务补偿
  - 复盘：时间线→根因→改进措施→责任认定→预防方案

  - 当前基线：资源使用率、数据增长趋势、业务峰值
  - 增长预测：按历史增长率×业务规划，预留buffer
  - 计算资源：CPU/内存/队列容量，按任务类型分类规划
  - 存储资源：HDFS/S3容量、副本策略、冷热分层
  - 网络资源：带宽需求、跨机房流量
  - 输出：季度/年度容量规划报告，扩容时间线
- 常见追问：如何应对突发增长？容量不足的应急方案？

**Q15：如何设计数据SLA监控**
- 核心考点：SLA监控的自动化
- 参考答案要点：
  - 指标采集：任务完成时间、数据产出时间、数据质量分
  - SLA规则：P0任务8:00前完成、数据准确率>99.9%
  - 告警链路：规则引擎→分级告警→值班响应→升级机制
  - 看板：SLA达成率趋势、违约任务Top10、部门排名
  - 自动化：自动重跑、自动降级、自动通知下游
- 常见追问：SLA违约后如何自动恢复？如何避免告警疲劳？

**Q16：如何做数据平台灰度发布**
- 核心考点：数据平台的发布策略
- 参考答案要点：
  - 灰度维度：按任务、按租户、按数据量、按时间
  - 流程：开发→测试→预发→灰度（5%→20%→50%）→全量
  - 验证：数据对比、性能对比、异常监控
  - 回滚：一键回滚、数据回滚、配置回滚
  - 工具：Feature Flag、流量路由、数据双跑
- 常见追问：数据任务的灰度怎么做？灰度期间数据不一致怎么办？

**Q17：如何做数据平台性能基线**
- 核心考点：性能基线的建立与维护
- 参考答案要点：
  - 采集：任务执行时间、资源消耗、Shuffle量、GC时间
  - 基线建立：取近30天P50值，排除异常值
  - 偏差检测：超过基线±20%触发告警
  - 归因：数据量变化、代码变更、资源竞争、集群状态
  - 优化：基于基线识别退化任务，推动优化
- 常见追问：基线如何自动更新？季节性波动怎么处理？

**Q18：如何设计数据平台多环境**
- 核心考点：开发/测试/生产环境管理
- 参考答案要点：
  - 环境划分：开发（Dev）→测试（Test）→预发（Staging）→生产（Prod）
  - 数据管理：生产脱敏→测试环境、数据量采样、定期刷新
  - 资源隔离：独立集群或共享集群隔离队列
  - 流程：Dev开发→Test验证→Staging预发→Prod上线
  - 挑战：环境一致性、数据脱敏、资源成本
- 常见追问：如何保证多环境配置一致？测试数据怎么管理？

**Q19：如何做数据平台文档体系**
- 核心考点：文档驱动的知识管理
- 参考答案要点：
  - 文档类型：架构文档、API文档、运维手册、故障手册、新人指南
  - 工具：Confluence/Notion + Git + 自动生成（Swagger/DBT Docs）
  - 原则：代码即文档、文档随代码更新、Review包含文档
  - 治理：定期Review、过期标记、Owner制度
  - 激励：文档贡献纳入绩效、文档质量奖
- 常见追问：如何保证文档和代码同步？新人如何快速上手？

**Q20：如何从技术专家转型技术管理**
- 核心考点：技术管理的转型路径
- 参考答案要点：
  - 思维转变：从自己做到推动别人做，从技术深度到技术广度
  - 管理技能：目标设定、任务分配、进度跟踪、绩效反馈
  - 沟通能力：向上管理、跨团队协作、技术翻译
  - 团队建设：招聘、培养、激励、文化建设
  - 常见误区：微观管理、不做技术、不敢决策
- 常见追问：如何平衡技术和管理？如何培养技术Leader？

## 五、面试实战技巧

### 5.1 自我介绍模板

**1分钟版本（电梯演讲）**：
> 我有X年大数据开发经验，主要技术栈是Spark/Flink/Kafka。在XX公司负责XX数据平台，日均处理XX TB数据，支撑XX业务方。最核心的成就是将XX任务从XX优化到XX，节省XX成本。目前寻求XX方向的岗位。

**3分钟版本（标准介绍）**：
> 1. 基本信息：姓名、工作年限、当前公司/岗位
> 2. 技术栈：核心技术和深度
> 3. 项目经验：2-3个重点项目，突出难点和成果
> 4. 技术亮点：开源贡献、技术分享、专利论文
> 5. 求职意向：为什么来面试、期望方向

**5分钟版本（深度介绍）**：
> 在3分钟版本基础上，增加：
> 1. 技术成长路径：从XX到XX的演进
> 2. 架构思考：对技术选型的思考过程
> 3. 团队影响：带团队、建规范、推变革
> 4. 未来规划：3-5年职业目标

### 5.2 项目经验讲述框架（STAR法则）

- **Situation（背景）**：业务场景、数据规模、技术挑战
- **Task（任务）**：你的具体职责、目标指标
- **Action（行动）**：技术方案、实施步骤、遇到的困难及解决
- **Result（结果）**：量化成果、业务价值、技术沉淀

**示例**：
- S：公司实时数仓缺失，业务方依赖T+1离线数据，决策延迟严重
- T：负责从0到1搭建实时数仓，要求核心指标延迟<5分钟
- A：选型Flink+Kafka+Doris，设计ODS→DWD→DWS实时分层架构，解决Exactly-Once和数据倾斜问题
- R：核心指标延迟从T+1降至3分钟，支撑15+业务方实时决策，数据准确率99.99%

### 5.3 系统设计面试流程

1. **需求澄清（5分钟）**：明确功能需求、非功能需求、约束条件
2. **容量估算（5分钟）**：数据量、QPS、存储、带宽
3. **高层设计（10分钟）**：核心组件、数据流、技术选型
4. **深入设计（15分钟）**：关键组件深入、权衡分析、异常处理
5. **总结回顾（5分钟）**：设计回顾、优化建议、扩展性讨论

**关键原则**：
- 先广后深，不要一上来就深入细节
- 主动和面试官沟通，确认方向
- 说清楚Why，不只是What
- 承认Trade-off，没有完美方案
- 画图辅助说明

### 5.4 薪资谈判策略

- **市场调研**：了解目标公司薪资范围、同级别市场价
- **锚定效应**：先报一个合理偏高的数字，给谈判留空间
- **总包思维**：Base + Bonus + Stock + Sign-on，不只看月薪
- **竞争Offer**：多个Offer互相制衡，但保持诚实
- **非薪资因素**：团队氛围、技术栈、成长空间、工作强度
- **谈判话术**："基于我的经验和市场情况，我期望XX范围"

### 5.5 反问环节准备

**技术方向**：
- 团队目前的技术栈和架构是怎样的？
- 团队面临的最大技术挑战是什么？
- 对这个岗位的期望和考核标准是什么？

**团队文化**：
- 团队的开发流程和协作方式？
- 技术决策是怎么做的？
- 团队的On-call制度是怎样的？

**个人发展**：
- 这个岗位的成长路径是怎样的？
- 公司对技术培训和学习有什么支持？
- 团队的技术分享频率？
