# L2 结业考核标准

> **所属阶段**：L2 中级工程师 | **考核周**：第22周 | **总时长**：2天

---

## 一、考核概览

| 考核项 | 方式 | 时长 | 权重 | 通过标准 |
|--------|------|------|------|----------|
| Kafka调优 | 现场实操 | 2h | 20% | 给定Topic达到基准吞吐量×1.5 |
| Flink开发 | 现场编码 | 1.5h | 20% | 1小时内完成带Window+Checkpoint的Flink任务 |
| 故障排查 | 模拟演练 | 1h | 20% | 30分钟内定位并解决讲师注入的故障 |
| 架构讲解 | 投影问答 | 1h | 20% | 完整讲解项目7的架构设计，回答5个Why |
| DDIA答辩 | PPT+问答 | 1.5h | 20% | DDIA读书笔记分享，回答"与实际工作的映射" |

> **总评**：5项全部通过 → 升入L3 | 3-4项通过 → 补考 | < 3项通过 → 重修

### 考核1：Kafka调优（现场实操）

```
考核形式:
  - 提供: 3节点Kafka集群 + 基准Topic（默认参数）
  - 任务: 在2小时内将吞吐量提升到基准值的1.5倍
  - 要求: 不能改变硬件配置，只能调参数

评分标准:
  [30分] 达到吞吐×1.5目标
  [25分] 能解释每个参数改变的原理
  [20分] 延迟不恶化（P99延迟不超过基准值的1.2倍）
  [15分] 提供调优报告（含实验数据和结论）
  [10分] 能给出不同场景下的推荐配置

基准数据:
  基准Topic: test-baseline
    Partitions: 6, Replication: 3
    Producer: acks=1, batch.size=16KB, linger.ms=0
    Consumer: 默认配置
  基准吞吐量: 约 200,000 rec/s（1KB消息）
  目标吞吐量: ≥ 300,000 rec/s

考核步骤:
  1. 先跑一次基准压测，确认基准值（15分钟）
  2. 分析瓶颈（CPU? 网络? 磁盘?）（15分钟）
  3. 设计调优方案并实施（60分钟）
  4. 跑最终压测验证（15分钟）
  5. 输出调优报告（15分钟）
```

  $ kafka-producer-perf-test.sh \

      --num-records 5000000 \

  预期输出:
  5000000 records sent, 196843.2 records/sec (192.3 MB/sec), 
  15.4 ms avg latency, 45.2 ms max latency,

  [10分] 场景推荐:

  Producer侧:
    batch.size: 16KB → 128KB (或512KB)
    linger.ms: 0 → 10 (或20)
    compression.type: none → lz4 (或 snappy)
    buffer.memory: 32MB → 128MB
  Broker侧:
    num.io.threads: 8 → 16
    socket.send.buffer.bytes: 100KB → 1MB
    num.replica.fetchers: 1 → 4
  Topic侧:
    partitions: 6 → 12

#### 10个备选故障场景详细清单

```
故障场景1: SPARK-001 Spark任务OOM
  注入方式: 讲师在Hive中准备一张倾斜分布的表（某Key占比50%+），
           学员提交的数据分析任务不做倾斜处理
  关键线索: Spark UI中Task输入量P99/P50比率 > 50
  期望排查路径:
    1. 查看Spark UI → 发现Task输入不均 → 数据倾斜
    2. Spark SQL查询Key分布 → 确认倾斜Key
    3. 方案: 加盐/AQE/Broadcast Join
  考核重点: 是否先看Spark UI而非直接改参数

故障场景2: KAFKA-001 Consumer Lag持续增长
  注入方式: 讲师在Consumer处理逻辑中注入sleep(200ms)模拟慢处理
  关键线索: kafka-consumer-groups --describe 的LAG持续增长
  期望排查路径:
    1. 检查Consumer Group Lag → 确认增长趋势
    2. 检查Consumer处理逻辑 → 发现sleep
    3. 方案: 优化处理逻辑/扩容Consumer
  考核重点: 是否检查了Consumer处理耗时

故障场景3: HIVE-001 查询变慢（分区裁剪失效）
  注入方式: 讲师将查询条件从WHERE dt=...改为WHERE DATE(create_time)=...
  关键线索: EXPLAIN中看不到partition信息
  期望排查路径:
    1. EXPLAIN查看执行计划 → 发现全表扫描

#### 5个Why深度解答示例

```
Why 1: 为什么选Kafka而不是Pulsar/RocketMQ？

  合格回答（60分）:
    Kafka生态更成熟，社区更大，文档更丰富

  优秀回答（90分+）:
    Kafka vs Pulsar: Pulsar的计算存储分离架构（BookKeeper做存储）
    使得扩缩容更灵活，但运维复杂度更高（需管理BookKeeper和Broker两套集群）。
    在我们的场景中，数据量在TB级以下，单集群足以支撑，Kafka的架构更简单。
    
    Kafka vs RocketMQ: RocketMQ在事务消息和顺序消息方面比Kafka更丰富，
    但我们主要用流式处理（Flink消费），不涉及复杂的消息路由逻辑，
    Kafka的Pull模型比RocketMQ的Push模型更适合我们（Flink Source控制消费速率）。
    
    关键决策因素:
    - Kafka与Confluent Schema Registry + Avro的生态整合
    - Kafka Connect的生态（Debezium CDC Connector直接用）
    - Flink的Kafka Connector成熟度远高于Pulsar/RocketMQ Connector
    - 团队已有Kafka运维经验，切换成本高

Why 2: 为什么选Flink而不是Spark Streaming？

  合格回答（60分）:
    Flink是真正的流处理，Spark Streaming是微批

  优秀回答（90分+）:
    Spark Streaming的DStream在2.0后被Structured Streaming取代，
    后者在Watermark支持、事件时间语义、Checkpoint机制上已经接近Flink。
    但关键差异在于:
    
    1. 延迟: Flink逐条处理，延迟在毫秒级；Structured Streaming微批，
       最小批次间隔约100ms，累计延迟更高
    2. Window触发: Flink的Window在Watermark到达时立即触发；
       Structured Streaming需要等下一个Micro-Batch到达
    3. 状态管理: Flink的RocksDB StateBackend + 增量Checkpoint在
       大状态场景下比Structured Streaming的State Store更高效
    4. Savepoint: Flink支持手动Savepoint + 从任意Savepoint恢复，
       这对于运维（如修改并行度后重启）非常重要
    
    选Flink的核心原因: 我们的"异常检测"需求要求秒级响应
    （大额订单检测延迟 < 3秒），Flink的EventTime + Watermark + 
    CEP组合是最优解。

Why 3: 为什么选ClickHouse而不是Doris/Druid？

  合格回答（60分）:
    ClickHouse查询快，社区活跃

  优秀回答（90分+）:
    ClickHouse vs Doris: Doris在MySQL协议兼容性和高并发点查方面更优，
    但我们的场景是OLAP分析查询（宽表扫描 + 聚合），并发只有10-20个Dashboard用户，
    ClickHouse的单表扫描性能远优于Doris。
    
    ClickHouse vs Druid: Druid是为时序数据预聚合设计的，写入时需要预定义聚合维度；
    ClickHouse是通用OLAP引擎，支持原始明细查询和任意维度聚合。
    我们的Grafana面板需要灵活的Ad-Hoc查询，不适合Druid的预聚合模型。
    
    关键决策: ReplicatedMergeTree + 分布式表 + TTL 72h
    数据量在100GB级别时，ClickHouse在30GB内存下即可达到亚秒级查询。

Why 4: 为什么宽表在Flink中Join而不是在ClickHouse中Join？

  合格回答（60分）:
    Flink Join可以实时完成，ClickHouse Join太慢

  优秀回答（90分+）:
    ClickHouse的JOIN性能是公认的弱点——它只有Hash Join一种策略，
    且不能利用索引加速JOIN。如果三张表（orders + order_items + users）
    都在ClickHouse中JOIN，每次Grafana刷新都需要一次三表JOIN，延迟不可接受。
    
    在Flink中预JOIN成宽表的好处:
    1. 计算前移：JOIN在数据写入时完成（流处理），查询时无需JOIN
    2. State管理：Flink的RocksDB StateBackend可以高效管理JOIN状态，
       通过KeyBy+connect实现双流JOIN
    3. 时效性：Flink流式JOIN的延迟 < 1秒，比T+1离线宽表更实时
    
    但也有Cost：Flink任务需要维护State（内存/磁盘），
    三流JOIN的Checkpoint大小可能达到GB级。

Why 5: 为什么用CDC而不是定时SELECT去拉取？

  合格回答（60分）:
    CDC可以实时捕获变更，定时SELECT有延迟

  优秀回答（90分+）:
    定时SELECT（如每5秒 SELECT * FROM orders WHERE update_time > last_time）的几个致命缺陷:
    1. 无法捕获DELETE操作（除非用逻辑删除flag，破坏了源表）
    2. 延迟不确定: 5秒轮询 → 平均延迟2.5秒（最坏5秒）
    3. 源库压力: 每次SELECT都需要扫描索引（即使只有1条变更也要扫）
    4. 数据一致性: 如果轮询期间发生了多次UPDATE，只能拿到最终值，
       丢失中间变更历史
    
    Debezium CDC的优势:
    1. 实时性: 基于Binlog，变更发生后毫秒级捕获
    2. 完整性: INSERT/UPDATE/DELETE全部捕获（含变更前后的值）
    3. 低侵入: 读Binlog完全不影响MySQL正常业务（不消耗MySQL线程池）
    4. Schema追踪: 自动感知源表DDL变更并同步Schema
    
    唯一注意点: 初始全量Snapshot + 增量Binlog的衔接
    （Debezium的snapshot.mode=initial模式自动处理）
```

## 九、故障排查考核——10个场景详细注入与排查路径

### 故障注入操作手册（给讲师用）

```
通用原则:
  1. 每次只注入1个故障
  2. 注入后验证学员能看到故障现象
  3. 如果学员15分钟内无法定位，给1个提示
  4. 如果学员25分钟内无法定位，给第2个提示
  5. 30分钟后终止考核，进入复盘

故障1: SPARK-001 (Spark任务OOM)
  注入命令:
    # 步骤1: 准备倾斜数据表
    spark-sql -e "
    CREATE TABLE IF NOT EXISTS skewed_sales AS
    SELECT
      CASE WHEN rand() < 0.6 THEN 'hot_key_A'
           ELSE concat('normal_key_', cast(rand()*1000 as int))
      END as product_key,
      cast(rand()*100 as double) as sale_amount
    FROM (SELECT explode(sequence(1,10000000)) as id);
    "
    # 步骤2: 提交不带倾斜处理的Spark任务
    spark-submit --class DataSkewTest --executor-memory 2g skewed-job.jar
  
  现象检查(学员视角):
    ① Spark UI → Stages → 某个Task的Shuffle Read Size远超其他
       P99/P50 > 50 → 确认数据倾斜
    ② Executors页面 → 某个Executor OOM被Kill
    ③ Driver日志: FetchFailedException 或 OutOfMemoryError
  
  排查路径评分:
    优秀路径 (30分):
      查看Spark UI → 定位倾斜Stage → 查看Task Metrics找到倾斜Key
      → 用countByKey确认分布 → 方案: 加盐/Salted Join/AQE
    
    一般路径 (20分):
      查看日志 → 发现OOM → 直接调大executor-memory → (可能)
      
    差路径 (10分):
      直接改配置 → 多次试错 → 超时
    
  评判提示级别:
    提示1(15分钟): "看看Spark UI的Stage详情，特别是Shuffle Read Size"
    提示2(25分钟): "你是否注意到某个Key的数据量远超其他？"

故障2: KAFKA-001 (Consumer Lag)
  注入命令:

    # 方法1: 修改Consume的代码加入sleep
    # 方法2: 用tc模拟网络延迟增加Consumer端到Broker的延迟
    # 方法3（最简单）: 通过iptables限制Consumer出流量

    # 注入上游高生产量
    kafka-producer-perf-test.sh --topic test-topic \
      --num-records 5000000 --record-size 1024 --throughput 50000 \
      --producer-props bootstrap.servers=localhost:9092 &
  
  现象检查:
    ① kafka-consumer-groups --describe → LAG持续增长
    ② Consumer处理日志: 每条消息处理时间 > 正常值
    ③ 如果在监控面板上: Consumer消费速率远低于生产速率
    
  排查路径评分:
    优秀路径:
      → 检查Consumer Group Lag (确认问题)
      → 检查Consumer处理日志(发现处理耗时增加)
      → 检查Consumer JVM GC (排查GC影响)
      → 检查Consumer与Broker间的网络延迟 (tc netem)
      → 定位根因: 网络延迟导致fetch慢
    
    一般路径:
      → 检查Lag → 直接扩容Consumer → 治标不治本
    
    差路径:
      → 重启Consumer → Lag暂时清零但马上又增长

故障3: HIVE-001 (分区裁剪失效)
  注入命令:

      order_id STRING, user_id STRING, amount DOUBLE
    ) PARTITIONED BY (dt STRING);

## 十一、架构讲解——5个Why案例分析

### Why 1: 为什么选Kafka而不是Pulsar/RocketMQ？

```
理想回答(90分+):
  "我们按四个维度做了对比分析：
   
   1. 功能匹配度: 我们的核心需求是'日志式消息流'——数据按顺序追加，
      Consumer按Offset顺序消费，支持多Group独立消费。Kafka的
      Partition + Consumer Group模型天然匹配这个模式。
      Pulsar虽然也支持，但其设计更适合'队列式'消息(Topic可以有
      多个Subscription，每个Subscription独立确认)，这在我们场景
      中是过度设计。
   
   2. 生态集成度: 
      - Kafka Connect(Debezium CDC) → Flink Connector 是最成熟的链路
      - Pulsar的Flink Connector在社区投票中被认为不如Kafka稳定
      - RocketMQ的Flink Connector只支持At-Least-Once
   
   3. 团队能力: 
      - 团队中3人有Kafka运维经验，0人有Pulsar经验
      - 迁移到Pulsar意味着至少3个月的学习和踩坑成本
   
   4. 性能对比(实测):
      在我们的1KB消息+6分区场景中:
      - Kafka: 320,000 rec/s, P99 35ms
      - Pulsar: 280,000 rec/s, P99 42ms (我们的benchmark)
      - RocketMQ: 210,000 rec/s, P99 55ms
      
  所以选Kafka不是因为它'最好'，而是因为它'最适合我们当前的约束'。"

  ② 没有体现对比分析能力

  ④ 暴露出技术选型没有方法论

### Why 2: 为什么选Flink而不是Spark Streaming？

```
理想回答(90分+):
  "Flink vs Spark Streaming的本质差异在于'流处理哲学'：
   
   1. 计算模型的根本不同:
      Flink: 逐条处理(每来一条消息就处理一次)
      Spark Streaming(Structured Streaming): 微批处理(攒一小批再处理)
      
      这导致了延迟差异:
      - Flink: 端到端延迟 100ms-1s (取决于网络的RTT)
      - Spark Streaming: 端到端延迟 500ms-5s (取决于trigger interval)
      
      在我们的异常检测场景中：
      '大额订单(>10000元)在3秒内告警' 是业务硬性要求。
      如果用Spark Structured Streaming，最小区间是1秒trigger，
      加上Kafka→Spark→ClickHouse→Grafana的链路延迟，无法保证3秒。
      
      Flink则可以在150ms内完成检测和告警。
   
   2. 窗口触发的精准度:
      Flink的EventTime + Watermark 可以在Watermark到达时立即触发
      窗口计算。Structured Streaming需要等下一个Micro-Batch到达
      才能输出，这意味着窗口输出的延迟是不确定的。
   
   3. 状态管理的深度:
      在我们的宽表Join场景中(多流Join)，Flink的RocksDB StateBackend
      可以高效管理数十GB的状态数据，支持增量Checkpoint。
      Structured Streaming的State Store虽然也支持RocksDB，但
      恢复机制和增量Checkpoint不如Flink成熟。
   
   4. Savepoint的运维价值:
      Flink支持从任意Savepoint恢复，这在以下场景中非常实用:
      - 修改Flink作业并行度后从Savepoint恢复 (每周可能做1-2次)
      - Flink版本升级后从Savepoint恢复
      - A/B测试: 用不同代码从同一Savepoint启动，对比结果
      
      Structured Streaming需要在代码中手动设置checkpointLocation
      和恢复逻辑，不如Flink方便。
   
   5. 我们的实测对比:
      测试场景: 20万条/秒, 5分钟窗口聚合
      Flink: 端到端延迟 800ms, 无OOM
      Structured Streaming: 端到端延迟 4.5s, 出现过1次OOM
      
  结论: Flink不是在所有场景都优于Spark Streaming，但在我们这种
  '低延迟+有状态+多流Join'的场景中是更优选择。"

不满意回答(40分):
  "Flink是真正的流处理，Spark Streaming是微批处理。"
  
  问题分析:
  ① 没有解释为什么微批处理会导致什么问题
  ② 没有结合具体业务场景说
  ③ 没有提供量化数据支持
  ④ 忽略了Spark Structured Streaming的进步

追问: "Spark Structured Streaming已经支持Continuous Processing mode
       (1ms延迟)，你怎么看？"
  
  优秀回答:
  "Continuous Processing是实验性feature，只支持map-like操作，
  不支持窗口聚合和Join。而且社区活跃度低(自Spark 2.4引入后
  几乎没有重大更新)。用实验性feature做生产核心链路是不负责的。"
```

### Why 3: 为什么选ClickHouse而不是Doris/Druid？

```
理想回答(90分+):

   1. 我们的查询模式分析:

      - 95%是聚合查询: SUM/COUNT/AVG + GROUP BY
      - 很少做JOIN (宽表已经预JOIN了)
      - 并发低: 10-20个用户同时看Dashboard
      - 数据量: 每天新增500GB，保留72小时(1.5TB)
      - 查询延迟要求: < 2秒(P99)
      
      ClickHouse在这种'大宽表扫描+聚合'场景下是最优的。
   
   2. ClickHouse vs Doris:
      Doris的优势在于MySQL协议兼容(工具链丰富)和多表Join性能。
      但我们的宽表不需要Join，单表扫描场景ClickHouse比Doris快30-40%。
      
      实测对比(1亿行宽表, 10列, 6个维度聚合):
      ClickHouse: 0.8s
      Doris: 1.2s
      差别不大，但ClickHouse的运维更简单(单机也能跑好，不像Doris必须多节点)。
   
   3. ClickHouse vs Druid:
      Druid是为时序预聚合设计的:
      - 写入时需要定义聚合维度和指标
      - 查询走预聚合结果，非常快
      - 但无法查询原始明细数据
      
      我们的Dashboard需要支持Ad-Hoc查询(如'某个时间范围内，某个用户
      的所有交易明细')，Druid做不到，ClickHouse可以。
      Druid适合'指标看板'，ClickHouse适合'分析看板'。
   
   4. ReplicatedMergeTree的优势:
      - 自动副本同步(基于ZK)
      - TTL自动清理过期数据
      - 压缩比高(ZSTD, 10:1)
      - 向量化执行引擎(SIMD加速)
   
   5. 成本考量:
      ClickHouse 3节点(16核64GB) ≈ 月费 ¥6,000
      Doris 3节点 ≈ 月费 ¥8,000
      Druid(含Historical+Broker+Coordinator) ≈ 月费 ¥12,000
      
  结论: 在'大宽表+聚合分析+低并发'的场景下，ClickHouse是最佳性价比选择。"

不满意回答(40分):

  问题分析:

  ② 没有分析查询模式
  ③ "开源免费"不是选择依据(Doris也开源)
  ④ 没有提到ClickHouse的弱点(如Join性能差)

  优秀回答:

### Why 4: 为什么宽表在Flink中Join而不是在ClickHouse中Join？

```
理想回答(90分+):
  "这是'计算前移'的经典案例：

   1. ClickHouse Join的性能瓶颈:
      ClickHouse的Join实现只有Hash Join一种策略。执行流程:
      ① 先构建右表的Hash表(全量加载到内存)
      ② 再扫描左表，对每行查Hash表
      
      问题:
      - 右表必须能全部装入内存
      - Hash表构建是单线程的
      - Join结果不会物化，每次查询都要重新Join
      
      在我们的场景中，如果3张表(订单+订单项+用户)都在ClickHouse中Join:
      - 每次Grafana刷新都需要一次3表Join
      - 如果宽表有1000万行，右表有500万行，Hash表构建时间 > 3秒
      - 加上Join计算时间，总查询时间 > 5秒 (远超2秒目标)
   
   2. Flink预Join的计算前移优势:
      Flink双流Join的执行方式:
      ① order流和order_item流按order_id进行KeyBy
      ② 两个流connect后使用CoProcessFunction
      ③ 用ValueState存储对方流的待匹配数据
      ④ 匹配成功后输出到宽表ClickHouse
      
      优势:
      - Join在写入时就完成了，查询时直接扫描宽表(无Join)
      - Flink的RocksDB StateBackend可以管理GB级的Join状态
      - 有Watermark的超时清理机制，防止State无限增长
   
   3. 成本分析:
      方案A: Flink预Join → ClickHouse宽表
        Flink计算成本: 每1000条消息约需10ms CPU (Join计算)
        ClickHouse存储成本: 宽表比3张表大约多30%(冗余字段)
        ClickHouse查询成本: 单表扫描0.8秒
        
      方案B: ClickHouse实时Join
        Flink计算成本: 0 (无Join，纯同步)
        ClickHouse存储成本: 3张表各存一份(节省30%空间)
        ClickHouse查询成本: 3表Join约5秒
        
      总结: 方案A查询快6倍，多花30%存储和少量Flink计算。
       30%存储 = 1.5TB × 0.3 = 450GB ≈ 月成本增加 ¥100
       换来查询延迟从5秒降到0.8秒，这是非常值得的。
   
   4. 可行性前提(必须解释):
      不是所有场景都适合Flink预Join，前提是:
      - Join的维度表不能太大(在State中能存下)
      - Join的维度表变更不频繁(每次变更需要更新State)
      - Join Key的数据分布不能过于倾斜
      
      在我们的场景中:
      - order_id是自然主键，分布均匀
      - 订单相关的维度表(user、商品)的数据量可控
      - 所以适合"

不满意回答(40分):
  "因为ClickHouse Join太慢了。"
  
  问题分析:
  ① 没有解释"为什么慢"（Hash Join的机制）
  ② 没有量化对比("太慢"是多慢？)
  ③ 没有分析计算前移的代价和前提
  ④ 单一因素分析，缺少系统思维

追问: "如果未来要Join的维度表有1亿行，Flink的State还能存下吗？"
  
  优秀回答:
  "1亿行的维度表在Flink State中确实是个挑战。解决方案取决于表的特征:
  ① 如果维度表变化慢(<1000次/天): 使用Flink的Broadcast State，
     将维度表全量加载到每个TaskManager的内存中。1亿行×100字节/行
     ≈ 10GB，在32GB的TaskManager中可以接受。
  ② 如果维度表变化频繁: 考虑用异步IO(AsyncIO)查询外部存储(Redis/HBase)，
     State只存必要的映射关系。
  ③ 如果对数据延迟不敏感: 考虑T+1的批处理预Join替代流处理Join。"
```

### Why 5: 为什么用CDC而不是定时SELECT去拉取？

```
理想回答(90分+):
  "CDC vs 定时SELECT的对比本质上是'推'vs'拉'的对比：
   
   1. 延迟对比:
      定时SELECT(5秒一轮):
      - 平均延迟: 2.5秒(一半的轮询间隔)
      - 最大延迟: 5秒(刚错过一次轮询)
      - 方差大: 0-5秒随机分布
      
      CDC(Binlog捕获):
      - 平均延迟: 50ms(Binlog写入→Debezium读取→Kafka→Flink)
      - P99延迟: 200ms
      - 延迟稳定(不会因数据量变化而波动)
      
      业务影响: 如果检测"大额订单异常"的延迟是5秒，用户已经完成
      支付了，告警失去了预防价值。50ms的延迟意味着我们可能在用户
      点击"支付"按钮的页面加载过程中就发现异常。
   
   2. 数据完整性:
      定时SELECT的致命缺陷:
      ① 无法捕获DELETE: SELECT只获取当前存在的行，被删的行的变更
         历史完全丢失。需要加is_deleted标记列(污染业务表Schema)
      ② 中间状态丢失: 如果一行在5秒内被UPDATE了3次，SELECT只能
         看到最终值。中间过程的变更历史丢失，无法做'状态变迁分析'
      ③ 时间窗口问题: SELECT ... WHERE update_time > last_time
         在高并发下可能出现: 同一秒插入10行，只拉到了前5行，
         因为last_time=第5行的update_time，后5行就被遗漏了
      
      CDC的完整性:
      ① INSERT → Binlog包含完整的新行数据
      ② UPDATE → Binlog包含变更前后(before + after)的完整行数据
      ③ DELETE → Binlog包含被删除行的完整数据
      ④ 所有变更按Binlog顺序严格有序
   
   3. 对源库的影响:
      定时SELECT:
      - 每次SELECT都需要扫描update_time索引
      - 如果表有1000万行，扫描索引(即使返回0行)也需要约10ms
      - 每5秒一次 = 每天17,280次索引扫描
      - 在业务高峰期(其他查询也在跑)会增加源库压力
      
      CDC:
      - Debezium读Binlog完全不消耗MySQL的工作线程
      - MySQL写Binlog是已经存在的操作(无论是否启用CDC)
      - Debezium只是读文件，不影响正常业务的读写
      - 唯一的开销: Binlog格式必须是ROW(不是STATEMENT)，会增加
        约10%的Binlog体积
   
   4. Schema变更处理:
      定时SELECT:
      - 表结构变更(加列/删列/改类型)会导致SELECT失败
      - 需要人工介入修改SELECT语句
      
      CDC(Debezium):
      - 自动检测DDL变更(ALTER TABLE)
      - Schema Registry中的Schema自动更新兼容版本
      - 消费端可以灵活选择使用新Schema或旧Schema
   
   5. 唯一的劣势和应对:
      CDC的劣势:
      ① 全量初始化(Snapshot)期间会对源库加锁
         → 解决: snapshot.locking.mode=minimal (仅短暂加锁)
      ② Binlog传输占用网络带宽
         → 实测: 1万条变更/秒 × 500字节/条 = 5MB/s (可接受)
      ③ 运维复杂度更高(需要维护Kafka Connect集群)
         → 解决: 使用Confluent Platform或MSK Connect托管服务
      
      定时SELECT的唯一优势:
      ① 不需要Binlog权限(某些企业的DBA不允许开放Binlog读取权限)
         → 这实际上是组织问题而非技术问题"
