## 二、故障排查总方法论

```
故障排查五步法（适用于所有场景）:

Step 1: 快速定位（1分钟）
  - 什么组件出问题了？
  - 什么时间段开始的？
  - 影响范围有多大？
  
Step 2: 信息收集（3分钟）
  - 收集：错误日志、监控截图、最近变更记录
  - 工具：grep, jstack, top, df, netstat
  
Step 3: 假设验证（5分钟）
  - 提出最可能的2-3个原因
  - 快速验证每个假设
  - 排除法缩小范围
  
Step 4: 解决问题（10分钟）
  - 执行修复操作
  - 验证修复效果
  - 记录修复过程
  
Step 5: 复盘预防（事后）
  - 写Postmortem
  - 添加监控告警
  - 更新Runbook
```

## 三、故障场景详解

### 场景1：Spark任务OOM

```
┌─────────────────────────────────────────────────────────────┐
│ 故障ID: SPARK-001                                           │
│ 严重级别: P2 (部分任务失败)                                   │
│ 影响范围: 离线ETL任务                                        │
└─────────────────────────────────────────────────────────────┘

现象描述:
  - Spark任务失败，Executor日志显示:
    "java.lang.OutOfMemoryError: Java heap space"
    或
    "Container killed by YARN for exceeding memory limits"
  
  - Spark UI中Task Metrics: Shuffle Read Size 严重不均衡
  - 某些Task处理了几GB数据，其他Task只有几MB

根因分析:
  数据倾斜是最常见原因（80%的情况）
  → 某个Key的数据量是其他Key的100+倍
  → 导致处理该Key的Task OOM
  
  其他可能原因:
  - executor.memory 配置过小
  - 大对象没有及时释放
  - broadcast变量过大

排查步骤:
  
  Step 1: 查看Spark UI → Stages → 失败的Stage
    → 看Task Metrics: 输入数据量是否严重不均?
    → 看Shuffle Read Size分布
    
  Step 2: 定位倾斜Key
    spark.sql("""
      SELECT key_column, COUNT(*) as cnt
      FROM source_table
      GROUP BY key_column
      ORDER BY cnt DESC
      LIMIT 20
    """).show()
    
  Step 3: 确认倾斜严重程度
    如果最大的Key占比 > 30% → 数据倾斜
    如果最大的Key占比 < 5% → 可能是其他原因

解决方案（3种）:

  方案1: 加盐打散（最常用，适合groupBy操作）
    ```python
    from pyspark.sql.functions import rand, col, floor
    
    salt_num = 100
    df_salted = df.withColumn("salt", floor(rand() * salt_num))
    
    # 第一轮：加盐聚合
    result_1 = df_salted.groupBy("salt", "key_col").agg(...)
    
    # 第二轮：去盐最终聚合
    result = result_1.groupBy("key_col").agg(sum("value"))
    ```

  方案2: Broadcast Join（适合大表JOIN小表）
    ```python
    from pyspark.sql.functions import broadcast
    
    # 小表(< 2GB) → broadcast
    result = big_df.join(broadcast(small_df), "key")
    
    # 确认Spark自动broadcast阈值
    spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 104857600) # 100MB
    ```

  方案3: AQE自动倾斜处理（Spark 3.0+，最简单）
    ```python
    spark.conf.set("spark.sql.adaptive.enabled", "true")
    spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
    spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")
    spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256MB")
    ```

预防措施:
  - 关键聚合查询前，先检查Key的数据分布
  - 开启AQE（Spark 3.0+）
  - 生产环境的executor.memory至少8GB
  - 添加监控：Spark Task输入数据量≥平均值的5倍 → 告警
```

#### 多种解决方案对比

| 方案 | 适用场景 | 代码改动量 | 性能影响 | 局限性 |
|------|----------|-----------|---------|--------|
| 加盐打散 | GroupBy/Agg | 中等 | 需两轮聚合，额外Shuffle | 不适用于需要精确结果的Join |
| Broadcast Join | 大表JOIN小表 | 小 | 无Shuffle，极快 | 小表必须 < 广播阈值（默认10MB，可调至8GB） |
| AQE倾斜处理 | Join场景 | 仅配置 | 自动处理，无额外开销 | 仅Spark 3.0+，对GroupBy无效 |
| 过滤倾斜Key | 倾斜Key无业务意义 | 小 | 最高效 | 如果倾斜Key是有效数据，不可用 |
| Segment部分重分区 | GroupBy + Join混合 | 大 | 较好 | 实现复杂，维护成本高 |

#### 真实案例故事

2023年双十一大促期间，某电商平台的数据团队发现每日的"用户行为汇总"ETL任务从平时的12分钟延长至2小时后仍无法完成，最终以OOM报错退出。经排查发现，"未登录用户"行为的user_id被统一记录为空字符串""，导致当天5亿条行为日志中有3.2亿条（64%）共享同一个user_id。Spark在Shuffle阶段将所有空字符串的数据路由到同一个Reduce Task，该Task需要处理3.2亿条记录导致Executor堆内存耗尽。首次快速修复采用了"加盐打散"方案：在第一轮聚合时对空user_id添加0-999的随机后缀，第二轮聚合时将加盐结果合并。处理时间从2小时未完成降至8分钟。根本性修复是修改日志采集端，为未登录用户分配基于设备指纹的临时ID（格式：anon_{device_fingerprint}），使数据分布均匀化。同时在Spark任务中添加了倾斜检测逻辑：如果P95输入数据量超过P50的3倍，自动触发加盐策略。这个案例的教训是：数据倾斜往往不是技术问题而是数据模型问题，根因在数据采集层而非计算层。

### 场景2：Kafka Consumer Lag持续增长

```
┌─────────────────────────────────────────────────────────────┐
│ 故障ID: KAFKA-001                                           │
│ 严重级别: P2 (数据处理延迟)                                   │
│ 影响范围: 实时流处理管道                                      │
└─────────────────────────────────────────────────────────────┘

现象描述:
  - Kafka Consumer Group Lag 持续增长
  - Grafana监控曲线: Lag 呈45度角上升
  - 消费速率 < 生产速率

根因分析:
  原因优先级排序:
  1. Consumer处理逻辑变慢 (新加的逻辑、DB慢查询等) → 最常见
  2. Consumer数量不足 (Partition > Consumer数)
  3. Producer突然增量 (大促、流量突增)
  4. Broker IO瓶颈 (磁盘慢、Page Cache不足)

排查步骤:
  
  Step 1: 确认Lag增长速度
    kafka-consumer-groups.sh \
      --bootstrap-server localhost:9092 \
      --group <group-id> \
      --describe
    
    CURRENT-OFFSET: 12500000
    LOG-END-OFFSET: 13000000
    LAG:           500000
    
    每30秒查一次，计算增长率
    
  Step 2: 计算恢复时间
    消费速率 = Consumer吞吐量 (条/秒)
    恢复时间 = LAG ÷ 消费速率
    如果 > 1小时 → 需要紧急扩容
    
  Step 3: 检查Consumer处理耗时
    添加日志记录每条消息的处理时间:
    long start = System.currentTimeMillis();
    processRecord(record);
    long elapsed = System.currentTimeMillis() - start;
    if (elapsed > 1000) log.warn("Slow processing: {}ms", elapsed);
    
  Step 4: 检查Broker IO
    使用 iostat 或 node_exporter 查看磁盘利用率
    iostat -x 1 | grep -E "Device|sd[a-z]"

解决方案:

  方案1: 扩容Consumer (最快)
    - 增加Consumer实例数量到Partition数
    - 注意：Consumer数不能超过Partition数
    
  方案2: 优化处理逻辑 (根本)
    - 异步化耗时操作（如外部API调用）
    - 批量化DB写入（减少DB连接开销）
    - 使用本地缓存减少远程查询
    
  方案3: 临时提升消费速率 (应急)
    props.put("fetch.max.bytes", "104857600");    // 100MB
    props.put("max.poll.records", "5000");         // 增加批量
    props.put("max.partition.fetch.bytes", "10485760"); // 10MB
    
  方案4: 如果来不及消费 (兜底)
    - 增加Topic Retention时间，防止数据过期
    kafka-configs.sh --alter \
      --topic <topic> \
      --add-config retention.ms=604800000 \
      --bootstrap-server localhost:9092
    
    - 启动应急Consumer Group专门追赶

预防措施:
  - Kafka Exporter + Prometheus监控Lag
  - 设置Lag告警阈值（如Lag > 100万 → 告警）
  - 生产速率和消费速率的基线监控
  - Consumer处理时长的P99监控
```

public class SlowConsumerJob {
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        
        KafkaSource<Order> source = KafkaSource.<Order>builder()
            .setBootstrapServers("localhost:9092")
            .setTopics("lag-test")
            .setGroupId("slow-consumer-group")
            .setStartingOffsets(OffsetsInitializer.earliest())
            .build();
        
        env.fromSource(source, WatermarkStrategy.noWatermarks(), "Kafka Source")
            .map(order -> {
                Thread.sleep(100);  // ⚠️ 故意延迟100ms，模拟慢处理
                return order;
            })
            .sinkTo(/* ... */);
        
        env.execute("Slow Consumer - Lag Reproduction");
    }
}
EOF

# 步骤4: 观察Lag增长（每30秒执行一次）

#### 多种解决方案对比

| 方案 | 生效时间 | 改动成本 | 效果持续 | 副作用 |
|------|----------|---------|---------|--------|
| 扩容Consumer | 秒级（Rebalance ~30s） | 中（需部署新实例） | 永久（前提→Partition数够） | 额外的进程管理成本 |
| 优化处理逻辑 | 天级（需编码+测试+上线） | 高（代码改动） | 永久 | 可能引入新Bug |
| 临时提升消费参数 | 秒级（配置热加载） | 低（改配置） | 临时（治标不治本） | 可能增加GC压力 |
| 增加Partition | 分钟级（Topic变更） | 中（Topic重建或扩展） | 永久 | 有Key的消息顺序性被打乱 |
| 应急追数Group | 分钟级（部署+启动） | 中 | 临时（用完即停） | 需协调两个Group的消费进度 |

#### 真实案例故事

2024年618大促期间，某金融科技公司的"实时风控管道"出现严重积压。凌晨1点，值班工程师被Grafana的Critical告警唤醒：Consumer Lag从平时的5万飙升至280万，并以每分钟8万的速度增长。初步排查发现，促销活动开始后订单量增长了4倍，但风控Consumer数量未变。更致命的是，前一天上线了一个"多头借贷查询"功能，每条消息需调用第三方征信API（平均响应时间350ms），导致单条消息处理时间从12ms剧增到380ms——原先6个Consumer各处理833条/秒（总数5000条/秒），现在变为26条/秒（总数156条/秒），远低于50000条/秒的生产速率。紧急处理：将Kafka Topic保留时间从24小时延长至72小时（防止数据过期），同时将Consumer数量从6个扩至24个（Topic的Partition数为24）。然而扩容后发现第三方API有QPS限制（并发上限100），24个Consumer同时调用导致大量429（Too Many Requests）。最终方案：在Consumer上加了本地聚合逻辑——先将1000笔订单聚合为1次API批量请求，将API调用频率从每秒24次降至约0.2次，同时将单个订单的处理时间从380ms降至约15ms（摊销）。10分钟后，Lag开始下降，2小时后完全恢复。事故的教训是：Consumer Lag排查不仅要看消费速率，还要追踪每条消息处理链路中各环节的耗时分布。

### 场景3：Hive查询超慢

```
┌─────────────────────────────────────────────────────────────┐
│ 故障ID: HIVE-001                                            │
│ 严重级别: P3 (数据分析延迟)                                   │
│ 影响范围: 离线报表生成                                        │
└─────────────────────────────────────────────────────────────┘

现象描述:
  - 一份平时5分钟的报表，今天跑了30分钟还没出结果
  - Hive日志显示有大量Map Task，但进度很慢
  - 资源使用率正常（不是资源不足）

根因分析:
  最常见5个原因:
  1. 分区裁剪失效 → 全表扫描
  2. JOIN时数据倾斜
  3. 使用了TextFile格式（而非Parquet/ORC）
  4. 小文件过多（Map Task数过多）
  5. 没有合适的统计信息（CBO优化器无法工作）

排查步骤:
  
  Step 1: 查看执行计划
    EXPLAIN EXTENDED
    SELECT ... FROM ...
    WHERE ...
    
    关注:
    - 扫描了多少Partition? (应该只扫需要的分区)
    - Stage数量是否合理? (每个JOIN多1-2个Stage)
    - 有没有MapJoin? (小表应该自动MapJoin)
    
  Step 2: 检查分区裁剪
    EXPLAIN DEPENDENCY
    SELECT ... FROM ...
    WHERE dt = '2024-01-15'
    
    确认input_partitions数量是否为1（而不是全表分区数）

#### 真实案例故事

### 场景4：Flink反压（Backpressure）

```
┌─────────────────────────────────────────────────────────────┐
│ 故障ID: FLINK-001                                           │
│ 严重级别: P2 (流处理延迟增加)                                  │
│ 影响范围: 实时数据处理管道                                     │
└─────────────────────────────────────────────────────────────┘

现象描述:
  - Flink UI中算子显示为 HIGH (红色) 反压状态
  - Checkpoint时间越来越长甚至超时
  - Kafka Source的消费速率下降
  - 端到端延迟持续增加

根因分析:
  反压传播链:
    Sink慢 → 反压上游Window算子 → 反压Map算子 → 反压Source
  
  最常见原因:
  1. Sink端写入慢（ClickHouse/DB写入慢）
  2. 数据处理逻辑中有阻塞操作
  3. 窗口状态过大导致访问慢
  4. 数据倾斜导致部分Task过载

排查步骤:
  
  Step 1: Flink UI定位瓶颈
    访问: http://jobmanager:8081
    → 找到当前运行的Job
    → 查看各算子的BackPressure状态
    → 找到第一个出现 HIGH 的算子（这就是瓶颈）
    
  Step 2: 检查Sink写入性能
    如果瓶颈在Sink:
    - 检查外部系统响应时间
    - 增大Sink并行度
    - 增加Sink批量写入大小
    
  Step 3: 检查数据倾斜
    如果瓶颈在Window算子:
    查看Subtask Metrics: numRecordsIn
    如果某些Subtask的输入量 >> 其他 → 数据倾斜

解决方案:
  
  方案1: Sink优化 (如果瓶颈在Sink)
    // 增大批量写入
    JDBCSink.builder()
      .setBatchSize(5000)         // 增大批量
      .setBatchIntervalMs(5000)   // 增大间隔
    
  方案2: 增加并行度
    env.setParallelism(8);  // 全局增加
    或
    stream.keyBy(...).setParallelism(16);  // 算子级别增加
    
  方案3: 异步IO (减少阻塞)
    AsyncDataStream.unorderedWait(
      stream,
      new AsyncDatabaseRequest(),
      1000, TimeUnit.MILLISECONDS,
      100  // 最大并发请求数
    );
    
  方案4: 数据倾斜处理
    // 加随机前缀分散Key
    stream.map(new SaltMapFunction(100))
      .keyBy(...)
      .process(new DesaltProcessFunction());

预防措施:
  - 配置反压监控告警
  - 设置合理的并行度（≥ 数据量的1/100）
  - Sink端必须做批量写入
  - 定期Review算子处理耗时
```

#### 真实案例故事

某物流公司的"实时运单轨迹监控"Flink任务在2024年春运期间出现严重反压。任务链路是：Kafka（运单事件，峰值10万/秒）→ Flink（窗口聚合，1分钟滚动窗口）→ PostgreSQL Sink（轨迹点宽表）。问题表现是端到端延迟从正常3秒飙升至45分钟，Flink UI中所有算子都是HIGH反压。第一步排查发现瓶颈在PostgreSQL Sink——它的写入TPS只有800，而Flink的聚合输出约为1200条/秒。为什么只有800TPS？深入分析PostgreSQL发现，轨迹点宽表有12个二级索引（用于支持各种查询维度），每次INSERT都需要同时更新12个索引，严重拖慢了写入速度。根本解决方案是"读写分离"：将写入目标的表改为无索引的"Append-Only表"，同时创建一个Materialized View（物化视图）来维护索引，写入速度从800TPS提升至12000TPS。应急方案：将Sink并行度从4增至16，配合Flink的批量写入（batchSize=5000），将写入TPS提升至4000。这个案例的教训是：反压问题往往是Sink端的外部系统瓶颈，排查时不仅要看Flink侧，还要深入外部系统的内部机制。

### 场景5：HDFS小文件问题

```
┌─────────────────────────────────────────────────────────────┐
│ 故障ID: HDFS-001                                            │
│ 严重级别: P3 (集群性能下降)                                   │
│ 影响范围: 所有HDFS读写操作                                    │

现象描述:

  - HDFS读写性能逐渐下降
  - `hdfs dfs -ls` 命令响应变慢
  - NameNode WebUI响应慢

根因分析:
  HDFS中每个文件/目录/块在NameNode中占用约150字节内存
  1000万个小文件 ≈ 1.5GB NameNode内存
  
  小文件来源:
  - Spark/Hive任务输出文件过多
  - 流式写入产生大量小文件
  - 动态分区产生大量分区目录

排查步骤:
  
  Step 1: 统计小文件数量
    hdfs fsck / -files -blocks -locations 2>/dev/null | 
      grep "Total files" | head -1
    hdfs fsck / -files -blocks -locations 2>/dev/null | 
      grep "Average block size" | head -1
    
  Step 2: 找到小文件集中的目录
    hdfs dfs -du -h /warehouse/ | sort -rh | head -20
    
  Step 3: 按文件大小分布分析
    # 需要Hive统计分析
    SELECT 
      CASE 
        WHEN size < 1048576 THEN '<1MB'
        WHEN size < 10485760 THEN '1-10MB'
        WHEN size < 104857600 THEN '10-100MB'
        ELSE '>100MB'
      END as size_range,
      COUNT(*) as file_count,
      SUM(size) / 1024 / 1024 / 1024 as total_gb
    FROM (
      SELECT INPUT__FILE__NAME, SUM(length(*)) as size
      FROM table_name GROUP BY INPUT__FILE__NAME
    ) t
    GROUP BY size_range;

  方案1: Hive合并小文件
    -- 输出时限制文件数

    SET hive.merge.mapredfiles=true;
    SET hive.merge.size.per.task=268435456; -- 256MB
    SET hive.merge.smallfiles.avgsize=16777216; -- 16MB
    
  方案2: Spark合并小文件
    df.coalesce(10).write.mode("overwrite").parquet("/path")
    -- 或
    df.repartition(10).write.mode("overwrite").parquet("/path")
    
  方案3: 历史小文件合并
    hadoop jar hadoop-extras.jar \
      org.apache.hadoop.tools.HDFSConcat \
      /path/to/dir/*.parquet /path/to/merged.parquet

    -- 每天凌晨合并前一天的小文件
    INSERT OVERWRITE TABLE target_table PARTITION(dt='2024-01-15')
    SELECT /*+ COALESCE(10) */ *
    FROM target_table
    WHERE dt = '2024-01-15';

预防措施:
  - 监控NameNode文件数增长趋势
  - 设置Hive/Spark输出文件数上限
  - 每日定时运行小文件合并任务

#### 真实案例故事

### 场景7：MySQL连接池耗尽

```
┌─────────────────────────────────────────────────────────────┐
│ 故障ID: DB-001                                              │

现象:

  - 或 "Timeout waiting for idle object"

  - 新连接被拒绝

根因:
  1. 连接泄漏: 获取了连接但未释放（最常见）
  2. 连接池太小: 并发请求超过了连接池上限
  3. 慢SQL占用了连接: 某些查询长时间执行

排查:
  -- 1. 查看当前连接数
  SHOW STATUS LIKE 'Threads_connected';
  
  -- 2. 查看连接分布
  SELECT USER, HOST, DB, COMMAND, TIME, STATE
  FROM information_schema.PROCESSLIST
  ORDER BY TIME DESC;
  
  -- 3. 检查是否有长时间运行的查询
  SELECT * FROM information_schema.PROCESSLIST
  WHERE TIME > 300 AND COMMAND != 'Sleep';

解决:
  紧急:
  -- 杀掉空闲连接
  SELECT CONCAT('KILL ', id, ';') 
  FROM information_schema.PROCESSLIST
  WHERE TIME > 600 AND COMMAND = 'Sleep';
  
  根本:
  // HikariCP正确配置
  HikariConfig config = new HikariConfig();
  config.setMaximumPoolSize(20);     // 不要太大
  config.setConnectionTimeout(30000);
  config.setIdleTimeout(600000);
  config.setMaxLifetime(1800000);
  config.setLeakDetectionThreshold(60000); // 60秒泄漏检测

预防:
  - 配置连接泄漏检测(leakDetectionThreshold)
  - 监控连接池使用率
  - 代码审查: 确保所有connection都在finally中关闭

某金融公司的Kafka集群在2024年初频繁出现"幽灵重启"现象：Flink作业每1-2小时自动重启一次，但查看Kafka自身日志,各Broker正常运行。深挖发现，根因是ZooKeeper集群被部署在Kubernetes中，且被调度到与多个Spark Executor共享物理节点上。当Spark任务提交后，物理节点内存被榨干，ZK容器被OOM Killer杀死并重启。ZK重启后，所有客户端的Session全部超时——Kafka Controller重新选举、Flink JobManager重新选举——导致了Flink作业的级联重启。修复方案：为ZK配置专用的Kubernetes节点（使用Node Selector + Taint），确保ZK与其他计算密集型工作负载隔离；同时将ZK JVM堆从1GB提升至4GB，GC策略从ParallelGC切换为G1GC。经过这些调整，ZK的Full GC从平均每20分钟一次降至每天不到一次。最根本的建议：ZK集群应该独立部署在专用物理机/虚拟机上，不适合与其他大数据组件共享计算资源。

#### 真实案例故事

某大型互联网公司的Hive平台在2024年春节期间经历了持续4小时的Metastore中断。起因是运维团队对Metastore的后端MySQL进行了在线DDL（增加一个索引），该操作对`TBLS`表加了元数据锁。由于`TBLS`表存储了所有Hive表的元数据，任何需要访问表元数据的查询（包括`SHOW TABLES`）都会排队等待锁释放。而那个DDL操作因为表太大（500万行）跑了超过2小时还没完成，导致积压了3000多个被阻塞的查询。紧急方案：使用`KILL`命令终止了DDL操作，重启Metastore清空连接池，并在MySQL中配置`lock_wait_timeout=10`秒防止类似事件。这次事故后，团队实现了Metastore HA（部署3个Metastore实例），并将元数据存储从单机MySQL迁移至MySQL Group Replication集群。教训：对Metastore DB的任何变更必须在变更窗口内执行，且必须有回滚预案；Metastore是无状态的，但后端DB是整个平台的单点。

### 场景10：Flink Checkpoint失败

```
┌─────────────────────────────────────────────────────────────┐
│ 故障ID: FLINK-002                                           │
│ 严重级别: P1 (丢失容错能力)                                   │
│ 影响范围: Flink流处理任务                                     │
└─────────────────────────────────────────────────────────────┘

现象:
  - Flink UI显示 Checkpoint 持续 FAILED
  - 日志: "Checkpoint expired before completing"
  - 或 "Checkpoint was declined"
  - 连续多个Checkpoint失败后，Job可能失败

根因:
  1. Checkpoint超时: 状态太大或存储慢
  2. Barrier对齐慢: 数据倾斜或反压
  3. 存储空间不足: HDFS/S3 空间满
  4. RocksDB状态损坏

排查:
  # 1. Flink UI查看Checkpoint History
  # 关注: Checkpoint Duration, State Size, Alignment Duration
  
  # 2. 查看JobManager日志
  grep "checkpoint" flink-jobmanager.log | tail -20
  
  # 3. 检查Checkpoint存储
  hdfs dfs -du -h /flink/checkpoints/<jobId>/

解决:
  紧急:
  # 1. 增大Checkpoint超时(如果状态大)
  env.getCheckpointConfig().setCheckpointTimeout(300000); # 5分钟
  
  # 2. 开启非对齐Checkpoint (Barrier对齐慢)
  env.getCheckpointConfig().enableUnalignedCheckpoints();
  
  # 3. 增大Checkpoint间隔(减少频率)
  env.enableCheckpointing(120000); # 2分钟

预防:
  - 设置tolerableCheckpointFailureNumber=3（允许偶发失败）
  - 监控Checkpoint耗时和大小
  - 定期清理旧的Checkpoint
  - 使用更快的存储（如本地SSD代替HDFS）
```

#### 完整复现步骤

```bash
# 步骤1: 创建一个状态非常大的Flink Job（容易超时）
cat > LargeStateJob.java << 'EOF'
public class LargeStateJob {
    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        
        // ⚠️ 故意设短Checkpoint超时
        env.enableCheckpointing(30000);  // 30秒间隔
        env.getCheckpointConfig().setCheckpointTimeout(5000);  // 5秒超时
        env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);
        
        DataStream<String> stream = env
            .addSource(new FlinkKafkaConsumer<>("checkpoint-test", schema, props))
            .keyBy(value -> value)
            .process(new KeyedProcessFunction<String, String, String>() {
                // ⚠️ 每个Key维护一个大状态
                private ValueState<List<String>> largeState;
                
                @Override
                public void open(Configuration parameters) {
                    ValueStateDescriptor<List<String>> descriptor = 
                        new ValueStateDescriptor<>("largeState", 
                            new ListTypeInfo<>(String.class));
                    largeState = getRuntimeContext().getState(descriptor);
                }
                
                @Override
                public void processElement(String value, Context ctx, 
                                           Collector<String> out) throws Exception {
                    List<String> list = largeState.value();
                    if (list == null) list = new ArrayList<>();
                    list.add(value);
                    if (list.size() > 1000000) {  // 100万条
                        list = new ArrayList<>();  // 重置但不清理状态
                    }
                    largeState.update(list);
                    out.collect(value);
                }
            });
        
        stream.addSink(new DiscardingSink<>());
        env.execute("Large State Checkpoint Test");
    }
}
EOF

# 步骤2: 填充HDFS空间（使Checkpoint存储空间不足）
dd if=/dev/zero of=/tmp/fill_hdfs bs=1M count=100000
hdfs dfs -put /tmp/fill_hdfs /tmp/fill1
hdfs dfs -put /tmp/fill_hdfs /tmp/fill2
# ... 直到HDFS空间 < 5%

# 步骤3: 观察Checkpoint失败
# 在Flink UI → Checkpoints → History:
# 预期看到连续的红色FAILED状态

#### 真实案例故事

某出行平台的"实时订单匹配"Flink任务在2024年初出现连续Checkpoint失败，累计失败18次后Job自动终止。该任务维护了一个全量的"司机位置状态"（约300万司机×位置信息，总计约12GB状态），配置的Checkpoint间隔为30秒、超时时间为2分钟。排查发现：随着用户量增长，状态从6GB膨胀至12GB，Checkpoint耗时从50秒增长至140秒（仍小于2分钟）；但业务的"早高峰效应"导致8:00-9:00间数据量暴涨3倍，Checkpoint Barrier的对齐时间从15秒飙升至90秒，使端到端Checkpoint时间达到230秒（140+90），超过2分钟超时。解决方案：在早高峰时段（7:30-9:30）动态调整Checkpoint配置——将超时从2分钟提升至5分钟，间隔从30秒调整至90秒，同时开启非对齐Checkpoint（`enableUnalignedCheckpoints`）消除Barrier对齐等待。非高峰时段恢复原配置。长期方案：实施状态分片——将全局司机状态按城市分片到不同的Flink Job中，每个Job只管理本城市的状态（单Job状态从12GB降至约1.5GB）。这个案例的教训：Checkpoint配置不应一成不变，需要根据数据流特征和时间模式动态调整。

## 六、告警分级标准

```yaml
告警级别定义:
  
  P0 - 紧急 (立即响应，5分钟):
    现象: 核心数据管道完全中断
    例子: 
      - Kafka所有Broker宕机
      - Flink JobManager挂了无HA
      - HDFS NameNode不可用
      - Hive Metastore不可用（所有查询失败）
    通知: 电话 + 短信 + 企业微信
    响应: 5分钟内开始处理
    升级: 15分钟无进展 → 通知技术总监
  
  P1 - 严重 (30分钟内响应):
    现象: 核心功能受影响但有降级方案
    例子:
      - 单个Broker宕机
      - Flink Job失败但有Checkpoint可恢复
      - Consumer Lag超过30分钟的数据量
      - MySQL连接池耗尽
      - ZooKeeper Session超时
    通知: 电话 + 企业微信
    响应: 30分钟内开始处理
    升级: 1小时无进展 → 通知技术经理
  
  P2 - 一般 (2小时内响应):
    现象: 非核心功能受影响
    例子:
      - Spark任务OOM（可重试）
      - Hive查询变慢
      - Flink反压（仍可处理，只是慢）
      - Grafana面板加载慢
    通知: 企业微信
    响应: 2小时内处理
    升级: 4小时无进展 → 通知值班主管
  
  P3 - 低 (下个工作日处理):
    现象: 不影响业务但有潜在风险
    例子:
      - HDFS小文件增多
      - Checkpoint偶尔失败一次
      - ClickHouse Merge积压缓慢增长
      - 磁盘使用率 > 70%
    通知: 邮件
    响应: 下个工作日处理
```
