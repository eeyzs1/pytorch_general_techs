# 贯穿式递进大项目：电商数据平台全周期演进

> **定位**：一个从L0到L4贯穿始终的递进式项目，让学员在同一个业务场景下体验数据平台从"脚本时代"到"云原生AI平台"的完整演进过程。每个阶段都在前一阶段的基础上迭代，而非推倒重来。

---

## 设计理念

- **同一业务场景，5个演进阶段**：始终围绕"电商平台"这一业务，从用户行为日志到交易数据，从离线报表到实时推荐，业务连续性贯穿始终
- **每个阶段产出可运行、可演示的系统**：不是纸上谈兵，而是真正能跑、能看、能用的系统
- **后一阶段必须复用前一阶段的数据和代码**：数据是核心资产，代码是技术积累，绝不推倒重来
- **每次升级都要回答"为什么要升级"和"升级解决了什么问题"**：技术选型必须有业务驱动力，拒绝为技术而技术

### 核心痛点（驱动升级到阶段1）

| # | 痛点 | 具体表现 | 影响范围 |
|---|------|----------|----------|

### 核心痛点（驱动升级到阶段2）

| # | 痛点 | 具体表现 | 影响范围 |

## 数据复用指南

每个阶段必须复用前一阶段的数据资产，这是贯穿项目的核心原则。以下是详细的复用路径：

### 阶段0 → 阶段1：MySQL数据上云

```
阶段0产出                    阶段1复用方式
─────────────────────────────────────────────────────────
MySQL电商数据库      ──▶     Sqoop全量导入 → HDFS ODS层
                           · orders表 → ods_order_inc
                           · users表 → ods_user_inc
                           · products表 → ods_product_inc

Nginx日志CSV文件     ──▶     上传HDFS → ODS层
                           · access.log → ods_log_inc

Python分析脚本逻辑   ──▶     迁移为Spark SQL
                           · PV/UV统计 → dws_traffic_daily
                           · 热门页面 → ads_hot_page

crontab定时规则      ──▶     迁移为Airflow DAG
                           · 0 2 * * * → daily_etl_dag
```

### 阶段1 → 阶段2：Hive表迁移到Iceberg

```
阶段1产出                    阶段2复用方式
─────────────────────────────────────────────────────────
Hive ODS/DWD/DWS表  ──▶     迁移为Iceberg表格式
                           · CREATE TABLE ... STORED BY Iceberg
                           · 数据文件无需移动，只改元数据

Sqoop导入链路        ──▶     CDC(Debezium)替代
                           · 全量导入 → binlog实时捕获
                           · 增量导入 → CDC事件流

Spark离线ETL作业     ──▶     保留+补充Flink实时作业
                           · 离线管道不变
                           · 新增实时管道并行运行

Hive ADS查询         ──▶     迁移到ClickHouse
                           · ADS层数据同步到ClickHouse
                           · 查询从分钟级降到秒级
```

### 阶段2 → 阶段3：数仓数据喂养ML

```
阶段2产出                    阶段3复用方式
─────────────────────────────────────────────────────────
Iceberg湖仓数据      ──▶     ML训练数据源
                           · DWS用户行为表 → 流失预测特征
                           · DWS商品交互表 → 推荐排序特征
                           · DWD订单明细表 → 特征工程

Kafka实时数据流      ──▶     Feast在线特征源
                           · 用户实时行为 → Redis在线特征
                           · 商品实时库存 → 特征实时更新

ClickHouse查询       ──▶     RAG助手数据后端
                           · 自然语言 → SQL → ClickHouse
                           · 查询结果 → 自然语言回答

Docker Compose配置   ──▶     迁移为K8s Helm Chart
                           · docker-compose.yaml → values.yaml
                           · 每个服务 → K8s Deployment
```

### 阶段3 → 阶段4：云原生平台化

```
阶段3产出                    阶段4复用方式
─────────────────────────────────────────────────────────
K8s单集群部署        ──▶     多租户Namespace隔离
                           · 每个业务线 → 独立Namespace
                           · ResourceQuota → 资源限制

MLflow+Feast         ──▶     AI租户独立部署
                           · 模型服务 → 独立K8s Service
                           · 特征存储 → 共享+权限隔离

散落的SQL/Spark作业  ──▶     dbt统一管理
                           · Spark SQL → dbt model
                           · 数据转换逻辑 → 版本控制

手动部署流程         ──▶     CI/CD自动化
                           · Git Push → 自动测试 → 自动部署
                           · 数据变更 → 数据测试 → 灰度发布
```
