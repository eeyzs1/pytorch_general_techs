#### 论文1：The Google File System (GFS) — 2003年

| 属性 | 内容 |
|------|------|
| **作者** | Sanjay Ghemawat, Howard Gobioff, Shun-Tak Leung (Google) |
| **一句话核心** | 为Google的网页爬取和大规模数据处理设计了一个可扩展的分布式文件系统 |
| **对应技术栈** | HDFS、Ceph、MinIO |

**阅读重点**：

1. **大块存储设计（Chunk Size = 64MB）**
   - 为什么选择64MB而不是更小的块？
   - 大块对元数据管理的影响（Master内存占用）
   - 大块对网络带宽的影响（减少了与Master的交互）
   - 大块导致的问题：小文件存储浪费、热点文件问题

2. **追加写语义（Append-Only）**
   - GFS为什么不支持随机写？
   - Atomic Append的实现：如何保证"至少追加一次"语义
   - 记录追加（Record Append）与普通追加的区别
   - 这种设计对上层应用（MapReduce）的影响

3. **Snapshot机制**
   - Copy-on-Write技术如何实现？
   - Snapshot在GFS中解决了什么问题？
   - 与现在文件系统的Snapshot对比

4. **单Master架构的权衡**
   - 为什么敢用单Master？（GFS的假设是什么）
   - Master故障恢复机制：Operation Log + Checkpoint
   - 这个设计的局限在哪里？（对比HDFS HA/联邦方案）

**批判性思考题目**：
- GFS假设"大多数文件很大（GB级别）"，这个假设在你的实际场景中成立吗？
- 如果GFS面对大量小文件（如IoT传感器每秒一条数据），设计需要怎么改？
- 为什么HDFS没有完全照搬GFS的设计？哪些地方不同？为什么不同？

**强制输出**：画GFS写流程图（标注Chunk Lease、Pipeline复制、Acknowledgement的顺序）

#### 论文2：MapReduce: Simplified Data Processing on Large Clusters — 2004年

| 属性 | 内容 |
|------|------|
| **作者** | Jeffrey Dean, Sanjay Ghemawat (Google) |
| **一句话核心** | 提出了一种简单而强大的编程模型，让普通程序员也能处理TB级数据 |
| **对应技术栈** | Hadoop MapReduce、Spark RDD的核心思想来源 |

**阅读重点**：

1. **MapReduce编程模型**
   - Map和Reduce的输入/输出类型约束（key-value对）
   - 为什么Map的输出要先按key分组（Shuffle阶段）再交给Reduce？
   - Combiner的作用：为什么Combiner必须是幂等的？
   - 分区函数（Partitioner）的默认实现和自定义场景

2. **容错策略**
   - Map任务失败: 重新调度到另一台机器（因为输出是临时文件，丢失了也无所谓）
   - Reduce任务失败: 重新执行（因为输出已写入GFS）
   - Master失败: 整个Job重做（为什么不做Master HA？）
   - Straggler处理: Backup Task机制——这个机制在Spark中也存在吗？

3. **Locality优化**
   - 如何将Map任务调度到数据所在的机器？
   - GFS的副本策略如何帮助Locality？
   - 如果数据所在机器都忙，怎么办？（迁移到同机架的机器 vs 跨机架）

4. **MapReduce的局限**
   - 为什么Spark要替代MapReduce？
   - 物化中间结果（写磁盘）导致的问题
   - 不适合迭代计算（机器学习算法需要多次迭代）

**批判性思考题目**：
- 如果数据量只有1GB，还需要用MapReduce吗？为什么？
- MapReduce的Shuffle阶段是瓶颈吗？什么情况下是？
- 你能想到哪些不能用MapReduce解决的分布式计算问题？

**强制输出**：手写一个WordCount的MapReduce伪代码 + 数据流图（标注每个阶段的key-value类型变化）

#### 论文5：Resilient Distributed Datasets (RDD) — 2012年

| 属性 | 内容 |
|------|------|
| **作者** | Matei Zaharia等 (UC Berkeley / Databricks) |
| **一句话核心** | 提出了RDD抽象——一个容错的、可并行的、支持粗粒度转换的数据结构，使得内存计算成为可能 |
| **对应技术栈** | Apache Spark核心 |

**阅读重点**：

1. **RDD的五大特性（必背！）**
   - Partition List：数据被分成哪些分区
   - Compute Function：如何从父RDD计算当前RDD
   - Dependency List：对父RDD的依赖关系
   - Partitioner（可选）：HashPartitioner vs RangePartitioner
   - Preferred Locations（可选）：数据在哪台机器上

2. **Lineage（血统）与容错**
   - Lineage如何实现容错？（重算失败的分区，不需要复制数据）
   - Lineage的粒度：一个分区 vs 一个RDD
   - Lineage过长的代价（能Chain多长？）
   - Checkpoint如何截断Lineage

3. **窄依赖 vs 宽依赖**
   - 窄依赖：一个父分区最多被一个子分区使用（map、filter、union）
   - 宽依赖：一个父分区被多个子分区使用（groupByKey、join、reduceByKey）
   - 为什么宽依赖会触发Shuffle？
   - Stage划分的规则：遇到宽依赖就切分

4. **RDD vs DSM（Distributed Shared Memory）**
   - 论文为什么强调RDD不是DSM？
   - 粗粒度转换 vs 细粒度更新的区别
   - Checkpoint的开销对比

**批判性思考题目**：
- RDD的Lineage容错比Checkpoint容错好在哪？什么场景下Checkpoint更好？
- 如果Lineage链条太长（100步），恢复一个分区的代价有多大？
- DataFrame/Dataset相比RDD做了什么优化？（Catalyst优化器 + Tungsten）

**强制输出**：画一个WordCount的DAG图（从textFile到collect），标注每个RDD的依赖类型和Stage边界

#### 论文12：Spanner: Google's Globally-Distributed Database (OSDI 2012)

| 属性 | 内容 |
|------|------|
| **作者** | James C. Corbett等 (Google) |
| **一句话核心** | 通过TrueTime API提供全球分布式数据库的外部一致性（External Consistency = 严格可串行化） |
| **对应技术栈** | Google Cloud Spanner、CockroachDB、TiDB（部分思想） |

**阅读重点**：

1. **TrueTime API**
   - TT.now() 返回的是一个时间区间 [earliest, latest]，而不是一个时间点

   - ε通常是多少？（1-7ms）
   - 为什么这种不确定性是可接受的？

2. **External Consistency（外部一致性）**
   - 比强一致性更强的保证
   - 如果事务T1在T2之前提交，那么T1的时间戳必须小于T2的时间戳
   - 如何用TrueTime实现？Commit Wait机制

3. **Paxos在Spanner中的角色**
   - 每个Paxos Group管理一个Tablet
   - Leader选举和日志复制
   - 为什么不用Raft？（历史原因）
   - Multi-Paxos的优化

4. **数据分布**
   - Directory（目录）作为数据放置单元
   - 如何在保证数据Locality的同时做负载均衡？
   - 全球部署的延迟挑战

**批判性思考题目**：
- TrueTime依赖原子钟和GPS，这意味着什么物理限制？
- 如果GPS被干扰，Spanner会怎样？论文有没有提到应对方案？
- 开源世界的CockroachDB和TiDB是如何在没有TrueTime的情况下实现类似保证的？

**强制输出**：画Spanner的事务提交时序图（包含Coordinator Leader和Participant Leaders的交互 + Commit Wait）

## 论文阅读方法论：三遍阅读法

### 第一遍：海量阅读（30分钟/篇）

**目标**：建立整体认知，知道这篇论文在解决什么问题

**阅读内容**（严格按此顺序）：
1. Title + Abstract（3分钟）：论文的核心贡献是什么？
2. Introduction（10分钟）：Motivation是什么？之前的方法有什么问题？
3. All Figures & Tables（10分钟）：图表通常包含最核心的设计思想
4. Conclusion（5分钟）：论文的最终主张和未来工作
5. 快速浏览Related Work（2分钟）：知道这篇论文在整个领域中的位置

**输出要求**：
- 一段话总结（100-200字）：解决了什么问题？核心思路是什么？
- 1个最让你印象深刻的图表：贴图 + 一句话解释

**不要做的事**：
- 不要纠结于算法的每个细节
- 不要试图理解所有数学公式
- 不要读Related Work全文（第一遍只需要知道位置）

---

### 第二遍：深度理解（2-4小时/篇）

**目标**：理解核心设计的每个细节，能向同事讲清楚

**阅读内容**：
1. 全文精读（特别注意System Design和Implementation部分）
2. 每遇到一个关键设计决策，停下来问自己"为什么这样设计？"
3. 对比你已知的技术系统（如：这里的做法和Kafka有什么异同？）
4. 标记不理解的部分（记下来，用于小组讨论）

**输出要求**：
- 结构化读书笔记（使用下面的模板）
- 能够给同事做30分钟的技术分享

**第二遍读书笔记模板**：

### 第三遍：批判思考（可选·精读，按需1-2小时）

**目标**：站在更高的维度评判论文——设计替代方案、质疑假设、找出局限

**具体做法**：
1. 重新审视论文的假设前提：这些假设在什么场景下不成立？
2. 尝试设计替代方案：如果不采用论文的方案，还有什么其他解法？
3. 对比开源实现：开源实现和论文描述有什么偏差？为什么有偏差？
4. 思考演进路线：这篇论文之后，这个方向又有什么新的发展？

**输出要求**：
- 1篇批判性评论文章（1000+字）
- 包含：至少1个论文未回答的问题 + 1个你自己的替代设计思路

## 论文阅读的"What/Why/How"框架

对于每篇论文的核心算法或设计，用以下框架快速梳理：

| 维度 | 问题 | 示例（以GFS为例） |
|------|------|-------------------|
| **What** | 他们做了什么？ | 设计了一个64MB大块的append-only文件系统 |
| **Why** | 为什么这样做？ | 减少元数据、利用顺序写性能、适配MapReduce |
| **How** | 具体怎么实现？ | Chunk Lease机制、Pipeline复制、Atomic Append |
| **Trade-off** | 牺牲了什么？ | 不支持随机写、单Master有瓶颈、小文件浪费 |
| **Application** | 可以怎么用？ | 大文件存储、日志存储、数据湖底层 |

## 提醒

- **不要跳过第一遍**：很多人一上来就想"读懂每一个细节"，结果读了一周连第一章都没读完。必须先建立整体认知。
- **不要在论文上过度内耗**：有些数学证明看不懂是正常的，先标记出来，后面再看。
- **强制输出是最好的学习方法**：你觉得自己"理解了"和你能写出来是两回事。每篇论文必须写笔记。
- **关心论文的"时代背景"**：2003年的GFS为什么选择64MB的Chunk？放到2024年这个选择还成立吗？
