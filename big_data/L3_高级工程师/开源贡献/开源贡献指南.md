# 开源贡献指南

> **目标**：完成至少1个Apache Spark/Flink/Kafka社区PR，从看Issue→修代码→Code Review→合入全流程
>
> **周期**：第31-32周（2周） | **产出**：至少1个PR链接 + 完整的社区互动记录

---

## 为什么做开源贡献？

开源贡献不是L3的"课外活动"，而是核心考核项目。以下是它的价值：

1. **技术深度的证明**：能在Apache项目级别提交代码，说明你真的读懂了源码
2. **英文协作能力**：用英文和全球开发者讨论技术方案
3. **Code Review的体验**：你的代码会被世界顶尖的工程师Review，这是很难得的学习机会
4. **简历的强力加分项**：一个被合入的Apache项目PR > 10个自嗨的Side Project
5. **社区人脉**：通过开源贡献认识社区的Committer和PMC成员

## 第1阶段：准备工作（2-3天）

### 1.1 注册社区账号

```yaml
必须注册:
  1. Apache JIRA账号:
     网址: https://issues.apache.org/jira/
     注册后记住用户名（JIRA ID）
  
  2. GitHub账号:
     网址: https://github.com/
     要求: 
       - 头像正规（不要卡通或空白）
       - Profile中有简短的自我介绍
       - 关联JIRA ID

  3. 订阅开发邮件列表:
     Spark: dev@spark.apache.org
     Flink: dev@flink.apache.org
     Kafka: dev@kafka.apache.org
     订阅方式: 发送邮件到 dev-subscribe@xxx.apache.org
     
     重要: 每天花15分钟浏览邮件列表，了解社区动态
```

### 1.2 Fork和Clone项目

```bash
# 以Spark为例
# 1. 在GitHub上Fork apache/spark 到你自己的账号
# 访问: https://github.com/apache/spark → 点击Fork按钮

# 2. Clone你Fork的仓库
git clone https://github.com/<你的GitHub用户名>/spark.git
cd spark

# 3. 添加upstream远程仓库
git remote add upstream https://github.com/apache/spark.git

# 4. 验证
git remote -v
# origin    https://github.com/<你的用户名>/spark.git (fetch)
# origin    https://github.com/<你的用户名>/spark.git (push)
# upstream  https://github.com/apache/spark.git (fetch)
# upstream  https://github.com/apache/spark.git (push)

# 5. 同步upstream最新代码
git checkout master  # 或 main
git fetch upstream
git rebase upstream/master
```

### 1.3 搭建本地开发环境

```bash
# Spark
./build/mvn -DskipTests clean package

# Flink
mvn clean package -DskipTests -Dfast

# Kafka
./gradlew jar
```

### 1.4 了解项目构建和测试

```bash
# 运行特定模块的测试（以Spark SQL为例）
./build/mvn test -pl sql/core -Dtest=none -DwildcardSuites=org.apache.spark.sql.DataFrameSuite

# 运行ScalaStyle/JavaStyle检查
./build/mvn scalastyle:check
./build/mvn checkstyle:check

# 运行单个测试方法
./build/mvn test -pl sql/core -Dtest=none -DwildcardSuites=org.apache.spark.sql.DataFrameSuite#testSelect
```

## 第2阶段：寻找合适的Issue（3-5天）

### 2.1 Issue筛选策略

在JIRA上搜索时使用以下过滤条件：

```yaml
优先级排序:
  第一选择: "Good First Issue" / "beginner" / "newbie" 标签
  第二选择: "Documentation" 或 "Improvement" 类型
  第三选择: Bug修复（状态为"Open"且未被Assign的）
  避免选择: 
    - 被标记为"Major"或"Blocker"的Bug（太难，需要深度熟悉代码）
    - 需要讨论设计方案的"Umbrella" Issue
    - 已经被其他人Assign的Issue
    - 超过3个月没更新的Issue（可能已过时）
```

**各项目Good First Issue搜索链接**：
- Spark: https://issues.apache.org/jira/issues/?filter=12343120
- Flink: https://issues.apache.org/jira/issues/?filter=12343700
- Kafka: https://issues.apache.org/jira/issues/?filter=12326078

### 2.2 Issue筛选Checklist

在Claim一个Issue之前，必须检查：
- [ ] Issue有清晰的问题描述
- [ ] Issue被确认是有效的Bug或合理的Improvement
- [ ] 最后的活动时间在3个月以内
- [ ] 没有人正在Work On This（检查Assignee字段和最新评论）
- [ ] 你理解这个问题，并能想到解决方案的雏形

### 2.3 可选方向推荐

| 难度 | 方向 | 示例 |
|------|------|------|
| ★☆☆☆☆ | 文档修正 | 修正拼写错误、补充缺失的文档 |
| ★★☆☆☆ | 错误信息优化 | 让报错信息更清晰、增加诊断建议 |
| ★★★☆☆ | 小Bug修复 | 边界条件处理不当、空指针等 |
| ★★★★☆ | 小功能新增 | 添加一个新的配置项、支持一种新类型 |
| ★★★★★ | 性能优化/核心改进 | 优化某个算子的实现 |

### 2.4 Claim Issue

```text
在JIRA Issue下评论，建议模板：

Hi, I'm interested in working on this issue. 
I've checked the codebase and I think the problem is [简要描述你的理解].
I plan to fix it by [简要描述你的方案].
Could someone assign this to me?

Thanks!
[你的名字]
```

**等待24-48小时**，如果有Committer回复确认，就可以开始动手了。

## 第3阶段：开发与测试（1周）

### 3.1 创建开发分支

```bash
# 分支命名规范
# 格式: [JIRA_ID]_[简短描述]
git checkout -b SPARK-12345_fix_null_pointer_in_catalyst
git checkout -b FLINK-67890_add_timeout_config
git checkout -b KAFKA-11111_improve_error_message

# 重要: 从最新的master/main创建分支
git checkout master
git pull upstream master
git checkout -b <你的分支名>
```

### 3.2 代码规范

**Spark代码规范**：
- Scala代码必须通过ScalaStyle检查
- 使用2空格缩进（不是Tab）
- 方法命名使用camelCase
- 必须有Apache License头

**Flink代码规范**：
```java
// 使用4空格缩进
// 所有public方法必须有JavaDoc
// import不能使用通配符（*）
```

**Kafka代码规范**：
- 遵循项目现有的checkstyle配置
- 主要在Scala源文件中编写

**通用规则**：
```yaml
不要做的事情:
  - 不要改动与Issue无关的代码
  - 不要做代码风格的大规模重构（除非Issue明确要求）
  - 不要引入新的依赖（除非绝对必要）
  - 不要在PR中包含多个不相关的修改

必须做的事情:
  - 添加单元测试（覆盖你的修改）
  - 运行现有测试确保不破坏任何东西
  - 如果你的修改改变了行为，更新相关文档
  - 写清楚的commit message
```

### 3.3 Commit Message规范

```
格式（Spark示例）:
[SPARK-12345][SQL] Fix NullPointerException in Catalyst when parsing empty expression

## What changes were proposed in this pull request?
简要描述改了什么

## Why are the changes needed?
为什么要改

## Does this PR introduce any user-facing change?
是否有用户可见的变化（是/否）

## How was this patch tested?
怎么测试的（单元测试/集成测试/手动验证）

Closes #XXXXX from <你的用户名>/<分支名>.
```

### 3.4 测试要求

```yaml
测试标准:
  最低要求:
    - 添加至少1个单元测试覆盖你的修改
    - 所有现有测试仍然通过
  
  推荐:
    - 覆盖边界条件（空输入、极限值）
    - 如果修改了异常处理，测试异常是否被正确抛出
    - 如果有性能影响，添加性能对比测试
  
  测试命令:
    Spark: ./build/mvn test -pl <模块名> -Dtest=<测试类名>
    Flink: mvn test -pl <模块名> -Dtest=<测试类名>
    Kafka: ./gradlew <模块名>:test --tests <测试类名>
```

## 第4阶段：提交PR与Code Review（1-2周）

### 4.1 创建PR

```bash
# 1. Push你的分支到你Fork的仓库
git push origin SPARK-12345_fix_null_pointer_in_catalyst

# 2. 在GitHub上创建PR
# 访问 https://github.com/apache/spark → Pull Requests → New Pull Request
# 选择 compare across forks → 选择你的分支 → Create Pull Request

# 3. 如果你需要修改PR（根据Review意见）:
git add .
git commit -m "Address review comments: 修改了XXX"
git push origin SPARK-12345_fix_null_pointer_in_catalyst
# PR会自动更新
```

### 4.2 PR标题和描述模板

```markdown
### Description
[清晰描述你的改动]

### Related Issue
Closes #JIRA_ID

### Motivation and Context
[为什么要做这个改动]

### How Has This Been Tested?
- [ ] 单元测试
- [ ] 集成测试
- [ ] 手动验证

### Types of Changes
- [ ] Bug fix (non-breaking change)
- [ ] New feature (non-breaking change)
- [ ] Documentation update
- [ ] Improvement (non-breaking change)

### Checklist
- [ ] 代码通过了style检查
- [ ] 添加了单元测试
- [ ] 所有现有测试通过
- [ ] 没有不相关的改动
```

### 4.3 如何处理Code Review

**Code Review中的常见反馈类型及应对**：

| 反馈类型 | 回复方式 | 示例 |
|----------|----------|------|
| 代码风格 | 快速修改，感谢Reviewer | "Good catch, fixed." |
| 逻辑错误 | 承认并修改，解释新方案 | "You're right, I've changed it to..." |
| 设计建议 | 如果同意→修改；如果不同意→礼貌讨论 | "I considered this approach but..." |
| 测试不足 | 补充测试用例 | "Added test cases for edge cases." |
| 文档缺失 | 补充文档 | "Updated the documentation." |
| 需要更多背景 | 解释为什么这样选择 | "The reason I chose this approach is..." |

**Code Review礼仪**：

```yaml
DO:
  - 每条Review意见都要回复（即使只是"Done"或"Fixed"）
  - 如果不同意，说"I respectfully disagree because..."
  - 感谢Reviewer的时间
  - 在修复后等待24-48小时再@Reviewer重新review
  - 如果一周没有回复，礼貌地ping一下

DON'T:
  - 与Reviewer争论（讨论是好的，争论不是）
  - 回复"你懂什么"或类似的语气
  - 忽略Review意见
  - 在没被要求的情况下Force Push覆盖Review历史
```

### 4.4 持续等待与推动

```text
PR合入的典型时间线:
  Day 1-3: 有人看到你的PR并开始Review
  Day 3-7: 第一轮Review完成，你可能需要修改
  Day 7-14: 后续Review和修改
  Day 14-21: 被Committer合入
  
  Kill Tips（如果一直没人看）:
  - 在PR下@项目的Committer（查看项目的Committer名单）
  - 在JIRA Issue下留言"PR is ready for review"
  - 在邮件列表上提到你的PR（注意语气：不要催，只说ready for review）
```

## 第5阶段：PR被合入后

### 5.1 合入后的工作

```yaml
立即:
  - 在LinkedIn/GitHub profile中更新你的贡献记录
  - 截图保存PR页面（作为证据）
  - 更新你的技术博客："我如何给Apache XXX贡献了第一个PR"

后续:
  - 继续关注这个Issue，看是否有人报告相关问题
  - 寻找下一个Issue（这次可以挑战更高难度）
  - 持续参与邮件列表讨论
```

### 5.2 如果PR最终没有被合入

这是常见情况（Apache项目的PR合入率可能低于50%）。不要气馁。

```yaml
PR没有被合入的可能原因:
  - 有人提出了更好的方案
  - 需求发生了变化
  - Issue本身被标记为Won't Fix
  - 你的方案不够完善

这些都算成功:
  - 你经历了完整的Code Review流程
  - 你学会了开源社区的协作规范
  - 你看到了世界级工程师是如何思考问题的
  - 你有实际的代码修改可以展示
```

## 常见问题与陷阱

### Q1: 我的PR放了2周没人看，怎么办？

A: 在PR下评论（@相关的Committer）："Hi @committer, this PR is ready for review. Is there anything I can do to help move it forward?"

### Q2: Reviewer让我做很大的改动，但我觉得没必要，怎么办？

A: 先说明你的理由（"The reason I chose this approach is... because..."），如果Reviewer坚持，通常按Reviewer说的做。作为新人，Reviewer的经验通常比你丰富。

### Q3: 我修复了一个Bug，但不知道怎么写测试？

A: 看项目中类似的测试用例是怎么写的，模仿即可。一个好的测试应该：
- 在你的修改之前会失败（Red）
- 在你的修改之后会通过（Green）

### Q4: 我的修改涉及多个文件，PR太大了怎么办？

A: 如果PR超过500行（不含测试），考虑拆分。但如果改动是逻辑上不可分割的整体，也可以一起提交。

### Q5: 如何运行CI？

A: Apache项目的CI（Jenkins/GitHub Actions）是在你提交PR后自动运行的。你不需要手动触发。关注PR页面上的检查状态。

## 最小实现路径（保底方案）

如果你找不到合适的Bug可以修，以下是"保底"方案：

```yaml
方案A: 改进错误信息（1-3天）
  找到项目中不清晰的错误信息 → 增加更有价值的诊断信息
  例如:
    原来: "NullPointerException"
    改为: "Column 'user_id' is null. Please check the input data for missing values."
  
  搜索方法: grep -r "throw new" | grep -v "test" | head -20

方案B: 文档完善（1-2天）
  找到官方文档中缺失的配置说明 → 补充
  找到代码注释中的拼写错误 → 修正
  
  搜索方法: 
    - 浏览项目的官方文档网站
    - 对比代码中的配置参数和文档中的说明

方案C: 测试补充（2-3天）
  找到覆盖率低的模块 → 补充测试用例
  查看JaCoCo报告或在GitHub上查看项目的CodeCov页面

方案D: 移除弃用代码（1-2天）
  找到标记为@Deprecated的代码 → 确认是否可以安全删除 → 删除
```
