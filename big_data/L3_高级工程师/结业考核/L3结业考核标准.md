# L3 结业考核标准

> **考核定位**：验证学员具备大数据高级工程师的技术深度——源码级理解、论文精读能力、调优攻坚能力和开源社区参与能力
>
> **考核周期**：第36周 | **总时长**：1天（上午实战 + 下午答辩） | **通过标准**：4项考核全部合格

|------|--------|------|------|------|----------|

## 一、考核1：源码阅读（现场问答，45分钟）

### 1.1 考核形式

学员提前15分钟随机抽取一段源码（约100-200行），准备后当面向评委：
1. 解释这段代码的功能和设计意图（5分钟）
2. 画流程图展示执行流程（5分钟）
3. 回答评委的追问（10分钟）

### 1.2 源码问答20题完整题库（含参考答案）

#### Spark Catalyst方向（8题）

```yaml
题目1: Dataset.select() 源码
  代码位置: sql/core/.../Dataset.scala, select() + selectUntyped()
  考点:
    - Column 如何转换为 NamedExpression
    - Project LogicalPlan 节点的构建过程
    - 为什么 select() 不立即执行（惰性求值）
  参考答案:
    select() 接收 Column* 参数 → 调用 .named 将 Column 转为 NamedExpression
    → 调用 selectUntyped() → withPlan(Project(expressions, logicalPlan))
    → 创建新 Dataset，携带新的 LogicalPlan（不可变设计）
    不立即执行是因为 Dataset 采用惰性求值，只有 Action（count/collect/show）
    才会触发 queryExecution 的懒加载和后续 Analysis→Optimization→Execution 流程

题目2: RuleExecutor.execute() 源码
  代码位置: sql/catalyst/.../rules/RuleExecutor.scala
  考点:
    - Batch 和 Rule 的关系
    - FixedPoint 策略的终止条件
    - Once 策略 vs FixedPoint 策略
  参考答案:
    execute() 按顺序遍历所有 Batch。每个 Batch 包含一组 Rule 和一个 Strategy。
    Once: 每个 Rule 只应用一次，按顺序执行
    FixedPoint(maxIterations): 反复应用所有 Rule 直到树不再变化（通过 fastEquals 比较），
    或达到 maxIterations 上限。终止条件：所有 Rule 连续一轮都不改变树。
    Batch 之间的顺序很重要——前一个 Batch 的输出是后一个 Batch 的输入。

题目3: PushDownPredicate 源码
  代码位置: sql/catalyst/.../optimizer/PushDownPredicates.scala
  考点:
    - Filter + Join 的拆分规则：哪些条件可下推、哪些不行
    - 下推后的 LogicalPlan 树变化
    - 为什么引用两表列的条件不能下推
  参考答案:
    PushDownPredicate 使用 transform 递归遍历 LogicalPlan 树。
    遇到 Filter + Join 组合时，将 Filter 的 condition 按 AND 拆分(splitConjunctivePredicates)。
    只引用左表列的条件 → 下推到左表；只引用右表列的条件 → 下推到右表；
    同时引用两表列的条件 → 保留在 Join 之后（因为需要在 Join 完成两表数据关联后才能求值）。

题目4: BroadcastHashJoinExec.doExecute() 源码
  代码位置: sql/core/.../joins/BroadcastHashJoinExec.scala
  考点:
    - Broadcast 变量在 Driver 端 collect 和广播的过程
    - HashedRelation 的构建和使用（O(1) Probe）
    - 与 SortMergeJoin 的本质区别（无 Shuffle）
  参考答案:
    doExecute() 流程: ① 确定 buildSide（被广播侧）和 streamSide（流式扫描侧）
    ② buildPlan.execute().collect() → 在 Driver 收集 build 侧所有数据
    ③ 构建 HashedRelation（以 Join Key 为键的 HashMap）
    ④ sc.broadcast(hashedRelation) → 广播到每个 Executor
    ⑤ streamPlan 在每个 Partition 内 probe HashedRelation → 返回 Join 结果
    优势: 不需要 Shuffle；劣势: build 侧必须全部装入 Driver 和每个 Executor 的内存

题目5: WholeStageCodegenExec 源码
  代码位置: sql/core/.../WholeStageCodegenExec.scala
  考点:
    - 代码生成 vs Volcano Iterator Model 的差异
    - Janino 编译器的作用
    - 生成代码中如何消除虚函数调用
  参考答案:
    WholeStageCodegenExec 将多个物理算子（如 Filter + Project + Aggregate）
    合并为一个"Whole Stage"，生成一个大的 Java while 循环。
    Volcano Model 每个算子都是一个迭代器，每次 next() 都是虚函数调用。
    代码生成将算子逻辑内联到一个循环中，消除虚函数调用和中间对象分配。
    Janino 将生成的 Java 源代码字符串编译为字节码并加载。

题目6: HashAggregateExec 源码
  代码位置: sql/core/.../aggregate/HashAggregateExec.scala
  考点:
    - Partial-Final 两阶段聚合的实现
    - Spill 机制的触发条件和处理
    - TungstenAggregationIterator 的内存管理
  参考答案:
    HashAggregateExec 支持 Partial(预聚合) 和 Final(最终聚合) 两阶段。
    Partial 阶段在每个 Map Task 上做本地聚合，输出部分聚合结果。
    Final 阶段在 Reduce Task 上合并 Partial 结果。
    当内存不足时触发 Spill：将 HashMap 的一部分溢写到磁盘，
    通过 FallbackIterator 在磁盘和多段内存 HashMap 之间合并。

题目7: QueryExecution 源码
  代码位置: sql/core/.../execution/QueryExecution.scala
  考点:
    - logical → analyzed → optimizedPlan → sparkPlan → executedPlan 的转换触发时机
    - 懒执行机制的实现（lazy val）
    - explain() 如何展示各阶段计划
  参考答案:
    QueryExecution 的核心阶段都是 lazy val：logical, analyzed, optimizedPlan,
    sparkPlan, executedPlan。第一个 Action 触发 executedPlan 的计算 → 连锁触发
    前面所有 lazy val。每个阶段的转换：Analyzer(RuleExecutor) → Optimizer(RuleExecutor)
    → SparkPlanner(Strategy 模式匹配) → preparations(CollapseCodegenStages 等)。

题目8: AdaptiveSparkPlanExec 源码
  代码位置: sql/core/.../adaptive/AdaptiveSparkPlanExec.scala
  考点:
    - AQE 的三大优化及其触发时机
    - 运行时如何收集统计信息
    - 动态切换 Join 策略的实现
  参考答案:
    AdaptiveSparkPlanExec 在运行时检测实际数据特征并动态调整执行计划。
    三大优化: ① OptimizeShuffleWithLocalRead(合并小分区) ② OptimizeJoinStrategy(切换Join策略)
    ③ OptimizeSkewedJoin(处理数据倾斜)。触发时机：每个 Stage 的 Shuffle 完成后，
    根据 MapOutputStatistics 决定是否优化。通过收集实际的 Shuffle 数据大小、
    分区分布等信息来替代编译时的估算。
```

#### Flink方向（6题）

```yaml
题目9: CheckpointCoordinator.triggerCheckpoint() 源码
  代码位置: flink-runtime/.../checkpoint/CheckpointCoordinator.java
  考点:
    - Checkpoint 的触发时机和条件（并发限制、最小间隔）
    - PendingCheckpoint 的生命周期管理
    - 与 Savepoint 的区别（触发方式、生命周期、存储位置）
  参考答案:
    triggerCheckpoint() 首先检查 pendingCheckpoints 数量是否超过 
    maxConcurrentCheckpointAttempts（默认1），超过则拒绝触发。
    然后生成递增的 checkpointId，创建 PendingCheckpoint 并注册到 pendingCheckpoints Map。
    通过 RPC 向所有 Source Task 注入 Checkpoint Barrier。启动超时定时器。
    Savepoint: 手动触发，长期保留，需要考虑格式兼容性；
    Checkpoint: 定时触发，故障恢复后可丢弃。

题目10: CheckpointBarrierAligner.processBarrier() 源码
  代码位置: flink-runtime/.../checkpoint/CheckpointBarrierAligner.java
  考点:
    - Barrier 对齐的通道阻塞机制（blockChannel + bufferStorage）
    - 对齐完成后的 notifyCheckpoint → snapshotState 触发
    - 阻塞期间数据缓存的实现（BufferStorage）
  参考答案:
    processBarrier() 维护 numBarriersReceived 计数器。
    第一个 Barrier: 阻塞该通道(barrierChannelStates[channel].setBlocked(true))，
    后续数据通过 processNonBarrier() 缓存到 bufferStorage。
    中间 Barrier: 递增计数器，阻塞对应通道。
    最后一个 Barrier: ① notifyCheckpoint() → 触发各算子的 snapshotState()
    ② 释放所有阻塞通道(setBlocked(null)) ③ bufferStorage.resumeConsumption() 释放缓存数据
    ④ 向下游转发 Barrier ⑤ 重置计数器。

题目11: HashMapStateBackend 的 snapshotState() 源码
  代码位置: flink-runtime/.../state/HashMapStateBackend.java
  考点:
    - KeyedState 的存储结构（内存中的 HashMap + CopyOnWriteStateTable）
    - 异步 Snapshot 的实现（使用异步线程池，不阻塞数据处理）
    - 与 RocksDB StateBackend 的序列化差异（Java序列化 vs 字节数组）
  参考答案:
    HashMapStateBackend 将 State 保存在 TaskManager 的堆内存中。
    snapshotState() 时对当前 State 做深拷贝 → 提交到异步线程池进行序列化和存储。
    CopyOnWriteStateTable 确保快照时不需要锁——快照读旧版本，新写入写新版本。
    适合 State 较小（<100MB）、需要低延迟访问的场景。

题目12: RocksIncrementalSnapshotStrategy 源码
  代码位置: flink-statebackend-rocksdb/.../snapshot/RocksIncrementalSnapshotStrategy.java
  考点:
    - 增量 Checkpoint 的核心原理（对比上次 Checkpoint，只上传新增的 SST 文件）
    - SharedStateRegistry 的引用计数管理
    - 什么情况下退化为全量 Checkpoint
  参考答案:
    流程: ① rocksDB.flush() 确保所有 MemTable 被刷到 SST 文件
    ② 获取当前 SST 文件列表(current) ③ 与上次 Checkpoint 的 SST 列表(last) 对比
    ④ newSstFiles = current - last，只上传新增的 SST 文件
    ⑤ 注册共享状态到 SharedStateRegistry，增加引用计数
    ⑥ 共享的 SST 文件只保存文件 Handle（不重复上传）。
    退化场景：Task 重启（RocksDB 被重新打开）、Major Compaction 替换了所有 SST。

题目13: TwoPhaseCommitSinkFunction 源码
  代码位置: flink-streaming-java/.../sink/TwoPhaseCommitSinkFunction.java
  考点:
    - beginTransaction / preCommit / commit / abort 的时序和调用时机
    - 恢复时如何决定 Commit 还是 Abort（从 Operator State 读取事务 ID）
    - 与 Kafka 事务的集成（KafkaProducer.initTransactions + commitTransaction）
  参考答案:
    invoke() 通过 currentTransaction 写入每条数据。
    snapshotState() 时: ① preCommit(当前事务) → Kafka producer.flush()
    ② 将当前事务移到 pendingCommitTransactions ③ 开启新事务
    ④ 保存事务 ID 到 Operator State。
    notifyCheckpointComplete(): 从 pendingCommitTransactions 中找到已完成的 Checkpoint
    对应的事务 → commit() → Kafka producer.commitTransaction()。
    notifyCheckpointAborted(): abort 对应的事务 → Kafka producer.abortTransaction()。

题目14: CheckpointBarrierUnaligner 源码
  代码位置: flink-runtime/.../checkpoint/CheckpointBarrierUnaligner.java
  考点:
    - 不对齐模式与对齐模式的核心差异（不阻塞 + Channel State + 更快的 Barrier 传播）
    - Channel State 的构成（记录每个通道在 Barrier 到达前后的 in-flight 数据）
    - 为什么不对齐的 State 更大
  参考答案:
    Unaligner 收到第一个 Barrier 时不阻塞任何通道，Barrier 立即向下游转发。
    同时开始追踪每个通道的 Channel State（第一个 Barrier 和最后一个 Barrier 之间到达的数据）。
    Snapshot = 算子 State + 所有通道的 Channel State。
    恢复时：先恢复算子 State，然后重放 Channel State 中的数据。
    State 更大是因为包含了 in-flight 的数据（而 Aligner 中这些数据被缓存在通道中，不在 State 里）。
```

#### Kafka方向（6题）

```yaml
题目15: KafkaProducer.doSend() 源码
  代码位置: clients/.../KafkaProducer.java
  考点:
    - 拦截器→序列化→分区→Accumulator 的完整流程
    - 异步发送模型的优势（sender 线程批量发送 + Callback 机制）
    - RecordAccumulator 内存管理（BufferPool + batch.size + linger.ms）
  参考答案:
    doSend() 流程: ① 拦截器链处理(onSend) ② 序列化 Key 和 Value
    ③ 计算分区(Partitioner.partition()) ④ 估算消息大小
    ⑤ 追加到 RecordAccumulator（按 topic-partition 分组的 ProducerBatch）
    ⑥ 如果 batch 满了或创建了新的 batch → 唤醒 sender 线程发送。
    异步模型: doSend 不等待 Broker 确认，而是注册 Callback。
    sender 线程批量从 RecordAccumulator 取数据发送，最大化吞吐。

题目16: RecordAccumulator.append() 源码
  代码位置: clients/.../RecordAccumulator.java
  考点:
    - ProducerBatch 的管理（Deque<ProducerBatch> per TopicPartition）
    - BufferPool 内存管理（free + waiting 双队列，条件变量等待）
    - 批量发送的触发条件（batch.size 满了 或 linger.ms 到了）
  参考答案:
    append() 尝试将消息追加到对应 TopicPartition 的最后一个未满 ProducerBatch。
    如果满了或不存在 → 尝试从 BufferPool 分配新 batch → 如无可用内存则等待（max.block.ms）。
    BufferPool 使用 free（可用内存块）和 waiters（等待者条件变量）管理内存。
    每创建一个新 batch 或 batch 满了 → 触发 sender 线程的唤醒信号。

题目17: Log.appendAsLeader() 源码
  代码位置: storage/.../Log.scala
  考点:
    - Segment Roll 的条件（segment.bytes 满了 或 segment.ms 到了）
    - Offset 递增检查（拒绝重复或乱序的 Offset）
    - Leader 写入和 Follower 写入的不同路径
  参考答案:
    appendAsLeader() ① 检查 offset 是否合法（必须等于当前 LEO） ② 追加到当前活跃 Segment
    ③ 如果 segment 大小超过 segment.bytes 或时间超过 segment.ms → Roll（创建新 Segment）
    ④ 返回 LogAppendInfo（包含 firstOffset, lastOffset, 写入位置等）。
    Leader 路径: 直接写入本地 Segment + 通知 ReplicaManager 复制到 Follower。
    Follower 路径: 从 Leader Fetch 数据后通过 appendAsFollower() 写入。

题目18: LogSegment 数据结构源码
  代码位置: storage/.../LogSegment.scala
  考点:
    - 稀疏索引的作用和实现（偏移索引: offset → position，每写入 N 字节记录一条）
    - 时间索引的实现（时间戳 → offset，用于按时间查询）
    - Segment 的 log/index/timeindex 三个文件的生命周期
  参考答案:
    LogSegment 包含: ① 数据文件(.log): 消息的实际数据
    ② 偏移索引(.index): offset → 文件物理位置，稀疏索引（约每4KB记录一条）
    ③ 时间索引(.timeindex): timestamp → offset，用于时间戳查找。
    查询流程: 给定 offset → 二分查找 .index → 获取近似物理位置 → 顺序扫描 .log 到目标 offset。
    Segment 删除: 当该 Segment 的所有消息都超过 retention 时间或大小限制时删除。

题目19: ReplicaManager.appendRecords() 源码
  代码位置: server/.../ReplicaManager.scala
  考点:
    - ISR 中的副本同步（向每个 ISR Follower 发送 Fetch 请求）
    - LEO 和 HW 的推进（HW = min(Leader LEO, 所有 ISR Follower 的 LEO)）
    - acks 配置影响（acks=all 必须等所有 ISR 确认）
  参考答案:
    appendRecords() ① 调用 Log.appendAsLeader() 写入本地 ② 根据 acks 配置决定何时响应
    acks=0: 不等待 → 立即返回；acks=1: 等待 Leader 写入成功 → 返回；
    acks=all(-1): 等待所有 ISR 副本确认 → 启动 DelayedProduce 等待 Follower 同步
    ③ 推进 HW: 取所有 ISR Follower 的 LEO 的最小值作为新的 HW。
    Follower 确认通过 Fetch 请求的响应隐式完成（Follower 告知自己的 LEO）。

题目20: KafkaApis.handleProduceRequest() 源码
  代码位置: server/.../KafkaApis.scala
  考点:
    - Reactor 模式的请求处理（RequestChannel → KafkaApis → 分发到具体 handler）
    - 零拷贝在响应中的应用（FileRecords.writeTo() → FileChannel.transferTo() → sendfile）
    - 请求配额和限流机制
  参考答案:
    handleProduceRequest() 从请求中解析 ProduceRequest → 验证事务 ID、Topic 权限
    → 调用 ReplicaManager.appendRecords() → 构建 ProduceResponse。
    Reactor 模式: Acceptor 接收连接 → Processor 读取请求放入 RequestChannel → 
    KafkaRequestHandler 从 RequestChannel 取请求 → 调用 KafkaApis 处理。
    消费者 Fetch 响应的零拷贝: FileRecords.writeTo() 使用 FileChannel.transferTo()
    将磁盘数据直接通过 sendfile 系统调用发送到 Socket。
```

## 二、考核2：论文复述（PPT答辩，30分钟）

### 2.1 考核形式

学员从L3论文清单中选择1篇论文，准备PPT（不超过10页），进行：
1. 论文核心内容讲解（15分钟）：解决了什么问题、核心设计是什么、实验如何验证
2. 评委提问（10分钟）：论文的局限、与实际技术的映射、批判性思考
3. 答辩总结（5分钟）

### 2.2 PPT内容要求

```yaml
PPT结构（10页以内）:
  Page 1: 标题页
    - 论文标题、作者、发表会议/年份
    - 你的名字

  Page 2: 问题背景（Motivation）
    - 这篇论文要解决什么问题？
    - 之前的方法有什么不足？
    - 为什么要关注这个问题？

  Page 3-5: 核心设计（至少3个关键设计）
    - 每个设计：做法 + 为什么这样做 + 效果
    - 配合图表说明（不能只有文字）

  Page 6: 实验评估
    - 最关键的实验结果
    - 实验结果说明了什么？
    - 实验设置的公平性评估

  Page 7: 与工业实践的映射
    - 这个设计在哪个开源系统中被使用？
    - 开源实现和论文有什么差异？
    - 为什么有这些差异？

  Page 8: 批判性思考
    - 论文的假设在什么场景下不成立？
    - 论文没有回答的问题
    - 如果你来重新设计，会怎么做？

  Page 9: 总结与启发
    - 3个核心Takeaway
    - 对你的实际工作有什么指导意义？

  Page 10: 参考资料
```

### 2.3 论文答辩评审表

## 三、考核3：性能调优（实战演示，60分钟）

### 3.1 考核形式

学员面对一个故意构造的"烂任务"，在60分钟内完成调优：
1. 快速诊断（10分钟）：找出性能瓶颈
2. 实施优化（30分钟）：按优先级依次优化
3. 效果展示（10分钟）：展示优化前后的数据对比
4. 原理解释（10分钟）：解释每个优化为什么有效

### 3.2 考核任务

```yaml
原始任务特征:
  数据规模: 约500GB（模拟数据，200+列宽表）
  问题清单:
    - 严重数据倾斜（某Key占60%数据量）
    - Shuffle分区数=200（默认，每个Task仅处理2.5GB/200≈12.5MB，调度开销大）
    - 至少1个Python UDF逐行处理（Row-at-a-time，序列化开销极大）
    - 没有合理的Cache策略（同一数据集被多次重复扫描）
    - Executor内存配置不合理（executor-memory=2g，GC频繁Full GC）
    - 输入为TextFile格式（无压缩、无列式存储优化）
    - 额外的: spark.sql.shuffle.partitions=200, spark.sql.autoBroadcastJoinThreshold=10m (默认)

调优目标:
  - 运行时间从≥30分钟 → <3分钟（10倍提升）
  - 成功率从<50% → 100%（解决OOM问题）
  - 能够合理解释每个优化的原理

可用资源:
  - 5个Executor（4核16GB）
  - Spark 3.x 集群
```
