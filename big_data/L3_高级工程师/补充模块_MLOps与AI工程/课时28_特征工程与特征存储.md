## 二、特征工程挑战

### 2.1 训练-服务偏差（Training-Serving Skew）

```
训练-服务偏差: 训练时和推理时使用的特征计算逻辑不一致

  ┌──────────────────────────────────────────────────────────┐
  │  训练时 (离线):                                           │
  │                                                          │
  │  Python脚本计算特征:                                      │
  │    user_avg_spend_7d = spark.sql("""                     │
  │      SELECT user_id, AVG(amount) as avg_spend            │
  │      FROM orders                                         │
  │      WHERE dt >= date_sub(current_date(), 7)             │
  │      GROUP BY user_id                                    │
  │    """)                                                  │
  │                                                          │
  │  问题:                                                    │
  │  1. 时间窗口: "最近7天"在训练时和推理时的含义不同           │
  │  2. 数据延迟: 离线数据T+1，在线数据实时                    │
  │  3. 聚合逻辑: SQL逻辑和Python逻辑可能有细微差异            │
  │  4. 缺失值处理: 训练时填充0，推理时填充NULL                │
  └──────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────────────┐
  │  推理时 (在线):                                           │
  │                                                          │
  │  Java服务计算特征:                                        │
  │    double avgSpend = orderService                        │
  │      .getRecentOrders(userId, 7)                         │
  │      .stream()                                           │
  │      .mapToDouble(Order::getAmount)                      │
  │      .average()                                          │
  │      .orElse(0.0);                                       │
  │                                                          │
  │  偏差来源:                                                │
  │  ✗ "7天"的定义: SQL用日期函数，Java用时间戳计算            │
  │  ✗ 数据范围: SQL查Hive全量，Java查MySQL增量               │
  │  ✗ 缺失值: SQL返回NULL，Java返回0.0                       │
  │  ✗ 精度: SQL用DECIMAL，Java用double                       │
  └──────────────────────────────────────────────────────────┘

  Feature Store的解决方案:
  ┌──────────────────────────────────────────────────────────┐
  │  训练时: Feature Store.get_historical_features()         │
  │  推理时: Feature Store.get_online_features()             │
  │                                                          │
  │  同一特征定义 → 同一计算逻辑 → 同一数据源 → 一致性保证    │
  └──────────────────────────────────────────────────────────┘
```

### 2.2 特征一致性

```
特征一致性问题的三种表现:

  问题1: 特征重复开发
  ┌──────────────────────────────────────────────────────────┐
  │  团队A: 用户7日平均消费额 (Python + Spark)                │
  │  团队B: 用户7日平均消费额 (Java + Flink)                  │
  │  团队C: 用户7日平均消费额 (SQL + Hive)                    │
  │                                                          │
  │  → 三份代码、三种逻辑、三个结果                            │
  │  → Feature Store: 一份定义，多处复用                      │
  └──────────────────────────────────────────────────────────┘

  问题2: 特征不可发现
  ┌──────────────────────────────────────────────────────────┐
  │  数据科学家: "有没有用户消费相关的特征？"                   │
  │  → 需要问人、翻代码、查Wiki                               │
  │  → Feature Store: 特征注册表，可搜索、可发现              │
  └──────────────────────────────────────────────────────────┘

  问题3: 特征不可追溯
  ┌──────────────────────────────────────────────────────────┐
  │  "这个特征是怎么算出来的？"                                │
  │  "这个特征的数据来源是什么？"                               │
  │  "谁在用这个特征？"                                       │
  │  → Feature Store: 特征血缘、特征文档、使用追踪            │
  └──────────────────────────────────────────────────────────┘
```

```
  │  │                   │    │                           │ │
  │  │  用途: 训练数据集  │    │  用途: 实时推理            │ │
  │  │  延迟: 分钟级      │    │  延迟: 毫秒级             │ │
  │  │  数据量: TB级      │    │  数据量: GB级              │ │
  │  │  成本: 低          │    │  成本: 中                  │ │
  │  └───────────────────┘    └───────────────────────────┘ │
  │           ↑                          ↑                   │
  │           │                          │                   │
  │     物化(Materialize)          物化(Materialize)         │
  │           │                          │                   │
  │  ┌────────┴────────┐    ┌───────────┴───────────────┐  │
  │  │  批处理Pipeline  │    │  实时计算Pipeline          │  │
  │  │  (Spark/Airflow) │    │  (Flink/Kafka)            │  │
  │  └─────────────────┘    └───────────────────────────┘  │
  └──────────────────────────────────────────────────────────┘
```

### 3.2 离线存储 vs 在线存储

| 维度 | 离线存储 | 在线存储 |
|------|---------|---------|
| 存储介质 | S3/Hive/Iceberg/BigQuery | Redis/DynamoDB/Cassandra |
| 数据延迟 | 分钟~小时级 | 毫秒级 |
| 查询方式 | SQL/Batch Scan | Key-Value Get |
| 数据量 | TB~PB级 | GB~TB级 |
| 成本 | 低（对象存储） | 中~高（内存/SSD） |
| 使用场景 | 模型训练、批量预测 | 实时推理、在线推荐 |
| 更新频率 | 每日/每小时 | 秒级/分钟级 |
| 一致性 | 最终一致 | 强一致 |

## 四、Feast实战

### 4.1 Feast核心概念

```
Feast 核心概念:

  Entity (实体): 特征的主键
    - user_id: 用户实体
    - item_id: 商品实体
    - order_id: 订单实体

  Feature View (特征视图): 一组相关特征的集合
    - 关联一个Entity
    - 定义特征名称和类型
    - 指定数据源(Source)
    - 可配置TTL(在线存储过期时间)

  Feature Service (特征服务): 多个Feature View的组合
    - 为特定模型/应用组合所需特征
    - 一次调用获取所有特征

  Data Source (数据源): 特征的原始数据
    - 批数据源: Hive/BigQuery/File
    - 流数据源: Kafka/Kinesis

  Materialization (物化): 将特征从离线存储同步到在线存储
    - 定时物化: 每小时/每日
    - 增量物化: 只同步变更数据
```

## 五、实时特征计算

### 5.1 Flink SQL计算实时特征

```sql
CREATE TABLE kafka_orders (
    order_id BIGINT,
    user_id BIGINT,
    amount DECIMAL(12, 2),
    order_status STRING,
    category STRING,
    create_time TIMESTAMP(3),
    WATERMARK FOR create_time AS create_time - INTERVAL '5' SECOND,
    PRIMARY KEY (order_id) NOT ENFORCED
) WITH (
    'connector' = 'upsert-kafka',
    'topic' = 'orders',
    'properties.bootstrap.servers' = 'localhost:9092',
    'key.format' = 'json',
    'value.format' = 'json'
);

CREATE TABLE kafka_user_actions (
    user_id BIGINT,
    action_type STRING,
    page_id STRING,
    item_id STRING,
    action_time TIMESTAMP(3),
    WATERMARK FOR action_time AS action_time - INTERVAL '5' SECOND
) WITH (
    'connector' = 'kafka',
    'topic' = 'user_actions',
    'properties.bootstrap.servers' = 'localhost:9092',
    'format' = 'json',
    'scan.startup.mode' = 'latest-offset'
);

CREATE TABLE realtime_user_transaction (
    user_id BIGINT,
    order_count_1h INT,
    total_amount_1h DECIMAL(12, 2),
    avg_amount_1h DECIMAL(12, 2),
    distinct_category_1h INT,
    window_end TIMESTAMP(3),
    PRIMARY KEY (user_id) NOT ENFORCED
) WITH (
    'connector' = 'jdbc',
    'url' = 'jdbc:mysql://localhost:3306/feast_online',
    'table-name' = 'realtime_user_transaction',
    'username' = 'root',
    'password' = 'root'
);

INSERT INTO realtime_user_transaction
SELECT
    user_id,
    COUNT(*) AS order_count_1h,
    SUM(amount) AS total_amount_1h,
    AVG(amount) AS avg_amount_1h,
    COUNT(DISTINCT category) AS distinct_category_1h,
    TUMBLE_END(create_time, INTERVAL '1' HOUR) AS window_end
FROM kafka_orders
GROUP BY
    user_id,
    TUMBLE(create_time, INTERVAL '1' HOUR);
```

### 5.2 Flink SQL写入Feast在线存储

```sql
CREATE TABLE feast_online_features (
    user_id BIGINT,
    order_count_1h INT,
    total_amount_1h DECIMAL(12, 2),
    avg_amount_1h DECIMAL(12, 2),
    distinct_category_1h INT,
    event_timestamp TIMESTAMP(3),
    PRIMARY KEY (user_id) NOT ENFORCED
) WITH (
    'connector' = 'redis',
    'redis.mode' = 'single',
    'database' = '0',
    'hostname' = 'localhost',
    'port' = '6379',
    'key.format' = 'json',
    'value.format' = 'json',
    'key.fields' = 'user_id',
    'command' = 'SET'
);

INSERT INTO feast_online_features
SELECT
    user_id,
    COUNT(*) AS order_count_1h,
    SUM(amount) AS total_amount_1h,
    AVG(amount) AS avg_amount_1h,
    COUNT(DISTINCT category) AS distinct_category_1h,
    CURRENT_TIMESTAMP AS event_timestamp
FROM kafka_orders
WHERE create_time >= CURRENT_TIMESTAMP - INTERVAL '1' HOUR
GROUP BY user_id;
```

### 6.3 特征质量监控

```python
import pandas as pd
import numpy as np
from datetime import datetime

def check_feature_quality(df, feature_name, rules):
    results = []
    value = df[feature_name]

    for rule_name, rule_config in rules.items():
        if rule_name == "not_null":
            null_rate = value.isnull().mean()
            passed = null_rate <= rule_config.get("max_null_rate", 0.01)
            results.append({
                "feature": feature_name,
                "rule": rule_name,
                "passed": passed,
                "value": f"null_rate={null_rate:.4f}",
                "threshold": f"max_null_rate={rule_config.get('max_null_rate', 0.01)}",
            })

        elif rule_name == "range":
            min_val = rule_config.get("min", float("-inf"))
            max_val = rule_config.get("max", float("inf"))
            out_of_range = ((value < min_val) | (value > max_val)).mean()
            passed = out_of_range <= rule_config.get("max_outlier_rate", 0.01)
            results.append({
                "feature": feature_name,
                "rule": rule_name,
                "passed": passed,
                "value": f"out_of_range_rate={out_of_range:.4f}",
                "threshold": f"range=[{min_val},{max_val}]",
            })

        elif rule_name == "unique":
            dup_rate = value.duplicated().mean()
            passed = dup_rate <= rule_config.get("max_dup_rate", 0.01)
            results.append({
                "feature": feature_name,
                "rule": rule_name,
                "passed": passed,
                "value": f"dup_rate={dup_rate:.4f}",
                "threshold": f"max_dup_rate={rule_config.get('max_dup_rate', 0.01)}",
            })

    return results

feature_rules = {
    "age": {
        "not_null": {"max_null_rate": 0.01},
        "range": {"min": 0, "max": 150, "max_outlier_rate": 0.01},
    },
    "order_count_7d": {
        "not_null": {"max_null_rate": 0.005},
        "range": {"min": 0, "max": 10000, "max_outlier_rate": 0.01},
    },
    "total_amount_7d": {
        "not_null": {"max_null_rate": 0.005},
        "range": {"min": 0, "max": 5000000, "max_outlier_rate": 0.01},
    },
}

df = pd.read_parquet("data/user_transaction_stats.parquet")

all_results = []
for feature_name, rules in feature_rules.items():
    if feature_name in df.columns:
        all_results.extend(check_feature_quality(df, feature_name, rules))

results_df = pd.DataFrame(all_results)
print(results_df.to_string(index=False))

failed = results_df[~results_df["passed"]]
if len(failed) > 0:
    print(f"\n⚠️ {len(failed)}项质量检查未通过!")
else:
    print(f"\n✅ 所有{len(results_df)}项质量检查通过!")
```

```python
from feast import FeatureStore
import time
import json

store = FeatureStore(repo_path=".")

def get_features_for_prediction(user_id):
    features = store.get_online_features(
        features=[
            "user_profile_features:age",
            "user_profile_features:gender",
            "user_profile_features:user_level",
            "user_transaction_stats_7d:order_count_7d",
            "user_transaction_stats_7d:total_amount_7d",
            "user_transaction_stats_7d:refund_count_7d",

test_user_ids = [1, 10, 50, 100, 500, 1000, 5000]

latencies = []
for uid in test_user_ids:
    start = time.time()
    features = get_features_for_prediction(uid)
    elapsed = (time.time() - start) * 1000
    latencies.append(elapsed)

avg_latency = sum(latencies) / len(latencies)
p99_latency = sorted(latencies)[int(len(latencies) * 0.99)]
print(f"\n在线特征服务延迟统计:")
```
