  一个数据分析任务需要30分钟才能完成，日志显示:
  - Spark UI中某个Task的Shuffle Read是其他Task的100倍
  - 有几个Task的执行时间占了总时间的80%
  - 频繁Full GC

考核要求:
  1. (30分) 诊断问题: 分析给出的资料，判断性能瓶颈在哪里
  2. (40分) 给出解决方案: 说出至少3种解决方案及原理
  3. (30分) 对比分析: 说明每种方案的适用场景和局限性

回答要点:
  "我看到Stage-2中Task-15的Shuffle Read为500MB，而其他Task
   平均只有5MB。这说明category_id=100存在严重数据倾斜。
   
   解决方案:
   1. 加盐打散: 给category_id=100加随机后缀，分散到多个Task处理

## 十、HDFS原理笔试题（10道）

> 以下10题为HDFS原理专项测试，每题10分，总分100分。用于考察学员对HDFS架构、读写流程、高可用机制的掌握程度。

### 题目1：HDFS架构组件（10分）

**题目**：请列出HDFS架构中的核心组件，并简述每个组件的职责。

**标准答案**：
```
组件1: NameNode (3分)
  - 职责: 管理文件系统命名空间
  - 维护文件系统树和所有文件/目录的元数据信息
  - 维护Block与DataNode的映射关系
  - 处理客户端读写请求
  - 管理DataNode的心跳检测和Block副本状态

组件2: Secondary NameNode (2分)
  - 职责: 辅助NameNode合并FsImage和EditLog
  - 定期从NameNode获取EditLog，与FsImage合并生成新的FsImage
  - 注意: 不是NameNode的热备节点，不能接管NameNode服务

组件3: DataNode (3分)
  - 职责: 存储实际的数据块(Block)
  - 响应客户端与NameNode的读写请求
  - 定期向NameNode发送心跳(默认3秒)和Block报告(默认6小时)
  - 执行Block的创建、删除、复制等操作

组件4: Client (2分)
  - 职责: 通过HDFS API与NameNode和DataNode交互
  - 文件写入: 先与NameNode交互获取Block分配，再与DataNode交互写数据
  - 文件读取: 先与NameNode交互获取Block位置，再与DataNode交互读数据
```

**评分标准**：
| 得分项 | 分值 |
|--------|------|
| 正确列出4个核心组件 | 4分 |
| 准确描述NameNode职责(至少3条) | 2分 |
| 准确描述DataNode职责(至少2条) | 2分 |
| 说明Secondary NameNode与NameNode的关系 | 2分 |

### 题目2：HDFS写数据流程（10分）

**题目**：详细描述HDFS写入一个文件(例如200MB)的完整流程，标注每个步骤涉及的组件和通信方式(RPC/Streaming)。

**标准答案**：
```
步骤1: 客户端调用 FileSystem.create()，向NameNode发起RPC请求创建文件。(1分)
步骤2: NameNode检查:
  - 文件是否已存在
  - 客户端是否有创建权限
  - 通过检查后，在命名空间中创建新文件记录(但不分配Block)(1分)
步骤3: 客户端调用 FSDataOutputStream.write() 开始写入数据。(1分)
步骤4: 数据先写入客户端本地缓冲区，当缓冲区满时，向NameNode
       请求分配新的Block。(1分)
步骤5: NameNode返回一组DataNode地址(副本放置策略):
  - 第一个副本: 客户端所在节点(如果客户端在集群内)或随机节点
  - 第二个副本: 第一个副本所在机架之外的节点
  - 第三个副本: 第二个副本所在机架内的另一个节点
  通过RPC返回。(2分)
步骤6: 客户端与第一个DataNode建立Pipeline连接，
       开始以Packet为单位的流式传输。(1分)
步骤7: DataNode接收数据后:
  - 写入本地磁盘
  - 同时转发给Pipeline中的下一个DataNode
  - 形成流水线复制(第一个→第二个→第三个)(2分)
步骤8: 每个DataNode完成写入后向上游发送ACK确认。(1分)
```

**评分标准**：
| 得分项 | 分值 |
|--------|------|
| 描述出NameNode创建文件记录 | 2分 |
| 描述出Block分配和副本放置策略 | 3分 |
| 描述出Pipeline流水线写入 | 3分 |
| 描述出ACK确认机制 | 2分 |

### 题目3：HDFS读数据流程（10分）

**题目**：详细描述HDFS读取一个文件的完整流程。如果读取过程中某个DataNode故障，会发生什么？

**标准答案**：
```
正常读取流程:
步骤1: 客户端调用 FileSystem.open()，向NameNode发起RPC请求。(1分)
步骤2: NameNode查询元数据:
  - 返回文件Block列表
  - 每个Block的DataNode位置列表(按与客户端的"距离"排序)(2分)
步骤3: 客户端选择最近的DataNode，建立Socket连接读取第一个Block。(1分)
步骤4: 以Packet为单位流式读取数据，客户端边读边校验Checksum。(1分)
步骤5: 第一个Block读完，关闭连接，对下一个Block重复步骤3-4。(1分)
步骤6: 全部Block读取完毕，客户端调用 close()。(1分)

DataNode故障处理:
  - 如果正在读取的DataNode发生故障，客户端标记该节点不可用
  - 自动切换到Block副本列表中的下一个DataNode继续读取
  - 读取完成后，客户端向NameNode报告该DataNode故障
  - NameNode会为该Block在其他节点上补充副本(3分)
```

### 题目6：FsImage和EditLog（10分）

**题目**：解释HDFS中FsImage和EditLog的概念，为什么需要两者协同工作？Secondary NameNode的Checkpoint流程是怎样的？

**标准答案**：
```
FsImage (3分):
  - 文件系统元数据的完整快照
  - 包含: 整个文件系统命名空间、目录结构、文件到Block的映射
  - 存储在NameNode本地磁盘
  - 启动时，NameNode将FsImage加载到内存
  - 特点: 完整但更新不及时

EditLog (3分):
  - 记录自上次FsImage以来所有元数据变更的事务日志
  - 每次元数据修改(创建/删除/重命名等)都会追加一条记录
  - 特点: 实时但只记录增量

协同工作原因 (2分):
  - HDFS元数据变更非常频繁，无法每次变更都更新FsImage(IO开销太大)
  - 所以采用WAL(Write-Ahead Log)设计:
    先写EditLog(顺序写,快速) → 定期合并到FsImage
  - 重启时: NameNode加载FsImage + 重放EditLog = 完整元数据

Checkpoint流程 (2分):
  1. Secondary NameNode通知NameNode开始Checkpoint
  2. NameNode停止写入当前EditLog，创建新的EditLog文件
  3. Secondary NameNode通过HTTP获取FsImage和旧的EditLog
  4. Secondary NameNode将EditLog合并到FsImage中
  5. Secondary NameNode将新的FsImage回传给NameNode
  6. NameNode用新的FsImage替换旧的

### 题目7：数据完整性校验（10分）

**题目**：HDFS如何保证数据不丢失和不损坏？请描述HDFS的数据完整性校验机制和数据恢复机制。

**标准答案**：
```
数据完整性校验(5分):

写入时校验:
  - 客户端在写入Block数据时，同时计算每个Packet的CRC32校验和
  - 校验和随数据一起发送给DataNode
  - DataNode存储数据Block和对应的校验和(在.meta文件中)
  - 写入完成后，DataNode验证校验和

读取时校验:
  - 客户端从DataNode读取数据时，也会读取对应的校验和
  - 客户端重新计算数据的校验和，与读取的校验和比对
  - 如果不匹配，说明数据已损坏
  - 客户端从另一个DataNode读取该Block的副本

定期校验:
  - DataNode后台线程(BlockScanner)定期扫描所有Block
  - 验证存储的数据与校验和是否一致
  - 发现损坏的Block立即报告给NameNode

数据恢复机制(5分):

副本恢复:
  - NameNode发现某个Block的副本数少于预期(replication factor)
  - 触发副本复制: 选择一个DataNode，从有该Block的DataNode复制
  - 副本选择: 优先同机架、其次跨机架

损坏Block恢复:
  - DataNode报告Block损坏
  - NameNode将该Block标记为损坏(从Block映射中移除)
  - 如果副本数不足，触发复制
  - 新副本从未损坏的副本复制

集群恢复(降级模式):
  - 如果所有副本都损坏 → 数据永久丢失
  - 如果Network Topology导致无法满足放置策略 → 降级暂时接受
```

**评分标准**：
| 得分项 | 分值 |
|--------|------|
| 写入时校验机制 | 2分 |
| 读取时校验机制 | 2分 |
| BlockScanner定期校验 | 1分 |
| 副本数恢复机制 | 3分 |
| 损坏Block恢复 | 2分 |

  - 将多个小文件合并到一个Map Task处理

### 题目10：纠删码Erasure Coding（10分）

**题目**：HDFS 3.0引入了纠删码(Erasure Coding)作为副本机制的替代。请对比EC和副本机制的优缺点，说明EC的适用场景。

**标准答案**：
```
EC原理(3分):
  - 数据分片: 将一个Block分为k个数据片(Data Cell)
  - 编码: 计算m个校验片(Parity Cell)
  - 存储: k+m个片分布在不同DataNode上
  - 容错: 任意m个片丢失都可以从剩余的k个片恢复
  - 常用: RS-6-3(6个数据片+3个校验片)，存储开销为(k+m)/k = 1.5x
          RS-10-4(10个数据片+4个校验片)，存储开销为1.4x

对比分析(5分):
┌──────────────┬─────────────────────┬─────────────────────┐
│    维度       │   三副本(3x)        │   EC(RS-6-3, 1.5x)  │
├──────────────┼─────────────────────┼─────────────────────┤
│ 存储开销      │ 200%                │ 50%                 │
│ 容错能力      │ 任意2个副本损坏     │ 任意3个片损坏       │
│ CPU开销       │ 几乎无              │ 编码/解码开销大     │
│ 读写延迟      │ 低                  │ 重构时需要解码      │
│ 小文件        │ 合适                │ 不合适(编码开销大)  │
│ 热数据        │ 合适                │ 不太合适(频繁读写)  │
│ 冷数据/归档   │ 浪费                │ 非常适合            │
└──────────────┴─────────────────────┴─────────────────────┘

适用场景(2分):
  EC适合:
    - 冷数据/归档存储(极少读取)
    - 大文件(编码开销可摊销)
    - 存储成本敏感的场景
  
  EC不适合:
    - 热数据/频繁读写
    - 小文件
    - 延迟敏感型应用
```

**评分标准**：
| 得分项 | 分值 |
|--------|------|

    .appName("WritePartitioned") \

    .getOrCreate()

### 题目1：数据倾斜的定义与原因（16.7分）

**题目**：什么是数据倾斜？导致数据倾斜的根本原因是什么？请结合Spark的Shuffle机制说明。

**标准答案**：
```
定义:
  数据倾斜是指在分布式计算中，由于数据分布不均匀，导致某些Task处理的数据量
  远多于其他Task，造成少数Task成为性能瓶颈，整个作业的完成时间取决于最慢的Task。

根本原因:
  1. 业务数据天然分布不均(如热门商品/热门城市/头部用户)
  2. Shuffle时Key分布不均 → 某些Key数量极大

Shuffle机制视角:
  在Spark中，groupByKey/reduceByKey/join等操作会触发Shuffle:
  1. Map端: 将数据按Key的Hash写入不同分区
  2. Reduce端: 拉取对应分区的数据(Shuffle Read)
  
  如果一个Key(Hash值相同)的数据量极大:
  - 对应Reduce Task的Shuffle Read会远大于其他Task
  - 该Task执行时间远大于其他Task(可能是100倍)
  - 其他Task完成后等待该Task → 资源浪费
  - 该Task所在Executor可能出现OOM/Full GC

举例:
  聚类分析中，category_id="手机"的数据量是其他品类的100倍。
  reduceByKey时，"手机"对应的Task处理数据量=其他Task × 100，
  该Task需要10分钟，而其他Task只需6秒。
```

**评分标准**：

| 根本原因分析正确 | 4分 |

### 题目3：加盐打散方案（16.7分）

**题目**：如果groupByKey操作出现数据倾斜，请详细描述"加盐打散"的解决方案，并写出完整代码。

**标准答案**：
```python
"""
加盐打散: 将倾斜的Key通过加随机后缀进行"分裂"，分散到多个Task计算后再合并。

原理:
  原key "手机" → 100万条数据 → 1个Task处理
  加盐后:
    "手机_0" → 10万条 → Task 1
    "手机_1" → 10万条 → Task 2
    ...          ...
    "手机_9" → 10万条 → Task 10
  每个Task处理10万条 → 均匀分布 → 性能提升10倍

代价: 需要两次聚合(Map端预聚合 + 去盐饿终聚合)
"""

from pyspark.sql import SparkSession
from pyspark.sql.functions import *

spark = SparkSession.builder \
    .appName("SaltSolution") \
    .enableHiveSupport() \
    .getOrCreate()

# ===== 方案实现 =====

# Step 1: 识别倾斜Key(通过采样)
sample_df = spark.table("dwd_user_log") \
    .filter(col("dt") == "20240331") \
    .sample(0.01)

key_stats = sample_df.groupBy("category_id").count() \
    .orderBy(col("count").desc())
key_stats.show(20)

# 假设category_id=100 是最倾斜的Key

# Step 2: 加盐打散
NUM_SALTS = 10  # 打散到10个分区

salted = spark.table("dwd_user_log") \
    .filter(col("dt") == "20240331") \
    .withColumn("salt",
        when(col("category_id") == 100,
             (rand() * NUM_SALTS).cast("int"))
        .otherwise(lit(0))
    ) \
    .withColumn("salted_key",
        when(col("category_id") == 100,
             concat(col("category_id"), lit("_"), col("salt")))
        .otherwise(col("category_id").cast("string"))
    )

# Step 3: 第一次聚合(加盐后)
agg1 = salted.groupBy("salted_key").agg(
    count("*").alias("partial_cnt")
)

# Step 4: 去盐 + 第二次聚合
result = agg1 \
    .withColumn("real_key",
        when(col("salted_key").contains("_"),
             split(col("salted_key"), "_")[0])
        .otherwise(col("salted_key"))
    ) \
    .groupBy("real_key").agg(
        sum("partial_cnt").alias("total_cnt")
    ) \
    .orderBy(col("total_cnt").desc())

result.show(20)
```

### 题目4：Broadcast Join方案（16.7分）

**题目**：什么场景适合使用Broadcast Join解决数据倾斜？描述其原理并写出代码，指出使用限制。

**标准答案**：
```python
"""
Broadcast Join原理:
  将小表广播到每个Executor的内存中，大表直接与内存中的小表做Map-Side Join，
  完全避免Shuffle，从而消除Shuffle带来的数据倾斜。

适用场景:
  1. 一个小表(如维度表) + 一个大表(如事实表)的Join
  2. 小表可完全放入Executor内存(默认<10MB, 可通过spark.sql.autoBroadcastJoinThreshold调大)
  3. 小表不可变(不需要在Join过程中更新)

限制:
  1. 小表必须足够小(几百MB以内)，否则广播开销和内存压力大
  2. Driver端需要收集小表全部数据到内存
  3. 不适合不停增长的流式小表
"""

from pyspark.sql import SparkSession
from pyspark.sql.functions import *

spark = SparkSession.builder \
    .appName("BroadcastJoin") \
    .config("spark.sql.autoBroadcastJoinThreshold", "104857600")  # 100MB
    .enableHiveSupport() \
    .getOrCreate()

# 场景: 大表(用户行为) + 小表(品类维度)
big_table = spark.table("dwd_user_log").filter(col("dt") == "20240331")
small_table = spark.table("dim_product")  # 约5MB, 1000条记录

# 方式1: 自动广播(使用hint)
from pyspark.sql.functions import broadcast
result = big_table.join(broadcast(small_table), "product_id", "left")

# 方式2: SQL中的hint
result_sql = spark.sql("""
    SELECT /*+ BROADCAST(d) */ f.*, d.category_name
    FROM dwd_user_log f
    LEFT JOIN dim_product d ON f.product_id = d.product_id
    WHERE f.dt = '20240331'
""")

# 方式3: RDD API广播变量
# small_dict = {row.product_id: row.category_name
#               for row in small_table.collect()}
# bc_small = spark.sparkContext.broadcast(small_dict)

result.groupBy("category_name").count().orderBy(col("count").desc()).show(20)

# 验证: 查看Execution Plan, 确认使用了BroadcastHashJoin
print("Execution Plan:")
result.explain(extended=True)
# 查找 "BroadcastHashJoin" 关键字
```

### 题目5：AQE自适应查询执行（16.7分）

**题目**：Spark 3.0的AQE(Adaptive Query Execution)如何自动解决数据倾斜？它的三个核心优化是什么？

**标准答案**：
```
AQE简介:
  Adaptive Query Execution (AQE) 是Spark 3.0引入的运行时查询优化框架。
  它在Shuffle MapStage完成后，根据实际的中间结果统计信息，动态调整后续的执行计划。

三个核心优化:

1. 自动分区合并 (Coalesce Shuffle Partitions) — 5分
  问题: 人为设置的分区数(如spark.sql.shuffle.partitions=200)可能不合理
  AQE做法: 在Shuffle Write完成后，检查实际数据量
    - 如果多个分区数据量很小 → 自动合并为一个大分区
    - 减少小Task调度的开销
  配置:
    spark.sql.adaptive.enabled=true
    spark.sql.adaptive.coalescePartitions.enabled=true
    spark.sql.adaptive.coalescePartitions.minPartitionSize=1MB

2. 倾斜Join优化 (Optimize Skew Join) — 5分
  问题: SortMergeJoin中，某个Key的数据倾斜导致个别Task很慢
  AQE做法: 检测到倾斜分区后，自动拆分为多个子分区
    - 识别: 分区大小 > 中位数大小 × skew factor (默认5)
    - 拆分: 将倾斜分区拆成多个同等大小的子分区
    - 广播: 将小表的对应数据广播到每个子分区
  配置:
    spark.sql.adaptive.skewJoin.enabled=true
    spark.sql.adaptive.skewJoin.skewedPartitionFactor=5
    spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes=256MB

3. 动态切换Join策略 (Switch Join Strategy) — 5分
  问题: 编译时判断小表做BroadcastJoin，但实际运行时表变大了
  AQE做法: 在Shuffle完成后再决定Join策略
    - 如果实际的Shuffle数据量 < broadcast阈值 → 改为BroadcastHashJoin
    - 提升性能: BroadcastHashJoin > SortMergeJoin
  配置:
    spark.sql.adaptive.localShuffleReader.enabled=true

总结对比: (1.7分)
┌──────────────┬───────────────────┬──────────────────────┐
│    问题       │   传统方案         │   AQE自动方案        │
├──────────────┼───────────────────┼──────────────────────┤
│ 分区数不合理  │ 手动调整shuffle数  │ 自动合并小分区        │
│ 数据倾斜     │ 手动加盐打散       │ 自动拆分倾斜分区      │
│ Join策略不当 │ 手动hint修改代码   │ 运行时自动切换        │
└──────────────┴───────────────────┴──────────────────────┘

AQE的局限:
  - 只支持SortMergeJoin的倾斜优化
  - 需要额外的Shuffle开销来收集统计信息
  - 仅Spark 3.0+版本可用
```
