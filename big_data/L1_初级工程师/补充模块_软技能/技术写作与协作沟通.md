# 技术写作与协作沟通

> **课时时长**：2小时（Workshop形式）
> **教学方式**：讲授 + 实战演练 + 互评
> **适用阶段**：L1初级工程师
> **前置要求**：已完成L1项目4（离线数仓构建）或有实际项目经验

---

## 一、教学目标

1. **技术文档写作**：能独立编写技术方案文档、README、CHANGELOG，掌握代码注释规范
2. **Code Review沟通**：能给出有建设性的Review意见，能理性回应他人Review
3. **跨团队沟通**：能与产品经理、运维、数据分析师高效沟通
4. **技术分享**：能准备并完成10分钟/30分钟/60分钟的技术分享

**实战案例：为L1项目4（离线数仓构建）写技术方案**

```markdown
# 离线数仓构建技术方案

## 1. 背景

当前业务数据散落在MySQL各业务库中，数据分析团队每次取数都需要:
- 找DBA申请数据库权限（平均等待2天）
- 手写复杂SQL（平均耗时4小时）
- 导出Excel做图表（平均耗时2小时）

月均取数需求约50次，总耗时约400人时/月。

**目标**: 构建离线数仓，将取数自助化率提升到80%，单次取数时间从6小时降到30分钟。

## 2. 方案设计

### 整体架构

```
MySQL业务库 → Sqoop导入 → HDFS(ODS层) → Hive ETL(ODS→DWD→DWS→ADS) → MySQL结果表 → Superset可视化
```

### 分层设计

| 层级 | 名称 | 说明 | 示例 |
|------|------|------|------|
| ODS | 操作数据层 | 原始数据1:1同步 | ods_order_di |
| DWD | 明细数据层 | 清洗、标准化 | dwd_order_detail_di |
| DWS | 汇总数据层 | 按主题轻度汇总 | dws_user_order_1d |
| ADS | 应用数据层 | 面向业务指标 | ads_user_retention |

## 3. 方案对比

| 维度 | 方案A: Hive全栈 | 方案B: Hive+Spark |
|------|----------------|-------------------|
| 计算引擎 | Hive on MapReduce | Hive元数据 + Spark计算 |
| ETL性能 | 慢（MR磁盘密集） | 快3-5倍（内存计算） |
| 学习成本 | 低（SQL为主） | 中（需学Spark） |
| 运维复杂度 | 低 | 中 |
| 适用场景 | 数据量<1TB/天 | 数据量>1TB/天 |

## 4. 选择理由

选择方案A（Hive全栈）:
- 当前日数据量约200GB，Hive MR可满足时效要求
- 团队以SQL为主，学习成本低
- 后续数据量增长可平滑迁移到Spark

## 5. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| Sqoop导入失败 | ODS层数据缺失 | 增量导入+失败重试+数据校验 |
| ETL任务延迟 | 报表产出延迟 | 任务监控+关键路径优先调度 |
| 数据质量差 | 分析结果不准 | DQC规则+数据质量报告 |

## 6. 实施计划

| 阶段 | 内容 | 时间 |
|------|------|------|
| 第1周 | ODS层搭建+Sqoop导入 | 5天 |
| 第2周 | DWD层ETL开发 | 5天 |
| 第3周 | DWS+ADS层开发 | 5天 |
| 第4周 | 报表开发+测试+上线 | 5天 |

## 7. 验收标准

- 功能: 核心报表10张全部产出
- 性能: 全部ETL在8:00前完成
- 质量: 数据准确率>99.9%
```

#### 代码注释规范

```

  ✅ 写"为什么"（Why），不写"是什么"（What）

  ✅ 写不明显的约束和假设
  ✅ 写TODO/FIXME/HACK标记

什么时候不写注释:
  ❌ 代码本身已经清楚表达意图的
  ❌ 注释和代码说的是同一件事

### 2.2 Code Review沟通（30分钟）

#### 如何给出有建设性的Review意见

```
Review意见的四个层次:

Level 1 - 指出问题（最差）
  "这段代码写得不好"

Level 2 - 指出问题+原因（及格）
  "这个SQL没有索引，查询会很慢"

Level 3 - 指出问题+原因+建议（良好）
  "这个SQL没有索引，查询会很慢。
   建议给user_id字段加索引，或者改用分区表。"

Level 4 - 指出问题+原因+建议+权衡（最佳）
  "这个SQL没有索引，查询会很慢。
   建议给user_id字段加索引，或者改用分区表。
   加索引会稍微影响写入性能，但这个表是日分区表，
   写入只在ETL时段，影响可忽略。"


Review意见分类标签:

  [必须修改]  影响正确性/安全性/性能的关键问题
  [建议修改]  可以提升代码质量但不影响功能
  [讨论]      不确定是否有问题，想讨论一下
  [风格]      代码风格/命名规范问题
  [赞]        值得学习的写法


Review意见模板:

  [标签] 简要描述问题
  位置: 文件名:行号
  原因: 为什么这是一个问题
  建议: 具体的修改建议
  示例: (如果需要) 修改后的代码片段
```

**Review意见示例**

```
❌ 不好的Review:
  "这个变量名太烂了"
  "你不懂SQL优化吗？"
  "重写吧"

✅ 好的Review:

  [建议修改] 变量名不够描述性
  位置: etl.py:42
  原因: 变量名tmp无法表达其含义，后续维护时需要阅读上下文才能理解
  建议: 改为user_order_summary或daily_agg_result
  示例:
    # 修改前
    tmp = spark.sql("SELECT ...")
    # 修改后
    user_order_summary = spark.sql("SELECT ...")

  [必须修改] SQL注入风险
  位置: query_builder.py:28
  原因: 使用f-string拼接SQL，存在SQL注入风险
  建议: 使用参数化查询
  示例:
    # 修改前
    sql = f"SELECT * FROM users WHERE id = '{user_id}'"
    # 修改后
    sql = "SELECT * FROM users WHERE id = %s"
    cursor.execute(sql, (user_id,))

  [赞] 巧妙使用窗口函数
  位置: analytics.py:55
  这个留存率的计算用窗口函数实现得很优雅，学到了！
```

#### 如何回应Review意见

```
回应Review的三个原则:

1. 不防御: 不要把Review意见当作人身攻击
   ❌ "我这样写是有原因的"（然后不解释原因）
   ✅ "确实，我改一下" 或 "我的考虑是xxx，你觉得呢？"

2. 不妥协: 不要盲目接受所有意见
   ❌ "好的好的，都改"（没有思考就接受）
   ✅ "这个建议很好，我改一下" 或 "这里我有不同看法，原因是xxx"

3. 要闭环: 每条意见都要有明确的回应
   - 接受: "已修改，请查看"
   - 拒绝: "我的考虑是xxx，因为xxx，你觉得呢？"
   - 讨论: "好问题，我查了一下xxx"


回应模板:

  对[必须修改]类意见:
    → 必须修改，回复"已修改"并说明改法

  对[建议修改]类意见:
    → 评估后决定:
      接受: "好建议，已修改"
      拒绝: "感谢建议，这里保持现状，因为xxx"
      延后: "记在TODO里，下个迭代处理"

  对[讨论]类意见:
    → 给出你的思考，寻求共识
    "我理解你的顾虑，我的考虑是xxx，你觉得这样是否可行？"

  对[赞]类意见:
    → 简单感谢即可
    "谢谢！"
```

### 2.3 跨团队沟通（25分钟）

#### 与产品经理沟通：需求澄清的5个问题

```
接到需求后，先问这5个问题:

1. 业务背景是什么？
   "这个需求是为了解决什么业务问题？"
   → 理解Why，避免做错方向

2. 用户是谁？使用场景是什么？
   "谁会用这个功能？在什么场景下用？"
   → 理解Who和Where，影响技术方案

3. 成功标准是什么？
   "怎么衡量这个功能做成功了？"
   → 理解How much，量化验收标准

4. 有什么约束？
   "有没有时间要求？数据量级？兼容性要求？"
   → 理解Constraints，影响技术选型

5. 不做会怎样？
   "如果不做这个功能，最坏的情况是什么？"
   → 理解Priority，判断优先级


沟通模板——需求评审邮件:

  主题: [需求评审] xxx功能 - 需澄清事项

  xxx（产品经理）:

  关于xxx需求，以下事项需要澄清:

  1. 业务背景
     理解: xxx
     疑问: xxx

  2. 数据量级
     理解: 日均xxx条
     疑问: 峰值是多少？增长预期？

  3. 时效要求
     理解: T+1
     疑问: 是否有实时需求？

  4. 验收标准
     理解: xxx
     疑问: 数据准确率的基线是多少？

  请在周三前回复，以便评估开发周期。

  谢谢！
```

#### 与数据分析师沟通：数据口径对齐

```
数据口径不一致是数据团队最常见的冲突:

  工程师: "日活用户 = 当天登录的用户数"
  分析师: "日活用户 = 当天有任何操作的用户数"
  产品:   "日活用户 = 当天至少停留5分钟的用户数"

  → 三个数字完全不同！


数据口径对齐模板:

  | 指标名称 | 定义 | 计算公式 | 数据来源 | 统计周期 | 更新频率 |
  |---------|------|---------|---------|---------|---------|
  | DAU | 日活跃用户数 | COUNT(DISTINCT user_id) WHERE event_date = today | dwd_user_event_di | 自然日 | T+1 |
  | 新增用户 | 首次注册用户数 | COUNT(DISTINCT user_id) WHERE is_first_login = 1 | dwd_user_register_di | 自然日 | T+1 |
  | 留存率 | N日后仍活跃的比例 | 回访用户数 / 新增用户数 * 100% | dws_user_retention | 自然日 | T+1 |

对齐流程:
  1. 工程师提出指标定义（基于数据可行性）
  2. 分析师确认业务含义（基于业务需求）
  3. 产品确认验收标准（基于业务目标）
  4. 三方签字确认，写入指标字典
  5. 任何修改需走变更流程
```

### 练习2：模拟一次Code Review（10分钟）

两人一组，一人扮演Reviewer，一人扮演Author。

**待Review的代码：**

```python
def get_user_orders(user_id):
    conn = pymysql.connect(host="localhost", user="root", password="123456", database="shop")
    cursor = conn.cursor()
    sql = f"SELECT * FROM orders WHERE user_id = '{user_id}'"
    cursor.execute(sql)
    results = cursor.fetchall()
    conn.close()
    return results

def calculate_retention(date):
    conn = pymysql.connect(host="localhost", user="root", password="123456", database="shop")
    cursor = conn.cursor()
    sql = "SELECT user_id FROM orders WHERE date = '" + date + "'"
    cursor.execute(sql)
    today_users = set(row[0] for row in cursor.fetchall())
    sql2 = "SELECT user_id FROM orders WHERE date = '" + str(int(date) - 1) + "'"
    cursor.execute(sql2)
    yesterday_users = set(row[0] for row in cursor.fetchall())
    conn.close()
    retention = len(today_users & yesterday_users) / len(yesterday_users) * 100
    return retention
```

**Reviewer任务：** 按Review清单找出所有问题，用[标签]格式写Review意见

**Author任务：** 对每条Review意见给出回应（接受/拒绝/讨论）

**参考答案（Review意见）：**

```
[必须修改] SQL注入风险
位置: get_user_orders:3, calculate_retention:4,6
原因: 使用f-string拼接SQL，存在SQL注入风险
建议: 使用参数化查询 cursor.execute("SELECT * FROM orders WHERE user_id = %s", (user_id,))

[必须修改] 数据库连接未使用上下文管理器
位置: get_user_orders:2, calculate_retention:2
原因: 如果cursor.execute()抛异常，conn.close()不会执行，导致连接泄漏
建议: 使用with语句或try/finally

[必须修改] 硬编码数据库密码
位置: get_user_orders:2, calculate_retention:2
原因: 密码明文写在代码中，存在安全风险
建议: 从环境变量或配置文件读取

[必须修改] 除零风险
位置: calculate_retention:8
原因: 如果yesterday_users为空集，len(yesterday_users)=0，会抛ZeroDivisionError
建议: 加判断 if not yesterday_users: return 0

[建议修改] 重复的数据库连接代码
位置: 两个函数都有相同的连接代码
建议: 提取为公共函数get_connection()

[建议修改] 日期计算逻辑错误
位置: calculate_retention:6
原因: int(date)-1 只对YYYYMMDD格式且非跨月/跨年时正确，1月1日会变成10100
建议: 使用datetime.timedelta计算

[风格] 函数缺少文档字符串
建议: 添加docstring说明参数、返回值
```
