### 2.1 OLAP vs OLTP — 两种数据库的根本差异（30min）

**OLTP（联机事务处理） vs OLAP（联机分析处理）：**

```
OLTP — 业务系统的事务处理:
  场景: 电商下单、银行转账、用户注册
  特征: 
    - 每次操作1~几行数据
    - 大量并发的小事务（INSERT/UPDATE/DELETE）
    - 要求强一致性（ACID）
    - 响应时间: 毫秒级
  代表: MySQL, PostgreSQL, Oracle

OLAP — 数据仓库的分析查询:
  场景: 销售报表、用户画像、趋势分析
  特征:
    - 每次扫描百万~亿行数据
    - 少量并发的大查询（SELECT + GROUP BY + AGG）
    - 只读或批量写入，极少UPDATE/DELETE
    - 响应时间: 秒级~分钟级
  代表: ClickHouse, StarRocks, DuckDB
```

**核心差异对比：**

| 维度 | OLTP | OLAP |
|------|------|------|
| 数据模型 | 3NF（规范化） | 星型/雪花模型（反规范化） |
| 写入模式 | 随机写入，逐行INSERT/UPDATE | 批量写入，追加INSERT |
| 查询模式 | 点查，按主键取1行 | 全表扫描，GROUP BY聚合 |
| 并发量 | 万级QPS | 十~百级QPS |
| 单次数据量 | 1行~几十行 | 百万~亿行 |
| 存储格式 | 行式存储 | 列式存储 |
| 索引 | B+Tree | 稀疏索引 + 跳数索引 |
| 响应时间 | <10ms | 100ms~几分钟 |

**为什么OLAP不能用MySQL做？**

```
场景: 1亿行订单表，按品类统计销售额

MySQL (行式存储):
  1. 读取1亿行完整数据（包括不需要的列）→ IO巨大
  2. B+Tree索引对全表扫描无帮助
  3. 单线程执行 → 无法利用多核
  4. 耗时: 5~30分钟

ClickHouse (列式存储):
  1. 只读取 category_id 和 amount 两列 → IO减少80%+
  2. 列存压缩率高 → 实际IO更少
  3. 向量化执行 + 多核并行 → CPU利用率高
  4. 耗时: 0.5~5秒
```

### 2.2 列式存储原理（20min）

**行式存储 vs 列式存储的物理布局：**

```
原始数据（4行3列）:
  ┌──────────┬────────────┬────────┐
  │ user_id  │ category   │ amount │
  ├──────────┼────────────┼────────┤
  │ 1        │ 电商       │ 299    │
  │ 2        │ 数码       │ 1599   │
  │ 3        │ 电商       │ 89     │
  │ 4        │ 家居       │ 450    │
  └──────────┴────────────┴────────┘

行式存储（MySQL/PostgreSQL）:
  磁盘上按行连续存储:
    [1, 电商, 299] [2, 数码, 1599] [3, 电商, 89] [4, 家居, 450]
    
  查询 SELECT SUM(amount) FROM orders:
    必须读取所有列的数据（包括不需要的user_id和category）
    → IO浪费

列式存储（ClickHouse/Parquet）:
  磁盘上按列连续存储:
    user_id:  [1, 2, 3, 4]
    category: [电商, 数码, 电商, 家居]
    amount:   [299, 1599, 89, 450]
    
  查询 SELECT SUM(amount) FROM orders:
    只读取amount列的数据
    → IO减少66%（3列中只读1列）
```

**列式存储的三大优势：**

```
优势1: 列裁剪（只读需要的列）
  查询: SELECT category, SUM(amount) FROM orders GROUP BY category
  100列的宽表，只需要2列 → IO减少98%

优势2: 高压缩率
  同一列的数据类型相同 → 压缩效果好
  - 数值列: Delta编码 + LZ4/ZSTD → 压缩率5:1~20:1
  - 字符串列: Dictionary编码 → 压缩率3:1~10:1
  实际效果: 1TB原始数据 → 100~200GB存储

优势3: 向量化执行（SIMD）
  列数据在内存中连续排列 → CPU可以一次处理多个值
  普通执行:  amount[0] + amount[1] + amount[2] + ...  (逐行)
  向量化:    AVX2指令一次处理4个int64 → 4倍加速
             AVX-512一次处理8个int64 → 8倍加速
```

### 2.3 ClickHouse核心（60min）

#### 2.3.1 MergeTree引擎家族

```
MergeTree是ClickHouse最核心的表引擎，所有其他引擎都基于它:

MergeTree家族:
  ├── MergeTree          → 基础引擎，按主键排序，数据按分区存储
  ├── ReplacingMergeTree → 去重引擎，自动删除相同主键的旧数据
  ├── SummingMergeTree   → 求和引擎，自动合并相同主键的数值列
  ├── AggregatingMergeTree → 聚合引擎，存储中间聚合状态
  └── CollapsingMergeTree  → 折叠引擎，用sign列抵消旧数据

选择指南:
  - 普通事实表 → MergeTree
  - 需要去重 → ReplacingMergeTree
  - 需要预聚合 → SummingMergeTree / AggregatingMergeTree
```

#### 2.3.2 ClickHouse建表实战

```sql
CREATE TABLE ecommerce.orders
(
    order_id        UInt64,
    user_id         UInt64,
    item_id         UInt64,
    category_id     UInt32,
    amount          Decimal(12, 2),
    behavior        LowCardinality(String),
    dt              Date,
    hour            UInt8,
    province        LowCardinality(String)
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(dt)
ORDER BY (category_id, dt, user_id)
SETTINGS index_granularity = 8192;
```

**建表关键参数解释：**

```
PARTITION BY toYYYYMM(dt):
  - 按月分区，每个月的数据存在独立目录
  - 查询时按分区裁剪，不扫描无关月份
  - 过期数据可按分区直接删除（比DELETE快1000倍）

ORDER BY (category_id, dt, user_id):
  - 数据在分区内部按此顺序排列（稀疏索引）
  - 查询条件包含排序列前缀时可以快速定位
  - 选择原则: 最常用的过滤字段放前面

index_granularity = 8192:
  - 每8192行生成一个索引标记（稀疏索引）
  - 稀疏索引占用空间小，可以常驻内存
  - 适合大数据量场景

LowCardinality(String):
  - 低基数字符串优化（如behavior只有pv/buy/cart/fav）
  - 内部使用Dictionary编码，节省存储和内存
```

#### 2.3.4 聚合查询示例

```sql
SELECT
    category_id,
    count() AS total_pv,
    uniq(user_id) AS total_uv,
    sum(amount) AS total_amount,
    avg(amount) AS avg_amount,
    sumIf(amount, behavior = 'buy') AS buy_amount,
    round(sumIf(amount, behavior = 'buy') / sum(amount) * 100, 2) AS buy_rate
FROM ecommerce.orders
WHERE dt >= '2024-01-01' AND dt <= '2024-12-31'
GROUP BY category_id
ORDER BY total_pv DESC
LIMIT 20;
```

**ClickHouse聚合函数速查：**

| 函数 | 说明 | 对比MySQL |
|------|------|-----------|
| count() | 计数 | COUNT(*) |
| uniq(x) | 近似去重计数（HyperLogLog） | COUNT(DISTINCT x) |
| uniqExact(x) | 精确去重计数 | COUNT(DISTINCT x) |
| sum(x) | 求和 | SUM(x) |
| avg(x) | 平均值 | AVG(x) |
| quantile(x, 0.95) | 分位数 | 无原生支持 |
| topK(x, 10) | 频率最高的TopN | 无原生支持 |
| groupArray(x) | 聚合为数组 | GROUP_CONCAT |

#### 2.3.5 物化视图加速常用查询

```
问题: 每次查询都要扫描百万/亿行数据做聚合 → 慢

物化视图的思路: 提前把聚合结果算好存起来
  - 类似于Spark的Cache，但是是持久化的
  - 数据写入时自动更新物化视图
  - 查询时自动命中物化视图（无需改SQL）
```

```sql
CREATE MATERIALIZED VIEW ecommerce.orders_daily_mv
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(dt)
ORDER BY (dt, category_id, province)
AS SELECT
    dt,
    category_id,
    province,
    count() AS total_pv,
    uniqState(user_id) AS uv_state,
    sum(amount) AS total_amount,
    sumIf(amount, behavior = 'buy') AS buy_amount
FROM ecommerce.orders
GROUP BY dt, category_id, province;
```

```sql
SELECT
    dt,
    category_id,
    total_pv,
    uniqMerge(uv_state) AS total_uv,
    total_amount,
    buy_amount,
    round(buy_amount / total_amount * 100, 2) AS buy_rate
FROM ecommerce.orders_daily_mv
WHERE dt >= '2024-01-01' AND dt <= '2024-03-31'
GROUP BY dt, category_id, total_pv, total_amount, buy_amount
ORDER BY total_pv DESC
LIMIT 20;
```

**物化视图注意事项：**

```
1. SummingMergeTree + uniqState/uniqMerge 是ClickHouse的常见模式
   - uniqState: 保存HyperLogLog中间状态
   - uniqMerge: 合并中间状态得到最终去重数

2. 物化视图是触发器模式，新数据写入时自动更新
   - 历史数据不会自动回填
   - 如果修改了物化视图定义，需要重建

3. 物化视图会增加写入开销
   - 每次INSERT都要更新所有相关物化视图
   - 物化视图越多，写入越慢
   - 建议: 只为高频查询创建物化视图
```

### 2.4 DuckDB — 嵌入式OLAP数据库（30min）

```
DuckDB的定位:
  - "数据分析领域的SQLite"
  - 嵌入式运行，无需部署服务端

适用场景:

  - 快速原型验证（不需要部署集群）
```

**DuckDB vs Pandas 性能对比：**

```python
import duckdb
import pandas as pd
import time

df = pd.read_parquet('data/orders.parquet')
print(f"数据量: {len(df)} 行")

start = time.time()
pandas_result = df[df['behavior'] == 'buy'].groupby('category_id').agg(
    total_amount=('amount', 'sum'),
    avg_amount=('amount', 'mean'),
    order_count=('amount', 'count')
).reset_index().sort_values('total_amount', ascending=False)
pandas_time = time.time() - start
print(f"Pandas 耗时: {pandas_time:.3f}s")

start = time.time()
duckdb_result = duckdb.sql("""
    SELECT
        category_id,
        sum(amount) AS total_amount,
        avg(amount) AS avg_amount,
        count(*) AS order_count
    FROM df
    WHERE behavior = 'buy'
    GROUP BY category_id
    ORDER BY total_amount DESC
""").fetchdf()
duckdb_time = time.time() - start
print(f"DuckDB 耗时: {duckdb_time:.3f}s")
print(f"DuckDB 加速比: {pandas_time / duckdb_time:.1f}x")
```

```
典型结果（100万行数据）:
  Pandas 耗时: 0.45s
  DuckDB 耗时: 0.05s
  DuckDB 加速比: 9.0x

原因:
  - DuckDB使用列式存储 + 向量化执行
  - Pandas是逐行处理（虽然底层有numpy优化）
  - 数据量越大，DuckDB优势越明显
```

### 2.5 StarRocks核心（25min）

```
StarRocks的定位:
  - 新一代MPP OLAP数据库（原DorisDB）
  - 兼容MySQL协议 → 可用MySQL客户端直接连接
  - 向量化执行 + CBO优化器
  - 支持实时更新（Unique Key模型）
  - 支持湖仓联邦查询（直接查Hive/Iceberg/Hudi）

核心优势:
  - MySQL兼容 → 学习成本低，BI工具无缝对接
  - 多表Join性能强（CBO + Shuffle Join + Colocate Join）
  - 实时+离线统一（既支持批量导入也支持流式写入）
```

#### 2.5.1 StarRocks建表

```sql
CREATE TABLE ecommerce.orders (
    order_id        BIGINT          NOT NULL COMMENT '订单ID',
    user_id         BIGINT          NOT NULL COMMENT '用户ID',
    category_id     INT             NOT NULL COMMENT '品类ID',
    amount          DECIMAL(12, 2)  NOT NULL COMMENT '金额',
    behavior        VARCHAR(20)     NOT NULL COMMENT '行为类型',
    dt              DATE            NOT NULL COMMENT '日期',
    province        VARCHAR(20)     NOT NULL COMMENT '省份'
)
ENGINE = OLAP
DUPLICATE KEY(order_id)
PARTITION BY RANGE(dt) (
    PARTITION p202401 VALUES [('2024-01-01'), ('2024-02-01')),
    PARTITION p202402 VALUES [('2024-02-01'), ('2024-03-01')),
    PARTITION p202403 VALUES [('2024-03-01'), ('2024-04-01'))
)
DISTRIBUTED BY HASH(user_id) BUCKETS 16
PROPERTIES (
    "replication_num" = "3",
    "storage_format" = "V2"
);
```

**StarRocks三种模型：**

```
1. Duplicate Key（明细模型）:
   - 数据不聚合，保留所有明细
   - 适合: 事实表、日志表
   - 排序列只用于排序，不做聚合

2. Aggregate Key（聚合模型）:
   - 相同Key的数据自动聚合
   - 可定义SUM/REPLACE/MAX/MIN聚合方式
   - 适合: 预聚合的汇总表

3. Unique Key（更新模型）:
   - 相同Key的数据保留最新版本
   - 适合: 需要更新的维度表、订单状态表
```

```sql
CREATE TABLE ecommerce.user_stats (
    user_id         BIGINT          NOT NULL,
    dt              DATE            NOT NULL,
    total_amount    DECIMAL(12, 2)  SUM COMMENT '总消费',
    order_count     INT             SUM COMMENT '订单数',
    last_active     DATE            REPLACE COMMENT '最后活跃日'
)
ENGINE = OLAP
AGGREGATE KEY(user_id, dt)
DISTRIBUTED BY HASH(user_id) BUCKETS 16
PROPERTIES ("replication_num" = "3");
```

### 2.6 选型对比：ClickHouse vs StarRocks vs Doris vs DuckDB（15min）

| 维度 | ClickHouse | StarRocks | Apache Doris | DuckDB |
|------|-----------|-----------|--------------|--------|
| **架构** | 分布式MPP | 分布式MPP | 分布式MPP | 嵌入式单机 |
| **协议** | 自有TCP/HTTP | MySQL兼容 | MySQL兼容 | 自有API |
| **Join能力** | 弱（大表Join性能差） | 强（CBO+Colocate） | 中等 | 中等 |
| **实时更新** | 弱（Mutation代价大） | 强（Unique Key模型） | 强（Unique Key模型） | 强（标准SQL） |
| **写入模式** | 批量追加最优 | 批量+流式均优 | 批量+流式均优 | 单机写入 |
| **物化视图** | 强（自动触发） | 强（异步刷新） | 弱 | 无 |
| **联邦查询** | 弱 | 强（Hive/Iceberg/Hudi） | 中等 | 强（Parquet/CSV/S3） |
| **运维复杂度** | 高（Zookeeper依赖） | 中 | 中 | 极低（无需部署） |
| **单查询性能** | 极高（单表扫描最快） | 高 | 高 | 高（单机） |
| **社区生态** | 最大 | 快速增长 | 成熟 | 快速增长 |
| **适用场景** | 日志分析、大单表 | 多表Join、实时报表 | 实时报表、即席查询 | 本地分析、数据科学 |

**选型决策树：**

```
if 需要单机本地分析:
    → DuckDB（零部署，直接pip install）

elif 主要是单表聚合（日志/指标分析）:
    → ClickHouse（单表扫描性能最强）

elif 需要多表Join + 实时更新:
    → StarRocks（Join性能好，支持实时更新）

elif 已有Doris集群 + MySQL生态:
    → Apache Doris（与StarRocks同源，更成熟稳定）

elif 需要湖仓联邦查询:
    → StarRocks（直接查Hive/Iceberg/Hudi）
```

**典型场景推荐：**

```
场景1: 日志分析平台（Nginx/业务日志）
  → ClickHouse
  原因: 单表写入快，查询快，物化视图成熟

场景2: 实时BI报表（多维度分析+实时更新）
  → StarRocks
  原因: MySQL兼容，多表Join强，支持实时更新

场景3: 数据科学家本地探索
  → DuckDB
  原因: 无需部署，直接查询Parquet，与Pandas互操作

场景4: 统一数仓（离线+实时）
  → StarRocks / Doris
  原因: 同时支持批量导入和流式写入，统一服务
```

### 练习4：用DuckDB直接查询本地Parquet文件（10min）

```python
import duckdb
import pandas as pd
import numpy as np

np.random.seed(42)
n = 1_000_000

df = pd.DataFrame({
    'user_id': np.random.randint(1, 100001, n),
    'category_id': np.random.randint(1, 1001, n),
    'amount': np.round(np.random.uniform(1, 5000, n), 2),
    'behavior': np.random.choice(['pv', 'buy', 'cart', 'fav'], n, p=[0.6, 0.1, 0.2, 0.1]),
    'dt': np.random.choice(pd.date_range('2024-01-01', '2024-12-31'), n)
})

df.to_parquet('data/orders.parquet', index=False)
print(f"已生成 {len(df)} 行Parquet文件")

con = duckdb.connect()

result1 = con.execute("""
    SELECT
        category_id,
        count(*) AS pv,
        sum(amount) AS total_amount
    FROM read_parquet('data/orders.parquet')
    GROUP BY category_id
    ORDER BY pv DESC
    LIMIT 10
""").fetchdf()
print("=== 品类PV TOP10 ===")
print(result1)

result2 = con.execute("""
    SELECT
        behavior,
        count(*) AS cnt,
        round(avg(amount), 2) AS avg_amount
    FROM read_parquet('data/orders.parquet')
    GROUP BY behavior
    ORDER BY cnt DESC
""").fetchdf()
print("\n=== 行为分布 ===")
print(result2)

result3 = con.execute("""
    SELECT
        date_trunc('month', dt) AS month,
        count(*) AS pv,
        sum(amount) AS total_amount
    FROM read_parquet('data/orders.parquet')
    WHERE behavior = 'buy'
    GROUP BY month
    ORDER BY month
""").fetchdf()
print("\n=== 月度购买趋势 ===")
print(result3)

con.close()
```
