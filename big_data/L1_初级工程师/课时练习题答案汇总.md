### 练习1：HDFS基本命令操作

**题目**：写出完成以下操作的完整HDFS命令行指令。

**答案**：

```bash
# 1. 查看根目录下的所有文件和目录
hdfs dfs -ls /

# 2. 递归查看 /user 目录
hdfs dfs -ls -R /user

# 3. 在HDFS上创建目录 /user/hadoop/exercise
hdfs dfs -mkdir -p /user/hadoop/exercise

# 4. 从本地上传文件到HDFS
hdfs dfs -put /local/data/file.txt /user/hadoop/exercise/

# 5. 从HDFS下载文件到本地
hdfs dfs -get /user/hadoop/exercise/file.txt /local/downloads/

# 6. 复制HDFS中的文件
hdfs dfs -cp /user/hadoop/exercise/file.txt /user/hadoop/backup/file.txt

# 7. 移动HDFS中的文件
hdfs dfs -mv /user/hadoop/exercise/file.txt /user/hadoop/archive/file.txt

# 8. 查看文件内容（前20行）
hdfs dfs -cat /user/hadoop/exercise/file.txt | head -20

# 9. 查看文件末尾内容（最后20行）
hdfs dfs -tail /user/hadoop/exercise/file.txt

# 10. 删除文件
hdfs dfs -rm /user/hadoop/exercise/file.txt

# 11. 删除目录（递归）
hdfs dfs -rm -r /user/hadoop/exercise

# 12. 查看文件大小和副本数
hdfs dfs -stat "%n %b %r %o" /user/hadoop/exercise/file.txt
# 输出格式: 文件名 文件大小(字节) 副本数 块大小

# 13. 查看磁盘使用情况
hdfs dfs -du -h /user/hadoop/
# -h: 人类可读格式

# 14. 设置文件副本数为2
hdfs dfs -setrep -w 2 /user/hadoop/exercise/file.txt
# -w: 等待操作完成

# 15. 查看文件的Block信息
hdfs fsck /user/hadoop/exercise/file.txt -files -blocks -locations
# -files: 显示文件信息
# -blocks: 显示Block信息
# -locations: 显示Block所在DataNode
```

**命令说明**：
- `hdfs dfs` 是所有HDFS文件系统操作的入口命令
- 大部分命令与Linux文件系统命令类似（ls、cp、mv、rm等）
- `-p` 参数表示递归创建父目录
- `-f` 参数可以强制操作（如强制删除）
- `-h` 参数以人类可读的格式显示大小（如KB、MB、GB）

### 练习2：HDFS体系架构分析

**题目**：分析以下架构图，回答问题。

**答案**：

```
问题1: NameNode存储什么？DataNode存储什么？
  
  NameNode存储（元数据）:
    - 文件系统命名空间（目录树结构）
    - 文件名 → Block ID的映射关系
    - Block ID → DataNode位置的映射关系
    - 文件权限、所有者、修改时间等属性
    - 这些都在内存中维护，同时持久化到FsImage和EditLog
  
  DataNode存储（实际数据）:
    - 文件数据块（Block，默认128MB）
    - 每个Block的校验和信息（.meta文件）
    - 定期向NameNode发送心跳和Block报告

问题2: 为什么HDFS不适合存储大量小文件？
  
  原因1: NameNode内存开销
    每个文件/目录/Block在NameNode内存中约占用150字节元数据。
    如果有1亿个小文件：
    1亿 × 150B = 15GB内存（仅元数据！）
    NameNode的JVM堆内存可能不够（通常配置32-64GB）

  原因2: MapReduce性能问题
    MapReduce默认每个文件至少对应一个Map Task。
    1亿个小文件 = 1亿个Map Task → 调度开销远大于数据处理开销。
    
  原因3: 磁盘寻道开销
    HDFS为顺序读写大文件优化。大量小文件意味着频繁的磁盘寻道，
    随机读取性能远差于顺序读取。

  解决方案：
    - Hadoop Archive (HAR)：将小文件打包
    - SequenceFile：Key-Value格式合并存储
    - CombineFileInputFormat：一个Map处理多个小文件
    - 将小文件存入HBase而非HDFS

问题3: 默认Block大小为什么是128MB？
  
  1. 减少寻道时间占比：
     如果Block只有1MB，顺序读取时间很短，寻道时间占比高。
     128MB可以保证传输时间远大于寻道时间。
  
  2. 减少元数据量：
     Block越大 → 文件需要的Block数越少 → NameNode内存中元数据越少。
     例如10GB文件：128MB Block → 80个Block记录；
                    1MB Block → 10240个Block记录。
  
  3. Map Task粒度：
     一个Map Task通常处理一个Block。128MB是一个合理的Map Task输入大小，
     足够大保证计算效益，又不会太大导致单Task处理时间过长。
```

### 练习3：HDFS读写流程

**题目**：描述HDFS的读写流程和数据完整性保障机制。

**答案**：

```
写数据流程（7步）:

Step 1: 客户端调用 DistributedFileSystem.create()
  → 向NameNode发起RPC请求

Step 2: NameNode检查
  → 文件是否已存在？客户端是否有写入权限？
  → 通过后在命名空间中创建新文件记录（此时无Block分配）

Step 3: 客户端调用 FSDataOutputStream.write()
  → 数据先写入客户端本地缓冲区（默认64KB Packet）

Step 4: 缓冲区满，客户端向NameNode请求新Block分配
  → NameNode返回DataNode列表（按副本放置策略）

Step 5: 客户端与第一个DataNode建立Pipeline
  → 以Packet为单位发送数据

Step 6: 流水线复制（Pipeline Replication）
  → D1收到数据 → 写入本地 → 转发给D2
  → D2收到数据 → 写入本地 → 转发给D3
  → D3收到数据 → 写入本地 → 返回ACK

Step 7: ACK回传
  → D3 → D2 → D1 → Client（确认所有副本写入成功）
  → 所有Block写入完成后，向NameNode发送完成信号

读数据流程（4步）:

Step 1: 客户端调用 DistributedFileSystem.open()
  → 向NameNode发起RPC请求

Step 2: NameNode返回Block位置列表
  → 按"网络距离"排序（本地 > 同机架 > 跨机架）

Step 3: 客户端连接最近的DataNode
  → 读取Block数据（边读边进行Checksum校验）
  → 读完一个Block后关闭连接，对下一个Block重复此步骤

Step 4: 全部Block读取完毕
  → 调用 FSDataInputStream.close()

数据完整性保障:

1. 写入时：客户端计算每个Packet的CRC32校验和，随数据发送
2. 存储时：DataNode保存数据的校验和到.meta文件中
3. 读取时：客户端验证读取数据的校验和，不匹配则从其他副本读取
4. 定期扫描：DataNode的BlockScanner定期扫描所有Block验证完整性
5. 损坏恢复：DataNode报告损坏 → NameNode标记 → 触发副本复制
```

### 练习4：NameNode高可用分析

**题目**：解释HDFS HA架构，说明故障切换过程。

**答案**：

```
HA架构组件:

   ┌─────────────┐         ┌─────────────┐
   │  ZooKeeper   │         │  ZooKeeper   │
   │    (ZK1)     │         │    (ZK2)     │
   └──────┬───────┘         └──────┬───────┘
          │        Quorum         │
          └──────────┬────────────┘
                     │
   ┌─────────────────┼─────────────────┐
   │                 │                 │
   ▼                 ▼                 ▼
┌────────┐    ┌──────────┐    ┌────────┐
│  JN1   │    │   JN2    │    │  JN3   │
│(EditLog)│   │(EditLog) │    │(EditLog)│
└────────┘    └──────────┘    └────────┘
   ▲                 ▲              ▲
   │    写入EditLog   │              │
   │                 │              │
┌──┴──────────┐     │    ┌─────────┴──────┐
│   Active    │     │    │    Standby     │
│  NameNode   │←同步─┘    │   NameNode    │
│             │  EditLog  │               │
│ 处理请求     │           │ 只读元数据     │
└─────────────┘           └───────────────┘

关键组件说明:

1. Active NameNode: 对外服务，处理所有客户端请求
2. Standby NameNode: 热备节点，通过读取JournalNode的EditLog保持同步
3. JournalNode (至少3个): 存储共享EditLog
   - Active写入 → QJM协议保证一致性 → Standby读取并重放
   - 奇数个（Paxos/Raft选举算法要求）
4. ZooKeeper: 
   - 监控NameNode健康状态
   - 管理Active/Standby选举
   - 通过ZKFC (ZK Failover Controller) 实现自动故障切换

故障切换流程:

1. ZKFC检测到Active NameNode心跳超时（不可用）
2. ZKFC通过Fencing机制确保旧Active已停止
   - 调用旧Active的HAServiceProtocol RPC
   - SSH到旧Active节点kill进程
3. ZKFC通知Standby NameNode切换为Active状态
4. 新Active从JournalNode读取剩余EditLog并重放
5. 新Active开始对外提供服务
6. DataNode自动向新Active发送心跳和Block报告

脑裂预防 (Fencing):
  脑裂是指两个NameNode同时认为自己是Active，同时对外服务。
  通过Fencing确保旧Active完全停止后才切换:
  - 方式1: SSH Fence - SSH到目标节点kill进程
  - 方式2: Shell Fence - 执行自定义脚本
  - 方式3: 共享存储Fence - 撤销旧Active的共享存储访问权限
```

## 课时12：MapReduce原理与实战 — 练习题答案

### 练习1：WordCount完整实现

**题目**：使用Java MapReduce API实现WordCount。

**答案**：

```java
import java.io.IOException;
import java.util.StringTokenizer;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.fs.Path;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.LongWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Job;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;
import org.apache.hadoop.mapreduce.lib.input.FileInputFormat;
import org.apache.hadoop.mapreduce.lib.output.FileOutputFormat;

/**
 * WordCount MapReduce实现
 *
 * 执行命令:
 * hadoop jar wc.jar WordCount /input/words.txt /output/wc_result
 */
public class WordCount {

    /**
     * Mapper类
     * 输入Key:   LongWritable  (文件偏移量)
     * 输入Value: Text          (一行文本)
     * 输出Key:   Text          (单词)
     * 输出Value: IntWritable   (计数1)
     */
    public static class TokenizerMapper
            extends Mapper<LongWritable, Text, Text, IntWritable> {

        private final static IntWritable one = new IntWritable(1);
        private Text word = new Text();

        @Override
        public void map(LongWritable key, Text value, Context context)
                throws IOException, InterruptedException {

            // 将一行文本按空格/制表符/换行符分割
            StringTokenizer tokenizer = new StringTokenizer(value.toString());

            while (tokenizer.hasMoreTokens()) {
                word.set(tokenizer.nextToken());
                context.write(word, one);
                // 输出: (hello, 1), (world, 1), (hello, 1), ...
            }
        }
    }

    /**
     * Reducer类
     * 输入Key:   Text          (单词)
     * 输入Value: Iterable<IntWritable> (该单词的所有计数[1,1,1,...])
     * 输出Key:   Text          (单词)
     * 输出Value: IntWritable   (总计数)
     */
    public static class IntSumReducer
            extends Reducer<Text, IntWritable, Text, IntWritable> {

        private IntWritable result = new IntWritable();

        @Override
        public void reduce(Text key, Iterable<IntWritable> values, Context context)
                throws IOException, InterruptedException {

            int sum = 0;
            for (IntWritable val : values) {
                sum += val.get();
            }
            result.set(sum);
            context.write(key, result);
            // 输出: (hello, 2), (world, 1), ...
        }
    }

    public static void main(String[] args) throws Exception {
        Configuration conf = new Configuration();
        Job job = Job.getInstance(conf, "word count");

        job.setJarByClass(WordCount.class);

        // 设置Mapper和Reducer
        job.setMapperClass(TokenizerMapper.class);
        job.setReducerClass(IntSumReducer.class);

        // 可选: 设置Combiner（Map端预聚合，减少Shuffle数据量）
        job.setCombinerClass(IntSumReducer.class);

        // 设置输出Key/Value类型
        job.setOutputKeyClass(Text.class);
        job.setOutputValueClass(IntWritable.class);

        // 设置输入输出路径
        FileInputFormat.addInputPath(job, new Path(args[0]));
        FileOutputFormat.setOutputPath(job, new Path(args[1]));

        // 提交作业并等待完成
        System.exit(job.waitForCompletion(true) ? 0 : 1);
    }
}
```

**MapReduce执行流程分析**：

```
Input: ["hello world", "hello hadoop"]

Map阶段:
  Map Task 1 处理 "hello world":
    → (hello, 1), (world, 1)
  Map Task 2 处理 "hello hadoop":
    → (hello, 1), (hadoop, 1)

Shuffle阶段 (MapReduce框架自动完成):
  1. Partition: 按Key的Hash值分区
  2. Sort: 每个分区内按键排序
  3. Group: 相同Key的Value归并为一个List
  → Reducer 1 收到: hello → [1, 1]
  → Reducer 2 收到: world  → [1]
  → Reducer 3 收到: hadoop → [1]

Reduce阶段:
  Reducer 1: (hello, 2)
  Reducer 2: (world, 1)
  Reducer 3: (hadoop, 1)

Output: hello:2  world:1  hadoop:1
```

### 练习2：Shuffle机制详解

**题目**：解释MapReduce Shuffle阶段的完整过程。

**答案**：

```
MapReduce Shuffle是指从Map输出到Reduce输入的整个过程:

┌──────────────────────────────────────────────────────────────┐
│                        Shuffle详细流程                        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Map Task                           Reduce Task              │
│  ┌──────────┐                       ┌──────────┐            │
│  │ 1. Map() │ 输出到内存缓冲区        │          │            │
│  │          │                       │          │            │
│  │ 2. 分区  │ Partitioner决定       │          │            │
│  │          │ 数据去哪个Reducer      │          │            │
│  │          │                       │          │            │
│  │ 3. 排序  │ 内存中按键排序         │          │            │
│  │          │                       │          │            │
│  │ 4. 溢写  │ 缓冲区满(默认100MB)   │          │            │

│         │                           │          │            │
│         └─────────── HTTP ──────────→│          │            │
│                                     │ 6. 拉取  │ 从Map端    │
│                                     │          │ 拉取分区数据│
│                                     │          │            │
│                                     │ 7. 合并  │ 合并不同   │
│                                     │          │ Map的数据   │
│                                     │          │            │
│                                     │ 8. 排序  │ 再次排序   │
│                                     │          │            │
│                                     │ 9. 分组  │ 同Key归并  │
│                                     │          │            │
│                                     │10.Reduce()│ 处理数据  │
│                                     └──────────┘            │
│                                                              │
└──────────────────────────────────────────────────────────────┘

关键配置参数:
  mapreduce.task.io.sort.mb     = 100   (内存缓冲区默认100MB)
  mapreduce.map.sort.spill.percent = 0.80 (缓冲区80%时开始溢写)
  mapreduce.task.io.sort.factor = 10    (一次合并的文件数)
  mapreduce.reduce.shuffle.parallelcopies = 5 (并行拉取线程数)

### 练习3：TopN实现

**题目**：使用MapReduce找出每个月份温度最高的3天。

**答案**：

```java
import java.io.IOException;
import java.util.TreeMap;
import org.apache.hadoop.io.IntWritable;
import org.apache.hadoop.io.NullWritable;
import org.apache.hadoop.io.Text;
import org.apache.hadoop.mapreduce.Mapper;
import org.apache.hadoop.mapreduce.Reducer;

/**
 * 数据格式: 2024-01-15 25°C
 * 输出: 每个月份温度最高的3天
 */
public class TopN {

    public static class TopNMapper
            extends Mapper<Object, Text, Text, IntWritable> {

        private Text month = new Text();
        private IntWritable temperature = new IntWritable();

        @Override
        public void map(Object key, Text value, Context context)
                throws IOException, InterruptedException {

            String[] fields = value.toString().split("\\s+");
            if (fields.length >= 2) {
                // 提取月份: "2024-01-15" → "2024-01"
                month.set(fields[0].substring(0, 7));

                // 提取温度: "25°C" → 25
                String tempStr = fields[1].replace("°C", "");
                temperature.set(Integer.parseInt(tempStr));

                context.write(month, temperature);
            }
        }
    }

    public static class TopNReducer
            extends Reducer<Text, IntWritable, Text, Text> {

        // TreeMap自动按Key(温度)排序，这里取降序
        private TreeMap<Integer, String> topN = new TreeMap<>(
            (a, b) -> b.compareTo(a)  // 降序
        );

        @Override
        public void reduce(Text key, Iterable<IntWritable> values, Context context)
                throws IOException, InterruptedException {

            topN.clear();
            int N = 3;

            // 遍历该月所有温度值
            for (IntWritable val : values) {
                topN.put(val.get(), "");
                // 超过N个就移除最小的
                if (topN.size() > N) {
                    topN.pollLastEntry();
                }
            }

            // 输出Top N结果
            StringBuilder result = new StringBuilder();
            for (Integer temp : topN.keySet()) {
                if (result.length() > 0) result.append(", ");
                result.append(temp).append("°C");
            }
            context.write(key, new Text(result.toString()));
        }
    }
}
```

### 练习4：Combiner优化分析

**题目**：解释Combiner的作用和使用注意事项。

**答案**：

```
Combiner = Map端的"小Reducer"

作用:
  1. 在Map端进行预聚合，减少Map输出的数据量
  2. 减少Shuffle时的网络传输量
  3. 减少Reducer端需要处理的数据量

示例（WordCount）:
  不使用Combiner:
    Map输出: (hello,1), (hello,1), (hello,1), (hello,1), (hello,1)
    → 5条记录需要Shuffle到Reducer
    
  使用Combiner后:
    Map端预聚合: (hello,5)
    → 1条记录需要Shuffle到Reducer
    → 网络传输量减少80%

使用注意事项:

  1. Combiner的输入/输出类型必须与Reducer一致
     → 可以复用Reducer类作为Combiner（如WordCount的IntSumReducer）

  2. 只适用于满足结合律和交换律的操作
     ✓ 可以使用: sum, max, min, count
     ✗ 不能使用: average (avg(avg(a,b), avg(c,d)) ≠ avg(a,b,c,d))
     ✗ 不能使用: distinct count

  3. Combiner不保证被调用
     → Hadoop可能调用0次、1次或多次Combiner
     → 因此Combiner不能改变最终结果的正确性

  4. 求平均值的正确处理方式:
     Mapper输出: (key, (sum, count))
     Combiner合并: (key, (sum1+sum2, count1+count2))
     Reducer计算: sum_total / count_total
```

## 课时13：Hive数据仓库入门 — 练习题20道答案

### 题目1-5：创建表与管理

**题目1**：创建一个外部表存储用户行为日志，按日期分区，使用Parquet存储格式和SNAPPY压缩。

**答案**：
```sql
CREATE EXTERNAL TABLE IF NOT EXISTS ods.ods_user_log (
    user_id    BIGINT COMMENT '用户ID',
    product_id BIGINT COMMENT '商品ID',
    behavior   STRING COMMENT '行为类型: pv/cart/fav/buy',
    ts         BIGINT COMMENT '事件时间戳(毫秒)'
)
COMMENT 'ODS-用户行为日志表'
PARTITIONED BY (dt STRING COMMENT '日期分区 yyyyMMdd')
STORED AS PARQUET
LOCATION '/warehouse/ods/ods_user_log'
TBLPROPERTIES (
    'parquet.compression' = 'SNAPPY',
    'parquet.block.size'  = '134217728'
);

-- 解释:
-- EXTERNAL: 外部表，删除表时不删除HDFS上的数据文件
-- PARTITIONED BY: 分区字段不能在普通列中重复定义
-- STORED AS PARQUET: 列式存储，支持列裁剪和谓词下推

SELECT

);

SELECT

    COUNT(*) AS cnt
FROM orders
GROUP BY city, gender WITH CUBE;

    COUNT(*) AS cnt

GROUP BY city, gender WITH ROLLUP;

SELECT

    IF(GROUPING(gender) = 1, '全部', gender) AS gender,
    COUNT(*) AS cnt
FROM orders
GROUP BY city, gender
GROUPING SETS ((), (city), (gender), (city, gender));

### 题目16-20：性能与优化

**题目16**：解释Hive中有哪些常见的优化手段。

**答案**：
```
Hive优化手段分类:

1. 分区裁剪（Partition Pruning）:
   WHERE dt = '20240331' → 只扫描指定分区
   效果: 100个分区中只读1个 → 数据量减少99%

2. 列裁剪（Column Pruning）:
   SELECT user_id, amount FROM ... (而非SELECT *)
   效果: 50列中只读2列 → IO减少96%

3. 谓词下推（Predicate Pushdown）:
   WHERE条件在存储层（Parquet/ORC）就执行过滤
   效果: 减少读取到内存的不必要数据

4. 小表Join大表优化:
   使用MapJoin(Broadcast)将小表广播到内存
   SET hive.auto.convert.join = true;
   SET hive.mapjoin.smalltable.filesize = 25000000; -- 25MB

5. 开启向量化执行:
   SET hive.vectorized.execution.enabled = true;
   批量处理32768行而非逐行处理

6. 使用ORC/Parquet列式存储:
   相比TextFile，存储空间减少75%，查询性能提升数倍

7. CBO基于成本的优化:
   SET hive.cbo.enable = true;
   Hive自动选择最优Join顺序

8. 合理设置并行度:
   SET mapreduce.job.reduces = 50;
   根据数据量调整Reducer数量

9. 避免数据倾斜:
   SET hive.groupby.skewindata = true;
   (已废弃，建议手动处理)

10. 使用Tez/Spark引擎替代MR:
    SET hive.execution.engine = spark;
    减少磁盘IO，DAG执行更高效
```

**题目17**：解释Hive中JOIN的常见类型和执行原理。

**答案**：
```
Hive JOIN类型:

1. Common/Shuffle Join (Reduce Side Join):
   - 两个大表Join时的默认方式
   - 执行: Map阶段读取 → Shuffle阶段按Join Key分区 → Reduce阶段Join
   - 缺点: 全量Shuffle，网络IO大

2. Map Join (Broadcast Join):
   - 小表Join大表时的优化方式
   - 执行: 小表广播到所有Map的内存 → Map阶段直接Join → 无Shuffle
   - 触发条件: 小表 < hive.mapjoin.smalltable.filesize (默认25MB)
   - 手动触发: SELECT /*+ MAPJOIN(small_table) */ ...

3. Bucket Map Join:
   - 两个表都按相同的Key分桶（Bucket）
   - 执行: 对应的Bucket直接Join → Shuffle数据量减少
   - 条件: 分桶数相同或成倍数

4. Sort Merge Bucket (SMB) Join:
   - Bucket Map Join + 表内已排序
   - 执行: 直接排序合并 → 最快速的Join方式

Join顺序优化:
  - 先Join产生中间结果最小的
  - 最后一个Join大表（"大表放后面"原则不绝对，取决于数据分布）

LEFT SEMI JOIN:
  - 相当于 IN (SELECT ...) 子查询
  - 只返回左表中在右表存在匹配的行
  - 不会返回右表的列
  - 比普通INNER JOIN + SELECT DISTINCT效率高

示例:
  -- 找出有购买记录的用户（不返回订单信息）
  SELECT u.* FROM dim_user u
  LEFT SEMI JOIN dwd_order_detail o
  ON u.user_id = o.user_id
  WHERE o.dt = '20240331';
```

**题目18**：给出Hive中对数据倾斜的处理方案。

**答案**：
```
数据倾斜场景与方案:

场景1: GROUP BY 倾斜
  问题: 某个Key的数据量占总量的大部分
  方案:
    SET hive.groupby.skewindata = true;
    (生成两个MR Job: 第一个随机分发预聚合，第二个最终聚合)
  
  或手动两阶段聚合:
    SELECT key, SUM(cnt) AS total
    FROM (
        SELECT key, CAST(RAND() * 100 AS INT) AS salt, COUNT(*) AS cnt
        FROM table
        GROUP BY key, CAST(RAND() * 100 AS INT)
    ) t
    GROUP BY key;

场景2: JOIN 倾斜
  问题: Join Key分布严重不均
  方案1: 如果小表 < 25MB → 使用MapJoin
    SELECT /*+ MAPJOIN(small) */ ...
  
  方案2: 倾斜Key单独处理
    -- 非倾斜数据正常Join
    SELECT ... FROM big JOIN small ON big.key = small.key
    WHERE big.key != 'hot_key'
    UNION ALL
    -- 倾斜数据使用MapJoin
    SELECT /*+ MAPJOIN(small_hot) */ ...
    FROM big_hot JOIN small_hot ON big_hot.key = small_hot.key

  方案3: 加盐打散
    给倾斜Key加随机后缀，分到多个Reducer处理后合并

场景3: COUNT DISTINCT 倾斜
  问题: 某些值重复度极高
  替代方案:
    使用 GROUP BY + COUNT 替代 COUNT(DISTINCT col)
    或用 approx_count_distinct 近似统计

场景4: 空值倾斜
  问题: 大量NULL值集中在一个Reducer
  方案: 过滤空值或随机化空值
    WHERE key IS NOT NULL
    或 COALESCE(key, CONCAT('null_', CAST(RAND()*100 AS INT)))
```

**题目19**：写出Hive表的分区和分桶的区别及各自适用场景。

**答案**：
```
┌──────────────────┬──────────────────┬──────────────────┐
│      维度         │      分区        │      分桶        │
├──────────────────┼──────────────────┼──────────────────┤
│ 物理表现          │ 目录层级         │ 文件(桶文件)     │
│ 语法              │ PARTITIONED BY   │ CLUSTERED BY     │
│ 数量控制          │ 手动指定值       │ 固定桶数         │
│ 裁剪方式          │ WHERE dt='xx'    │ TABLESAMPLE      │
│ 数据分布          │ 按业务逻辑分组   │ 按Hash均匀分布   │
│ 基数要求          │ 低基数(<10000)   │ 无限制           │
│ 典型字段          │ dt, country      │ user_id, order_id│
│ 适用查询          │ 全量/范围扫描    │ 采样/Join优化     │
└──────────────────┴──────────────────┴──────────────────┘

分区适用场景:
  - 按日期/小时分区: WHERE dt = '2024-03-31' → 只读一天数据
  - 按国家/城市分区: WHERE country = 'CN'
  - 按业务类型分区: WHERE biz_type = 'order'
  注意: 分区数不要超过几千个，否则NameNode元数据压力大

分桶适用场景:
  - 采样查询: TABLESAMPLE(BUCKET 1 OUT OF 100 ON user_id)
  - 高效Join: 两个表都按相同Key分桶 → Bucket Map Join
  - 优化groupBy: 分桶后同key数据在同一文件 → 局部聚合更高效
  注意: 分桶数应该是2的幂次，方便数据分配

组合使用（分区+分桶）:
  CREATE TABLE orders (
      order_id BIGINT,
      user_id BIGINT,
      amount DOUBLE
  )
  PARTITIONED BY (dt STRING)
  CLUSTERED BY (user_id) INTO 256 BUCKETS
  STORED AS ORC;
  
  分区: 按日期隔离数据（时间维度裁剪）
  分桶: 按用户ID均匀分布（Join优化 + 采样）
```

## 课时14：Spark RDD核心原理 — 练习题答案

### 练习1：RDD五大特性

**题目**：列出RDD的五大特性并解释每个特性的含义。

**答案**：
```
RDD五大特性:

1. 一系列分区 (A list of partitions)
   - RDD由多个分区组成，每个分区分布在不同节点上
   - 分区是Spark并行计算的基本单位
   - 分区数决定并行度
   - 通过 getPartitions 方法获取

2. 计算函数 (A function for computing each split)
   - 每个分区由一个计算函数处理
   - 如map/filter/flatMap等
   - 通过 compute 方法执行

3. 依赖关系 (A list of dependencies on other RDDs)
   - 记录了当前RDD是如何从父RDD转换而来的
   - 窄依赖(Narrow): 一个父RDD分区被一个子RDD分区使用
   - 宽依赖(Wide): 一个父RDD分区被多个子RDD分区使用（需Shuffle）

4. 分区器 (Optionally, a Partitioner for key-value RDDs)
   - 对于Key-Value类型的RDD，可指定分区器
   - 如HashPartitioner、RangePartitioner
   - 控制数据如何分布到不同分区

5. 首选位置 (Optionally, a list of preferred locations)
   - 记录每个分区数据的优先计算位置
   - 如HDFS文件的Block所在节点
   - 用于数据本地性优化（移动计算而非移动数据）
```

**窄依赖 vs 宽依赖示例**：
```python
from pyspark import SparkContext

sc = SparkContext()

# 窄依赖: map, filter, flatMap, union (一个父分区→一个子分区)
rdd1 = sc.parallelize([1, 2, 3, 4, 5])
rdd2 = rdd1.map(lambda x: x * 2)      # 窄依赖
rdd3 = rdd2.filter(lambda x: x > 5)    # 窄依赖

# 宽依赖: reduceByKey, groupByKey, sortByKey, join (需要Shuffle)
pairs = rdd3.map(lambda x: (x % 2, x))
rdd4 = pairs.reduceByKey(lambda a, b: a + b)  # 宽依赖

sc.stop()
```

### 练习2：Transformation和Action的区别

**题目**：列出10个常用的Transformation和5个常用的Action，说明区别。

**答案**：
```
Transformation (转换-惰性求值):
  特性: 不会立即执行，只是记录运算逻辑，构建DAG
  
  常用Transformation:
  ┌──────────────┬──────────────────────────────────────┐
  │ map          │ 一对一映射：rdd.map(lambda x: x*2)  │
  │ filter       │ 过滤：rdd.filter(lambda x: x>0)     │
  │ flatMap      │ 一对多映射：rdd.flatMap(str.split)   │
  │ reduceByKey  │ 按Key聚合：rdd.reduceByKey(lambda a,b:a+b)│
  │ groupByKey   │ 按Key分组：rdd.groupByKey()         │
  │ mapValues    │ 只对Value映射：rdd.mapValues(f)     │
  │ sortBy       │ 排序：rdd.sortBy(lambda x: x[1])    │
  │ join         │ 连接：rdd1.join(rdd2)               │
  │ union        │ 合并：rdd1.union(rdd2)              │
  │ distinct     │ 去重：rdd.distinct()                │
  │ coalesce     │ 减少分区：rdd.coalesce(2)           │
  │ repartition  │ 重新分区：rdd.repartition(10)       │
  │ sample       │ 采样：rdd.sample(False, 0.1)        │
  │ mapPartitions│ 按分区映射：rdd.mapPartitions(f)    │
  └──────────────┴──────────────────────────────────────┘

Action (动作-触发计算):
  特性: 触发整个DAG的执行，产生最终结果
  
  常用Action:
  ┌──────────────┬──────────────────────────────────────┐
  │ collect      │ 收集全部数据到Driver：rdd.collect()  │
  │ count        │ 计数：rdd.count()                   │
  │ first        │ 取第一个元素：rdd.first()           │
  │ take(n)      │ 取前n个：rdd.take(10)               │
  │ reduce       │ 全局聚合：rdd.reduce(lambda a,b:a+b)│
  │ foreach      │ 遍历：rdd.foreach(print)            │
  │ saveAsTextFile│ 保存到文件：rdd.saveAsTextFile(p)  │
  │ countByKey   │ 按键计数：rdd.countByKey()          │
  │ takeOrdered  │ 取排序后前n个：rdd.takeOrdered(10)  │
  └──────────────┴──────────────────────────────────────┘

区别:
  Transformation是惰性的(Lazy):
    - 只记录运算逻辑（构建DAG）
    - 不会触发实际计算
    - 可以链式调用，Spark自动优化（如filter pushdown）
    
  Action是立即的(Eager):
    - 触发整个DAG的执行
    - 从数据源读取 → 执行所有Transformation → 产生结果
    - 结果返回到Driver或写入外部存储
```

### 练习3：reduceByKey vs groupByKey

**题目**：详细对比reduceByKey和groupByKey，写出代码示例并分析性能差异。

**答案**：
```python
from pyspark import SparkContext

sc = SparkContext()

data = [("a", 1), ("b", 2), ("a", 3), ("b", 4), ("a", 5)]

# ===== reduceByKey =====
# 原理: 先在Map端做预聚合(Combine)，再Shuffle
rdd_reduce = sc.parallelize(data) \
    .reduceByKey(lambda a, b: a + b)

# 执行过程:
# Map端预聚合（每个分区内先合并）:
#   Partition 1: (a,1), (b,2) → Combine → (a,1), (b,2)
#   Partition 2: (a,3), (b,4), (a,5) → Combine → (a,8), (b,4)
# Shuffle后Reduce:
#   a: [1, 8] → 9
#   b: [2, 4] → 6

# ===== groupByKey =====
# 原理: 直接将所有数据Shuffle到Reducer，在Reducer端聚合
rdd_group = sc.parallelize(data) \
    .groupByKey() \
    .mapValues(lambda values: sum(values))

# 执行过程:
# Map端不预聚合:
#   Partition 1: (a,1), (b,2)
#   Partition 2: (a,3), (b,4), (a,5)
# Shuffle: 全部数据发送到Reducer
#   a: [1, 3, 5] → sum → 9
#   b: [2, 4] → sum → 6

# 性能对比:
print("=== reduceByKey ===")
print(rdd_reduce.collect())  # [('b', 6), ('a', 9)]

print("=== groupByKey ===")
print(rdd_group.collect())   # [('b', 6), ('a', 9)]

"""
性能差异分析:

┌──────────────────┬──────────────────────┬──────────────────────┐
│      维度         │    reduceByKey        │     groupByKey      │
├──────────────────┼──────────────────────┼──────────────────────┤
│ Map端预聚合       │ ✓ (Combiner)         │ ✗                   │
│ Shuffle数据量     │ 少(预聚合后减少)      │ 多(全量Shuffle)      │
│ 内存使用          │ 低                   │ 高(需暂存所有Value)  │
│ OOM风险           │ 低                   │ 高                   │
│ 网络开销          │ 低                   │ 高                   │
│ 适用场景          │ 聚合操作(sum/count)  │ 需要访问所有Value    │
└──────────────────┴──────────────────────┴──────────────────────┘

最佳实践:
  1. 能用reduceByKey的地方绝不用groupByKey
  2. 如果确实需要访问所有Values，考虑使用aggregateByKey或combineByKey
  3. groupByKey + mapValues 可以替换为 reduceByKey/aggregateByKey
"""
```

### 练习4：共享变量（累加器与广播变量）

**题目**：解释累加器(Accumulator)和广播变量(Broadcast)的作用、使用场景和注意事项。

**答案**：
```python
from pyspark import SparkContext

sc = SparkContext()

# ===== 累加器 (Accumulator) =====
# 作用: 在分布式计算中聚合计数器和求和等操作

# 示例: 统计有效记录和空行数
valid_count = sc.accumulator(0)
empty_count = sc.accumulator(0)

lines = sc.parallelize([
    "user001,pv,2024-01-01",
    "",
    "user002,buy,2024-01-02",
    "",
    "user003,cart,2024-01-03"
])

def process_line(line):
    global valid_count, empty_count
    if line.strip() == "":
        empty_count.add(1)
        return None
    else:
        valid_count.add(1)
        return line.split(",")

result = lines.flatMap(lambda x: [process_line(x)] if process_line(x) else []).collect()

print(f"有效记录: {valid_count.value}")   # 有效记录: 3
print(f"空行: {empty_count.value}")        # 空行: 2

# 累加器注意事项:
# 1. Executor只能add，不能读取值（只能Driver读取.value）
# 2. 在Transformation中: 作业失败重试可能导致重复计数
# 3. 在Action中: 每次Action都会累加一次
# 4. 自定义累加器需要实现AccumulatorV2接口

# ===== 广播变量 (Broadcast) =====
# 作用: 将只读变量缓存到每个Executor的内存中，避免重复传输

# 示例: 将小维度表广播
user_map = {1: "Alice", 2: "Bob", 3: "Charlie", 4: "Diana"}
bc_users = sc.broadcast(user_map)

order_data = sc.parallelize([
    (1, 100.0), (1, 200.0),
    (2, 150.0), (3, 300.0),
    (1, 50.0),  (4, 80.0)
])

def enrich_with_name(record):
    user_id, amount = record
    name = bc_users.value.get(user_id, "Unknown")
    return (name, amount)

result = order_data.map(enrich_with_name).collect()
for r in result:
    print(r)

# 广播变量注意事项:
# 1. 数据从Driver广播到所有Executor
# 2. 广播变量是只读的，任务中不能修改
# 3. 广播后建议调用 unpersist() 释放内存
# 4. 适用于小维表、配置信息、机器学习模型参数等
# 5. 广播变量大小: 建议<几百MB（受Executor内存限制）

# 释放广播变量
bc_users.unpersist()
sc.stop()
```

### 练习5：DAG与Stage划分

**题目**：根据以下代码，画出DAG图并划分Stage。

```python
rdd1 = sc.textFile("hdfs://data.txt")
rdd2 = rdd1.flatMap(lambda x: x.split())
rdd3 = rdd2.map(lambda x: (x, 1))
rdd4 = rdd3.reduceByKey(lambda a, b: a + b)
rdd5 = rdd4.filter(lambda x: x[1] > 3)
rdd6 = rdd5.join(small_rdd)
rdd6.collect()
```

**答案**：
```
DAG图和Stage划分:

Stage 0 (包含: textFile → flatMap → map → reduceByKey的Map端):
┌─────────┐    ┌─────────┐    ┌─────────┐
│textFile  │ → │flatMap   │ → │  map    │
└─────────┘    └─────────┘    └─────────┘
                                    │
                           Shuffle Write
                           (partition by key hash)
──────────────────────────────────────────────────
                           Shuffle Read
                                    │
Stage 1 (包含: reduceByKey的Reduce端 → filter):
┌──────────────┐    ┌─────────┐
│ reduceByKey   │ → │ filter  │
└──────────────┘    └─────────┘
                           │
                    ┌──────┴──────┐
                    │   (No Shuffle if small_rdd is broadcasted)   │
                    └──────┬──────┘
                           │
Stage 2 (包含: join → collect):
┌─────────┐    ┌─────────┐
│  join    │ → │ collect │
└─────────┘    └─────────┘

Stage划分规则:
  1. 遇到宽依赖（如reduceByKey, groupByKey, join等Shuffle操作）
     → 切分为新的Stage
  2. 窄依赖的操作（如map, filter, flatMap）可以在同一Stage内
     pipeline（流水线）执行
  
  本例中:
  - reduceByKey产生宽依赖 → Stage 0和Stage 1的分界
  - join: 如果small_rdd被广播 → 窄依赖，在同一Stage
         如果两个RDD都大 → 产生宽依赖，切新Stage

  Shuffle过程:
  Stage 0 → Shuffle Write (写本地磁盘) → Stage 1
  Stage 1 → Shuffle Read (从各节点拉取)
```

class Median:
    """计算中位数（简单实现）"""
    def __init__(self):
        self.values = []

    def update(self, value):

            self.values.append(value)

        self.values.extend(other.values)
        return self

    def finish(self):

            return None
        self.values.sort()
        n = len(self.values)

### 练习5：Catalyst优化器

**题目**：解释Spark SQL的Catalyst优化器的4个阶段。

**答案**：
```
Catalyst优化器4阶段:

阶段1: 分析 (Analysis) — 解析和绑定
  输入: 未解析的逻辑计划 (Unresolved Logical Plan)
  处理:
    1. 使用Catalog解析表名、列名（resolveRelations, resolveReferences）
    2. 确定数据类型（resolveTypes）
    3. 检查函数是否存在
    4. 检查列是否在表中
  输出: 已解析的逻辑计划 (Resolved Logical Plan)

阶段2: 逻辑优化 (Logical Optimization) — 规则优化
  输入: 已解析的逻辑计划
  处理（应用一系列优化规则）:
    1. 谓词下推 (Predicate Pushdown): WHERE → 数据源层
    2. 列裁剪 (Column Pruning): SELECT → 只读需要的列
    3. 常量折叠 (Constant Folding): 编译时计算常量表达式
    4. 投影合并/消除: 消除无用子查询
    5. 布尔表达式简化: (a AND a) → a
    6. Join重排序: 先Join小表
  输出: 优化后的逻辑计划 (Optimized Logical Plan)

阶段3: 物理计划 (Physical Planning) — 策略选择
  输入: 优化后的逻辑计划
  处理:
    1. 为每个逻辑算子选择物理算子
       - Join: BroadcastHashJoin / SortMergeJoin / ShuffledHashJoin
       - 聚合: HashAggregate / SortAggregate
    2. 为每个物理算子生成执行策略
    3. 生成多个候选物理计划
  输出: 多个物理计划 (Physical Plans)

阶段4: 代码生成 (Code Generation) — 字节码生成
  输入: 选中的物理计划
  处理（Whole-Stage Code Generation）:
    1. 将多个算子合并为一个Java函数（函数折叠）
    2. 使用Janino编译器生成JVM字节码
    3. 减少虚函数调用和内存分配
  输出: 最终可执行的RDD DAG

示例:

原始SQL:
  SELECT name, SUM(salary * 1.1)
  FROM employees
  WHERE age > 30 AND dept = 'IT'
  GROUP BY name

经过Catalyst优化:
  分析阶段: 解析表名employees、列名name/salary/age/dept
  逻辑优化: 
    - 谓词下推: age>30 AND dept='IT' → 在扫描时过滤
    - 常量折叠: salary * 1.1 → 编译时处理
    - 列裁剪: 只读取name, salary, age, dept（不读其他列）
  物理计划: 
    - 选择HashAggregate（内存足够时）
  代码生成:
    - 生成一个函数: scan + filter + project + aggregate → 一次执行
```

### 练习4：资源配置调优

**题目**：为一个Spark作业推荐资源配置（数据量：100GB）。

**答案**：
```
场景: 处理100GB数据（Parquet格式），包含Shuffle操作

推荐配置:
  --driver-memory 4g              # Driver内存4GB
  --driver-cores 2                # Driver核数2
  --executor-memory 8g            # 每个Executor 8GB
  --executor-cores 4              # 每个Executor 4核
  --num-executors 10              # 共10个Executor
  --conf spark.executor.memoryOverhead=2g  # 堆外内存2GB

计算依据:
  总资源: 10×4 = 40 cores, 10×(8+2) = 100GB 内存
  分区数: 100GB / 128MB = 800 partitions（建议）

Executor内存分配:
  8GB JVM堆内存的分配:
    - Spark内部预留: 约2.4GB (spark.memory.fraction=0.6, storage=0.5)
      └── Storage内存: 1.2GB (缓存RDD/DataFrame)
      └── Execution内存: 1.2GB (Shuffle/Aggregation执行内存)
    - 用户内存: 约1.6GB (spark.memory.fraction剩余)
    - Reserved: 约300MB
    预留了约4GB供实际JVM开销

  1. 小Executor(1核4GB): Task并行度不足，需太多Executor

关键配置参数:
  spark.executor.memoryOverhead: YARN模式下堆外内存
    = max(executorMemory * 0.1, 384MB)
  
  spark.sql.shuffle.partitions: Shuffle分区数
    = 数据量 / 目标分区大小
    100GB/128MB ≈ 800
    
  spark.default.parallelism: RDD默认并行度
    = executor数 × cores × 2-3
    = 10 × 4 × 2 = 80
```
