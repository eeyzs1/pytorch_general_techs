# 系统设计题8：数据血缘追踪系统

## 一、需求分析

### 1.1 功能性需求

| 维度 | 需求详情 |
|------|----------|
| **血缘类型** | 表级血缘（表→表的依赖关系）、列级血缘（字段→字段的转换关系）、作业级血缘（ETL任务→输入表→输出表） |
| **覆盖范围** | Hive SQL（主要）、Spark SQL/PySpark、Flink SQL、数据管道（Kafka→Flink→ClickHouse）、BI报表→底层表 |
| **数据规模** | 5000+表，20000+列，日5000+ ETL任务 |
| **查询能力** | 正向追踪（此表影响下游哪些表/报表）、反向溯源（此报表数据来自上游哪些表/字段） |
| **变更影响分析** | 修改表A，自动分析影响多少下游表/任务/报表 |
| **用户角色** | 数据开发（查看血缘）、架构师（影响分析）、数据治理（冗余分析）、业务方（理解数据来源） |

### 1.2 非功能性需求

```
性能:
  - 血缘解析: 1000行SQL < 1秒
  - 血缘查询: < 500ms (10层深度的关系查询)
  - 影响分析: < 1秒 (1000级下游)

可用性:
  - 血缘服务: 99.9%
  - 元数据采集: 不阻塞正常ETL流程

准确性:
  - 表级血缘准确率 > 99%
  - 列级血缘准确率 > 95%（SQL解析的固有局限）

扩展性:
  - 支持新增计算引擎（只需添加新的SQL解析器）
  - 支持自定义血缘关系（手动声明）
```

- Atlas社区推进缓慢，版本更新不活跃
- HBase索引在大规模场景下性能衰减

## 五、最终选择：方案B（DataHub） + SQLGlot列级血缘增强

### 选择理由

**1. DataHub是LinkedIn在Atlas基础上吸取教训的新一代产品**
DataHub解决了Atlas的运维复杂度问题（去掉了HBase/Solr的强依赖），同时提供了现代化的Web UI体验。LinkedIn在数据治理领域有深厚的积累。

**2. 列级血缘是关键差异化**
我们的核心场景是"修改表A的字段X，评估影响范围"。Atlas仅支持表级血缘无法满足这个需求。DataHub可以通过Ingestion Framework集成SQLGlot实现列级血缘，补全这个能力。

**3. 社区活力**
DataHub是当前数据治理领域最活跃的开源项目之一，GitHub 10k+ Stars，有商业化公司Acryl Data背后的持续投入。Atlas社区已经显著放缓。

**4. 渐进式落地**
先部署DataHub实现表级血缘+数据发现，再逐步通过SQLGlot增强列级血缘。相比自研方案，DataHub提供了更多开箱即用的能力（数据搜索、标签、Owner管理等）。

## 六、核心架构设计

### 6.1 SQLGlot列级血缘解析

```python
import sqlglot
from sqlglot import exp
from sqlglot.lineage import lineage

def extract_column_lineage(sql: str, default_db: str = "dwd"):
    """解析SQL，提取列级血缘"""
    
    # 解析SQL为AST
    parsed = sqlglot.parse_one(sql)
    
    # 获取所有源表
    source_tables = []
    for table in parsed.find_all(exp.Table):
        db = table.db or default_db
        source_tables.append(f"{db}.{table.name}")
    
    # 获取目标表 (INSERT/CREATE TABLE AS)
    target_table = None
    if isinstance(parsed, exp.Insert):
        target_table = str(parsed.this.this)
    elif isinstance(parsed, exp.Create):
        target_table = str(parsed.this.this)
    
    # 使用sqlglot.lineage分析列级血缘
    if target_table:
        lin = lineage(
            sql, 
            schema={t: {"columns": {}} for t in source_tables},
            dialect="hive"  # 支持Hive SQL方言
        )
        
        # 提取列级映射
        column_mappings = {}
        for col_name, col_lineage in lin.items():
            column_mappings[col_name] = [
                str(path) for path in col_lineage
            ]
        
        return {
            "target_table": target_table,
            "source_tables": source_tables,
            "column_lineage": column_mappings
        }
    
    return None

# 使用示例
sql = """
INSERT INTO ads.user_order_summary
SELECT 
    u.user_id,
    u.user_name,
    SUM(o.amount) as total_amount,
    COUNT(o.order_id) as order_count
FROM dwd.users u
JOIN dwd.orders o ON u.user_id = o.user_id
WHERE o.dt = '2024-01-15'
GROUP BY u.user_id, u.user_name
"""

result = extract_column_lineage(sql)
# 输出: {
#   "target_table": "ads.user_order_summary",
#   "source_tables": ["dwd.users", "dwd.orders"],
#   "column_lineage": {
#     "user_id": ["dwd.users.user_id"],
#     "user_name": ["dwd.users.user_name"],
#     "total_amount": ["dwd.orders.amount"],
#     "order_count": ["dwd.orders.order_id"]
#   }
# }
```

### 6.3 Neo4j血缘图查询

```cypher
// 查询1: 正向追踪 - 表A影响哪些下游表/报表？
MATCH (source:Table {name: 'dwd.orders'})-[:PRODUCES*1..10]->(downstream)
RETURN downstream.name, downstream.type, length(path) AS distance
ORDER BY distance

// 查询2: 反向溯源 - 报表X的数据来自上游哪些表和字段？
MATCH (report:Chart {name: 'GMV日报'})-[:CONSUMES*1..10]->(upstream)
WHERE upstream.type IN ['Table', 'Column']
RETURN upstream.name, upstream.type

// 查询3: 影响分析 - 删除字段orders.amount影响范围
MATCH (col:Column {name: 'dwd.orders.amount'})-[:DERIVED_FROM*1..10]->(downstream)
RETURN downstream.name, downstream.type

// 查询4: 冗余检测 - 找出被多个任务重复计算的表
MATCH (t:Table)<-[:CONSUMES]-(job:Job)
WITH t, count(job) AS consumer_count
WHERE consumer_count >= 3
RETURN t.name, consumer_count
ORDER BY consumer_count DESC
```

## 七、数据采集设计

### 7.1 SQL采集方案

```
采集方式优先级:

P0: 解析SQL文件 (Git仓库 / 调度系统)
  覆盖: 所有ETL SQL
  准确率: 最高 (静态分析完整SQL)
  局限: 只覆盖代码仓库中的SQL

P1: Hive/Spark Hook (运行时采集)
  覆盖: Ad-hoc查询
  准确率: 高 (运行时Input/Output表关系精确)
  局限: 列级血缘缺失

P2: Metastore审计日志
  覆盖: 所有Hive表访问记录
  准确率: 中等 (仅能推断表级关系)
  局限: 时序关系推断可能存在误判
```

## 八、监控与告警

```
血缘采集:
  - 日采集成功率 (告警: < 95%)
  - 血缘更新延迟 (告警: T+1 > 12:00)
  - SQL解析失败率 (告警: > 5%)

血缘质量:
  - 孤立节点比例 (无上下游血缘的表, 告警: > 10%)
  - 循环依赖检测 (告警: 发现即告警)
  - 血缘深度异常 (链路超过20层告警)

图数据库:
  - Neo4j查询延迟 (告警: > 1秒)
  - 图数据库磁盘使用率
  - 节点/边数量异常增长
```
