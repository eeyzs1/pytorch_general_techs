# 系统设计题1：日处理100TB的实时数仓

## 一、需求分析

### 1.1 功能性需求

| 维度 | 需求详情 |
|------|----------|
| **数据源** | MySQL业务数据库（订单、用户、商品）、Nginx访问日志、IoT传感器数据 |
| **数据量** | 日处理100TB原始数据，覆盖300+张业务表，5000+埋点事件类型 |
| **时效性** | DWD明细层5分钟内可见；DWS/ADS汇总层15分钟内可见 |
| **存储周期** | 原始数据7天，DWD层30天，DWS层1年，ADS层3年 |
| **查询要求** | 固定报表查询<3秒；自助分析Hive/Spark SQL<30秒；实时大屏<1秒刷新 |

### 1.2 非功能性需求

```
性能:
  - 端到端延迟: DWD < 5分钟, ADS < 15分钟
  - 查询延迟: P99 < 3秒（报表）, P95 < 30秒（自助分析）
  - 写入吞吐: 峰值 50万条/秒

可用性:
  - 整体SLA: 99.9%（月宕机 < 43分钟）
  - 核心链路（实时大屏）: 99.95%
  - RTO < 10分钟, RPO < 1分钟

可扩展性:
  - 数据量年增长 50%
  - 支持水平扩展，新增节点后自动负载均衡

一致性:
  - 实时链路: 最终一致性（延迟 < 5分钟）
  - 离线链路: 每日对账确保 T+1 最终一致
  - 关键指标（GMV）: 实时和离线对账差异 < 0.1%

安全性:
  - Kerberos + Ranger 认证授权
  - PII数据脱敏（手机号、身份证）
  - 全链路审计日志
```

## 二、容量估算

### 2.1 数据量估算

```
每日数据增量:
  MySQL CDC (订单/用户/商品): 500万条 × 2KB = 10GB/天
  MySQL CDC (日志/流水): 2亿条 × 1KB = 200GB/天
  Nginx访问日志: 20亿条 × 300B = 600GB/天
  IoT传感器数据: 100亿条 × 100B = 1TB/天
  埋点事件日志: 50亿条 × 500B = 2.5TB/天
  第三方API数据: 100万条 × 5KB = 5GB/天
  ────────────────────────────────────────────
  原始数据日增量: ~4.3TB/天（压缩后约1.5TB）
  考虑峰值（大促×3）: ~13TB/天原始 → ~4.5TB压缩
```

### 2.2 存储容量（3副本）

```
分层存储预估:
  ODS层（原始数据, 保留7天）:
    4.3TB × 7天 × 3副本 = 90TB

  DWD层（明细数据, 保留30天, 压缩比3:1）:
    1.5TB × 30天 × 3副本 = 135TB

  DWS层（汇总数据, 保留365天）:
    估算日汇总量: 100GB × 365天 × 3副本 = 110TB

  ADS层（应用数据, 保留3年）:
    估算日产出: 20GB × 1095天 × 3副本 = 66TB

  查询加速索引: 约50TB
  临时中间数据: 约100TB

  总存储需求: ~550TB（当前）, 预留1PB（考虑年增长50%）
```

### 2.3 计算资源估算

```
实时计算(Flink):
  处理吞吐: 50万条/秒峰值
  每条处理复杂度: 含3次JOIN + 5次聚合
  预估TM数量: 20个TaskManager × 8核16GB = 160核320GB

离线计算(Spark):
  日常跑批: 100个任务 × 平均4核8GB = 400核800GB
  大促峰值: ×2资源
  预留弹性: ×1.5

OLAP查询(ClickHouse/StarRocks):
  并发查询: 200 QPS峰值
  单节点8核32GB, 预估10节点集群

总计算资源:
  CPU: ~800核 (含弹性buffer)
  内存: ~2TB
```

### 2.4 网络带宽

```
数据入站:
  日均 4.3TB / 86400s ≈ 400 Mbps(平均)
  峰值(白天×3) ≈ 1.2 Gbps

数据出站（查询结果）:
  200 QPS × 平均1MB/查询 = 200MB/s ≈ 1.6 Gbps

推荐:
  入站: 万兆网卡 (10 Gbps)
  出站: 双万兆网卡绑定
```

## 五、最终选择：方案B（StarRocks + Hudi）

### 选择理由

**1. 多表JOIN是核心需求**
该数仓有300+张业务表，实时宽表构建需要频繁的多表JOIN（订单JOIN用户JOIN商品JOIN地区）。StarRocks的分布式Colocate Join和Bucket Shuffle Join在JOIN场景下性能远超ClickHouse的"先打宽表再查询"模式。

**2. CDC场景的Upsert需求**
MySQL CDC数据需要支持UPDATE/DELETE操作。Hudi的MoR（Merge on Read）表原生支持Upsert，配合Flink CDC可以实现分钟级的增量更新。而ClickHouse的ReplacingMergeTree在数据量大时，后台Merge压力很大。

**3. 查询层统一**
StarRocks既可以作为实时OLAP引擎，也可以通过External Catalog直接查询Hudi/Iceberg数据湖中的离线数据，实现一套SQL覆盖实时+离线。减少Trino这一层，降低架构复杂度。

**4. MySQL协议兼容**
StarRocks原生兼容MySQL协议和SQL语法，BI工具（Tableau/Superset/Metabase）可以直接连接，无需额外适配。数据分析师的学习成本几乎为零。

**5. 不过度设计的权衡**
虽然ClickHouse在单表聚合场景下性能更极致，但我们的场景需要大量JOIN。为了JOIN能力而增加ClickHouse + Trino的组合，不如用StarRocks一站式解决，牺牲少许单表性能换取更简洁的架构。

### 风险与应对

| 风险 | 应对方案 |
|------|----------|
| StarRocks写入吞吐在大促场景不足 | 设置合理的攒批参数+开启Pipeline并行写入 |
| Hudi社区不如Iceberg活跃 | 跟踪社区动态，核心引擎层用Flink抽象，切换成本可控 |
| 团队StarRocks经验不足 | 安排2周专项培训+引入外部咨询+POC验证期 |

### ADR 记录

```markdown
# ADR-002: 实时数仓OLAP引擎选择 StarRocks 而非 ClickHouse

## 状态
已采纳

## 决策
选择 StarRocks 作为实时OLAP引擎。

## 理由
```

## 六、核心数据模型设计

### 6.1 数仓分层

```
ODS层（原始数据）:
  ods_order_info_df          -- 订单主表（全量快照）
```

### 6.2 StarRocks 建表示例

```sql
-- DWS层：用户日订单汇总（StarRocks主键模型）
CREATE TABLE dws_trade_user_order_1d (
    dt DATE,
    user_id BIGINT,
    order_count BIGINT SUM DEFAULT '0',
    order_amount DECIMAL(20,2) SUM DEFAULT '0.00',
    first_order_time DATETIME MIN DEFAULT '9999-12-31',
    last_order_time DATETIME MAX DEFAULT '1970-01-01',
    paid_order_count BIGINT SUM DEFAULT '0',
    refund_order_count BIGINT SUM DEFAULT '0'
) ENGINE = OLAP
PRIMARY KEY(dt, user_id)
PARTITION BY RANGE(dt) (
    PARTITION p20240101 VALUES LESS THAN ('2024-01-02'),
    PARTITION p20240102 VALUES LESS THAN ('2024-01-03')
)
DISTRIBUTED BY HASH(user_id) BUCKETS 32
PROPERTIES (
    "replication_num" = "3",
    "storage_format" = "DEFAULT",
    "compression" = "LZ4"
);

-- ADS层：实时GMV（StarRocks聚合模型）
CREATE TABLE ads_gmv_realtime (
    dt DATE,
    hour TINYINT,
    minute TINYINT,
    category_id BIGINT,
    gmv DECIMAL(20,2) SUM DEFAULT '0.00',
    order_count BIGINT SUM DEFAULT '0',
    user_count HLL HLL_UNION DEFAULT '0'
) ENGINE = OLAP
AGGREGATE KEY(dt, hour, minute, category_id)
PARTITION BY RANGE(dt) ()
DISTRIBUTED BY HASH(category_id) BUCKETS 16
PROPERTIES (
    "replication_num" = "3"
);
```

## 七、关键链路设计

### 7.1 实时链路（延迟 < 5秒）

```
MySQL ──CDC──→ Kafka ──→ Flink ETL ──→ StarRocks
                    1. 字段标准化         Stream Load
                    2. 数据去重
                    3. 维表JOIN
                    4. 宽表输出
```

### 7.2 近实时链路（延迟 < 5分钟）

```
Kafka ──→ Flink SQL ──→ Hudi (MoR表)
              │
              ├── 1分钟Micro-batch写入Hudi
              ├── 生成增量Compaction
              └── StarRocks External Catalog查询Hudi
```

### 7.3 离线链路（T+1）

```
HDFS ──→ Spark ETL ──→ Hudi (CoW表) ──→ StarRocks External Catalog
数据文件     │
            ├── ODS → DWD (Spark SQL)
            ├── DWD → DWS (聚合)
            └── DWS → ADS (应用报表)
```

```
关键监控指标:
  数据延迟:
    - Kafka Consumer Lag（告警阈值: > 100万或增长速度 > 10万/分钟）

    - 行数对账: 实时vs离线差异（告警阈值: > 1%）

    - NULL值比例（告警阈值: > 5%）

  系统健康:
    - Flink Checkpoint失败率（告警阈值: > 10%）
    - StarRocks BE存活数（告警阈值: < 总节点数 - 1）
    - HDFS Block丢失数（告警阈值: > 0）
```

## 十一、数据一致性保障机制

### 11.1 实时vs离线对账架构

```
每日凌晨2:00执行对账流程:

┌──────────────────────┐
│   Step 1: 数据采样    │
│   随机抽取10万条/表   │
└────────┬─────────────┘
         │
┌────────▼─────────────┐
│ Step 2: 双源对比      │
│ 实时链路(StarRocks)   │←→ 离线链路(Hudi)
│   vs 离线链路(Hudi)   │
└────────┬─────────────┘
         │
┌────────▼─────────────┐
│ Step 3: 差异分析      │
│ 按表统计:             │
│  - 行数差异           │
│  - 聚合值差异(GMV)    │
│  - 抽样对比差异       │
└────────┬─────────────┘
         │
┌────────▼─────────────┐
│ Step 4: 差异修复      │
│ 差异>0.1%: 手动排查   │
│ 差异<0.1%: 自动修复   │
└──────────────────────┘
```

### 11.2 对账SQL实现

```sql
-- 1. 行数对账
SELECT 
    'dwd_order_fact' AS table_name,
    (SELECT COUNT(*) FROM starrocks.dwd.dwd_order_fact WHERE dt = '${yesterday}') AS rt_count,
    (SELECT COUNT(*) FROM hive.dwd.dwd_order_fact WHERE dt = '${yesterday}') AS offline_count,
    ABS(rt_count - offline_count) AS diff,
    CASE WHEN ABS(rt_count - offline_count) / offline_count > 0.001 THEN 'FAIL' ELSE 'PASS' END AS result;

-- 2. GMV对账
SELECT
    (SELECT SUM(order_amount) FROM starrocks.dws.dws_trade_order WHERE dt = '${yesterday}') AS rt_gmv,
    (SELECT SUM(order_amount) FROM hive.dws.dws_trade_order WHERE dt = '${yesterday}') AS offline_gmv,
    ABS(rt_gmv - offline_gmv) / offline_gmv AS diff_rate,
    CASE WHEN ABS(rt_gmv - offline_gmv) / offline_gmv > 0.001 THEN 'FAIL' ELSE 'PASS' END AS result;

-- 3. 抽样明细对比
WITH rt_sample AS (
    SELECT order_id, order_amount, user_id, product_id
    FROM starrocks.dwd.dwd_order_fact 
    WHERE dt = '${yesterday}' 
    ORDER BY RAND() LIMIT 10000
),
offline_sample AS (
    SELECT order_id, order_amount, user_id, product_id
    FROM hive.dwd.dwd_order_fact 
    WHERE dt = '${yesterday}' 
    ORDER BY RAND() LIMIT 10000
)
SELECT 
    COUNT(DISTINCT r.order_id) AS rt_unique_orders,
    COUNT(DISTINCT o.order_id) AS offline_unique_orders,
    COUNT(*) FILTER(WHERE r.order_id = o.order_id) AS matched_orders,
    COUNT(*) FILTER(WHERE r.order_id = o.order_id AND r.order_amount = o.order_amount) AS exact_matched
FROM rt_sample r
FULL OUTER JOIN offline_sample o ON r.order_id = o.order_id;
```

### 11.3 数据回补策略

```
场景分类:
  1. 实时链路遗漏数据（Kafka消费Lag过大导致跳过）:
     → 扫描Hudi离线表，补充StarRocks缺失数据
     → 触发条件: 对账差异 > 1000行/表

  2. 离线链路延迟（Spark任务失败）:
     → 实时链路数据作为临时可信源
     → 离线链路恢复后，覆盖StarRocks数据

  3. 两端数据同时丢失（极端情况）:
     → 回放MySQL Binlog（保留7天）
     → 重新消费Kafka原始消息

回补工具:
  StarRocks补数据: INSERT INTO ... SELECT FROM hive_catalog.dwd.xxx WHERE dt = '${date}'
  Hudi补数据: Spark任务重跑指定分区
```

## 十二、10个故障场景处理SOP

### 故障1：Kafka Broker宕机

```
严重级别: P0
影响范围: 所有实时链路中断
检测方式: Kafka Under-Replicated Partitions > 0 且持续 > 2分钟

处理SOP:
  1min: 确认宕机节点数量（< ISR min则拒绝写入）
  2min: 检查是否自动Failover（ISR中其他Broker接手Leader）
  3min: 若未自动恢复 → 手动将宕机Broker Partition迁移到健康节点
  5min: 恢复写入 + 监控Consumer Lag回追速度
  10min: 检查无数据丢失

预防措施:
  - replication.factor=3, min.insync.replicas=2
  - Broker跨机架部署
  - 定期演练Broker宕机恢复
```

### 故障2：Flink Checkpoint持续失败

```
严重级别: P0
影响范围: 故障恢复后无法从Savepoint恢复，需从最早Offset重放
检测方式: Flink Checkpoint失败率 > 10% 持续5分钟

处理SOP:
  1min: 确认失败原因（State Backend磁盘满/RocksDB异常/网络超时）
  3min: 扩容Checkpoint存储空间或重启RocksDB
  5min: 触发手动Savepoint + 停作业
  10min: 从Savepoint恢复作业
  15min: 确认消费恢复正常

预防措施:
  - Checkpoint超时 < 10分钟
  - 保留最近10个成功的Checkpoint
  - 每日自动触发Savepoint
```

### 故障3：StarRocks BE节点宕机

```
严重级别: P1
影响范围: 查询性能下降（数据副本重新分布）
检测方式: SHOW PROC '/backends' 中 Alive=false

处理SOP:
  1min: 确认宕机BE数量（< 副本数则数据完整）
  3min: 检查是否自动Failover（FE自动调度副本迁移）
  5min: 若未自动恢复 → 手动Decommission宕机BE
  10min: 观察副本迁移进度
  30min: 副本迁移完成，集群恢复正常

预防措施:
  - replication_num=3
  - 监控磁盘/内存/网络使用率，提前预警
```

### 故障4：MySQL Binlog风暴

```
严重级别: P1
影响范围: Kafka消息挤压，端到端延迟增加
检测方式: Kafka Topic消息速率 > 正常值3倍 且 持续时间 > 5分钟

处理SOP:
  1min: 确认Binlog暴增原因（大批量更新/DDL操作）
  3min: 临时增加Kafka Partition（若允许动态扩容）
  5min: 扩容Flink TM（水平扩展消费能力）
  10min: 开启Kafka消息过期策略（临时缩短保留时间）

预防措施:
  - 业务方大批量操作需提前通知
  - Kafka Partition预留30% buffer
  - Flink Auto Scaling（K8s部署时）
```

### 故障5：HDFS NameNode脑裂

```
严重级别: P0
影响范围: 所有离线任务中断，数据不可读写
检测方式: 两个NameNode都声称Active

处理SOP:
  1min: 立即隔离两个NameNode（断网一个）
  3min: 确定哪个是真正的Active（基于JournalNode最新edit log）
  5min: 将错误Active强制转为Standby
  10min: 重启整个HDFS集群验证
  20min: 恢复离线任务

预防措施:
  - NameNode HA + QJM
  - ZooKeeper自动Failover（不要手动触发）
  - 定期验证HA切换功能
```

### 故障6：数据倾斜导致Flink Task反压

```
严重级别: P2
影响范围: 部分Key数据处理延迟
检测方式: Flink BackPressure Status = HIGH 且 单个SubTask处理时间是中位数的5倍

处理SOP:
  1min: 识别倾斜Key（通过Flink Metrics查看各SubTask处理量）
  3min: 临时方案: 增加并行度（稀释倾斜SubTask）
  5min: 根本方案: 对倾斜Key加盐分桶（salt_key = key + random(0,N)）
  15min: 重启Flink作业（使用Savepoint恢复）

预防措施:
  - 使用Flink SQL的LOCAL聚合（两阶段聚合）
  - 对已知热点Key（如大促活动ID）预设加盐策略
```

### 故障7：StarRocks查询雪崩

```
严重级别: P1
影响范围: 所有查询超时，大屏断流
检测方式: StarRocks查询队列深度 > 1000 且 P99延迟 > 30秒

处理SOP:
  1min: 识别慢查询来源（SHOW PROCESSLIST）
  3min: Kill慢查询（KILL QUERY <query_id>）
  5min: 开启查询限流（FE配置: query_queue_capacity）
  10min: 分析慢查询原因（全表扫描/无分区裁剪/JOIN爆炸）
  30min: 添加查询超时限制 + 优化慢查询

预防措施:
  - 查询超时默认300秒
  - 物化视图预聚合
  - 大屏走预计算结果表，不走实时聚合
```

### 故障8：Hudi Compaction失败

```
严重级别: P2
影响范围: MOR表查询性能逐渐下降
检测方式: Hudi Timeline上Compaction Plan长时间Pending

处理SOP:
  1min: 检查Compaction失败日志（磁盘空间/RocksDB异常）
  3min: 手动触发Compaction: hoodie-cli compaction schedule && compaction run
  5min: 若手动失败 → 增加Compaction资源(YARN/Spark executor)
  15min: 监控Compaction完成进度

预防措施:
  - 异步Compaction（Inline Compaction=false）
  - 设置Compaction触发阈值（Delta Commit数 > 5）
  - 监控Compaction队列长度
```

### 故障9：跨表对账GMV差异 > 1%

```
严重级别: P0
影响范围: 实时大屏数据不准确，可能影响运营决策
检测方式: 每日对账报告

处理SOP:
  1min: 确认差异范围（是否所有日期都差异/仅今天）
  3min: 定位差异来源:
    - 对比源端MySQL与Kafka消息数（是否丢失）
    - 对比Kafka消费进度与写入StarRocks数（是否消费遗漏）
    - 对比Hudi与StarRocks按小时数据（定位具体小时）
  5min: 临时修复: 用Hudi离线数据覆盖StarRocks
  15min: 根因修复: 根据差异来源修复管道
  30min: 通知业务方数据已修正

预防措施:
  - 实时对账Dashboard（每小时自动对比）
  - 关键业务指标双重校验（GMV同时计算实时+离线两路）
  - 数据异常自动告警（小时级波动 > 10%）
```

### 故障10：全链路网络分区

```
严重级别: P0
影响范围: 所有系统无法通信
检测方式: 各组件间心跳超时

处理SOP:
  1min: 确认网络分区范围（是交换机故障/光纤中断/路由错误）
  3min: 通知网络团队紧急排查
  5min: 确认各组件是否进入安全模式:
    - Kafka: 停止接受写入（ISR不足）
    - HDFS: 进入Safe Mode
    - Flink: 全部Failover
  10min: 网络恢复后，按顺序重启:
    - HDFS退出Safe Mode
    - Kafka恢复
    - Flink从Savepoint恢复
    - StarRocks恢复写入
  30min: 全链路健康检查 + 数据丢失验证

预防措施:
  - 核心交换机双活
  - 网络链路冗余（至少2条物理路径）
  - 网络监控 + 自动Failover
```
