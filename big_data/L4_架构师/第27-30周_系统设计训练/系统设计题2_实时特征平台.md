# 系统设计题2：支持10万QPS的实时特征平台

## 一、需求分析

### 1.1 功能性需求

| 维度 | 需求详情 |
|------|----------|
| **特征类型** | 用户特征（画像标签）、商品特征（热度/评分）、上下文特征（时间/位置/天气）、交叉特征（用户-商品交互） |
| **特征来源** | MySQL业务库、Kafka实时日志、Hive离线宽表、第三方API |
| **更新频率** | 实时特征 < 1秒；近线特征 < 5分钟；离线特征 T+1 |
| **查询模式** | 在线点查（推荐/广告/搜索）：单次请求查询100-500个特征，P99 < 10ms |
| **QPS** | 峰值10万QPS，均值3万QPS |
| **特征数量** | 初期5000+特征，远期20000+特征 |

### 1.2 非功能性需求

```
性能:
  - 在线查询: P99 < 10ms, P999 < 20ms
  - 特征更新: 实时特征写入到可查询延迟 < 1秒
  - 特征同步: 离线特征批量导入 < 1小时

可用性:
  - 在线服务: 99.99%（年宕机 < 52分钟）
  - 离线计算: 99.9%
  - 降级策略: 关键特征Redis兜底，非关键特征返回默认值

一致性:
  - 在线特征: 最终一致性（< 1秒延迟）
  - 离线特征: T+1强一致（每日全量对账）
  - 特征版本: 支持回滚到前N个版本

扩展性:
  - 水平扩展：支持50万QPS远期目标
  - 特征数量：从5000扩展到20000+无性能衰减
```

## 二、容量估算

### 2.1 数据量

```
特征存储:
  用户数量: 2亿
  每用户特征数: 5000
  每特征平均大小: 8 bytes
  单用户特征数据: 5000 × 8B = 40KB
  总数据量: 2亿 × 40KB = 8TB
  副本数: 3
  实际存储: 8TB × 3 = 24TB

写入流量:
  日新增/更新特征: 5000万用户 × 100特征变化 × 8B = 40GB/天
  峰值写入: 10万QPS × 50特征/次 × 8B = 40MB/s
```

### 2.2 QPS分析

```
查询QPS:
  推荐场景: 5万QPS × 200特征/次 = 1000万特征读取/秒
  广告场景: 3万QPS × 300特征/次 = 900万特征读取/秒
  搜索场景: 2万QPS × 100特征/次 = 200万特征读取/秒
  合计: 2100万特征读取/秒峰值

点查性能目标:
  单次查询100-500特征，P99 < 10ms
  → 需要单节点支撑: 5000 QPS × 500特征 = 250万特征读取/秒
  → 集群规模: 2100万 / 250万 = 8-10节点（加buffer用15节点）
```

## 三、候选方案对比

### 方案A：Redis Cluster + Feature Store (Feast)

```
┌──────────────────────────────────────────────────────────┐
│                    离线特征计算                           │
│  Hive ──→ Spark ──→ Feast Offline Store (Parquet)       │
│                              │                           │
│                    Feast Materialize                     │
│                              │                           │
│                              ▼                           │
│                    Feast Online Store                    │
│                    (Redis Cluster)                       │
│                    ◄──────────                          │
│                    ┌──────────┘                         │
│  Flink ──→ Redis   │  特征查询 API                       │
│  (实时特征写入)      │  P99 < 10ms                        │
└──────────────────────────────────────────────────────────┘
```

**优点**:
- Redis P99延迟 < 1ms，性能极佳
- Feast提供特征注册、血缘、版本管理能力
- Redis Cluster原生支持水平扩展
- 社区成熟，运维经验丰富

**缺点**:
- Redis纯内存存储，24TB数据内存成本极高（~$50万/月）
- 无特征自动过期机制，需手动管理TTL
- 特征变更需要全量重建，运维成本高
- Redis Cluster重分片时影响可用性

### 方案B：Aerospike + 自研Feature Store

```
┌──────────────────────────────────────────────────────────┐
│                    离线特征计算                           │
│  Spark ──→ 批量写入 Aerospike (SSD + 内存混合存储)       │
│                                                          │
│                    实时特征计算                           │
│  Flink ──→ Aerospike (实时更新)                          │
│                                                          │
│                    特征查询                               │
│  Aerospike Client ──→ P99 < 5ms                         │
│                                                          │
│             存储: SSD为主 + 内存缓存                      │
│             成本: ~$8万/月                               │
└──────────────────────────────────────────────────────────┘
```

**优点**:
- SSD+内存混合存储，成本比纯内存方案低60%+
- 单集群可支撑百万级TPS
- 强一致性配置下P99 < 5ms
- 支持自动数据过期(TTL)和数据压缩

**缺点**:
- 社区版功能受限（如XDR跨集群复制需企业版）
- 开源生态不如Redis丰富
- 运维需要专业技能，人才市场招聘困难
- 中文文档和社区支持少

### 方案C：自研KV存储 + RocksDB

```
┌──────────────────────────────────────────────────────────┐
│                    离线 + 实时特征计算                    │
│           Flink / Spark ──→ 自研KV存储                   │
│                                                          │
│              自研KV存储 (C++ / Rust)                     │
│    ┌──────────────────────────────────────┐             │
│    │  内存热缓存(LRU) → 90%命中率          │             │
│    │  RocksDB(SSD持久化) → 全量数据        │             │
│    │  Raft共识 → 多副本强一致               │             │
│    │  gRPC接口 → 特征查询                  │             │
│    └──────────────────────────────────────┘             │
│            P99 < 5ms (缓存命中) / < 20ms (IO)            │
│            成本: ~$3万/月                                │
└──────────────────────────────────────────────────────────┘
```

**优点**:
- 技术自主可控，无供应商锁定
- 成本最优（SSD存储，内存仅做缓存）
- 可深度定制的查询优化（如特征批量预取）
- 长期技术壁垒积累

**缺点**:
- 开发周期长（6-12个月），人力成本高（需3-5人专职团队）
- 需要自建监控、运维、故障恢复体系
- 当公司不擅长底层系统开发时，风险极高
- 与大厂方案（如字节ByteKV）有代差

## 五、最终选择：混合方案（方案B + 方案A的Feast）

### 选择理由

**1. 存储层选Aerospike（方案B）**
- 24TB特征数据如果用纯内存Redis，成本约50万/月，不可接受
- Aerospike SSD+内存混合存储，成本约8万/月，性价比最优
- P99 < 5ms满足10ms SLA要求
- 原生支持水平扩展和自动数据过期

**2. 特征管理层选Feast**
- Feast提供特征注册、元数据管理、特征血缘、版本回溯
- 无需自研特征管理系统
- 与TensorFlow/PyTorch等ML框架无缝集成
- 社区活跃，Feature Store已成为ML Infra标配

**3. 为什么不选方案C（自研）**
- 团队当前无底层系统开发经验，6-12个月开发周期风险太高
- 业务需要3个月内上线特征平台
- 自研KV投入3-5人，机会成本远超采购Aerospike企业版

## 六、核心数据模型

### 6.1 特征存储模型

```
Aerospike Key设计:
  Key: "feat:user:{user_id}"  (用户特征)
  Key: "feat:item:{item_id}"  (商品特征)
  Key: "feat:cross:{user_id}:{item_id}"  (交叉特征)

Value格式 (Protobuf):
  message FeatureBundle {
    int64 version = 1;              // 特征版本号
    int64 updated_at = 2;           // 更新时间戳
    map<string, FeatureValue> features = 3;  // 特征名→特征值
  }

  message FeatureValue {
    oneof value {
      int64 int_val = 1;
      double float_val = 2;
      string string_val = 3;
      IntList int_list_val = 4;
      FloatList float_list_val = 5;
    }
  }

Aerospike Namespace配置:
  namespace feature_store {
    replication-factor 3
    memory-size 64G          # 每个节点内存缓存
    storage-engine device {
      device /dev/nvme0n1    # NVMe SSD
      write-block-size 128K
    }
    default-ttl 30D          # 30天自动过期
  }
```

## 七、查询服务设计

### 7.1 查询优化策略

```
1. 批量合并查询:
   将同一用户的多特征请求合并为单次Aerospike查询
   
2. 多级缓存:
   L1: 本地内存缓存 (Caffeine, 容量1GB, TTL 60s)
       命中率: ~70%
   L2: Aerospike (SSD+内存)
       命中率: ~29%
   L3: 默认值降级 (缓存未命中+存储不可用)
       命中率: ~1%

3. 特征预取:
   推荐请求处理前，批量预取用户/商品/交叉特征

4. 连接池优化:
   长连接 + Async客户端 + Pipeline批量操作
```

### 7.2 典型查询流程

```
客户端请求: {user_id: 123, feature_names: ["age", "gender", "ltv", "active_days"]}

1. Feature Query Service收到请求
2. 拼接Key: "feat:user:123"
3. 查询本地缓存 → 命中则返回
4. 未命中 → 查询Aerospike
5. Aerospike返回全量特征bundle → 本地过滤需要的特征
6. 更新本地缓存
7. 返回结果
```

## 九、特征计算引擎详细对比

### 9.1 Flink vs Spark Streaming vs Storm vs 自研引擎

| **适合场景** | 实时特征+SQL | 近实时特征+ML | 简单流处理 | 核心链路极致性能 |

## 十、在线服务降级策略完整实现

### 10.1 降级层次设计

```
Level 0: 全功能模式
  - 所有特征从Aerospike获取
  - P99延迟 < 5ms
  - 特征完整度 100%

Level 1: 缓存优先模式 (Aerospike P99 > 10ms 或 错误率 > 1%时触发)
  - 热点特征从Caffeine本地缓存获取
  - 非热点特征从Aerospike获取
  - P99延迟 < 8ms
  - 特征完整度 100%

Level 2: 降级模式 (Aerospike集群不可用时触发)
  - 仅从Caffeine缓存获取
  - 缓存未命中的特征返回默认值
  - P99延迟 < 3ms
  - 特征完整度 ~85%

Level 3: 最小模式 (缓存也无法使用时触发)
  - 全部返回离线预计算的特征快照
  - P99延迟 < 2ms
  - 特征完整度 ~60%
  - 模型效果下降但服务可用

Level 4: 熔断模式 (全链路故障)
```

### 10.2 降级触发条件与自动切换

```java
// 降级状态机实现
public class DegradationController {
    
    // 滑动窗口监控
    private SlidingTimeWindow<Metrics> window = new SlidingTimeWindow<>(60_000, 10_000); // 1分钟窗口, 10秒滑动
    
    public DegradationLevel evaluateDegradation() {
        Metrics current = window.getCurrentMetrics();
        
        // Level 0 → Level 1: Aerospike延迟退化
        if (current.aerospikeP99LatencyMs > 10 || current.aerospikeErrorRate > 0.01) {
            log.warn("Aerospike degradation detected: P99={}ms, errorRate={}", 
                current.aerospikeP99LatencyMs, current.aerospikeErrorRate);
            return DegradationLevel.LEVEL_1_CACHE_PRIORITY;
        }
        
        // Level 1 → Level 2: Aerospike不可用
        if (current.aerospikeErrorRate > 0.05 || current.aerospikeUnavailableCount > 3) {
            log.error("Aerospike unavailable: errorRate={}", current.aerospikeErrorRate);
            return DegradationLevel.LEVEL_2_CACHE_ONLY;
        }
        
        // Level 2 → Level 3: 缓存命中率过低
        if (current.cacheHitRate < 0.5) {
            log.error("Cache hit rate too low: {}", current.cacheHitRate);
            return DegradationLevel.LEVEL_3_OFFLINE_SNAPSHOT;
        }
        
        // Level 3 → Level 4: 全链路故障
        if (current.offlineSnapshotAvailable == false) {
            log.error("Offline snapshot unavailable, entering circuit breaker mode");
            return DegradationLevel.LEVEL_4_CIRCUIT_BREAKER;
        }
        
        // 自动恢复检查
        if (current.aerospikeP99LatencyMs < 5 && current.aerospikeErrorRate < 0.001) {
            log.info("Aerospike recovered, restoring to full mode");
            return DegradationLevel.LEVEL_0_FULL;
        }
        
        return DegradationLevel.LEVEL_0_FULL;
    }
}
```

## 十一、特征一致性正确性验证方案

### 11.1 验证层次

```
层次1: 特征值对比 (离线 vs 在线)
  方式: 随机抽取1000个用户，对比离线Spark生成的T+1特征与在线Aerospike存储的实时特征
  阈值: 差异率 < 5%
  频率: 每日凌晨

层次2: 特征分布对比
  方式: 对比在线特征的全局分布 vs 离线特征的全局分布
  阈值: KL散度 < 0.1, JS散度 < 0.05
  频率: 每日凌晨

层次3: 模型效果一致性
  方式: 同一批用户，分别用在线特征和离线特征进行推荐打分，对比Top100推荐结果重叠率
  阈值: Top100重叠率 > 80%
  频率: 每周

层次4: 业务指标对比
  方式: A/B实验，对比使用在线特征vs离线特征的模型在线CTR/CVR
  阈值: CTR差异 < 5%
  频率: 每次特征变更时
```

### 11.2 验证实现

```python
# 特征一致性验证脚本
import numpy as np
from scipy.stats import entropy
from feast import FeatureStore

class FeatureConsistencyValidator:
    def __init__(self):
        self.fs = FeatureStore(repo_path="/data/feature_repo")
    
    def validate_point_check(self, feature_name, sample_size=10000):
        """点查验证: 随机用户对比离线vs在线特征值"""
        users = self._random_sample_users(sample_size)
        
        offline_features = self.fs.get_historical_features(
            entity_df=users,
            features=[f"user_features:{feature_name}"],
        ).to_df()
        
        online_features = self.fs.get_online_features(
            entity_rows=[{"user_id": uid} for uid in users],
            features=[f"user_features:{feature_name}"],
        ).to_df()
        
        merged = offline_features.merge(online_features, on="user_id", suffixes=("_offline", "_online"))
        
        # 数值特征对比
        mismatches = merged[merged[f"{feature_name}_offline"] != merged[f"{feature_name}_online"]]
        mismatch_rate = len(mismatches) / len(merged)
        
        return {
            "feature": feature_name,
            "sample_size": sample_size,
            "mismatch_rate": mismatch_rate,
            "passed": mismatch_rate < 0.05
        }
    
    def validate_distribution(self, feature_name):
        """分布验证: KL散度"""
        offline_dist = self._get_offline_distribution(feature_name)
        online_dist = self._get_online_distribution(feature_name)
        
        # 平滑处理避免log(0)
        offline_dist = offline_dist + 1e-10
        online_dist = online_dist + 1e-10
        
        kl_div = entropy(offline_dist, online_dist)
        js_div = 0.5 * (entropy(offline_dist, (offline_dist+online_dist)/2) + 
                        entropy(online_dist, (offline_dist+online_dist)/2))
        
        return {
            "feature": feature_name,
            "kl_divergence": kl_div,
            "js_divergence": js_div,
            "passed": js_div < 0.05
        }
    
    def validate_model_consistency(self, top_k=100):
        """模型效果一致性验证"""
        users = self._random_sample_users(10000)
        
        # 分别用在线和离线特征打分
        online_scores = self.model.score(users, source="online")
        offline_scores = self.model.score(users, source="offline")
        
        online_topk = set(online_scores[:top_k])
        offline_topk = set(offline_scores[:top_k])
        
        overlap = len(online_topk & offline_topk) / top_k
        
        return {
            "top_k": top_k,
            "overlap_rate": overlap,
            "passed": overlap > 0.8
        }
```

## 十二、热点特征缓存详细设计

### 12.1 热点识别算法

```java
/**
 * 基于W-TinyLFU的热点特征识别
 * 结合频率(Frequency Sketch)和时间衰减(Temporal Decay)
 */
public class HotFeatureDetector {
    
    // Count-Min Sketch用于频率估计
    private final CountMinSketch frequencySketch;
    
    // 时间衰减权重
    private final int windowSizeSeconds = 300; // 5分钟窗口
    private final double decayFactor = 0.5;    // 半衰期5分钟
    
    /**
     * 识别热点特征:
     * 1. Window Count > threshold (绝对频率高)
     * 2. Window Count / Global Count > ratio (相对频率高)
     */
    public Set<String> detectHotFeatures(int topN) {
        Map<String, WindowStats> stats = new HashMap<>();
        long currentTime = System.currentTimeMillis();
        
        for (String featureId : allFeatureIds) {
            long windowCount = frequencySketch.getCount(featureId, windowSizeSeconds);
            long globalCount = frequencySketch.getTotalCount(featureId);
            double recency = Math.pow(decayFactor, 
                (currentTime - lastAccessTime.get(featureId)) / 1000.0 / (windowSizeSeconds / 60.0));
            
            double hotScore = windowCount * recency * (1 + windowCount / (double)(globalCount + 1));
            
            stats.put(featureId, new WindowStats(windowCount, hotScore));
        }
        
        return stats.entrySet().stream()
            .sorted((a, b) -> Double.compare(b.getValue().hotScore, a.getValue().hotScore))
            .limit(topN)
            .map(Map.Entry::getKey)
            .collect(Collectors.toSet());
    }
}
```

```python
class CacheWarmer:
    
    def warm_up_on_startup(self):
        """服务启动时的缓存预热"""
        hot_features = self._load_yesterday_hot_features()
        for feature in hot_features:
            value = self.aerospike_client.get(feature.key)
            self.caffeine_cache.put(feature.key, value)

    def warm_up_on_pattern_change(self):
        """特征热度变化时的增量预热"""
        # 检测热度模式变化(如大促期间商品特征变热)
        new_hot_features = self.hot_detector.detect_shifting_hotness()

        for feature in new_hot_features:
            self.executor.submit(lambda: self._prefetch(feature))
    
    def warm_up_before_peak(self):

        predicted_hot_features = self.predictor.predict_peak_features()
        # 分批预热，避免打垮Aerospike
```

### 12.4 缓存一致性保证

```
写操作时缓存更新策略:
  Cache-Aside Pattern (旁路缓存)

更新流程:
  1. 特征计算完成 → 写入Aerospike
  2. 发布特征变更事件 → Kafka Topic: "feature_updates"
  3. Feature Query Service消费事件 → 删除本地Caffeine缓存
  4. 下次读取时自动回源Aerospike获取最新值

一致性保证:
  - 更新延迟: < 1秒(步骤1-4的总延迟)
  - 缓存失效窗口: 约50ms(步骤2-3的延迟)
  - 极端情况: 缓存Server重启后从Aerospike全量恢复(约30分钟)

防缓存击穿:
  - 对同一个特征Key的并发回源请求合并(通过CompletableFuture)
  - 使用互斥锁: 第一个请求回源，后续请求等待结果
```
