# 系统设计题5：超大规模离线计算平台

## 一、需求分析

### 1.1 功能性需求

| 维度 | 需求详情 |
|------|----------|
| **任务类型** | Spark SQL ETL、PySpark数据处理、Hive SQL报表、Python ML训练、数据迁移/备份 |
| **任务规模** | 日运行5000+任务，其中200个核心ETL链路，50个ML训练任务 |
| **调度模式** | 定时调度（Cron/依赖触发）、事件驱动（文件到达/Kafka消息）、手动触发 |
| **资源规模** | 日常2000核/4TB内存，大促峰值5000核/10TB内存 |
| **SLA** | 核心链路必须在凌晨6:00前完成（窗口4小时） |
| **数据量** | 日均处理100TB数据，扫描量500TB+ |

### 1.2 非功能性需求

```
性能:
  - 任务排队时间 < 5分钟（非高峰期）/ < 30分钟（高峰期）
  - 单个Spark Job运行时间: 核心链路 < 2小时
  - 调度延迟: 从触发到Application Master启动 < 30秒

可用性:
  - 调度器: 99.99% HA（主备切换 < 1分钟）
  - 计算资源: 99.9%

可扩展性:
  - 支持K8s + YARN双调度模式
  - 支持弹性扩缩（凌晨自动扩容、白天自动缩容）
  - 任务数量年增长30%

安全性:
  - 多租户隔离（队列/namespace）
  - 敏感任务资源配额限制
  - 代码审查流水线（防止生产事故）
```

```
日任务分布:
  Core ETL链路: 200个 (22:00-06:00 时间窗口)
  报表任务: 800个 (02:00-07:00)

  加上大任务(20核40GB): +1000核 = 3000核
  Buffer(×1.3): ~4000核
```

```
中间数据存储:
  每日中间Shuffle数据: ~50TB

日志存储:
  每个任务日志: ~100MB
  日任务5000个: ~500GB/天
  保留30天: ~15TB
```

## 三、候选方案对比

### 方案A：YARN + Airflow (传统方案)

```
┌─────────────────────────────────────────────────────┐
│                  Airflow 调度层                       │
│   Web Server │ Scheduler │ Worker │ DAG解析          │
└──────────────────────┬──────────────────────────────┘
                       │
           ┌───────────┼───────────┐
           ▼           ▼           ▼
      ┌─────────┐ ┌─────────┐ ┌─────────┐
      │  YARN   │ │  YARN   │ │  YARN   │
      │ Queue A │ │ Queue B │ │ Queue C │
      │ (Core)  │ │ (Report)│ │ (ML)    │
      │ 60%资源 │ │ 30%资源 │ │ 10%资源 │
      └─────────┘ └─────────┘ └─────────┘
```

**优点**:
- 业界最成熟方案，无数大厂验证
- YARN资源管理和多租户隔离成熟
- Airflow DAG编排灵活，Python定义Pipeline
- 运维经验丰富，人才市场充足

**缺点**:
- Airflow Scheduler性能瓶颈（5000+ DAG后调度延迟增大）
- YARN资源分配粒度粗（Container级别），资源利用率一般60-70%
- 弹性扩缩能力弱（NodeManager扩容需要分钟级别）
- DAG解析在高任务量下变慢（需解析所有DAG）

### 方案B：Kubernetes + Spark Operator + DolphinScheduler

```
┌─────────────────────────────────────────────────────┐
│             DolphinScheduler 调度层                   │
│   Master │ Worker │ API │ 可视化DAG编排              │
└──────────────────────┬──────────────────────────────┘
                       │
           ┌───────────┼───────────┐
           ▼           ▼           ▼
      ┌─────────┐ ┌─────────┐ ┌─────────┐
      │K8s NS A │ │K8s NS B │ │K8s NS C │
      │Spark Op │ │Spark Op │ │Spark Op │
      │(弹性)   │ │(弹性)   │ │(弹性)   │
      └─────────┘ └─────────┘ └─────────┘
            ▲           ▲           ▲
            └───────────┼───────────┘
                        │
                  ┌─────▼─────┐
                  │ Cluster   │
                  │ Autoscaler│
                  │ (Karpenter│
                  │  /CAS)    │
                  └───────────┘
```

**优点**:
- K8s原生支持弹性扩缩（HPA/VPA/Cluster Autoscaler），资源利用率可达80%+
- Spark Operator声明式管理Spark作业（有状态+自愈）
- DolphinScheduler去中心化架构，调度能力优于Airflow（已验证支持10万+ DAG）
- 统一基础设施（K8s同时管理在线服务和离线任务）

**缺点**:
- K8s上运行Spark的稳定性不如YARN（大规模场景下Shuffle性能可能略差）
- DolphinScheduler社区不如Airflow成熟，文档和教程较少
- K8s网络开销略高于YARN（Pod间通信走CNI）
- 需要团队具备K8s运维能力

### 方案C：混合方案（YARN + K8s + Airflow）

```
┌─────────────────────────────────────────────────────┐
│                   Airflow 调度层                      │
└───────┬─────────────┬──────────────┬────────────────┘
        │             │              │
        ▼             ▼              ▼
┌───────────┐  ┌───────────┐  ┌───────────┐
│  YARN     │  │  K8s      │  │  K8s      │
│  核心ETL  │  │  报表Spark│  │  ML训练   │
│  (稳定优先)│  │  (弹性优先)│  │  (GPU优先)│
└───────────┘  └───────────┘  └───────────┘
```

**优点**:
- 核心链路用YARN保障稳定性，非核心链路用K8s获得弹性
- 渐进式迁移，降低风险
- ML训练可利用K8s GPU调度能力

**缺点**:
- 双资源管理器运维复杂度最高
- 两套集群成本高
- 团队需要同时掌握YARN和K8s

## 五、最终选择：方案B（K8s + DolphinScheduler + Spark Operator）

### 选择理由

**1. 弹性扩缩节省50%+计算成本**
离线计算有明显的潮汐特征（凌晨忙、白天闲）。K8s的Cluster Autoscaler + Karpenter可以在凌晨自动扩容到4000核，白天缩容到500核（保留常驻服务），资源利用率从YARN的60%提升至80%+，计算成本节省30-50%。

**2. 调度能力匹配未来3年增长**
Airflow在DAG数超过5000后调度延迟明显增加（Scheduler扫描所有DAG）。DolphinScheduler的去中心化Master-Worker架构天然支持水平扩展，已验证10万+任务场景。日5000任务且年增长30%，3年后将达到1万+，选DolphinScheduler是提前为未来布局。

**3. K8s统一基础设施**
公司目前已将在线服务全面K8s化，离线计算上K8s可以实现统一基础设施管理、统一监控、统一CI/CD，减少运维团队的心智负担。

**4. Spark on K8s日趋成熟**
Spark 3.x对K8s的支持已非常成熟，Remote Shuffle Service解决了Shuffle性能问题，社区也逐渐从YARN转向K8s作为默认资源管理器。

### ADR记录

```markdown

- 风险：大规模Spark on K8s的极端场景需充分压测
```

## 六、核心架构设计

### 6.1 任务调度DAG示例

```python
# DolphinScheduler Python API定义ETL DAG
from pydolphinscheduler.tasks import Spark, Shell, Dependent
from pydolphinscheduler.core.process_definition import ProcessDefinition

with ProcessDefinition(
    name="core_etl_daily",
    schedule="0 2 * * *",
    tenant="data_platform",
) as pd:
    
    # 第1层: 数据抽取（并行）
    extract_mysql = Spark(
        name="extract_mysql",
        main_class="com.example.ExtractMySQL",
        main_package="hdfs:///jars/etl.jar",
        driver_cores=2, driver_memory="4G",
        executor_cores=4, executor_memory="8G",
        executor_nums=10,
    )
    
    extract_logs = Spark(
        name="extract_logs",
        main_class="com.example.ExtractLogs",
        main_package="hdfs:///jars/etl.jar",
        driver_cores=2, driver_memory="4G",
        executor_cores=4, executor_memory="8G",
        executor_nums=20,
    )
    
    # 第2层: DWD加工（依赖第1层全部完成）
    dwd_orders = Spark(
        name="dwd_orders",
        ...,
        deps=[extract_mysql, extract_logs],
    )
    
    dwd_users = Spark(
        name="dwd_users",
        ...,
        deps=[extract_mysql],
    )
    
    # 第3层: DWS汇总
    dws_daily = Spark(
        name="dws_daily",
        ...,
        deps=[dwd_orders, dwd_users],
    )
    
    # 第4层: ADS + 数据质量检查
    ads_report = Spark(name="ads_report", ..., deps=[dws_daily])
    dq_check = Shell(name="dq_check", command="python dq_check.py", deps=[dws_daily])
    
    # 第5层: 通知
    notify = Shell(
        name="notify",
        command="python notify.py",
        deps=[ads_report, dq_check],
    )
    
    pd.submit()
```

### 6.2 Spark Operator配置

```yaml
apiVersion: sparkoperator.k8s.io/v1beta2
kind: SparkApplication
metadata:
  name: dwd-orders-daily
  namespace: data-platform
spec:
  type: Scala
  mode: cluster
  image: registry.example.com/spark:3.5.0
  mainClass: com.example.DwdOrdersDaily
  mainApplicationFile: s3a://jars/dwd-orders.jar
  arguments:
    - "--dt={{ ds }}"
    - "--env=prod"
  driver:
    cores: 2
    coreLimit: "2.5"
    memory: "4g"
    serviceAccount: spark-operator
    labels:
      cost-center: data-platform
  executor:
    cores: 4
    coreLimit: "4.5"
    memory: "8g"
    instances: 20
    labels:
      cost-center: data-platform
  restartPolicy:
    type: OnFailure
    onFailureRetries: 3
    onFailureRetryInterval: 600
  batchScheduler: volcano  # 使用Volcano批量调度
```

### 6.3 弹性扩缩策略

```
Karpenter配置 (凌晨扩容、白天缩容):

凌晨扩缩计划(K8s CronJob):
  22:00: 预扩到1000核 (核心链路启动)
  00:00: 扩到2500核 (报表+ML任务启动)
  02:00: 扩到4000核 (峰值)
  06:00: 缩到2000核 (核心链路完成)
  09:00: 缩到500核  (常驻基线)

Spot实例策略:
  离线任务优先使用Spot实例（成本降低70%）
  核心链路使用On-Demand实例兜底
  Karpenter自动管理Spot中断时重新调度
```

## 十、Spot实例中断的详细处理流程

### 10.1 Spot实例中断机制

```
Spot实例生命周期:

┌─────────────────────────────────────────────────────────────────┐
│                        Spot实例完整生命周期                       │
│                                                                  │
│  [创建] ──→ [运行] ──→ [中断通知] ──→ [优雅终止] ──→ [释放]    │
│    │         │           │               │              │        │
│    │         │           │ 提前2分钟      │ 30秒时间窗口  │        │
│    │         │           │ 收到通知       │ 保存状态      │        │
│    │         │           │               │              │        │
│    │         │           └─ 无通知: 直接终止(极少数情况)         │
│    │         │                                                  │
│    └─ 创建失败: 当前出价低于市场价 → 等待出价回升后重试          │
└─────────────────────────────────────────────────────────────────┘

中断原因:
  1. 市场价超过设定出价 (最常见, 85%+)
  2. 资源池容量不足 (阿里云回收, 10%)
  3. 宿主机维护/故障 (5%)
```

```
│  │                                                             │  │
```

### F-01: 核心ETL链路超过SLA时间窗口

```
故障描述: 凌晨6:00前核心ETL(200个任务)未全部完成，影响早间报表

根因分析 (按概率排序):
  1. (40%) 上游数据源产出延迟(MySQL主库备份延时)
  2. (25%) 集群资源竞争(报表任务与ETL同时运行争夺资源)
  3. (15%) 数据量突增(大促/活动导致数据量翻倍)
  4. (10%) 某任务执行时间异常(代码变更引入性能退化)
  5. (10%) 基础设施问题(网络抖动/磁盘慢/节点故障)

处理SOP:
┌─────────────────────────────────────────────────────────────────┐
│ Step 1 [5min]: 快速定位瓶颈任务                                 │
│   - 查看DolphinScheduler Dashboard → 找出DAG中最慢的节点       │
│   - 检查该节点的Spark UI: Stage耗时分布、数据倾斜情况          │
│   - 并行执行: 同时检查K8s集群资源使用率                        │
│                                                                  │
│ Step 2 [10min]: 应急恢复                                        │
│   方案A - 资源倾斜 (如果资源足够):                               │
│     $ kubectl patch sparkapp dwd-orders-daily --type=merge -p ' │
│       {"spec":{"executor":{"instances":40}}}'                    │
│     将瓶颈任务的Executor从20扩到40                               │
│                                                                  │
│   方案B - 降级执行 (如果数据量异常大):                           │
│     - 跳过非核心维度表JOIN (如跳过用户画像标签表)               │
│     - 添加WHERE条件只处理高优先级数据                           │
│     - 设置Spark超时参数: spark.sql.adaptive.advisoryPartitionSizeInBytes=256MB │
│                                                                  │
│ Step 3 [10min]: 通知下游                                        │
│   - 发送"报表延迟通知"到相关群组                                │
│   - 给出新的预计完成时间                                        │
│                                                                  │
│ Step 4 [5min]: 事后复盘                                         │

└─────────────────────────────────────────────────────────────────┘

  - 提前30分钟预热水位检查: 每日05:30自动巡检
```

### F-02: Spark Driver OOM反复重启

```
故障描述: Spark Driver Pod反复因OOM被杀(CrashLoopBackOff)

根因分析:
  1. (50%) collect()/take()操作拉取过多数据到Driver内存
  2. (30%) Broadcast变量过大(超过Driver内存限制)
  3. (15%) DAG中有大量Task，元数据占用Driver内存过多
  4. (5%) Driver内存配置不足

处理SOP:
┌─────────────────────────────────────────────────────────────────┐

│                                                                  │
│ Step 2 [5min]: 临时修复                                         │
│   方案A - 增加Driver内存 (首选):                                 │
│     kubectl patch sparkapp <name> --type=merge -p '{            │
│       "spec":{"driver":{"memory":"8g","coreLimit":"4"}}}'       │
│                                                                  │
│   方案B - 禁用Broadcast (如果确认是Broadcast过大):               │
│     spark.sql.autoBroadcastJoinThreshold=-1  # 关闭自动Broadcast│
│                                                                  │
│ Step 3 [5min]: 代码修复 (如果问题持续)                           │
│   替换: df.collect() → df.write.parquet() + read               │
│   替换: 大Broadcast → 普通Shuffle Join                         │
│                                                                  │
│ Step 4 [3min]: 验证重跑                                         │
│   - 重新提交作业，观察Driver内存                                 │
│   - 确认Spark UI中无OOM相关WARN                                 │
└─────────────────────────────────────────────────────────────────┘

  小作业(<100GB数据): driver_memory=2g
  中等作业(100GB-1TB): driver_memory=4g
  大作业(>1TB): driver_memory=8g-16g
  超大作业(>10TB): driver_memory=16g-32g (但尽量避免)
```

### F-03: 数据倾斜导致个别Task运行超时

```
故障描述: Spark Stage中99%的Task在5分钟内完成，但有1-2个Task运行超过2小时

识别方法:
  Spark UI → Stages → 查看Task Duration分布
  如果 max(Shuffle Read Size) / median > 5: 确认数据倾斜

常见倾斜KEY示例:
  - user_id = NULL (大量空值聚集在一个Task)
  - 北京/上海的用户数据远超其他城市
  - 大促期间某热门商品订单量是平均水平的100倍

处理SOP:
┌─────────────────────────────────────────────────────────────────┐
│ 方案A: 加盐(Salting) - 适用于聚合倾斜                           │
│   -- 倾斜Key加随机盐                                            │
│   SELECT /*+ REPARTITION(200) */                                │
│     CONCAT(user_id, '_', CAST(RAND()*10 AS INT)) as salted_key, │
│     SUM(amount) as total_amount                                 │
│   FROM orders                                                   │
│   GROUP BY salted_key                                           │
│   -- 然后再聚合去掉盐值                                          │
│                                                                  │
│ 方案B: AQE自动倾斜处理 (Spark 3.2+)                              │
│   spark.sql.adaptive.enabled=true                               │
│   spark.sql.adaptive.skewJoin.enabled=true                      │
│   spark.sql.adaptive.skewJoin.skewedPartitionFactor=5           │
│   spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes=256MB │
│                                                                  │
│ 方案C: 广播小表避免Shuffle - 适用于JOIN倾斜                      │
│   -- 如果维表<广播阈值，自动广播                                │
│   spark.sql.autoBroadcastJoinThreshold=104857600  # 100MB       │
│                                                                  │
│ 方案D: 两阶段聚合 - 适用于GroupBy倾斜                            │
│   -- 第一阶段: 加随机前缀聚合                                   │
│   -- 第二阶段: 去掉前缀再聚合                                   │
└─────────────────────────────────────────────────────────────────┘
```

### F-04: Kafka Offset积压过大

```
故障描述: Kafka Consumer Lag持续增长，实时管道延迟

根因分析:
  1. (50%) Flink消费速度 < Kafka生产速度
  2. (30%) Flink作业重启导致从最早Offset消费
  3. (15%) Flink处理逻辑变慢(外部调用超时)
  4. (5%) Kafka Broker性能问题

处理SOP:
┌─────────────────────────────────────────────────────────────────┐
│ Step 1 [2min]: 确认积压规模                                      │
│   kafka-consumer-groups --bootstrap-server localhost:9092 \     │
│     --group flink-user-pipeline --describe                      │
│   → 记录 CURRENT-OFFSET 与 LOG-END-OFFSET 的差值               │
│                                                                  │
│ Step 2 [5min]: 应急追赶                                         │
│   方案A - 增加Flink并行度 (首选):                                │
│     kubectl patch flinkdeployment realtime-pipeline --type=merge -p '{          │
│       "spec":{"job":{"parallelism":48}}}'                       │
│                                                                  │
│   方案B - 临时降低处理复杂度:                                     │
│     - 关闭非核心的CEP规则                                        │
│     - 减少窗口聚合的维度数                                       │
│                                                                  │
│   方案C - 跳过部分历史数据 (仅非关键管道):                        │
│     kafka-consumer-groups --reset-offsets \                     │
│       --to-latest --group flink-user-pipeline --execute         │
│                                                                  │
│ Step 3 [3min]: 监控追赶进度                                      │
│   while true; do                                                │
│     kafka-consumer-groups --describe --group flink-user-pipeline │
│     sleep 60                                                    │
│   done                                                          │
│   预期: Lag每5分钟减少约500万条(取决于Flink吞吐)                 │
└─────────────────────────────────────────────────────────────────┘

预防措施:
  - 设置Flink的checkpoint间隔: 60s
  - 配置Kafka Topic保留时间: 7天 (足够恢复时间)
  - Flink作业设置 restart-strategy: fixed-delay
```


```yaml
# alertmanager-config.yaml
global:
  resolve_timeout: 5m
  smtp_smarthost: 'smtp.example.com:587'
  smtp_from: 'alertmanager@example.com'
  smtp_auth_username: 'alertmanager@example.com'
  smtp_auth_password: '******'

# 告警路由树
route:
  group_by: ['alertname', 'severity']
  group_wait: 10s          # 等待更多告警聚合
  group_interval: 5m       # 同组告警发送间隔
  repeat_interval: 4h      # 同一条告警重复间隔
  
  # 默认接收者
  receiver: 'default-receiver'
  
  routes:
  # P0/Critical告警 → 电话+企微群
  - match:
      severity: critical
    receiver: 'critical-oncall'
    continue: true
    
  # P1/Warning告警 → 企微群+邮件
  - match:
      severity: warning
    receiver: 'warning-channel'
    
  # ETL核心链路 → 数据工程专属群
  - match:
      component: etl
    receiver: 'data-engineering-team'
    
  # 成本告警 → 管理群
  - match:
      component: cost
    receiver: 'management-channel'

receivers:
- name: 'default-receiver'
  webhook_configs:
  - url: 'http://wecom-bot.example.com/alert'

- name: 'critical-oncall'
  webhook_configs:
  - url: 'http://wecom-bot.example.com/critical'
  - url: 'http://pagerduty.example.com/trigger'
    send_resolved: true

- name: 'warning-channel'
  email_configs:
  - to: 'data-platform@example.com'
  webhook_configs:
  - url: 'http://wecom-bot.example.com/warning'

- name: 'data-engineering-team'
  webhook_configs:
  - url: 'http://wecom-bot.example.com/data-eng'

- name: 'management-channel'
  email_configs:
  - to: 'cto@example.com, vp-eng@example.com'

# 告警抑制规则
inhibit_rules:
- source_match:
    severity: critical
  target_match:
    severity: warning
  equal: ['alertname', 'instance']
```
