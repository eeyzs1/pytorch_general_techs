# 系统设计题3：跨多云的数据联邦查询引擎

## 一、需求分析

### 1.1 功能性需求

| 维度 | 需求详情 |
|------|----------|
| **数据源** | AWS RDS MySQL（交易数据）、阿里云 Hologres（实时数仓）、Azure Blob Storage（日志归档）、自建Hive（历史数据）、GCP BigQuery（广告数据） |
| **查询方式** | 统一SQL入口，跨数据源JOIN和聚合，无需数据搬迁 |
| **性能** | 简单查询 < 3秒，跨源JOIN < 30秒，大规模扫描 < 5分钟 |
| **并发** | 支持50个并发联邦查询 |
| **安全** | 统一的认证授权，跨云的传输加密 |

### 1.2 非功能性需求

```
性能:
  - P50查询延迟: < 3秒（单源）/ < 30秒（跨源JOIN）
  - P99查询延迟: < 10秒 / < 60秒
  - 并发查询数: 50个并发，排队不超过100个

可用性:
  - 联邦查询引擎自身: 99.9%
  - 下游数据源故障时：优雅降级（跳过该源或返回缓存数据）

安全性:
  - 统一认证（LDAP+OAuth）
  - 列级别权限控制
  - 跨云传输TLS 1.3加密
  - 完整审计日志

成本:
  - 尽量减少跨云数据传输（数据传输成本高）
  - 优先在数据所在云执行计算（计算靠近数据）
```

## 三、候选方案对比

### 方案A：Trino + 多Catalog Connector

```
┌──────────────────────────────────────────────────────────┐
│                    Trino Coordinator                      │
│                   (查询解析/优化/调度)                     │
└──────────┬──────────┬──────────┬──────────┬──────────────┘
           │          │          │          │
    ┌──────▼──┐ ┌─────▼───┐ ┌───▼────┐ ┌──▼──────┐
    │  MySQL  │ │Hologres │ │  Hive  │ │BigQuery │
    │Connector│ │Connector│ │Connector│ │Connector│
    │  (AWS)  │ │(阿里云) │ │(自建)  │ │  (GCP)  │
    └─────────┘ └─────────┘ └────────┘ └─────────┘
```

**优点**:
- 丰富的Connector生态（30+数据源），开箱即用
- 内存中Pipeline执行模型，查询性能优异
- 社区极度活跃（Presto→Trino继承了FB的贡献）
- 支持跨源JOIN和联邦查询

**缺点**:
- 跨云场景下，大表JOIN会产生大量跨云数据传输
- 无内置查询结果缓存，重复查询浪费资源
- Connector质量参差不齐（如Hologres Connector可能不成熟）
- 需要自建Trino集群，运维复杂度高

### 方案B：Presto + Alluxio缓存加速

```
┌──────────────────────────────────────────────────────────┐
│                    Presto Coordinator                     │
└──────────┬──────────┬──────────┬──────────┬──────────────┘
           │          │          │          │
    ┌──────▼──────────▼──────────▼──────────▼──────┐
    │              Alluxio 分布式缓存                │
    │     (跨云数据本地缓存 + 数据亲和调度)          │
    └──────┬──────────┬──────────┬──────────┬──────┘
           │          │          │          │
    ┌──────▼──┐ ┌─────▼───┐ ┌───▼────┐ ┌──▼──────┐
    │  MySQL  │ │Hologres │ │  Hive  │ │BigQuery │
    │  (AWS)  │ │(阿里云) │ │(自建)  │ │  (GCP)  │
    └─────────┘ └─────────┘ └────────┘ └─────────┘
```

**优点**:
- Alluxio跨云数据缓存，大幅减少重复跨云传输
- 数据亲和调度：优先在数据所在节点执行计算
- 支持缓存热数据，冷数据按需拉取
- 适合频繁查询相同数据集的场景

**缺点**:
- 引入Alluxio增加了架构和运维复杂度
- 缓存一致性管理复杂（源数据更新后缓存失效策略）
- Alluxio本身也需要存储资源（SSD/内存）
- 查询延迟受缓存命中率影响波动大

## 五、最终选择：方案B（Presto/Trino + Alluxio）

### 选择理由

**1. 跨云数据传输是核心瓶颈**
联邦查询的最大挑战不是查询引擎本身，而是跨云数据传输延迟和成本。方案B通过Alluxio在"数据消费端"建立缓存层，将频繁访问的热数据缓存在本地，避免每次查询都跨云拉取数据。

**2. 成本可控**
方案C（Starburst）商业许可按vCPU计费，50并发查询至少需要200 vCPU的集群，年许可费用可能超过$30万。方案B全开源，成本仅为硬件+运维人力，预估年成本~$10万。

**3. 数据亲和调度**
Alluxio的数据亲和调度确保计算任务优先在数据所在节点执行，减少跨网络数据传输。例如，如果AWS RDS上的用户表经常与GCP BigQuery上的广告表JOIN，可以将用户表的缓存副本与BigQuery数据调度到同一节点计算。

**4. 灵活的缓存策略**
- 热数据（最近7天的交易数据）：Alluxio内存+SSD缓存
- 温数据（30天内的报表）：Alluxio SSD缓存
- 冷数据（历史归档）：按需从Azure Blob拉取

### ADR记录

```markdown
# ADR-004: 联邦查询引擎选择 Presto/Trino + Alluxio

## 状态
已采纳

## 决策
使用开源Presto/Trino作为查询引擎，Alluxio作为跨云数据缓存层。

## 理由
1. 跨云数据传输延迟是大表JOIN的性能瓶颈，Alluxio缓存有效降低重复传输
2. 全开源方案年成本~$10万 vs Starburst商业许可~$30万
3. 数据亲和调度减少跨网络传输，提升JOIN性能
4. 灵活的缓存策略适配热/温/冷数据场景

## 后果
- 正面：成本可控，性能达标，无供应商锁定
- 负面：运维复杂度增加（需同时管理Trino+Alluxio集群）
- 风险：缓存一致性需精细管理，避免返回过期数据

## 替代方案
- Starburst：功能完善但商业许可费用高
- 自建Trino无缓存：跨云JOIN性能不可接受
```

### 6.2 Alluxio缓存策略

```
缓存层级:
  L1 (RAM): 128GB/节点, 缓存最近1小时内访问的数据
  L2 (SSD): 4TB/节点, 缓存最近30天访问的数据

缓存写入策略:
  - ASYNC_THROUGH: 异步写穿（写入Alluxio同时异步写回源）
  - CACHE: 仅缓存（只读场景，源数据不变）
  
缓存淘汰策略:
  - LRU (默认)
  - 按表级别设置TTL: 
    * 实时交易表: TTL=1h
    * 日报表: TTL=24h
    * 历史归档: TTL=7d
```

## 九、查询优化策略详解

### 9.1 谓词下推（Predicate Pushdown）

谓词下推是联邦查询引擎最核心的优化手段。核心思想是：将WHERE条件尽可能"推"到数据源端执行，而不是将所有数据都拉到Trino再做过滤。

```sql
-- 原始联邦查询
SELECT u.name, o.order_amount, o.order_date
FROM mysql.aws.users u
JOIN hive.legacy.orders o ON u.user_id = o.user_id
WHERE u.region = 'APAC'
  AND u.status = 'active'
  AND o.order_date >= '2024-01-01'
  AND o.order_date < '2024-02-01';

-- Trino优化器会将谓词分解并下推:
-- MySQL端执行的等价查询:
--   SELECT user_id, name FROM users WHERE region='APAC' AND status='active'
-- Hive端执行的等价查询:
--   SELECT user_id, order_amount, order_date FROM orders
--   WHERE order_date >= '2024-01-01' AND order_date < '2024-02-01'
-- Trino端仅执行: JOIN(user_id) on 已过滤的小结果集
```

**谓词下推的支持矩阵**:

| Connector | 简单谓词下推 | 复杂谓词下推 | LIKE下推 | IN下推 | 函数下推 |
|-----------|------------|------------|---------|-------|---------|
| MySQL | ✅ | ✅ (AND/OR) | ✅ | ✅ | ❌ |
| PostgreSQL | ✅ | ✅ | ✅ | ✅ | 部分 |
| Hive | ✅ | ✅ (分区键) | ✅ | ✅ | ❌ |
| BigQuery | ✅ | ✅ | ✅ | ✅ | ✅ |
| Hologres | ✅ | ✅ | ✅ | ✅ | ✅ |
| MongoDB | ✅ | ✅ | ❌ | ✅ | ❌ |

**验证谓词下推效果**:

```sql
-- 使用EXPLAIN ANALYZE查看谓词下推情况
EXPLAIN ANALYZE
SELECT u.name, COUNT(*) as order_cnt
FROM mysql.aws.users u
JOIN hive.legacy.orders o ON u.user_id = o.user_id
WHERE u.region = 'APAC' AND o.order_date >= '2024-01-01'
GROUP BY u.name;

-- 执行计划中关键指标:
-- 1. "ScanFilterProject" → 看filter是否在TableScan内部
-- 2. 各分片的输入行数 vs 输出行数 → 判断下推是否生效
-- 3. "Exchange" 数据量 → 如果交换数据量接近原始表大小，说明下推未生效
```

**谓词下推不生效的常见场景**:

```
场景1: 跨源JOIN的JOIN条件无法下推
  JOIN条件中的列来自不同数据源，无法直接下推到任一源执行

场景2: 函数包裹列导致下推失败
  WHERE UPPER(u.name) = 'JOHN'  -- UPPER函数不一定被下推
  优化: 在ETL阶段预计算UPPER值存入新列

场景3: OR条件中包含不同表的列
  WHERE u.region = 'APAC' OR o.amount > 1000
  优化器通常无法拆分OR条件到不同数据源

场景4: Connector未实现对应下推接口
  MongoDB Connector不支持LIKE下推
  解决方案: 在Connector端实现ConnectorMetadata.getTableStatistics()
```

### 9.2 分区裁剪（Partition Pruning）

对于Hive/Iceberg/Hudi等分区表，分区裁剪是避免全表扫描的关键优化。Trino通过分析WHERE条件中的分区列，自动跳过不相关分区。

```sql
-- Iceberg分区表定义
CREATE TABLE hive.legacy.orders (
    order_id BIGINT,
    user_id BIGINT,
    amount DECIMAL(20,2),
    order_date DATE,
    status STRING
) WITH (
    partitioning = ARRAY['days(order_date)', 'bucket(user_id, 32)']
);

-- 查询时的分区裁剪
SELECT * FROM hive.legacy.orders
WHERE order_date BETWEEN DATE '2024-01-01' AND DATE '2024-01-07'
  AND user_id = 12345;

-- Trino优化器自动裁剪:
-- 1. 时间分区裁剪: 只扫描2024-01-01~2024-01-07共7个分区 (全表有365个分区)
-- 2. Bucket裁剪: user_id=12345 → hash%32 → 只扫描1个bucket文件
-- 裁剪比例: 7/365 × 1/32 ≈ 0.06% 的原始数据量

-- 使用EXPLAIN验证分区裁剪
EXPLAIN SELECT count(*) FROM hive.legacy.orders
WHERE order_date = DATE '2024-01-15';

-- 期望输出:
-- ScanProject[table = orders, 
--   partitions = 1/365,    <-- 只扫描1个分区
--   splits = 32,            <-- splits数量=worker并行度
--   filterPredicate = ...]
```

**分区裁剪失效的排查与修复**:

```
问题1: 谓词中使用了函数导致分区裁剪失效
  ❌ WHERE DATE(order_timestamp) = '2024-01-01'  -- DATE()函数阻止分区裁剪
  ✅ WHERE order_date = DATE '2024-01-01'         -- 直接比较分区字段

问题2: OR条件跨分区
  ❌ WHERE order_date = '2024-01-01' OR user_type = 'VIP'
  → 优化方案: 拆分为UNION ALL
  ✅ SELECT * FROM orders WHERE order_date = '2024-01-01'
     UNION ALL
     SELECT * FROM orders WHERE user_type = 'VIP' AND order_date != '2024-01-01'

问题3: 动态分区裁剪不生效
  -- 当分区列来自JOIN的另一侧时，需要运行时动态裁剪
  SELECT /*+ DYNAMIC_PARTITION_PRUNING */ o.*
  FROM hive.legacy.orders o
  JOIN mysql.aws.hot_dates d ON o.order_date = d.dt
  WHERE d.is_promotion = true;
  -- 提示词仅在支持的版本生效，需要Trino 356+
```

### 9.3 列裁剪（Column Pruning）

列裁剪确保只从数据源读取实际需要的列，减少网络传输和内存消耗。

```sql
-- 查询只需要3列，但表有50列
SELECT user_id, name, email FROM mysql.aws.users WHERE region = 'APAC';

-- Trino扫描层优化:
-- 1. 列裁剪: 只从MySQL读取 user_id, name, email 三列
-- 2. 行过滤: region='APAC' 下推到MySQL
-- 实际传输: region='APAC'过滤后的3列数据 < 原始表的0.1%

-- Parquet/Iceberg等列式存储的额外收益:
--   Parquet文件的Row Group内按列存储
--   未查询的47列对应的数据块完全不读取
--   IO节省: ~94% (47/50列)
```

**列裁剪在跨源场景的效果量化**:

```
场景: 查询BigQuery的100TB ads表，但只需3列(共80列)
  - 无列裁剪: 需要传输100TB × 80列/80列 = 100TB (跨GCP公网)
  - 有列裁剪: 需要传输100TB × 3列/80列 ≈ 3.75TB
  - 传输成本: GCP出向 $0.12/GB → 节省 (100-3.75)TB × $0.12/GB ≈ $11,550/查询

场景: Parquet存储加成
  - 未查询列完全不从磁盘读取 → IO吞吐提升 80/3 ≈ 26倍
  - Parquet列编码压缩率不同，实际效果可能有差异
```

### 9.4 JOIN优化策略

**小表广播 vs 大表重分布**:

```sql
-- 场景: users表(10万行, 约50MB) JOIN orders表(10亿行, 约500GB)
SELECT u.name, o.amount
FROM mysql.aws.users u
JOIN hive.legacy.orders o ON u.user_id = o.user_id;

-- 优化器决策流程:
-- 1. 统计信息收集: users表估算50MB << 广播阈值(默认32MB可用session调大)
-- 2. 决策: 广播users表到所有Worker节点
-- 3. 执行: orders表按user_id重分布到各Worker, 与本地广播的users做HashJoin
--
-- 若未广播(默认阈值限制):
--   users表也需要按user_id重分布 + 跨网络传输
--   → 增加一次全量数据的网络Shuffle

-- 手动控制广播阈值
SET SESSION join_distribution_type = 'BROADCAST';
-- 或调整阈值
SET SESSION join_max_broadcast_table_size = '100MB';
```

**跨源JOIN的延迟分解（实测模型）**:

```
场景: MySQL(AWS) users 10万行 JOIN BigQuery(GCP) ads 500万行
条件: user_id = user_id, 结果集约50万行

延迟分解:
  Phase 1 - 扫描阶段:
    MySQL扫描10万行(users): 2秒
    BigQuery扫描500万行(ads): 8秒 (含BigQuery作业启动)
    
  Phase 2 - 数据传输阶段:
    MySQL → Trino: 50MB → 经AWS→Trino内网: 0.5秒
    BigQuery → Trino: 2.5GB → 经GCP→Trino公网: 15秒 (瓶颈!)
    
  Phase 3 - JOIN计算阶段:
    HashBuild(小表): 1秒
    HashProbe(大表): 3秒
    
  Phase 4 - 结果返回:
    50万行序列化返回: 1秒

总延迟: ~30秒 (数据跨云传输占50%)
  
优化后(Alluxio缓存ads表):
  Phase 1: 同前, 但ads从Alluxio读取: 3秒 (本地SSD)
  Phase 2: BigQuery→Trino传输: 0 (已缓存)
  Phase 3-4: 同前: 5秒
  
总延迟: ~10秒 (降低67%)
```

### 10.2 缓存写入与读取策略

**读取路径（Read Path）**:

```
1. 客户端发起读请求 → Alluxio Worker
2. Worker检查L1 RAM → 命中则直接返回
3. L1未命中 → 检查L2 SSD → 命中则返回 + 提升到L1 (Promote)
4. L2未命中 → 检查L3 HDD → 命中则返回 + 提升到L2
5. 全部未命中 → 从UFS(底层存储)拉取 → 写入L2 + 返回到L1

提升策略(Promotion):
  - 访问频率阈值: 1小时内被访问超过3次的数据 → 自动Promote到上层
  - 手动预热: 大促前通过DistributedLoad命令预热关键表
  - pin策略: 核心维表永久pin在内存层
```

**写入策略对比**:

| 策略 | 写入行为 | 适用场景 | 数据一致性 |
|------|---------|---------|-----------|
| MUST_CACHE | 仅写Alluxio，不写UFS | 临时中间结果 | 弱 |
| CACHE_THROUGH | 同步写Alluxio和UFS | 写入量小的场景 | 强 |
| ASYNC_THROUGH | 写Alluxio + 异步写UFS | 高吞吐写入 | 最终一致 |
| THROUGH | 仅写UFS(穿透) | 不缓存写入 | 强 |

### 11.2 延迟代价模型

```
跨源JOIN延迟公式:
Total_Latency = max(各源扫描时间) + max(数据传输时间) + JOIN计算时间 + 结果返回时间

单源扫描时间模型:
  T_scan = 数据量 / (并行度 × 单Worker扫描吞吐)
  
  例: MySQL 10万行, 并行度4, 扫描吞吐10万行/秒
    T_scan = 10万 / (4 × 10万) = 0.25秒

数据传输时间模型 (跨云):
  T_transfer = 数据量 × 8 / (可用带宽 × 带宽利用率)
  
  例: 2.5GB数据, 跨云专线1Gbps, 利用率70%
    T_transfer = 2.5GB × 8 / (1Gbps × 0.7) = 20Gb / 0.7Gbps ≈ 28.6秒

不同数据量级的跨源JOIN延迟 (实测基准):

┌──────────────────────┬──────────┬──────────┬──────────┬──────────┐

关键发现:
  1. 小数据量(< 100MB): 延迟主要由网络RTT主导, 缓存收益有限
```

### 11.3 查询优化器决策模型

```python
# Trino优化器的简化决策伪代码
class CostBasedOptimizer:
    """
    CBO (Cost-Based Optimizer) 在联邦查询中的关键决策点
    """
    
    # 统计信息需求
    table_stats = {
        'mysql.aws.users': {
            'row_count': 10_000_000,
            'data_size': '5GB',
            'column_ndv': {'region': 10, 'status': 3},
            'null_fraction': {'email': 0.05}
        },
        'bigquery.gcp.ads': {
            'row_count': 500_000_000,
            'data_size': '150GB',
            'column_ndv': {'user_id': 200_000_000, 'campaign': 500},
        }
    }
    
    def decide_join_strategy(self, left_table, right_table, join_condition, predicates):
        left_size = self.estimate_filtered_size(left_table, predicates)
        right_size = self.estimate_filtered_size(right_table, predicates)
        
        # 决策1: 广播 vs 重分布
        if left_size < BROADCAST_THRESHOLD:  # 默认32MB
            return 'BROADCAST_LEFT'
        elif right_size < BROADCAST_THRESHOLD:
            return 'BROADCAST_RIGHT'
        
        # 决策2: 是否合并谓词到JOIN条件
        if self.can_push_join_to_single_source(join_condition, predicates):
            return 'PUSHED_JOIN'  # JOIN完全下推到单个数据源
        
        # 决策3: 是否使用SortMergeJoin (当两边数据都很大)
        if left_size > SORT_MERGE_THRESHOLD and right_size > SORT_MERGE_THRESHOLD:
            return 'SORT_MERGE_JOIN'
        
        return 'HASH_JOIN'
    
    def estimate_filtered_size(self, table, predicates):
        """基于统计信息的过滤后大小估算"""
        base_rows = table_stats[table]['row_count']
        selectivity = 1.0
        
        for pred in predicates:
            if pred.type == 'EQUAL':
                ndv = table_stats[table]['column_ndv'].get(pred.column, 100)
                selectivity *= 1.0 / ndv
            elif pred.type == 'RANGE':
                selectivity *= 0.25  # 经验值
        
        return base_rows * selectivity

    def estimate_network_cost(self, table, filter_selectivity):
        """跨云传输成本估算"""
        base_size = table_stats[table]['data_size']
        filtered_size = base_size * filter_selectivity
        
        # 检查是否在本地缓存中
        cache_hit_rate = self.get_cache_hit_rate(table)
        network_transfer = filtered_size * (1 - cache_hit_rate)
        
        # 检查对应的云出向成本
        cloud = table.split('.')[0]  # 'aws' / 'gcp' / 'azure'
        egress_cost = CLOUD_EGRESS_PRICE[cloud] * network_transfer
        
        return {
            'transfer_size': network_transfer,
            'egress_cost': egress_cost,
            'estimated_time': network_transfer / AVAILABLE_BANDWIDTH
        }
```

## 十二、失败场景与降级策略

### 12.1 数据源故障分级

```
故障等级定义:

L1 - 致命故障:
  数据源完全不可用(所有查询失败)
  用户无法获取任何数据
  → 需要紧急降级 + 5分钟内通知 + 30分钟内恢复或启动容灾

L2 - 严重故障:
  数据源部分不可用(某些表/分区查询失败)
  部分用户受影响
  → 自动降级 + 15分钟内通知 + 2小时内解决

L3 - 中等故障:
  数据源响应变慢(延迟超过SLA × 3)
  用户体验下降但功能可用
  → 自动超时降级 + 1小时内通知

L4 - 轻微故障:
  数据源偶尔超时(< 5%请求)
  大部分用户无感知
  → 日志记录 + 日报中体现
```

### 12.2 各数据源故障处理方案

#### 场景1: AWS RDS MySQL 故障

```
故障现象: MySQL主库宕机/网络不通/连接池耗尽

影响面分析:
  - 涉及表: users, products, shops (5张核心表)
  - 影响查询: 所有需要用户/商品/店铺维度的联邦查询(~40%总查询量)
  - 影响的业务场景: 用户行为分析, 商品销售报表, 推荐特征查询

降级方案:

Level 1 - 自动降级 (引擎层自动执行):
  ┌──────────────────────────────────────────────────┐
  │ 1. 连接超时检测: TCP连接超时 → 标记Connector状态  │
  │ 2. 查询重试: 指数退避重试 (1s, 2s, 4s, 8s)      │
  │ 3. 切换读副本: 自动切换到MySQL只读副本            │
  │ 4. 若读副本也不可用 → 进入L2降级                  │
  └──────────────────────────────────────────────────┘

Level 2 - 缓存兜底:
  ┌──────────────────────────────────────────────────┐
  │ 1. Alluxio中user表的最新缓存作为降级数据          │
  │ 2. 响应头添加: X-Data-Source: alluxio-cache      │
  │ 3. 响应头添加: X-Cache-Age: 3600 (缓存1小时前)   │
  │ 4. 仅返回缓存的列，可能缺失新增列                 │
  └──────────────────────────────────────────────────┘

Level 3 - 业务降级:
  ┌──────────────────────────────────────────────────┐
  │ 1. 跳过MySQL数据源的查询部分                      │
  │ 2. 返回部分结果 + 标识不可用字段                  │
  │ 3. 示例: 用户分析报表 → "用户维度"显示"N/A"      │
  │ 4. 降级期间不保证数据完整性                       │
  └──────────────────────────────────────────────────┘

具体实现代码:

-- Trino Session属性控制降级行为
SET SESSION mysql_connection_timeout = '10s';
SET SESSION mysql_query_timeout = '30s';
SET SESSION allow_cache_fallback = true;

-- 查询结果中的降级标识
-- 正常响应: { "user_name": "张三", "source": "mysql" }
-- 降级响应: { "user_name": "张三", "source": "cache", "cache_age_seconds": 3600 }
```

#### 场景2: 阿里云 Hologres 故障

```
故障现象: Hologres实例不可用/查询超时

影响面: 实时数仓查询不可用(~30%查询量)

降级方案:

短期 (< 1小时):
  1. Hologres查询自动路由到Trino + Iceberg备份表
     ┌──────────────────────────────────────────┐
     │ Hologres(实时) ──故障──→ Iceberg(T+1)   │
     │ 延迟: < 1秒         → 延迟: T+1          │
     │ 数据: 最新          → 数据: 昨天的快照   │
     └──────────────────────────────────────────┘
  
  2. 查询改写规则:
     原查询: SELECT * FROM hologres.aliyun.realtime_gmv
     改写后: SELECT * FROM iceberg.warehouse.dws_gmv_1d WHERE dt = CURRENT_DATE - 1

中期 (1-24小时):
  1. 如果Hologres有异地灾备实例 → DNS切换
  2. 如果没有灾备 → 持续使用Iceberg兜底 + 页面提示"数据延迟1天"

长期 (> 24小时):
  1. 启动Hologres恢复流程 (阿里云工单)
  2. 通知所有下游业务方数据时效降级

优雅降级查询实现:

-- 通过Trino UDF实现自动降级路由
CREATE OR REPLACE FUNCTION safe_hologres_query(
    query_template VARCHAR,
    fallback_query VARCHAR
) RETURNS TABLE (...)

-- Session中的降级标记
SET SESSION hologres_fault_tolerance = 'FALLBACK_TO_ICEBERG';
SET SESSION hologres_max_wait_ms = 5000;  -- 超时5秒后降级
```

#### 场景3: Azure Blob Storage 故障

```
故障现象: Azure Storage Account不可用/限流

影响面: 历史归档数据查询不可用(~5%查询量，但数据量最大)

降级方案:

短期 (查询重试):
  1. 指数退避重试: 1s → 2s → 4s (最多3次)
  2. 重试间切换Storage Account (如有跨Region副本)
  3. 若所有重试失败 → 进入中期降级

中期 (跳过不可用分区):
  1. 返回部分结果(排除Azure不可用分区的数据)
  2. 结果中添加 Warning 元数据
  
  示例查询:
  SELECT partition_id, COUNT(*) as cnt
  FROM hive.azure.logs_archive
  WHERE dt BETWEEN '2023-01-01' AND '2024-01-01'
  GROUP BY partition_id;
  
  降级后结果:
  partition_id  │ cnt    │ status
  ──────────────┼────────┼─────────
  2024-01-01    │ 1.2M   │ OK
  2024-01-02    │ 1.1M   │ OK
  2024-01-03    │ 0      │ UNAVAILABLE  ← Azure Blob故障
  ...
  
  返回HTTP Header:
  X-Query-Completeness: 98.5%  ← 数据完整度
  X-Excluded-Partitions: 7     ← 跳过的分区数
  X-Excluded-Reason: azure_storage_unavailable

长期 (数据恢复):
  1. 从异地灾备副本恢复(OSS跨Region复制)
  2. 恢复期间限制Azure源的查询并发(排队等待)
```

#### 场景4: GCP BigQuery 故障

```
故障现象: BigQuery Job queue满/配额耗尽/API限流

影响面: 广告数据分析不可用(~15%查询量)

降级方案:

短期 (限流保护):
  1. 降低BigQuery并发查询数 (从20降到5)
  2. 排队的查询展示进度条 + 预估等待时间
  3. 优先级调度: 核心业务报表优先于临时分析

中期 (预计算结果替换):
  1. 关键广告指标(KPI)每小时定时汇总到Iceberg
     - CTR, CPC, CPM, 消耗金额 等20个核心指标
     - 每小时JOB: Flink实时计算+每小时checkpoint到Iceberg
  2. BigQuery故障时直接用Iceberg汇总数据
  3. 非核心查询直接拒绝，返回预计算的Dashboard链接

长期 (降级SOP):
  1. 申请BigQuery配额提升 (GCP工单)
  2. 评估是否需要购买BigQuery预留Slot(固定费用)
  3. 容量规划: 预留Slot 1000 = $29,000/月

BigQuery连接池配置:
# Trino BigQuery Connector配置
bigquery.max-connections=20          # 最大连接数
bigquery.connection-timeout=30s      # 连接超时
bigquery.query-timeout=10m           # 单查询超时
bigquery.backoff-max-delay=32s       # 最大退避时间
bigquery.retry-limit=3               # 重试次数
```

### 12.4 降级监控与告警

```
降级相关监控指标:

# 数据源健康度
datasource_status{name="mysql_aws"} 0/1          # 0=不健康
datasource_latency_p99{name="mysql_aws"}         # P99延迟
datasource_error_rate{name="mysql_aws"}          # 错误率

# 降级触发
degradation_triggered_total{source="mysql_aws", level="L2"}  # 降级次数
degradation_active{source="mysql_aws"} 0/1       # 当前降级状态

# 缓存兜底
cache_fallback_hit_total{source="mysql_aws"}     # 降级缓存命中次数
cache_fallback_staleness_seconds{source="mysql_aws"}  # 缓存数据陈旧度

告警规则:
  - datasource_status=0 持续 > 1分钟 → P1告警
  - degradation_triggered_total 1小时内增加 > 5 → P2告警
  - cache_fallback_staleness_seconds > 3600 → P3告警(数据过期风险)
```
