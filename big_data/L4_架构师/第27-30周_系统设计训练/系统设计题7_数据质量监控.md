# 系统设计题7：数据质量监控平台

## 一、需求分析

### 1.1 功能性需求

| 维度 | 需求详情 |
|------|----------|
| **监控对象** | Hive表（5000+）、Kafka Topic（200+）、ClickHouse表（100+）、MySQL表（50+）、API数据源（20+） |
| **监控维度** | 完整性（行数/字段非空率）、准确性（业务规则校验）、一致性（跨表对账）、及时性（数据到达时间）、唯一性（主键去重） |
| **监控频率** | 实时监控（数据管道中）、准实时（每5分钟）、T+1（每日产出后） |
| **告警方式** | 企业微信/钉钉/邮件/电话、分级告警（P0-P3） |
| **用户角色** | 数据开发（配置规则）、数据分析师（查看质量报告）、SRE（处理告警）、管理者（质量趋势仪表盘） |

### 1.2 非功能性需求

```
性能:
  - 单表质量检查: < 30秒 (1亿行/100列级别)
  - 全量检查: 5000+表每天凌晨4小时内完成
  - 实时检查: 延迟 < 1分钟

可用性:
  - 监控平台: 99.9%
  - 告警通道: 99.99% (多通道冗余)

扩展性:
  - 新增监控规则零代码（配置化）
  - 支持自定义检查逻辑（Python/SQL UDF）

成本:
  - 质量检查任务的计算成本不应超过被检查任务的10%
```

## 五、最终选择：方案B（自研数据质量平台）

### 选择理由

**1. 数据安全是第一优先级**
作为电商平台，用户交易数据绝对不能上传到第三方SaaS。这是选择自研方案的决定性因素。

**2. Web UI降低全公司使用门槛**
数据质量不应该只是数据工程师的事。运营、产品、财务团队也需要能查看自己关心的数据质量。自研Web UI可以做到让非技术人员每天看一眼"数据质量雷达图"就知道数据是否可信。

**3. 实时监控是刚需**
电商大促期间，实时管道的数据质量直接影响运营决策。方案A的Great Expectations仅支持离线检查，无法满足实时监控需求。

**4. 投入产出合理**
2-3人团队3个月开发MVP，6个月完善。全年人力成本约$30万。对比方案C商业工具$5-10K/月 + 数据安全风险，自研方案1.5年可回本。

```json
  },

  "enabled": true
}
```

### 6.2 规则类型定义

```
规则模板（零代码配置，下拉选择）：

完整性检查:
  - row_count_range: 行数最大值/最小值
  - not_null_rate: 字段非空率
  - distinct_count: 去重值数量
  - partition_exists: 分区是否存在

准确性检查:
  - value_in_set: 字段值在规定集合内
  - value_range: 数值字段范围
  - regex_match: 正则匹配（手机号/邮箱格式）
  - custom_sql: 自定义SQL校验（AND逻辑）

一致性检查:
  - cross_table_compare: 跨表对账（如订单表金额SUM vs 支付表金额SUM）
  - cross_source_compare: 跨源对账（Hive vs ClickHouse）

及时性检查:
  - data_arrival_time: 数据到达时间 < 指定时间
  - data_freshness: 数据新鲜度（max(event_time) < now - N小时）
```

### 6.3 执行引擎

```python
# 离线检查执行器 (PySpark)
class OfflineQualityChecker:
    def __init__(self, spark_session, rule_config):
        self.spark = spark_session
        self.config = rule_config
    
    def check_row_count(self, target, params):
        df = self.spark.table(f"{target['database']}.{target['table']}")
        count = df.count()
        passed = params['min_rows'] <= count <= params['max_rows']
        return {
            'rule_id': self.config['rule_id'],
            'actual_value': count,
            'expected_range': [params['min_rows'], params['max_rows']],
            'passed': passed,
            'severity': self.config['severity']
        }
    
    def check_not_null_rate(self, target, params):
        df = self.spark.table(f"{target['database']}.{target['table']}")
        total = df.count()
        not_null = df.filter(col(params['column']).isNotNull()).count()
        rate = not_null / total if total > 0 else 0
        passed = rate >= params['min_rate']
        return {
            'rule_id': self.config['rule_id'],
            'actual_rate': rate,
            'expected_min_rate': params['min_rate'],
            'passed': passed,
        }
```

### 6.4 实时监控

```sql
-- Flink SQL 实时数据质量检查
CREATE TABLE kafka_source (
    event_time TIMESTAMP(3),
    user_id BIGINT,
    order_id BIGINT,
    amount DECIMAL(10,2),
    WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
) WITH ('connector' = 'kafka', ...);

-- 实时异常检测: 金额为负数的订单
SELECT 
    TUMBLE_START(event_time, INTERVAL '1' MINUTE) AS window_start,
    COUNT(*) AS total_orders,
    SUM(CASE WHEN amount < 0 THEN 1 ELSE 0 END) AS negative_amount_orders,
    SUM(CASE WHEN user_id IS NULL THEN 1 ELSE 0 END) AS null_user_orders
FROM kafka_source
GROUP BY TUMBLE(event_time, INTERVAL '1' MINUTE)
HAVING SUM(CASE WHEN amount < 0 THEN 1 ELSE 0 END) > 0
    OR SUM(CASE WHEN user_id IS NULL THEN 1 ELSE 0 END) > 0;
```

### 6.5 告警分级与处理

```
P0 - 紧急（立即处理，电话+企微+钉钉）:
  - 核心GMV表行数为0
  - 核心Kafka Topic无数据超过10分钟
  - 数据延迟超过SLA 2倍
  
P1 - 严重（30分钟内响应，企微+钉钉）:
  - 表行数偏差超过20%
  - 关键字段非空率低于90%
  - 跨表对账差异超过1%

P2 - 一般（2小时内响应，钉钉）:
  - 表行数偏差超过10%
  - 非关键表数据延迟
  - 标签覆盖率下降

P3 - 提示（日报汇总，不单独告警）:
  - 非关键字段非空率轻微下降
  - 新增表未注册质量规则
  - 质量规则执行超时

告警收敛:
  - 同一规则5分钟内只告警1次
  - 同一表的多条规则合并为1条告警
  - P0告警持续未恢复：每15分钟升级提醒
```

## 九、30+数据质量规则完整定义

### 9.1 完整性规则（Completeness）- 10条

```sql
-- 规则1: 表行数范围检查
-- 描述: 检查分区表行数是否在合理范围内
-- 适用: 所有业务表
SELECT 'rule_row_count_range' AS rule_id, '${dt}' AS check_date,
       COUNT(*) AS actual_count,
       CASE WHEN COUNT(*) BETWEEN ${min_rows} AND ${max_rows} THEN 'PASS' ELSE 'FAIL' END AS result
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则2: 列非空率检查
-- 描述: 检查关键字段的非空比例是否达标
SELECT 'rule_not_null_rate' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ${column} IS NOT NULL THEN 1 ELSE 0 END) AS not_null_rows,
       ROUND(SUM(CASE WHEN ${column} IS NOT NULL THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS not_null_rate,
       CASE WHEN (SUM(CASE WHEN ${column} IS NOT NULL THEN 1 ELSE 0 END) * 100.0 / COUNT(*)) >= ${min_rate}
            THEN 'PASS' ELSE 'FAIL' END AS result
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则3: 列完整度检查（多列同时非空）
-- 描述: 组合字段同时非空的行比例
SELECT 'rule_column_completeness' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ${col1} IS NOT NULL AND ${col2} IS NOT NULL AND ${col3} IS NOT NULL 
           THEN 1 ELSE 0 END) AS complete_rows,
       ROUND(SUM(CASE WHEN ${col1} IS NOT NULL AND ${col2} IS NOT NULL AND ${col3} IS NOT NULL 
           THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS completeness_rate
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则4: 分区存在性检查
-- 描述: 确认指定日期的分区是否生成
SELECT 'rule_partition_exists' AS rule_id,
       CASE WHEN EXISTS(
           SELECT 1 FROM ${database}.${table} WHERE dt = '${dt}' LIMIT 1
       ) THEN 'PASS' ELSE 'FAIL' END AS result;

-- 规则5: 分区连续性检查
-- 描述: 检查连续N天是否都有分区
SELECT 'rule_partition_continuity' AS rule_id,
       dt, LAG(dt) OVER (ORDER BY dt) AS prev_dt,
       DATEDIFF(dt, LAG(dt) OVER (ORDER BY dt)) AS gap_days
FROM (
    SELECT DISTINCT dt FROM ${database}.${table} ORDER BY dt
) t
WHERE DATEDIFF(dt, LAG(dt) OVER (ORDER BY dt)) > 1;

-- 规则6: 去重值数量检查
-- 描述: 检查某列去重值数量是否达标（如活跃用户数）
SELECT 'rule_distinct_count' AS rule_id,
       COUNT(DISTINCT ${column}) AS distinct_count,
       CASE WHEN COUNT(DISTINCT ${column}) >= ${min_distinct} 
            THEN 'PASS' ELSE 'FAIL' END AS result
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则7: 数据覆盖率检查（用户ID是否存在）
-- 描述: 检查事实表的外键是否在维表中存在
SELECT 'rule_fk_coverage' AS rule_id,
       COUNT(DISTINCT f.${fk_column}) AS total_fk,
       SUM(CASE WHEN d.${pk_column} IS NOT NULL THEN 1 ELSE 0 END) AS matched_fk,
       ROUND(SUM(CASE WHEN d.${pk_column} IS NOT NULL THEN 1 ELSE 0 END) * 100.0 
             / COUNT(DISTINCT f.${fk_column}), 2) AS coverage_rate
FROM ${fact_table} f
LEFT JOIN ${dim_table} d ON f.${fk_column} = d.${pk_column}
WHERE f.dt = '${dt}';

-- 规则8: 记录数零值检查
-- 描述: 确保关键表的分区不为空
SELECT 'rule_non_zero_records' AS rule_id,
       COUNT(*) AS row_count,
       CASE WHEN COUNT(*) > 0 THEN 'PASS' ELSE 'FAIL' END AS result
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则9: 增量数据量波动检查
-- 描述: 检查今日增量是否在历史均值的合理范围
WITH daily_stats AS (
    SELECT dt, COUNT(*) AS row_count
    FROM ${database}.${table}
    WHERE dt >= DATE_SUB('${dt}', 30)
    GROUP BY dt
)
SELECT 'rule_daily_volume_stability' AS rule_id,
       today.row_count AS today_count,
       AVG(his.row_count) AS avg_30d,
       STDDEV(his.row_count) AS stddev_30d,
       CASE WHEN ABS(today.row_count - AVG(his.row_count)) <= 3 * STDDEV(his.row_count)
            THEN 'PASS' ELSE 'FAIL' END AS result
FROM (SELECT row_count FROM daily_stats WHERE dt = '${dt}') today
CROSS JOIN (SELECT row_count FROM daily_stats WHERE dt < '${dt}') his;

-- 规则10: 多分区行数一致性（同一表不同分区行数差异不应过大）
SELECT 'rule_partition_row_consistency' AS rule_id,
       dt, COUNT(*) AS row_count
FROM ${database}.${table}
WHERE dt >= DATE_SUB('${dt}', 7)
GROUP BY dt
HAVING COUNT(*) < (SELECT AVG(cnt) * 0.5 FROM (
    SELECT dt, COUNT(*) AS cnt FROM ${database}.${table}
    WHERE dt >= DATE_SUB('${dt}', 7) GROUP BY dt
) t);
```

### 9.2 准确性规则（Accuracy）- 8条

```sql
-- 规则11: 枚举值检查
-- 描述: 检查字段值是否在预定义的合法集合内
SELECT 'rule_value_in_set' AS rule_id,
       ${column} AS invalid_value,
       COUNT(*) AS invalid_count
FROM ${database}.${table}
WHERE dt = '${dt}'
  AND ${column} NOT IN (${allowed_values})
GROUP BY ${column};

-- 规则12: 数值范围检查
-- 描述: 检查数值字段是否在合理范围内
SELECT 'rule_value_range' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ${column} < ${min_value} OR ${column} > ${max_value} 
           THEN 1 ELSE 0 END) AS outlier_count,
       ROUND(SUM(CASE WHEN ${column} < ${min_value} OR ${column} > ${max_value} 
           THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS outlier_rate
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则13: 正则表达式匹配检查（手机号格式）
-- 描述: 检查手机号是否符合标准格式
SELECT 'rule_regex_match_phone' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ${column} REGEXP '^1[3-9]\\d{9}$' 
           THEN 1 ELSE 0 END) AS valid_rows,
       ROUND(SUM(CASE WHEN ${column} REGEXP '^1[3-9]\\d{9}$' 
           THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) AS valid_rate
FROM ${database}.${table}
WHERE dt = '${dt}' AND ${column} IS NOT NULL;

-- 规则14: 业务规则校验（订单金额 = 单价 × 数量）
-- 描述: 检查业务计算逻辑是否正确
SELECT 'rule_business_logic' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ABS(${amount} - ${unit_price} * ${quantity}) > 0.01 
           THEN 1 ELSE 0 END) AS logic_error_count
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则15: 日期逻辑校验（结束时间 > 开始时间）
SELECT 'rule_date_logic' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ${end_time} <= ${start_time} THEN 1 ELSE 0 END) AS error_count
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则16: 统计异常检测（Z-Score离群值）
WITH stats AS (
    SELECT AVG(${column}) AS mean_val, STDDEV(${column}) AS stddev_val
    FROM ${database}.${table} WHERE dt = '${dt}'
)
SELECT 'rule_statistical_outlier' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ABS((${column} - mean_val) / NULLIF(stddev_val, 0)) > 3 
           THEN 1 ELSE 0 END) AS outlier_count
FROM ${database}.${table} CROSS JOIN stats
WHERE dt = '${dt}';

-- 规则17: 主键唯一性检查
SELECT 'rule_pk_unique' AS rule_id,
       COUNT(*) AS total_rows,
       COUNT(DISTINCT ${pk_columns}) AS unique_keys,
       COUNT(*) - COUNT(DISTINCT ${pk_columns}) AS duplicate_count,
       CASE WHEN COUNT(*) = COUNT(DISTINCT ${pk_columns}) THEN 'PASS' ELSE 'FAIL' END AS result
FROM ${database}.${table}
WHERE dt = '${dt}';

-- 规则18: 嵌套数据完整性（如地址包含省/市/区）
SELECT 'rule_nested_completeness' AS rule_id,
       COUNT(*) AS total_rows,
       SUM(CASE WHEN ${province_col} IS NOT NULL AND ${city_col} IS NOT NULL 
           THEN 1 ELSE 0 END) AS complete_nested
FROM ${database}.${table}
WHERE dt = '${dt}';
```

### 9.3 一致性规则（Consistency）- 7条

```sql
-- 规则19: 跨表行数对账（订单表 vs 订单明细表）
SELECT 'rule_cross_table_row_count' AS rule_id,
       (SELECT COUNT(*) FROM ${database}.order_header WHERE dt = '${dt}') AS header_count,
       (SELECT COUNT(*) FROM ${database}.order_detail WHERE dt = '${dt}') AS detail_count,
       ABS((SELECT COUNT(*) FROM ${database}.order_header WHERE dt = '${dt}') - 
           (SELECT COUNT(*) FROM ${database}.order_detail WHERE dt = '${dt}')) AS diff;

-- 规则20: 跨表金额对账（订单金额SUM = 支付金额SUM）
SELECT 'rule_cross_table_amount' AS rule_id,
       (SELECT ROUND(SUM(order_amount), 2) FROM ${database}.orders WHERE dt = '${dt}') AS order_total,
       (SELECT ROUND(SUM(payment_amount), 2) FROM ${database}.payments WHERE dt = '${dt}') AS payment_total,
       ROUND(ABS(
           (SELECT SUM(order_amount) FROM ${database}.orders WHERE dt = '${dt}') - 
           (SELECT SUM(payment_amount) FROM ${database}.payments WHERE dt = '${dt}')
       ) / NULLIF((SELECT SUM(order_amount) FROM ${database}.orders WHERE dt = '${dt}'), 0) * 100, 4) AS diff_pct;

-- 规则21: 跨源对账（Hive vs ClickHouse）
-- 在Hive端列出总行数，在ClickHouse端也用类似SQL
-- 对比结果通过Python脚本汇总

-- 规则22: 时间序列数据连续性（无断点）
SELECT 'rule_timeseries_continuity' AS rule_id,
       event_time, LAG(event_time) OVER (ORDER BY event_time) AS prev_time,
       TIMESTAMPDIFF(SECOND, LAG(event_time) OVER (ORDER BY event_time), event_time) AS gap_seconds
FROM ${database}.${table}
WHERE dt = '${dt}'
  AND TIMESTAMPDIFF(SECOND, LAG(event_time) OVER (ORDER BY event_time), event_time) > ${max_gap};

-- 规则23: 上下游一致性（ODS行数 >= DWD行数）
SELECT 'rule_upstream_downstream_row' AS rule_id,
       (SELECT COUNT(*) FROM ods.${table} WHERE dt = '${dt}') AS ods_count,
       (SELECT COUNT(*) FROM dwd.${table} WHERE dt = '${dt}') AS dwd_count,
       CASE WHEN (SELECT COUNT(*) FROM dwd.${table} WHERE dt = '${dt}') >= 
                 (SELECT COUNT(*) FROM ods.${table} WHERE dt = '${dt}') * 0.95
            THEN 'PASS' ELSE 'FAIL' END AS result;

-- 规则24: 维度一致性（事实表中的维度值在维表中存在）
SELECT 'rule_dimension_consistency' AS rule_id,
       a.${dim_key},
       COUNT(*) AS orphan_count
FROM ${fact_table} a
LEFT JOIN ${dim_table} b ON a.${dim_key} = b.${dim_key}
WHERE a.dt = '${dt}' AND b.${dim_key} IS NULL
GROUP BY a.${dim_key};

-- 规则25: 聚合一致性（明细聚合 = 汇总表）
WITH detail_agg AS (
    SELECT ${group_col}, SUM(${agg_col}) AS detail_sum
    FROM ${detail_table} WHERE dt = '${dt}' GROUP BY ${group_col}
)
SELECT 'rule_aggregation_consistency' AS rule_id,
       d.${group_col},
       d.detail_sum,
       s.${agg_col_name} AS summary_val,
       ABS(d.detail_sum - s.${agg_col_name}) AS diff
FROM detail_agg d
JOIN ${summary_table} s ON d.${group_col} = s.${group_col} AND s.dt = '${dt}'
WHERE ABS(d.detail_sum - s.${agg_col_name}) > ${tolerance};
```

```sql
            THEN 'PASS' ELSE 'FAIL' END AS result
FROM ${database}.${table};

SELECT 'rule_kafka_lag' AS rule_id,
```

```sql
WHERE table_schema = '${database}'
GROUP BY table_name;
```

### 11.2 DQ Score计算公式

```
DQ Score = 加权综合评分 (0-100)

组成:
  完整性得分(C) = Σ(各表完整性检查通过率 × 表权重) / Σ表权重
    权重: 核心表×5, 重要表×3, 普通表×1
    通过率: PASS规则数 / 总规则数

  准确性得分(A) = Σ(各表准确性检查通过率 × 表权重) / Σ表权重
  
  一致性得分(O) = Σ(各跨表对账检查通过率 × 对账权重) / Σ对账权重
  
  及时性得分(T) = Σ(各ETL按时完成率 × 任务权重) / Σ任务权重

DQ Score = C × 30% + A × 30% + O × 20% + T × 20%

等级:
  A级: ≥ 95分 (优秀)
  B级: 85-94分 (良好)
  C级: 70-84分 (合格)
  D级: 60-69分 (待改进)
  F级: < 60分 (不合格)
```

### 12.1 告警升级策略

```
告警升级时间线:

T+0min: 质量检查失败 → 企业微信机器人（开发群）
T+5min: 未确认 → 钉钉群通知 + @owner
T+15min: P0级未确认 → 升级到数据平台Leader
T+30min: P0级未确认 → 升级到数据总监
T+60min: P0级未确认 → 电话通知 + CTO群

升级条件:
  P0: 15分钟内未确认 → 立即升级
  P1: 30分钟内未确认 → 升级
  P2: 2小时内未确认 → 升级
  P3: 非工作时间不升级（在日报告中汇总）
```

### 12.2 数据阻断策略

```python
# 数据管道阻断器
class DataPipelineBlocker:
    """
    当数据质量检查严重失败时，阻断下游管道
    
    阻断策略:
      - 核心表P0失败: 阻断所有下游ETL（硬阻断）
      - 核心表P1失败: 阻断关键下游（软阻断，可手动跳过）
      - 非核心表P0失败: 仅阻断自身下游（不影响其他表）
    """
    
    def should_block_downstream(self, check_result):
        table_tier = self._get_table_tier(check_result.table)
        severity = check_result.severity
        
        if table_tier == "CRITICAL" and severity == "P0":
            return BlockDecision.BLOCK_ALL
        elif table_tier == "CRITICAL" and severity == "P1":
            return BlockDecision.BLOCK_CRITICAL_DOWNSTREAM
        elif table_tier == "IMPORTANT" and severity == "P0":
            return BlockDecision.BLOCK_OWN_DOWNSTREAM
        else:
            return BlockDecision.ALLOW_WITH_WARNING

# 阻断实施流程:
# 1. DQ Check失败 → 计算阻断决策
# 2. 写入阻断标记表: block_rules(table, downstream, reason, expiry)
# 3. 下游ETL启动前检查阻断标记表 → 发现阻断 → 不执行
# 4. DQ问题修复 + 重新验证 → 清理阻断标记
# 5. 重新触发下游ETL
```

### 12.4 自动修复机制

```
自动修复场景:

场景1: 分区延迟（已确认上游数据最终会到达）
  → 等待30分钟后自动重检
  → 最多重试3次（每次间隔30分钟）
  → 3次后仍失败 → 自动降级为P2告警

场景2: 数据量轻微偏差（实际值在历史3σ范围内但超出设定阈值）
  → 自动调整阈值（基于历史30天均值±3σ）
  → 记录自动调整日志
  → 通知owner审核新阈值

场景3: 非空率轻微下降（≥95%但 < 99%）
  → 自动降低告警级别（P1→P2）
  → 连续3天不达标才升级为P1

场景4: 跨表对账差异
  → 自动触发数据回补流程
  → 回补完成后自动重检
  → 回补失败则升级告警
```

## 十三、DQ SLA定义和度量方法

### 13.1 SLA定义

| SLA指标 | 定义 | 目标值 | 度量方式 | 报告频率 |
|---------|------|--------|----------|----------|
| **数据完整性** | 数据到达且行数在预期范围内 | ≥ 99.5% | (PASS表数 / 总检查表数) × 100% | 日/周/月 |
| **数据准确性** | 业务规则和格式校验全部通过 | ≥ 99.9% | (PASS规则数 / 总检查规则数) × 100% | 日/周/月 |
| **数据及时性** | 在规定SLA时间前数据就绪 | ≥ 99.5% | (按时就绪表数 / 总表数) × 100% | 日/周/月 |
| **数据可用性** | 数据在承诺时间内可供查询 | ≥ 99.9% | 可用时间 / 总时间 | 月/季 |
| **告警响应时间** | P0告警从触发到响应的时间 | < 15分钟 | 中位数(P50) | 月 |
| **问题解决时间** | 从问题发现到修复完成的时间 | P0: < 2小时<br>P1: < 8小时<br>P2: < 24小时 | 中位数(P50) | 月 |

### 13.2 SLA度量方法

```python
# SLA度量计算器
class DataQualitySLA:
    
    def calculate_completeness_sla(self, date_range):
        """数据完整性SLA"""
        total_checks = self.get_dq_check_results(date_range).count()
        passed_checks = self.get_dq_check_results(date_range, result='PASS').count()
        return {
            'metric': 'data_completeness',
            'target': 99.5,
            'actual': round(passed_checks / total_checks * 100, 2),
            'achieved': (passed_checks / total_checks * 100) >= 99.5
        }
    
    def calculate_timeliness_sla(self, date_range):
        """数据及时性SLA: 每张表在规定时间前完成"""
        tables = self.get_all_monitored_tables()
        on_time_tables = 0
        
        for table in tables:
            sla_deadline = self._get_table_sla_deadline(table)
            actual_completion = self._get_actual_completion_time(table)
            if actual_completion <= sla_deadline:
                on_time_tables += 1
        
        return {
            'metric': 'data_timeliness',
            'target': 99.5,
            'actual': round(on_time_tables / len(tables) * 100, 2),
            'achieved': (on_time_tables / len(tables) * 100) >= 99.5
        }
    
    def calculate_availability_sla(self, date_range):
        """数据可用性SLA: 查询服务可用时间"""
        total_minutes = self._get_total_minutes(date_range)
        downtime_minutes = self._get_downtime_minutes(date_range)
        availability = (total_minutes - downtime_minutes) / total_minutes * 100
        return {
            'metric': 'data_availability',
            'target': 99.9,
            'actual': round(availability, 3),
            'achieved': availability >= 99.9
        }
    
    def calculate_alert_response_sla(self, date_range):
        """告警响应SLA"""
        p0_alerts = self._get_alerts(date_range, severity='P0')
        response_times = [alert.ack_time - alert.trigger_time for alert in p0_alerts]
        p50_response = np.percentile(response_times, 50)
        return {
            'metric': 'alert_response_p50',
            'target_seconds': 900,  # 15分钟
            'actual_seconds': p50_response.total_seconds(),
            'achieved': p50_response.total_seconds() <= 900
        }
```
