# 系统设计题6：实时用户画像系统

## 一、需求分析

### 1.1 功能性需求

| 维度 | 需求详情 |
|------|----------|
| **画像维度** | 基础属性（年龄/性别/地域）、行为偏好（品类/时段/价格敏感度）、消费能力（LTV/RFM）、生命周期（新客/活跃/沉睡/流失）、实时行为（最近浏览/加购/下单） |
| **标签类型** | 统计类标签（频次/金额）、规则类标签（条件判断）、模型类标签（ML预测）、实时类标签（最近N分钟行为） |
| **更新频率** | 实时标签 < 1秒；准实时标签 < 5分钟；离线标签 T+1 |
| **查询场景** | 推荐引擎（<10ms）、运营平台（<100ms）、数据分析（< 1秒） |
| **用户规模** | 2亿注册用户，5000万月活，500万日活 |
| **标签数量** | 初期500+标签，远期3000+标签 |

### 1.2 非功能性需求

```
性能:
  - 实时标签更新: 端到端 < 1秒 (从行为产生到标签可查)
  - 标签查询: P99 < 10ms (推荐引擎调用)
  - 离线标签: T+1 早8点前全量产出

可用性:
  - 在线查询: 99.99%
  - 离线计算: 99.9%

一致性:
  - 实时标签: 最终一致 (< 1秒)
  - 离线标签: T+1强一致

数据质量:
  - 标签覆盖率 > 90%
  - 标签准确率 > 95%
```

## 三、候选方案对比

### 方案A：Flink + Redis + HBase (经典方案)

```

└────────────────────────┬─────────────────────────────────┘
                         │
┌────────────────────────▼─────────────────────────────────┐
│  实时计算层 (Flink)                                       │
│  ┌─────────────┐  ┌──────────────┐  ┌──────────────┐    │
│  │ 规则引擎    │  │ 窗口聚合     │  │ CEP事件匹配  │    │
│  │ (Drools)   │  │ (5min/1h/1d)│  │ (用户行为链) │    │
│  └─────────────┘  └──────────────┘  └──────────────┘    │
└──────────┬────────────────────┬──────────────────────────┘
           │                    │
     ┌─────▼─────┐       ┌──────▼──────┐
     │   Redis   │       │    HBase    │
     │ 实时标签  │       │  全量标签   │
     │ (TTL 7天) │       │ (永久存储)  │
     └─────┬─────┘       └──────┬──────┘
           │                    │
           └────────┬───────────┘
                    │
           ┌────────▼────────┐
           │  标签查询服务    │
           │  (Redis热数据   │
           │   + HBase兜底)  │
           └─────────────────┘
```

**优点**:
- Flink CEP + 规则引擎实时性最好
- Redis性能极佳，满足10ms SLA
- HBase全量存储，标签可回溯
- 架构经典，业界大量验证

**缺点**:
- 三层存储（Flink State + Redis + HBase），运维复杂
- Redis纯内存成本高（2TB数据需要大量内存）
- HBase在大量随机读场景下GC压力大
- 实时/离线两套标签可能不一致

### 方案B：Flink + Hologres/StarRocks (统一OLAP)

```
┌──────────────────────────────────────────────────────────┐
│  Kafka ──→ Flink ETL ──→ Hologres / StarRocks           │
│                             │                             │
│                      ┌──────▼──────┐                     │
│                      │  标签查询    │                     │
│                      │  (SQL→API)  │                     │
│                      └─────────────┘                     │
│                                                          │
│  T+1: Spark ──→ 批量更新标签宽表                          │
└──────────────────────────────────────────────────────────┘
```

**优点**:
- 统一存储，实时+离线同一份数据
- SQL查询灵活，新增标签无需修改代码
- StarRocks P99 < 10ms可能

**缺点**:
- 2亿用户 × 500标签 主键模型更新压力大
- OLAP引擎10ms延迟不稳定（GC波动）
- 实时写入与查询资源竞争

### 方案C：自研 Bitmap + RocksDB 方案 (极致性能)


**优点**:
- Bitmap圈人查询性能极致（百万级人群交集 < 10ms）
- 位图存储高效（500标签仅需12.5GB内存）
- 标签点查同样快速（RocksDB + 内存缓存）
- 成本最低（无需Redis/HBase集群）

**缺点**:
- 开发周期长（6-12个月），需要C++/Rust专业能力
- 非标准标签（Float/String）Bitmap不支持
- 运维工具需要全部自建
- 不适合非大厂团队

## 五、最终选择：方案A增强版（Flink + Redis + HBase + Bitmap）

### 选择理由

**1. 分层存储，各取所长**
- Redis存储最近7天活跃用户的热标签（~500万用户，50GB数据），满足10ms SLA
- HBase存储全量2亿用户标签，支持按需加载
- 在查询层增加Bitmap索引，加速人群圈选

**2. 经验曲线短**
方案A是业界最成熟的画像系统架构，团队可以1个月内完成MVP。方案C的自研方案虽然性能最优但风险太高。

**3. 渐进式优化**
先用方案A上线，后续逐步优化：
- 阶段1：引入本地缓存（Caffeine），减少Redis调用
- 阶段2：引入Bitmap索引，加速圈人查询
- 阶段3：评估是否切换HBase到更优存储（如TiKV）

### 6.2 Redis + HBase 存储模型

```
Redis存储 (热数据, 7天TTL):
  Key: "user_tag:{user_id}"
  Value: Protobuf序列化的活跃标签

  "tag:gender:male" → [01001101...] (2亿bit)
  "tag:city:beijing" → [10010010...]
  "tag:consume:high" → [00101100...]

  圈选 [北京] AND [高消费] AND [男]:
    bit_and(city_beijing, consume_high, gender_male)
    → 毫秒级返回目标用户ID列表
```

### 6.4 查询服务

```go
// 标签查询服务 (Go, 无状态)
func QueryUserTags(ctx context.Context, userIds []int64, tagNames []string) (map[int64]map[string]interface{}, error) {
    result := make(map[int64]map[string]interface{})
    missedIds := []int64{}
    
    // L1: 本地缓存 (Caffeine/LRU)
    for _, uid := range userIds {
        if tags, ok := localCache.Get(uid); ok {
            result[uid] = filterTags(tags, tagNames)
        } else {
            missedIds = append(missedIds, uid)
        }
    }
    
    // L2: Redis (热数据)
    if len(missedIds) > 0 {
        redisResults := redis.MGet(ctx, toRedisKeys(missedIds))
        stillMissed := []int64{}
        for i, val := range redisResults {
            if val != nil {
                result[missedIds[i]] = parseTags(val)
                localCache.Set(missedIds[i], result[missedIds[i]])
            } else {
                stillMissed = append(stillMissed, missedIds[i])
            }
        }
        missedIds = stillMissed
    }
    
    // L3: HBase (全量数据)
    if len(missedIds) > 0 {
        hbaseResults := hbase.BatchGet(ctx, "user_tags", missedIds)
        for uid, tags := range hbaseResults {
            result[uid] = tags
            localCache.Set(uid, tags)
            // 回写Redis热缓存
            redis.Set(ctx, "user_tag:"+uid, tags, 7*24*time.Hour)
        }
    }
    
    return result, nil
}
```

## 八、Bitmap 方案详细实现

### 8.1 Bitmap 存储模型设计

Bitmap(位图)是用户画像系统中实现人群圈选的高效数据结构。核心思想：为每个标签值维护一个bit数组，数组中每一位对应一个用户，1表示该用户有此标签。

```
存储结构设计:

用户ID映射:
  内部ID: 0 ~ 200,000,000 (连续整数，与外部用户ID映射表关联)
  每个bit代表一个用户: 2亿bit = 25MB (单个标签值的位图大小)
  
标签编码:
  tag_id 编码规则: namespace + tag_name + tag_value
  示例:
    tag:1:gender:male      → 内部ID 10001, Bitmap: 25MB
    tag:1:gender:female    → 内部ID 10002, Bitmap: 25MB
    tag:2:city:beijing     → 内部ID 20001, Bitmap: 25MB
    tag:3:consume:high     → 内部ID 30001, Bitmap: 25MB
    tag:4:age:25_35        → 内部ID 40001, Bitmap: 25MB
    
  500个标签 × 25MB/标签 = 12.5GB (全量标签位图内存)
  
  考虑压缩:
    RoaringBitmap: 实际使用压缩后约 3-5GB (取决于标签稀疏度)
    压缩率: 大部分标签只覆盖少数用户 (如"已购iPhone"可能只有0.1%用户)
```

### 8.2 RoaringBitmap 核心数据结构

RoaringBitmap是目前最高效的位图压缩实现，被广泛应用于Lucene、Spark、Druid、ClickHouse等系统中。

```
RoaringBitmap三层结构:

┌────────────────────────────────────────────────────────────┐
│                   RoaringBitmap                            │
│                                                             │
│  ┌──────────────────────────────────────────────────┐     │
│  │  Container[0]                                     │     │
│  │  key=0 (高16位 = 0)                              │     │
│  │  ┌─────────────────┐                             │     │
│  │  │ ArrayContainer   │ ← 稀疏(≤4096值)用short数组 │     │
│  │  │ [3, 5, 127, ...]│   2B/value, 空间 = 2×count │     │
│  │  └─────────────────┘                             │     │
│  ├──────────────────────────────────────────────────┤     │
│  │  Container[1]                                     │     │
│  │  key=1 (高16位 = 1)                              │     │
│  │  ┌─────────────────────────────┐                 │     │
│  │  │ BitmapContainer              │ ← 密集(>4096)  │     │
│  │  │ [0010101101...]  (8KB)      │   用bitmap存储  │     │
│  │  └─────────────────────────────┘                 │     │
│  ├──────────────────────────────────────────────────┤     │
│  │  Container[2]                                     │     │
│  │  key=2 (高16位 = 2)                              │     │
│  │  ┌────────────────────────────┐                  │     │
│  │  │ RunContainer                │ ← 连续值         │     │
│  │  │ [start=100, len=5000, ...] │   用RLE编码      │     │
│  │  └────────────────────────────┘                  │     │
│  └──────────────────────────────────────────────────┘     │
│                                                            │
│  关键概念:                                                  │
│   - 将32位int的高16位作为Container Key (最多65536个容器)   │
│   - 低16位的值存储在Container内部                           │
│   - 根据数据密度自动选择最优Container类型                   │
│   - 内存极小但运算极快(CPU cache友好)                       │
└────────────────────────────────────────────────────────────┘

Container自动选型:
  稀疏(≤4096个值): ArrayContainer → 2B/value
  密集(>4096个值): BitmapContainer → 固定8KB
  连续序列: RunContainer → 2 short/run
```

### 8.3 RoaringBitmap Java 实现

```java
import org.roaringbitmap.RoaringBitmap;
import org.roaringbitmap.buffer.ImmutableRoaringBitmap;
import java.util.concurrent.ConcurrentHashMap;
import java.util.List;
import java.util.Map;

/**
 * Bitmap画像索引服务
 * 核心能力: 标签值 → Bitmap映射, 人群圈选(Bitmap AND/OR/NOT), 查询缓存
 */
public class BitmapProfileIndex {

    // 标签值 → RoaringBitmap
    private final ConcurrentHashMap<String, RoaringBitmap> tagBitmaps;
    
    // 用户外部ID → 内部ID 映射
    private final ConcurrentHashMap<String, Integer> userToInternalId;
    
    // 内部ID → 用户外部ID 反向映射
    private final String[] internalIdToUser;
    
    // 全局用户位图 (所有已注册用户 = 1)
    private RoaringBitmap allUsersBitmap;
    
    public BitmapProfileIndex(int maxUsers) {
        this.tagBitmaps = new ConcurrentHashMap<>();
        this.userToInternalId = new ConcurrentHashMap<>(maxUsers);
        this.internalIdToUser = new String[maxUsers];
        this.allUsersBitmap = new RoaringBitmap();
    }
    
    /**
     * 注册新用户: 分配内部ID
     */
    public synchronized int registerUser(String externalUserId) {
        if (userToInternalId.containsKey(externalUserId)) {
            return userToInternalId.get(externalUserId);
        }
        int internalId = userToInternalId.size();
        userToInternalId.put(externalUserId, internalId);
        internalIdToUser[internalId] = externalUserId;
        allUsersBitmap.add(internalId);
        return internalId;
    }
    
    /**
     * 批量设置标签位图
     * 场景: T+1离线Spark任务产出标签 → 批量加载到内存
     */
    public void batchSetTagBitmap(String tagKey, byte[] serializedBitmap) {
        RoaringBitmap bitmap = new RoaringBitmap();
        try {
            bitmap.deserialize(java.nio.ByteBuffer.wrap(serializedBitmap));
            tagBitmaps.put(tagKey, bitmap);
        } catch (Exception e) {
            throw new RuntimeException("Failed to deserialize bitmap for tag: " + tagKey, e);
        }
    }
    
    /**
     * 单用户标签实时更新
     * 场景: Flink实时流计算出新标签 → 实时打标
     */
    public void setUserTag(int internalId, String tagKey, boolean value) {
        RoaringBitmap bitmap = tagBitmaps.computeIfAbsent(tagKey, k -> new RoaringBitmap());
        if (value) {
            bitmap.add(internalId);
        } else {
            bitmap.remove(internalId);
        }
    }
    
    /**
     * 人群圈选核心: Bitmap 逻辑运算
     * 
     * 输入: ["tag:gender:male", "tag:city:beijing", "tag:consume:high"]
     * 操作: 三个Bitmap做 AND 操作
     * 输出: 满足所有标签的用户Bitmap
     */
    public RoaringBitmap crowdSelection(List<String> andTags, List<String> orTags, List<String> notTags) {
        RoaringBitmap result = allUsersBitmap.clone();
        
        // AND 操作: result = result AND tag_bitmap_1 AND tag_bitmap_2 ...
        for (String tag : andTags) {
            RoaringBitmap tagBitmap = tagBitmaps.get(tag);
            if (tagBitmap == null) {
                return new RoaringBitmap(); // 标签不存在 → 空结果
            }
            result.and(tagBitmap);
        }
        
        // OR 操作: orResult = tag_bitmap_a OR tag_bitmap_b ...
        if (orTags != null && !orTags.isEmpty()) {
            RoaringBitmap orResult = new RoaringBitmap();
            for (String tag : orTags) {
                RoaringBitmap tagBitmap = tagBitmaps.get(tag);
                if (tagBitmap != null) {
                    orResult.or(tagBitmap);
                }
            }
            result.and(orResult);
        }
        
        // NOT 操作: result = result AND (NOT tag_bitmap)
        if (notTags != null) {
            for (String tag : notTags) {
                RoaringBitmap tagBitmap = tagBitmaps.get(tag);
                if (tagBitmap != null) {
                    result.andNot(tagBitmap);
                }
            }
        }
        
        return result;
    }
    
    /**
     * 人群圈选 + 用户ID回翻
     */
    public List<String> crowdSelectionWithUserIds(List<String> andTags, List<String> orTags, List<String> notTags) {
        RoaringBitmap result = crowdSelection(andTags, orTags, notTags);
        return result.stream()
            .mapToObj(internalId -> internalIdToUser[internalId])
            .collect(java.util.stream.Collectors.toList());
    }
    
    /**
     * 人群基数估算 (不展开用户ID，仅count)
     * RoaringBitmap.getCardinality() = O(容器数) ≈ 微秒级
     */
    public long estimateCrowdSize(List<String> andTags, List<String> orTags, List<String> notTags) {
        RoaringBitmap result = crowdSelection(andTags, orTags, notTags);
        return result.getLongCardinality();
    }
    
    /**
     * Bitmap内存占用统计
     */
    public Map<String, Long> getMemoryStats() {
        Map<String, Long> stats = new HashMap<>();
        long totalBytes = 0;
        for (Map.Entry<String, RoaringBitmap> entry : tagBitmaps.entrySet()) {
            long sizeInBytes = entry.getValue().getSizeInBytes();
            stats.put(entry.getKey(), sizeInBytes);
            totalBytes += sizeInBytes;
        }
        stats.put("__TOTAL__", totalBytes);
        return stats;
    }

    /**
     * Bitmap持久化与恢复
     * 场景: 服务重启时从磁盘快速恢复Bitmap，避免全量重建
     */
    public void saveToDisk(String path) {
        for (Map.Entry<String, RoaringBitmap> entry : tagBitmaps.entrySet()) {
            String fileName = path + "/" + sanitizeFileName(entry.getKey()) + ".rbm";
            try (java.io.FileOutputStream fos = new java.io.FileOutputStream(fileName)) {
                entry.getValue().serialize(java.nio.channels.Channels.newChannel(fos));
            } catch (Exception e) {
                throw new RuntimeException("Failed to save bitmap: " + entry.getKey(), e);
            }
        }
    }
    
    public void loadFromDisk(String path) {
        java.io.File dir = new java.io.File(path);
        for (java.io.File file : dir.listFiles((d, name) -> name.endsWith(".rbm"))) {
            String tagKey = file.getName().replace(".rbm", "");
            RoaringBitmap bitmap = new RoaringBitmap();
            try (java.io.FileInputStream fis = new java.io.FileInputStream(file);
                 java.nio.channels.ReadableByteChannel channel = java.nio.channels.Channels.newChannel(fis)) {
                bitmap.deserialize(channel);
                tagBitmaps.put(tagKey, bitmap);
            } catch (Exception e) {
                throw new RuntimeException("Failed to load bitmap: " + file.getName(), e);
            }
        }
    }

    private String sanitizeFileName(String key) {
        return key.replaceAll("[^a-zA-Z0-9_\\-]", "_");
    }
}
```

### 8.4 Bitmap 多值标签方案

```
问题: Bitmap天然只支持布尔型标签(有/无)，对于多值标签如何处理?

多值标签类型:
  1. 枚举型多值 (品类偏好: 服装/数码/美妆)
    → 拆分为多个布尔Bitmap:
      tag:category:clothing   → Bitmap (喜欢服装的用户)
      tag:category:digital    → Bitmap (喜欢数码的用户)
      tag:category:beauty     → Bitmap (喜欢美妆的用户)
    
    圈选: "喜欢服装或数码" 
    → bit_or(category_clothing, category_digital)

  2. 数值型多值 (年龄: 整数值)
    → 分桶:
      tag:age:0_17    → Bitmap
      tag:age:18_24   → Bitmap
      tag:age:25_35   → Bitmap
      tag:age:36_50   → Bitmap
      tag:age:50_plus → Bitmap

  3. 有序多值 (消费等级: L1<L2<L3<L4<L5)
    → 分层Bitmap (高层级包含了低层级):
      tag:consume:L1  → [消费≥L1的用户]
      tag:consume:L3  → [消费≥L3的用户]
    
    圈选: "消费L3及以上"
    → 直接使用 tag:consume:L3 Bitmap (已包含L3+L4+L5)

  4. 位图矩阵 (Bit-Sliced Index)
    → 用于精确数值，如精确年龄、精确金额
    
    示例: 年龄 0-150岁, 用8个bit表示 (2^8=256足够)
    Bit-0: 年龄值的第0位为1的用户
    Bit-1: 年龄值的第1位为1的用户
    ...
    Bit-7: 年龄值的第7位为1的用户
    
    查询: 年龄=25 (二进制 00011001)
    → bit_and的8个bit: 按25的二进制组合8个bit slice
    → 复杂度: O(log N) vs 全表扫描 O(N)
```

```
    - 剩余预算 < 20%: 冻结所有发布 + 集中修复稳定性问题
```

### 11.2 降级策略代码实现

```java
/**
 * 标签查询服务 - 多层降级实现
 */
public class ProfileQueryService {

    private final Cache<Long, UserProfile> localCache;      // L1: Caffeine
    private final RedisClient redisClient;                   // L2: Redis
    private final HBaseClient hbaseClient;                   // L3: HBase
    private final BitmapProfileIndex bitmapIndex;             // Bitmap索引
    private final UserProfile defaultProfile;                 // 兜底画像
    
    // 降级状态
    private volatile DegradationLevel currentLevel = DegradationLevel.NORMAL;
    
    // 熔断器
    private final CircuitBreaker redisBreaker;
    private final CircuitBreaker hbaseBreaker;
    
    public ProfileQueryService() {
        this.localCache = Caffeine.newBuilder()
            .maximumSize(5_000_000)     // 500万用户
            .expireAfterWrite(30, TimeUnit.MINUTES)
            .recordStats()
            .build();
        
        this.redisBreaker = CircuitBreaker.ofDefaults("redis");
        this.hbaseBreaker = CircuitBreaker.ofDefaults("hbase");
        this.defaultProfile = loadDefaultProfile();
        
        // 启动降级状态监控
        startDegradationMonitor();
    }
    
    /**
     * 获取用户标签 - 核心降级逻辑
     */
    public UserProfile getUserTags(long userId, List<String> tagNames) {
        UserProfile profile = null;
        
        // === Layer 0: 本地缓存 ===
        profile = localCache.getIfPresent(userId);
        if (profile != null) {
            metrics.increment("cache.hit.local");
            return profile.filterTags(tagNames);
        }
        metrics.increment("cache.miss.local");
        
        // === Layer 1: Redis (带熔断) ===
        if (currentLevel.ordinal() <= DegradationLevel.REDIS_DEGRADED.ordinal()) {
            try {
                profile = redisBreaker.executeSupplier(
                    () -> redisClient.getUserProfile(userId)
                );
                if (profile != null) {
                    localCache.put(userId, profile);
                    metrics.increment("cache.hit.redis");
                    profile.setDataFreshness(Freshness.REALTIME);
                    return profile.filterTags(tagNames);
                }
            } catch (CircuitBreakerOpenException e) {
                log.warn("Redis circuit breaker OPEN, falling back to HBase");
                metrics.increment("degradation.redis_fallback");
            } catch (Exception e) {
                log.warn("Redis query failed for user {}, error: {}", userId, e.getMessage());
                metrics.increment("degradation.redis_error");
            }
        }
        
        // === Layer 2: HBase (带熔断) ===
        if (currentLevel.ordinal() <= DegradationLevel.STORAGE_DEGRADED.ordinal()) {
            try {
                profile = hbaseBreaker.executeSupplier(
                    () -> hbaseClient.getUserProfile(userId)
                );
                if (profile != null) {
                    localCache.put(userId, profile);
                    metrics.increment("cache.hit.hbase");
                    profile.setDataFreshness(Freshness.OFFLINE_T1);
                    // 异步回写Redis(非阻塞)
                    asyncRestoreRedisCache(userId, profile);
                    return profile.filterTags(tagNames);
                }
            } catch (CircuitBreakerOpenException e) {
                log.warn("HBase circuit breaker OPEN, falling back to default");
                metrics.increment("degradation.hbase_fallback");
            } catch (Exception e) {
                log.warn("HBase query failed for user {}, error: {}", userId, e.getMessage());
                metrics.increment("degradation.hbase_error");
            }
        }
        
        // === Layer 3: 默认画像兜底 ===
        if (currentLevel.ordinal() <= DegradationLevel.STATIC_FALLBACK.ordinal()) {
            metrics.increment("degradation.default_profile");
            profile = defaultProfile.clone();
            profile.setUserId(userId);
            profile.setDataFreshness(Freshness.DEGRADED_DEFAULT);
            return profile.filterTags(tagNames);
        }
        
        // 理论上到不了这里
        return defaultProfile;
    }
    
    /**
     * 人群圈选 - 降级逻辑
     */
    public CrowdSelectionResult selectCrowd(List<String> andTags, 
                                             List<String> orTags, 
                                             List<String> notTags,
                                             int limit) {
        if (currentLevel.ordinal() <= DegradationLevel.STORAGE_DEGRADED.ordinal()) {
            try {
                // 正常路径: Bitmap运算
                RoaringBitmap resultBitmap = bitmapIndex.crowdSelection(andTags, orTags, notTags);
                long totalCount = resultBitmap.getLongCardinality();
                
                List<String> userIds;
                if (limit > 0 && totalCount > limit) {
                    // 只返回前limit个用户
                    userIds = resultBitmap.stream()
                        .limit(limit)
                        .mapToObj(bitmapIndex::internalToExternal)
                        .collect(Collectors.toList());
                } else {
                    userIds = bitmapIndex.crowdSelectionWithUserIds(andTags, orTags, notTags);
                }
                
                return new CrowdSelectionResult(totalCount, userIds, Freshness.REALTIME);
            } catch (Exception e) {
                log.error("Bitmap crowd selection failed", e);
                metrics.increment("degradation.bitmap_error");
            }
        }
        
        // Bitmap不可用: 降级到基于Redis/HBase的扫描(慢路径)
        log.warn("Bitmap unavailable, falling back to scan-based crowd selection");
        metrics.increment("degradation.crowd_scan_fallback");
        return scanBasedCrowdSelection(andTags, orTags, notTags, limit);
    }
    
    /**
     * 降级状态自动监控与恢复
     */
    @Scheduled(fixedDelay = 10000) // 每10秒检查
    public void checkDegradationRecovery() {
        if (currentLevel == DegradationLevel.NORMAL) return;
        
        // 尝试恢复Redis
        if (currentLevel.ordinal() <= DegradationLevel.REDIS_DEGRADED.ordinal()) {
            try {
                redisClient.ping();
                redisBreaker.transitionToClosedState();
                log.info("Redis recovered, circuit breaker reset");
            } catch (Exception ignored) {}
        }
        
        // 尝试恢复HBase
        if (currentLevel.ordinal() <= DegradationLevel.STORAGE_DEGRADED.ordinal()) {
            try {
                hbaseClient.ping();
                hbaseBreaker.transitionToClosedState();
                log.info("HBase recovered, circuit breaker reset");
            } catch (Exception ignored) {}
        }
        
        // 评估是否需要提升降级级别
        updateDegradationLevel();
    }
    
    private void updateDegradationLevel() {
        DegradationLevel newLevel = DegradationLevel.NORMAL;
        
        if (redisBreaker.getState() == CircuitBreaker.State.OPEN) {
            newLevel = DegradationLevel.REDIS_DEGRADED;
        }
        if (hbaseBreaker.getState() == CircuitBreaker.State.OPEN) {
            newLevel = DegradationLevel.STORAGE_DEGRADED;
        }
        if (!redisBreaker.isCallPermitted() && !hbaseBreaker.isCallPermitted()) {
            newLevel = DegradationLevel.STATIC_FALLBACK;
        }
        
        if (newLevel != currentLevel) {
            log.warn("Degradation level changed: {} → {}", currentLevel, newLevel);
            alertingService.sendAlert(
                String.format("画像服务降级级别变更: %s → %s", currentLevel, newLevel),
                newLevel.ordinal() >= DegradationLevel.STORAGE_DEGRADED.ordinal() 
                    ? Severity.CRITICAL : Severity.WARNING
            );
            currentLevel = newLevel;
        }
    }
    
    /**
     * 手动强制降级/恢复 (通过Admin API)
     */
    public void forceDegradation(DegradationLevel level) {
        log.warn("Manual degradation forced: {} → {}", currentLevel, level);
        this.currentLevel = level;
        if (level == DegradationLevel.NORMAL) {
            redisBreaker.transitionToClosedState();
            hbaseBreaker.transitionToClosedState();
        }
    }
    
    enum DegradationLevel {
        NORMAL,              // 全部正常
        REDIS_DEGRADED,      // Redis降级
        STORAGE_DEGRADED,   // 存储层降级
        STATIC_FALLBACK,    // 静态兜底
        CLIENT_ONLY         // 仅客户端缓存
    }
}
```
