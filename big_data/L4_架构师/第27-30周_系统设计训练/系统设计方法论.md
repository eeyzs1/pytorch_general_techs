# 系统设计方法论——架构设计五步法

## 概述

系统设计是大数据架构师的核心能力。本方法论基于 Alex Xu《系统设计面试》和大厂面试实战经验总结，将架构设计抽象为可复用的五步流程。无论面对什么类型的系统设计题（实时数仓、特征平台、数据湖仓等），都可以套用这个框架。

## 五步法全景

```
第1步：需求澄清 ──→ 第2步：容量估算 ──→ 第3步：高层设计
                                              │
                                              ▼
第5步：权衡分析 ←── 第4步：深入设计
```

---

## 第1步：需求澄清（最重要的一步）

> **核心原则：问10个问题再动笔。80%的设计失败源于需求理解偏差。**

### 1.1 功能性需求（Functional Requirements）

必须明确回答的问题：

| 维度 | 关键问题 |
|------|----------|
| 数据源 | 有哪些数据源？数据格式？数据量级？更新频率？ |
| 数据处理 | 需要做什么样的处理？ETL/ELT？实时/离线/近实时？ |
| 数据输出 | 最终数据给谁用？什么形式？什么频率？ |
| 查询模式 | 查询类型（点查/范围查/聚合查）？并发量？延迟要求？ |
| 数据质量 | 需要哪些质量校验？缺失值处理？重复数据处理？ |
| 安全合规 | 有哪些安全要求？数据脱敏？访问控制？审计日志？ |

**实战技巧——需求澄清清单**：

```
□ 谁在使用这个系统？（角色：数据分析师/运营/算法工程师/外部客户）
□ 使用场景是什么？（报表/自助分析/实时大屏/ML训练/API调用）
□ 数据从哪里来？（业务库/埋点日志/IoT/第三方API/外部文件）
□ 数据量有多大？（每日增量/总量/峰值/增长速度）
□ 时效性要求？（实时秒级/近实时分钟级/离线小时级/T+1天级）
□ 一致性要求？（强一致/最终一致/弱一致）
□ 可用性要求？（99.9%/99.99%/99.999%）
□ 数据保留多久？（原始数据/汇总数据/归档数据）
□ 预算限制？（硬件预算/云资源预算/人力预算）
□ 有什么特殊约束？（合规/数据主权/遗留系统/团队技能栈）
```

### 1.2 非功能性需求（Non-Functional Requirements）

```
性能（Performance）:
  - 数据延迟：端到端数据处理延迟 < X 分钟
  - 查询延迟：P99 查询延迟 < Y 毫秒
  - 吞吐量：支持 Z QPS 的并发查询

可用性（Availability）:
  - SLA：99.9%（年宕机 < 8.76小时）到 99.999%（年宕机 < 5.26分钟）
  - RTO（恢复时间目标）：故障后 X 分钟内恢复
  - RPO（恢复点目标）：最多丢失 Y 分钟数据

可扩展性（Scalability）:
  - 数据量年增长 X%
  - 是否支持水平扩展
  - 是否需要多机房/多云部署

一致性（Consistency）:
  - CAP 三角的选择：CP 还是 AP？
  - 是否需要事务支持？
  - 跨表/跨系统的数据一致性问题如何处理

安全性（Security）:
  - 认证方式（LDAP/Kerberos/OAuth）
  - 授权粒度（库/表/列/行级别）
  - 数据加密（传输加密/存储加密）
  - 审计日志

成本（Cost）:
  - 存储成本 + 计算成本 + 网络成本 + 运维人力成本
  - 是否有成本优化空间
```

### 1.3 利益相关者分析

```
角色              关注点                    沟通方式
─────────────────────────────────────────────────────
CEO/CTO           ROI、战略价值              精简汇报（1页纸）
业务负责人         业务指标、数据时效           业务语言
产品经理          功能完整性、用户体验          原型演示
数据工程师         技术可行性、维护成本          技术方案文档
运维/SRE          稳定性、监控、告警            SLA文档
安全合规           数据安全、合规性             安全审计报告
```

## 第2步：容量估算

> **核心原则：用信封背面的计算（Back-of-the-Envelope）验证方案可行性。**

### 2.1 数据量估算

```
每日数据量 = Σ(各数据源每日增量)

示例：
  MySQL订单数据: 100万订单/天 × 2KB/条 = 2GB/天
  Nginx日志: 5亿PV/天 × 500B/条 = 250GB/天
  埋点日志: 5000万事件/天 × 1KB/条 = 50GB/天
  ─────────────────────────────────────────
  每日总增量: ~302GB/天
  每月: 302GB × 30 ≈ 9TB/月
  每年: 302GB × 365 ≈ 110TB/年
```

### 2.2 存储容量估算

```
总存储 = 原始数据 × 副本数 × 保留周期 + 中间数据 + 归档数据

不同数据类型：
  - 原始数据（RAW）：全量保存，副本数3，保留7天
  - ODS层：全量保存，副本数3，保留30天
  - DWD层：清洗后保存，副本数2，保留90天
  - DWS层：汇总数据，副本数2，保留3年
  - ADS层：应用数据，副本数2，保留永久

存储分层成本：
  - 热存储（SSD）：$0.1/GB/月
  - 温存储（HDD）：$0.03/GB/月
  - 冷存储（归档）：$0.01/GB/月
```

### 2.3 QPS与并发估算

```
读QPS = 活跃用户数 × 人均查询次数 / 时间窗口

示例：
  200个数据分析师，每人每小时查询10次
  → 读QPS ≈ 200 × 10 / 3600 ≈ 0.56 QPS（平均）
  → 峰值QPS = 平均QPS × 3 = 1.7 QPS

写QPS：
  订单写入: 100万/天 / 86400秒 ≈ 12 QPS
  日志写入: 5亿/天 / 86400秒 ≈ 5787 QPS（集中在白天，白天的QPS约12000）
```

### 2.4 带宽估算

```
带宽 = 日均数据量 / 86400 × 峰值倍数

示例：
  日均数据量 302GB → 平均带宽 302GB/86400s ≈ 35Mbps
  峰值倍数 ×3 → 峰值带宽 ≈ 105Mbps
```

### 2.5 常用数字速查表（背下来！）

```
1KB = 10^3 bytes       1MB = 10^6 bytes       1GB = 10^9 bytes
1TB = 10^12 bytes      1PB = 10^15 bytes

常见操作的耗时数量级：
  L1 Cache访问: 0.5 ns
  L2 Cache访问: 7 ns
  内存访问: 100 ns
  SSD随机读: 100 μs
  HDD随机读: 10 ms
  网络往返(同机房): 0.5 ms
  网络往返(跨地域): 100 ms

常见吞吐量：
  HDD顺序读: 150 MB/s
  SSD顺序读: 500 MB/s - 3 GB/s
  万兆网卡: 1.25 GB/s
  Kafka单Partition: 10-50 MB/s（取决于配置）
```

## 第3步：高层设计

> **核心原则：先画盒子，再连线。不要一上来就陷入细节。**

### 3.1 子系统划分

根据数据流向，将系统划分为逻辑子系统：

```
典型大数据平台子系统划分：

┌──────────────────────────────────────────────────────┐
│                    数据采集层                          │
│  CDC(MySQL→Kafka) │ 日志采集(Filebeat→Kafka) │ API接入│
└─────────────────────────┬────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────┐
│                    消息队列层                          │
│              Kafka / Pulsar / RocketMQ                │
└─────────────────────────┬────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────┐
│                    计算处理层                          │
│        Flink (实时)  │  Spark (离线)  │  Trino (查询)  │
└─────────────────────────┬────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────┐
│                    存储层                              │
│  HDFS/OSS │ ClickHouse │ HBase │ Iceberg │ Redis      │
└─────────────────────────┬────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────┐
│                    应用服务层                          │
│      BI报表  │  自助查询  │  API服务  │  ML平台       │
└──────────────────────────────────────────────────────┘
```

### 3.2 数据流向图

绘制端到端的数据流向，标注每个环节的数据格式和延迟：

```
[MySQL] ──CDC──→ [Kafka] ──→ [Flink ETL] ──→ [Kafka(结果)]
  2GB/天          原始      实时清洗           实时结果
  <1s             <100ms     <5s              <1s
                                                    │
                    ┌───────────────────────────────┘
                    ▼
              [ClickHouse] ←── [Flink SQL] ←── [Kafka(结果)]
               实时OLAP          实时聚合
               <50ms             <10s
```

### 3.3 接口定义

```
子系统间接口清晰定义：

采集层 → 消息队列:
  协议: Kafka Protocol
  格式: Avro (Schema Registry管理)
  压缩: lz4
  分区策略: 按user_id的hash

消息队列 → 计算层:
  协议: Kafka Consumer API
  消费模式: Consumer Group
  提交策略: At-Least-Once + 幂等处理

计算层 → 存储层:
  写入方式: Batch Insert / Stream Load
  格式: Parquet (离线) / MergeTree (实时)
  分区: 按天分区
```

## 第4步：深入设计

> **核心原则：每个关键组件至少对比3个候选方案，给出明确的选择理由。**

### 4.1 组件选型框架

对每个关键组件，使用以下框架进行对比：

```
对比维度：
  1. 功能满足度：是否满足当前和未来18个月的需求？
  2. 性能基准：在目标数据量下的性能表现
  3. 运维复杂度：部署/监控/升级/故障处理的难度
  4. 社区活跃度：GitHub Stars/Contributors/Issue响应时间
  5. 团队技能匹配：团队现有技能栈是否匹配？
  6. 成本：许可证费用 + 基础设施成本 + 运维人力成本
  7. 生态兼容性：与现有技术栈的集成难度
  8. 供应商锁定风险：是否依赖特定云厂商？
```

### 4.2 数据模型设计

```
存储格式选型决策树：

需要频繁UPDATE/DELETE？
  ├── 是 → 主键索引存储 (ClickHouse ReplacingMergeTree / Hudi MOR)
  └── 否 → 需要高压缩比？
            ├── 是 → 列式存储 (Parquet / ORC)
            └── 否 → 需要Schema灵活？
                      ├── 是 → Avro / JSON
                      └── 否 → Protobuf

分区策略选择：
  - 时间分区：适合按时间范围查询（最常用）
  - 业务分区：适合按业务维度查询（如按地区、品类）
  - Hash分区：适合均匀分布、避免热点
  - 多级分区：时间 + 业务（dt + region）

索引策略：
  - 主键索引：ClickHouse的稀疏索引
  - Bloom Filter：快速过滤不存在的数据
  - Bitmap索引：低基数列的高效过滤
  - 倒排索引：Elasticsearch/Lucene全文检索
```

### 4.3 高可用设计

```
各层高可用方案：

采集层：
  - CDC工具：Debezium分布式模式（多实例 + ZooKeeper选主）
  - 日志采集：Filebeat + Kafka(acks=all, min.insync.replicas=2)

消息队列层：
  - Kafka ISR机制：replication.factor=3, min.insync.replicas=2
  - 跨AZ部署：3个Broker分布在3个可用区

计算层：
  - Flink：HA JobManager + Checkpoint(S3/RocksDB)
  - Spark：YARN HA / K8s自动重启

存储层：
  - ClickHouse：ReplicatedMergeTree + 3副本 + ZooKeeper协调
  - HDFS：NameNode HA + 3副本
  - HBase：RegionServer自动Failover + HLog

容灾方案：
  - 同城双活：RPO < 1分钟, RTO < 5分钟
  - 异地灾备：RPO < 1小时, RTO < 30分钟
```

## 第5步：权衡分析

> **核心原则：没有完美的架构，只有最适合的 Trade-Off。必须能说出"为什么不选B"。**

### 5.1 常见 Trade-Off 清单

```
一致性 vs 可用性（CAP）:
  场景：实时用户画像系统
  → 选择AP（牺牲一致性）
  → 理由：用户画像允许短暂不一致（最终一致性可接受），但服务不能中断

延迟 vs 吞吐量:
  场景：实时推荐系统
  → 优先低延迟（<50ms）
  → 理由：推荐延迟直接影响用户体验和转化率

开发速度 vs 运维复杂度:
  场景：初创公司大数据平台
  → 选择托管服务（如AWS EMR + Athena）
  → 理由：团队小，优先交付业务价值，运维能力可以后续补

成本 vs 性能:
  场景：日志分析系统
  → 选择HDD存储 + 冷热分层
  → 理由：日志查询频率低，用SSD性价比低

灵活性 vs 标准化:
  场景：数据模型设计
  → 选择宽表（大宽表 vs 星型模型）
  → 理由：分析场景下宽表查询效率高，虽然存储有冗余
```

### 5.2 风险识别与应对

```
技术风险：
  风险：新技术不成熟（如Iceberg/Data Lakehouse）
  应对：POC验证 → 灰度上线 → 核心链路逐步迁移
  监控：关键指标（延迟、准确率、成本）对比

性能风险：
  风险：方案在10倍数据量下性能衰减严重
  应对：预留2-3倍容量buffer，设计水平扩展方案
  监控：定期压测，关注性能衰减曲线

组织风险：
  风险：团队技能栈不匹配（如从Spark迁移到Flink）
  应对：培训计划 + 外部咨询 + 分阶段迁移
  监控：团队成员学习进度 + 故障处理时间

数据风险：
  风险：上游数据源变更导致管道断裂
  应对：Schema Registry + 数据契约 + 变更通知机制
  监控：数据延迟 + 数据量波动告警
```

### 5.3 决策记录模板（ADR）

每个关键决策都应记录为 ADR（Architecture Decision Record）：

```markdown
# ADR-001: 消息队列选择 Kafka 而非 Pulsar

## 状态
已采纳

## 背景
需要为实时数据管道选择消息队列。候选方案：Apache Kafka、Apache Pulsar、AWS Kinesis。

## 决策
选择 Apache Kafka。

## 理由
1. 社区成熟度：Kafka 社区活跃度远高于 Pulsar（GitHub Stars 28k vs 14k）
2. 团队经验：团队有2名Kafka认证工程师
3. 生态兼容：Flink/Spark/Debezium 对 Kafka 的支持最完善
4. Pulsar分层存储虽好，但我们的数据保留周期（7天）不需要

## 后果
- 正面：快速落地，运维成熟度高
- 负面：未来如需多租户隔离和分层存储，需要额外方案
- 风险：Confluent社区版功能限制，需关注KIP演进

## 替代方案
- Pulsar：分层存储和多租户更强，但运维复杂度和社区成熟度不足
- Kinesis：AWS锁定，跨云迁移成本高
```

## 系统设计面试时间分配（参考）

```
总时间：45-60分钟

0-5分钟：需求澄清
  - 问3-5个最关键的问题
  - 明确功能性和非功能性需求
  - 在白板上写下关键约束

5-10分钟：容量估算
  - 快速估算数据量和QPS
  - 验证方案可行性（不能超过物理限制）

10-25分钟：高层设计 + 数据流
  - 画出子系统盒子图
  - 标注数据流向和格式
  - 解释关键组件的选择和职责

25-40分钟：深入设计
  - 挑选2-3个核心子系统深入
  - 说明技术选型理由
  - 讨论数据模型和分区策略

40-50分钟：Trade-Off分析
  - 说明2-3个关键权衡
  - 讨论风险和应对
  - 回答"How would you handle X?"

50-55分钟：总结
  - 回顾整体方案
  - 补充遗漏点

55-60分钟：Q&A
```

## 常见错误与规避

| 错误 | 表现 | 正确做法 |
|------|------|----------|
| 跳过需求澄清 | 上来就画架构图 | 先问5个问题，明确约束 |
| 容量估算缺失 | 方案在1000QPS下可行，实际100万QPS | 先估算，再设计 |
| 只给一个方案 | "我们用Flink就够了" | 至少对比2-3个候选，说明取舍 |
| 忽略非功能需求 | 只考虑功能实现 | 列出性能/可用性/安全性约束 |
| 设计过度 | 为10年后扩展做设计 | YAGNI原则：只设计未来18个月需要的 |
| 不考虑运维 | 设计了完美的架构但没人会运维 | 设计时就考虑监控/告警/部署/升级 |

## 容量估算速查表

### 一、数据库性能基准数据

| 数据库 | 读QPS（单节点） | 写QPS（单节点） | 典型P99延迟 | 存储容量上限 | 适合场景 |
|--------|---------------|---------------|------------|-------------|----------|
| MySQL (InnoDB) | 5K-15K | 2K-8K | 1-10ms | TB级 | OLTP事务处理 |
| PostgreSQL | 8K-20K | 3K-10K | 1-8ms | TB级 | 复杂查询OLTP |
| MongoDB | 10K-30K | 5K-15K | 1-5ms | TB级 | 文档型NoSQL |
| Redis | 100K-200K | 80K-150K | <1ms | 内存限制(百GB) | 缓存/计数器 |
| ClickHouse | 50K-200K(聚合) | 50K-500K | 10-200ms | PB级 | OLAP分析 |
| StarRocks | 30K-150K | 30K-300K | 5-100ms | PB级 | 实时OLAP |
| Elasticsearch | 5K-20K | 3K-10K | 10-200ms | TB级 | 全文检索/日志 |
| HBase | 10K-50K | 10K-50K | 1-10ms | PB级 | 大宽表点查 |
| Cassandra | 20K-80K | 20K-60K | 1-5ms | PB级 | 时序/高写 |
| TiDB | 10K-30K | 5K-20K | 2-10ms | TB-PB级 | 分布式OLTP |

### 二、消息队列/Kafka吞吐基准

| 参数 | 值 | 说明 |
|------|-----|------|
| 单Partition写入吞吐 | 10-50 MB/s | 默认配置，优化可达80MB/s |
| 单Partition消费吞吐 | 20-100 MB/s | 取决于Consumer处理能力 |
| 单Broker吞吐(万兆) | ~500 MB/s | 受磁盘顺序写能力限制 |
| 单Broker吞吐(千兆) | ~100 MB/s | 网络带宽瓶颈 |
| 消息大小最优值 | 1KB-10KB | 太小浪费吞吐，太大延迟增加 |
| 推荐Partition数/Broker | 1000-4000 | 超过会导致ZooKeeper性能问题 |
| 端到端延迟最小值 | 2-5ms | Producer→Broker→Consumer(同DC) |
| 数据保留磁盘需求 | 消息速率×保留时间×副本数 | 如: 100MB/s×7天×3=180TB |
| Pulsar单Partition吞吐 | 5-20 MB/s | 默认配置略低于Kafka |
| RocketMQ单队列吞吐 | 2-10 MB/s | 阿里内部优化可达更高 |

### 三、Flink/Spark计算性能基准

| 场景 | Flink吞吐(核) | Spark吞吐(核) | 说明 |
|------|-------------|--------------|------|
| 简单ETL(Map/Filter) | 10K-50K条/秒 | 8K-30K条/秒 | 无状态处理 |
| 窗口聚合(1min) | 5K-20K条/秒 | 2K-8K条/秒 | 有状态处理 |
| 双流JOIN | 2K-10K条/秒 | 1K-5K条/秒 | 需要缓存两边数据 |
| 维表JOIN(Async IO) | 5K-25K条/秒 | 3K-10K条/秒 | 依赖外部存储延迟 |
| 大窗口(24H) | 1K-5K条/秒 | 500-2K条/秒 | RocksDB IO瓶颈 |
| Checkpoint开销 | 10-30秒 | 20-60秒 | 取决于State大小 |

**Flink资源估算公式**:
```
TM数量 = CEIL(峰值处理QPS / 单核处理QPS) × (100 + 反压Buffer%) / 100 / 单TM核数
State大小估算 = Key数量 × 每Key状态大小 × 窗口内Key数
Checkpoint存储 = State大小 × 2（至少保留2个Checkpoint）
```

**Spark资源估算公式**:
```
Executor数量 = CEIL(总数据量 / 单Executor处理能力) / 并行度
Executor内存 = 数据处理内存 + Shuffle内存(30%) + 预留内存(40%)
Shuffle数据量 = 输入数据量 × Shuffle系数(0.3-1.0)
```

### 四、HDFS/对象存储性能基准

| 参数 | HDD模式 | SSD模式 | 说明 |
|------|---------|--------|------|
| 顺序写吞吐 | 100-200 MB/s | 300-800 MB/s | 单DataNode |
| 顺序读吞吐 | 100-200 MB/s | 400-1000 MB/s | 单DataNode |
| 随机读IOPS | 50-100 | 5000-50000 | 小文件场景 |
| Block Size建议 | 128-256MB | 128-256MB | 大文件用256MB |
| 小文件问题阈值 | < 1个Block | < 1个Block | 数百万小文件严重拖慢NN |
| NN内存需求 | 1GB/百万文件 | 1GB/百万文件 | 元数据全在内存 |
| DataNode磁盘使用率上限 | 70-80% | 70-80% | 超过后性能下降明显 |
| S3写入延迟(P99) | - | 100-500ms | 对象存储统一接口 |
| S3读延迟(P99) | - | 50-200ms | 首字节延迟 |

### 五、网络带宽速查

| 网络类型 | 带宽 | 延迟 | 典型场景 |
|----------|------|------|----------|
| 同机器进程间 | ~10 GB/s | <0.01ms | IPC |
| 同机架内 | 10 Gbps | 0.1-0.5ms | 集群内部通信 |
| 同数据中心跨机架 | 10 Gbps | 0.5-2ms | 跨交换机通信 |
| 同城跨数据中心 | 1-10 Gbps | 2-5ms | 同城双活 |
| 跨地域(国内) | 100 Mbps-1 Gbps | 10-50ms | 异地灾备 |
| 跨地域(国际) | 10-100 Mbps | 50-300ms | 全球部署 |
| 公网(国内) | 1-100 Mbps | 10-100ms | API网关 |
| 专线 | 1-10 Gbps | 间接距离决定 | 混合云连接 |

### 决策树1：消息队列选型

```
需要消息队列？
├── 需要Exactly-Once语义？
│   ├── 是 → Kafka (需要Kafka Streams或Flink配合)
│   └── 否 → 需要多租户隔离和分层存储？
│           ├── 是 → Pulsar
│           └── 否 → 需要低延迟推模式？
│                   ├── 是 → 需要事务消息？
│                   │       ├── 是 → RocketMQ
│                   │       └── 否 → RabbitMQ
│                   └── 否 → Kafka
└── 低成本/轻量级场景？
    └── Redis Streams (适合小规模/已有Redis的场景)
```

### 决策树2：OLAP引擎选型

```
需要实时OLAP？
├── 查询模式主要是单表聚合？
│   ├── 是 → 数据量 < 10TB？
│   │       ├── 是 → ClickHouse (极致单表性能)
│   │       └── 否 → ClickHouse集群 + 分布式表
│   └── 否 → 需要多表JOIN？
│           ├── 是 → 频繁UPDATE/DELETE？
│           │       ├── 是 → StarRocks (主键模型)
│           │       └── 否 → 数据量 > 100TB？
│           │               ├── 是 → StarRocks (Colocate Join)
│           │               └── 否 → ClickHouse (打宽表)
│           └── 否 → Doris (ClickHouse兼容 + JOIN增强)
└── 需要高并发点查 + 聚合混合？
    ├── 是 → 数据模型简单？
    │       ├── 是 → Apache Pinot / Apache Druid
    │       └── 否 → StarRocks
    └── 否 → 仅需离线分析？
            └── Presto/Trino + Hive/Iceberg
```

### 决策树3：流处理引擎选型

```
需要实时流处理？
├── 编程语言偏好？
│   ├── Java/Scala → Flink (最完整的生态)
│   └── Python → Spark Structured Streaming (PySpark支持好)
├── 延迟要求？
│   ├── 毫秒级 → Flink (事件驱动, 原生流处理)
│   ├── 秒级 → Spark Structured Streaming (Micro-batch)
│   └── 分钟级 → Spark Batch + 调度
├── 是否需要SQL？
│   ├── 是 → Flink SQL (最完整)
│   └── 否 → 需要Python原生库？
│           ├── 是 → Spark Structured Streaming
│           └── 否 → Flink DataStream API
└── 团队技能？
    ├── 有Flink经验 → Flink
    ├── 有Spark经验 → Spark Structured Streaming
    └── 都没有 → Flink（学习曲线类似但未来方向更明确）
```

### 决策树4：数据湖格式选型

```
需要数据湖格式？
├── 主要写模式是？
│   ├── Append Only → 需要Time Travel？
│   │                 ├── 是 → Iceberg (最成熟的Time Travel)
│   │                 └── 否 → Hive/Parquet (最简单的方案)
│   ├── Upsert/Merge → 写入延迟要求？
│   │                  ├── 近实时(< 5min) → Hudi (MoR模式)
│   │                  ├── T+1 → Hudi (CoW模式) / Iceberg v2
│   │                  └── 需要流读 → Delta Lake (与Spark生态绑定)
│   └── CDC同步 → Hudi（最佳CDC支持）
├── 查询引擎？
│   ├── 主要用Spark → Hudi（Spark原生支持最好）
│   ├── 主要用Trino → Iceberg（Trino支持最成熟）
│   └── 主要用Flink → Hudi（Flink CDC无缝集成）
└── 云环境？
    ├── AWS → Iceberg（Athena/Glue/EMR原生支持）
    ├── Azure → Delta Lake（Databricks原生支持）
    └── GCP → Iceberg（BigLake支持）
```

### 决策树5：缓存方案选型

```
需要缓存？
├── 数据量？
│   ├── < 10GB且单机够用 → Caffeine/Guava Cache(本地缓存)
│   ├── 10GB-500GB → Redis Cluster
│   └── > 500GB → 需要持久化？
│                 ├── 是 → Aerospike / Pika(RocksDB+Redis协议)
│                 └── 否 → Redis Cluster + 数据淘汰策略
├── 延迟要求？
│   ├── < 1ms → 本地Caffeine
│   ├── < 5ms → Redis(同DC)
│   └── < 50ms → Aerospike / Redis(跨DC)
└── 数据结构？
    ├── 简单KV → Redis / Memcached
    ├── 复杂数据结构(Set/ZSet/Hash) → Redis
    └── 大Value(>1MB) → Aerospike / 对象存储+CDN
```

## 常见反模式案例集（10个真实案例）

### 反模式1：过早优化——为"未来可能的"需求设计复杂架构

**案例**: 初创电商平台，日均订单仅1000单，架构师设计了Flink+Kafka+ClickHouse+Iceberg+Hudi的实时数仓。系统上线后，每天的计算成本是业务收入的3倍。

**问题**:
- 当前业务规模完全不需要实时数仓（T+1批处理足够）
- 过度设计导致极高的运维成本和人力投入
- 复杂架构让团队疲于运维，无法专注业务开发

**启示**:
- 方案要与当前业务规模匹配，可预留扩展接口但不要提前实现
- YAGNI原则：You Aren't Gonna Need It
- 用"18个月原则"：只设计未来18个月可能需要的功能

---

### 反模式2：技术选型跟风——用最新的技术替代成熟方案

**案例**: 某数据团队看到社区热议Data Lakehouse，决定将已稳定运行的Hive+Spark数仓迁移到Iceberg+Trino。迁移后发现：Trino对已有200+张Hive表的兼容性问题、Iceberg的Compaction导致存储成本翻倍、老SQL的大量改写工作。

**问题**:
- 仅看到新技术的优势，没评估迁移成本
- Hive+Spark在当前场景下完全满足需求
- 新技术的运维经验不足，问题处理效率低

**启示**:
- 技术选型要回答"不迁移的代价是什么"，而不仅是"新技术有多好"
- 新技术先在新项目/POC中验证，不要轻易迁移核心链路
- 迁移成本 = 开发改写 + 测试验证 + 数据迁移 + 回滚方案 + 团队学习

---

### 反模式3：单点设计——所有流量打到同一个组件

**案例**: 特征平台初期将20000维特征全部存储在单个Redis Cluster。大促期间，热Key导致单分片CPU 100%，整个Redis Cluster不可用，全站推荐系统瘫痪30分钟。

**问题**:
- 没有识别热点特征和冷特征，混合存储导致热Key打爆单分片
- 单一存储层的降级能力不足
- 没有针对热Key的拆分和本地缓存方案

**启示**:
- 识别系统中的热数据和冷数据，分层存储
- 热Key提前加盐/拆分/本地缓存
- 多级缓存 + 降级策略是标配

---

### 反模式4：忽略非功能需求——只关注"能不能做"，不关注"做得好不好"

**案例**: 某数据同步平台从100个数据源扩展到2000个后，全量同步从2小时变成20小时。排查发现：所有同步任务串行执行，扩展性为零；没有断点续传，任务失败后从零重跑。

**问题**:
- 初期设计没有考虑水平扩展（任务增加时如何扩容）
- 没有设计断点续传机制（假设同步总是成功的）
- 没有容量模型（不知道加机器能提升多少）

**启示**:
- 设计阶段就定义好SLA和扩展方式
- 用Back-of-the-Envelope计算验证方案在目标规模下的可行性
- 为失败而设计（Design for Failure）

---

### 反模式5：重复造轮子——自研市场上已有的成熟方案

**案例**: 某公司花了6个月自研了一套工作流调度引擎（类似Airflow的功能），用于编排200+个ETL任务。上线后发现：DAG可视化缺失、任务重跑逻辑有Bug、没有日历调度功能。最终改用Airflow + 自定义Operator。

**问题**:
- 低估了调度引擎的复杂度（不仅仅是按顺序执行）
- 没有评估自己的团队是否适合做底层系统开发
- 6个月的机会成本（Airflow 2周就能上线）

**启示**:
- 问自己三个问题再决定自研：市场上是否已有？我们能否做得更好？差异化价值是什么？
- 自研的前提：团队在该领域有深厚积累 + 商业方案不可行 + 核心链路差异化需求
- 绝大多数场景下，复用 > 自研

---

### 反模式6：忽视数据质量——认为"数据进来就好用"

**案例**: 某推荐系统上线后发现效果很差，排查发现：特征平台的用户年龄字段有30%的NULL值、商品类目字段有15%的值不在预定义集合中、实时特征更新延迟有时超过10分钟。但之前没有人做数据质量监控。

**问题**:
- 假设上游数据是完美的（"Garbage In, Garbage Out"的铁律）
- 没有在数据管道中嵌入质量检查
- 问题发现太迟（已经影响了线上模型）

**启示**:
- 每一条数据管道都应该有数据质量检查
- 数据质量应该是架构设计的一部分，而非事后补救
- 关键数据需要双重校验（源头+目标）

---

### 反模式7：缺乏可观测性——上线后不知道系统在做什么

**案例**: 某实时数仓上线3个月后，业务方投诉数据不对。排查发现：Flink某个Task半年前就因为数据倾斜处于反压状态，部分数据被丢弃。但因为没有监控告警，没人发现。业务方基于错误数据做了3个月的决策。

**问题**:
- 上线后没有设置监控和告警
- 团队成员没有定期检查系统状态的习惯
- 没有业务指标的对账机制（实时 vs 离线）

**启示**:
- 系统上线 = 代码上线 + 监控上线 + 告警上线 + Dashboard上线
- 关键业务指标必须有双重校验（两条独立的计算链路）
- 设计时就要考虑"如何证明数据是对的"

---

### 反模式8：技术债务积累——明知有问题但一直延后修复

**案例**: 某数据平台上线时为了赶进度使用了硬编码的数据库连接串、没有Schema版本管理、手动管理分区。2年后，平台有200+个硬编码配置、分区管理全靠人记、表结构改了没人同步。一次数据库迁移导致50%的ETL任务失败。

**问题**:
- 为了速度牺牲质量，且从不回头还债
- 没有把"临时方案"标记为技术债务并纳入计划
- 缺乏配置中心/Schema Registry等基础设施

**启示**:
- 每个迭代留20%时间还技术债务
- 所有临时方案打上//TODO和JIRA标签，设定修复时间
- 基础设施（配置中心/监控/Schema管理）必须第一个Phase建设

---

### 反模式9：团队能力和方案不匹配——设计了团队驾驭不了的架构

**案例**: 某公司技术负责人设计了基于Flink+Kafka+Hudi+StarRocks的实时数仓，但团队只有Spark+Hive经验。项目上线后：Flink Checkpoint频繁失败没人能处理、Hudi Compaction导致的查询延迟问题无法解决、StarRocks Colocate Join配置理解错误导致数据分布不均。6个月后，团队放弃实时链路，退回Spark T+1方案。

**问题**:
- 方案设计完美但脱离了团队实际能力
- 没有评估学习曲线和过渡期
- 没有引入外部咨询或分阶段渐进式迁移

**启示**:
- 架构设计要考虑团队的技术栈和成长速度
- 新技术引入需要POC验证 + 培训 + 外部支持
- 分阶段迁移：先用Spark实现核心功能 → 逐步引入Flink优化延迟 → 最后用Hudi优化增量

---

### 反模式10：忽视成本——设计"无限预算"的架构

**案例**: 某日志分析平台将所有日志存储在Elasticsearch SSD集群中，保留90天。日均100亿条日志导致ES集群需200个节点，每月成本$50万。但实际上：90%的查询都在最近3天内，超过7天的日志几乎无人查询。

**问题**:
- 没有分析查询模式（冷热数据分离）
- 所有数据同等对待，无差别地使用最贵的存储
- 没有成本模型和预算控制

**启示**:
- 设计时就列出成本估算（计算/存储/网络/人力）
- 利用冷热分层：热数据SSD → 温数据HDD → 冷数据对象存储/归档
- 设置成本告警，超出预算自动触发Review

### ADR最佳实践

```
1. ADR是"为什么"的记录，不是"是什么"的文档
   - 糟糕的ADR: "我们使用Kafka"（没有理由）
   - 好的ADR: "我们使用Kafka而非Pulsar，因为团队有3人具备Kafka认证，且下游Flink/Spark
     Connector的Kafka支持最成熟"（有理由有对比）

2. ADR要实事求是地记录缺点
   - 不要只写优点不写缺点
   - "选择Kafka的代价是失去多租户隔离能力，待后续补充"

3. ADR可以而且应该被废弃
   - 当环境变化时（团队成长、新版本释放新特性、业务方向调整），大胆废弃旧ADR
   - "ADR-005 替代 ADR-002: 团队经过6个月实践，具备了Pulsar运维能力，现切换到Pulsar"

4. ADR存放位置
   - 放在代码仓库的 /docs/adr/ 目录下
   - 文件名: ADR-{序号}-{短描述}.md，如 ADR-001-选择Kafka作为消息队列.md
   - 索引文件: /docs/adr/README.md 列出所有ADR及状态

5. ADR讨论流程
   提议 → 团队讨论 → 修改 → 投票 → 采纳/拒绝
   讨论介质: 代码仓库的PR/MR评论（避免聊天记录丢失）

6. 何时需要写ADR
   以下任一条件满足时:
   - 选择了一个新的技术栈组件
   - 否决了一个合理的替代方案
   - 对一个已有架构做重大变更
   - 跨团队的技术决策
   - 成本超过$10K的采购/开发决策
```

## 面试高频追问速查

### 追问1: "如果数据量是现在的10倍，架构哪里会先撑不住？"

**回答框架**:
1. 按数据流路径逐层分析瓶颈
2. 给每层评分（1-5，5=最容易成为瓶颈）

```
采集层: ★★☆☆☆ (Kafka加Partition即可扩展)
计算层: ★★★★☆ (Flink State膨胀、Checkpoint超时最危险)
存储层: ★★★☆☆ (HDFS加节点可扩展，但小文件问题会加剧)
查询层: ★★★★★ (OLAP引擎JOIN爆炸、查询并发最危险)
```

### 追问2: "如果给你砍掉50%的预算，你会砍什么？"

**回答框架**:
1. 区分"必须保留"和"可以砍掉"
2. 计算每项砍掉后的影响

```
不可砍的核心（20%预算）:
  - Kafka核心链路（数据不能丢）
  - 核心表存储（GMV等关键指标）

可以降级的（30%预算）:
  - ClickHouse/StarRocks从SSD降级到HDD+SSD缓存
  - Kafka保留期从7天缩短到3天
  - 副本数从3降到2（核心仍保持3）

可以砍掉的（50%预算）:
  - 非核心查询引擎（Trino→直接用Spark SQL）
  - 开发/测试环境从全量降到采样（10%数据量）
  - 冷数据从SSD迁移到对象存储
```
