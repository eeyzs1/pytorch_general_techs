## 项目背景

### 公司现状

**E-Shop（模拟名称）** 是一家中型电商平台，主营业务为服装、3C数码、美妆等品类。公司成立5年，业务增长迅速，但数据基础设施严重滞后。

```
当前数据环境:
  - 数据源: MySQL × 15 (订单/用户/商品/支付/物流等), MongoDB × 5 (商品详情/评论), Redis × 8 (缓存/会话), Elasticsearch × 2 (商品搜索)
  - 数据量: MySQL ~5TB, MongoDB ~2TB, ES ~500GB
  - 数据管道: Python Cron脚本定时拉取数据到本地，经常中断无人知晓
  - 数据分析: 数据团队5人，主要用Excel + Python Pandas手动分析
  - 报表时效: T+7（每周一出上周周报），大促期间需求爆仓
  - 数据质量: 没有统一口径，"同一个GMV三个部门三个数"
```

### 业务痛点

1. **决策滞后**：CEO周一才能看到上周的数据，无法快速响应市场变化
2. **数据不一致**：运营部、财务部、市场部各自拉数据口径不同，开会时争论"谁的数据对"
3. **效率低下**：数据分析师80%时间在"找数据、洗数据、对口径"，只有20%时间做分析
4. **无法支撑算法**：推荐团队想要用户行为数据训练模型，但数据分散在各系统
5. **扩展性差**：当前方案完全依赖手工，预计明年业务增长50%时彻底崩盘

### 业务目标

```
量化目标:
  - 报表时效: T+7 → T+0 (当天可看前一天的完整数据)
  - 数据质量: 数据一致性问题从"每周3次争论"降到"每月<1次"
  - 自助分析: 支持20人同时自助查询，分析师效率提升300%
  - 实时能力: 大促期间实时GMV大屏延迟 < 1分钟
  - 成本效率: 总投资 < 500万/年，3年ROI > 200%
```

### 1.2 核心业务指标定义

```
北极星指标:
  - GMV (Gross Merchandise Volume): 平台总交易额
    定义: 已支付订单金额总和 - 已退款金额
    口径: 以支付完成时间为准，排除测试订单(金额<0.01元)

关键业务指标:
  交易域:
    - 订单量: 日/周/月订单数 (口径: 已支付)
    - 客单价: GMV / 订单量
    - 复购率: 30天内重复购买用户数 / 总购买用户数
    - 退款率: 退款订单数 / 总订单数

  流量域:
    - PV (页面浏览量): 所有页面的总浏览次数
    - UV (独立访客): 按设备ID去重的访问用户数
    - 转化率: 下单UV / 访问UV
    - 跳出率: 只浏览1个页面即离开的Session占比

  用户域:
    - DAU/MAU: 日/月活跃用户数
    - 新用户注册数: 当日新增注册用户
    - 留存率: 注册后第N天仍有行为的用户占比 (次日/7日/30日)
    - LTV: 用户生命周期总消费金额 (按180天窗口计算)

  商品域:
    - 动销率: 有销量的SKU数 / 总SKU数
    - 库存周转天数: 平均库存 / 日均销量
    - TOP N品类/商品: 按GMV/销量排名

  实时大屏 (仅大促期间):
    - 实时GMV (秒级刷新)
    - 实时订单量/下单速度
    - 实时热销商品TOP10
    - 各品类交易分布
```

### 2.2 组件选型对比

#### 2.2.1 消息队列

| 维度 | Apache Kafka | Apache Pulsar | RocketMQ |
|------|-------------|---------------|----------|
| 吞吐量 | ★★★★★ (百万/秒) | ★★★★☆ | ★★★★☆ |
| 延迟 | < 5ms | < 5ms | < 1ms |
| 多租户 | ★★★☆☆ | ★★★★★ | ★★★★☆ |
| 生态兼容 | ★★★★★ (Flink/Spark/Debezium) | ★★★☆☆ | ★★★☆☆ (阿里生态) |
| 运维复杂度 | ★★★★☆ | ★★★☆☆ | ★★★★☆ |
| 社区活跃度 | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| 推荐度 | ✅ 首选 | 备选 | 仅阿里云用户 |

**选择: Apache Kafka**
理由: 生态最完善，Flink/Spark/Debezium原生支持，团队有1名成员有Kafka经验。

#### 2.2.2 计算引擎

| 维度 | Apache Flink (实时) | Apache Spark (离线) | Apache Storm |
|------|---------------------|---------------------|--------------|

| Exactly-Once | ★★★★★ | ★★★☆☆ | ★★★☆☆ |
| SQL支持 | ★★★★★ | ★★★★★ | ★☆☆☆☆ |

### 2.3 数据模型设计

#### 2.3.1 数仓分层

```
ODS层 (Operational Data Store) - 原始数据层:
  目标: 1:1镜像源数据，不做任何清洗
  存储: Iceberg表, Parquet格式
  分区: 按天全量快照 + 增量日志

  ods_order_info_df          -- 订单主表(全量快照)
  ods_order_detail_df        -- 订单明细
  ods_user_info_df           -- 用户信息
  ods_product_info_df        -- 商品信息
  ods_payment_flow_df        -- 支付流水
  ods_logistics_df           -- 物流信息
  ods_user_behavior_log_df   -- 用户行为日志(APP+小程序+Web埋点)
  ods_nginx_access_log_df    -- Nginx访问日志

DWD层 (Data Warehouse Detail) - 明细数据层:
  目标: 数据清洗、标准化、维度关联
  原则: 构建以业务过程为中心的明细宽表

  dwd_trade_order_detail_di  -- 交易订单明细宽表 (关联用户+商品+支付+物流)
  dwd_traffic_page_log_di    -- 页面流量日志明细

  dwd_user_register_di       -- 用户注册明细
  dwd_product_info_full      -- 商品信息全量表(合并MySQL + MongoDB)

DWS层 (Data Warehouse Summary) - 轻度汇总层:
  目标: 按主题域进行日粒度轻度汇总

  -- 交易主题
  dws_trade_user_order_1d    -- 用户日订单汇总
  dws_trade_sku_order_1d     -- 商品日订单汇总
  dws_trade_cate_order_1d    -- 品类日订单汇总

  -- 流量主题
  dws_traffic_page_view_1d   -- 页面日PV/UV汇总
  dws_traffic_session_1d     -- Session级别行为汇总
  
  -- 用户主题
  dws_user_behavior_1d       -- 用户日行为汇总

ADS层 (Application Data Service) - 应用数据层:
  目标: 面向具体业务场景的报表/看板/API

  ads_user_profile_tags      -- 用户画像标签宽表
```

#### 2.3.2 核心表设计示例

```sql

CREATE TABLE dwd.dwd_trade_order_detail_di (
    order_id BIGINT NOT NULL COMMENT '订单ID',
    user_id BIGINT NOT NULL COMMENT '用户ID',
    product_id BIGINT NOT NULL COMMENT '商品ID',
    shop_id BIGINT COMMENT '店铺ID',
    
    -- 用户维度
    user_name STRING COMMENT '用户名',
    user_city STRING COMMENT '用户城市',
    user_register_days INT COMMENT '注册天数',
    user_member_level STRING COMMENT '会员等级',
    
    -- 商品维度
    product_name STRING COMMENT '商品名称',
    category_id BIGINT COMMENT '品类ID',
    category_name STRING COMMENT '品类名称',
    brand_id BIGINT COMMENT '品牌ID',
    product_price DECIMAL(10,2) COMMENT '商品单价',
    
    -- 订单维度
    order_status STRING COMMENT '订单状态',
    order_amount DECIMAL(20,2) COMMENT '订单金额',
    discount_amount DECIMAL(10,2) COMMENT '优惠金额',
    actual_amount DECIMAL(20,2) COMMENT '实付金额',
    payment_method STRING COMMENT '支付方式',
    
    -- 时间维度
    create_time TIMESTAMP COMMENT '创建时间',
    pay_time TIMESTAMP COMMENT '支付时间',
    dt DATE COMMENT '分区日期'
) USING iceberg
PARTITIONED BY (dt)

### 2.5 高可用设计

```
多层高可用:

数据采集层:
  - Debezium: 分布式模式，ZooKeeper选主，Failover < 30秒
  - Filebeat: 多实例部署，Kafka作为缓冲

消息队列层:
  - Kafka: 3节点, replication.factor=3, min.insync.replicas=2
  - 允许1节点宕机不影响写入

计算层:
  - Flink: HA模式 (ZooKeeper + S3 Checkpoint), RTO < 1分钟
  - Spark on K8s: Pod自动重启, Driver HA (需要额外配置)
  - 任务调度: Airflow HA (CeleryExecutor + Redis + PostgreSQL HA)

存储层:
  - MinIO: 纠删码 EC 8+2 (允许2节点同时故障)
  - StarRocks: 3副本, FE 3节点HA
  - Kafka: 3副本

容灾:
  同城双活:
    - MinIO Site Replication同步到备用集群
    - RPO < 1分钟, RTO < 15分钟
  异地灾备:
    - OSS跨区域复制 (DWS/ADS汇总数据)
    - RPO < 1小时, RTO < 4小时
```

脱敏 (Masking):
  - 手机号: 138****1234
  - 身份证: 3301**********1234
  - 地址: 只保留到城市级别

加密:
  - 传输: TLS 1.3 (所有组件间通信)
  - 存储: MinIO SSE-S3 (服务端加密)
```

## 第6章：遗留系统平滑迁移方案

### 6.1 迁移总体策略

```
迁移核心原则:

1. 双写并行 (Dual Write)
   - 新旧系统同时运行至少3个月
   - 新旧数据每日自动对账
   - 对账通过后逐步切换流量

2. 业务无损
   - 数据迁移期间业务正常运行
   - 任何一次迁移都可快速回滚
   - 用户无感知切换

3. 渐进替换
   - 按数据源优先级分批迁移 (P0 → P1 → P2)
   - 每个数据源单独验证、单独切换
   - 每批次间隔至少2周(观察期)

4. 数据校验
   - 迁移前: 全量行数+抽样值对比
   - 迁移中: 实时数据校验
   - 迁移后: 7天并行验证期
```

│  │ 示例: SELECT SUM(order_amount) FROM orders WHERE dt='昨天'│   │
│  │ 阈值: 差异 > 0.1% → P1告警 (GMV差异)                      │   │

│  ┌───────────────────────────────────────────────────────────┐   │
│  │ 随机抽样1000条记录, 逐字段比较值                            │   │
│  │ 阈值: 任意一行差异 > 0 → P3告警(记录)                      │   │
│  │ 抽样策略: 分层抽样(高价值订单+随机订单)                    │   │
│  └───────────────────────────────────────────────────────────┘   │
│                                                                     │
│  对账报告: 每日自动发送到数据质量频道(企微)                          │
│  存储: 所有对账记录存入数据质量表, 保留180天                       │
└───────────────────────────────────────────────────────────────────┘

回滚机制:

  如果任何阶段出现严重问题(数据差异 > 1%, 或核心报表错误):
  
  Step 1 [立即]: 停止新平台的报表访问(返回503+维护页)
  Step 2 [15min内]: 恢复旧Cron脚本运行(脚本已保留)
  Step 3 [30min内]: 用旧流程出报表, 发给业务方
  Step 4 [2h内]: 定位问题根因, 修复新平台Bug
  Step 5 [验证后]: 重新灰度上线

  回滚触发条件:
    - GMV差异 > 1% (连续2天)
    - 核心报表Delay > 4小时
    - 数据质量门禁 > 3项FAIL
    - 业务方联合投诉(≥3个部门)
