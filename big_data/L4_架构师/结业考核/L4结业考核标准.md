# L4结业考核标准

## 考核概述

| 项目 | 详情 |
|------|------|
| **考核阶段** | L4 架构师：全局设计 |
| **考核周期** | 2周（第43-44周） |
| **考核形式** | 系统设计面试 + 架构方案答辩 + 成本优化评估 + Mentor成果评估 |
| **通过标准** | 四项考核全部通过 |
| **答辩评委** | CTO评审团（3位以上L5级评委） |

---

## 考核一：系统设计面试

### 考核方式

```
形式: 现场白板设计 + 口头讲解
时长: 2小时
选题: 评委从题库中随机抽取1题（不在学员已练习的8题中）

流程:
  0-15分钟: 需求澄清
    - 学员主动提问，明确功能性和非功能性需求
    - 评委扮演"业务方"回答
    - 学员将关键约束写到白板上
  
  15-25分钟: 容量估算
    - 快速估算数据量/QPS/存储
    - 验证方案在物理上是否可行
  
  25-60分钟: 方案设计
    - 画出高层架构图（盒子+箭头+数据流）
    - 深入2-3个核心子系统
    - 说明每个关键组件的选择理由
  
  60-90分钟: 深入讨论
    - 评委追问"What if..."问题
    - 讨论Trade-Off和替代方案
    - 应对极端场景（10倍流量、机房断电等）
  
  90-105分钟: 方案总结
    - 学员用3分钟回顾整体方案
    - 补充遗漏点
  
  105-120分钟: Q&A + 评委打分
```

### 评分标准

| 评分维度 | 权重 | 优秀(5分) | 合格(3分) | 不合格(1分) |
|----------|------|-----------|-----------|-------------|
| 需求澄清 | 20% | 主动提出8+关键问题，准确识别隐含约束 | 提出5+问题，覆盖主要约束 | 没有提问就动笔设计 |
| 容量估算 | 15% | 估算准确，考虑峰值和增长，给出量级验证 | 有估算过程，数量级基本正确 | 没有估算或估算严重错误 |
| 架构设计 | 30% | 架构清晰分层，子系统职责明确，数据流完整标注 | 架构合理，主要模块覆盖 | 架构混乱，关键模块缺失 |
| 技术选型 | 20% | 每个关键组件有2+候选方案对比，选择理由充分 | 有候选方案意识，选择基本合理 | 仅提一个方案，无对比分析 |
| Trade-Off分析 | 10% | 深入分析2+关键权衡，理解代价和风险 | 能说出主要Trade-Off | 未体现权衡思考 |
| 沟通表达 | 5% | 逻辑清晰，图文并茂，能引导讨论节奏 | 表达基本清晰，能回答问题 | 表达混乱，答疑困难 |

**通过线**: 总分 ≥ 3.5 分（满分5分）

### 题库（部分）

```
1. 设计一个支持100万QPS的实时异常检测系统
2. 设计一个大数据平台的元数据管理系统
3. 设计支持100亿条/天的日志分析平台
4. 设计一个跨10个业务线的数据服务网关
5. 设计电商大促的实时库存扣减+对账系统
6. 设计一个支持2000个数据源的数据同步平台
7. 设计一个面向分析师的自然语言查询系统 (Text-to-SQL)
8. 设计一个支持1000+表的数据治理平台
```

## 考核二：架构方案答辩

### 考核方式

```
形式: PPT汇报 + 评委提问
时长: 汇报30分钟 + 提问30分钟
内容: 基于项目11（企业大数据平台架构设计）的方案

PPT结构要求 (不超过30页):
  Page 1: 标题页 + 方案摘要（1句话）
  Page 2-3: 业务背景与痛点（用数字说话）
  Page 4-6: 核心需求与非功能目标
  Page 7-10: 整体架构图 + 数据流（重点！）
  Page 11-18: 关键技术选型对比（5个关键组件×3方案对比）
  Page 19-21: 数据模型设计（分层+核心表结构）
  Page 22-23: 实施路线图（分Phase+里程碑）
  Page 24-25: 成本预算与ROI
  Page 26-27: 风险与应对
  Page 28-29: 总结 + 下一步行动
  Page 30: Q&A
```

### 评分标准

| 评分维度 | 权重 | 评估要点 |
|----------|------|----------|
| 需求理解 | 15% | 是否抓住核心痛点？是否量化了业务目标？ |
| 架构合理性 | 25% | 整体架构是否清晰？分层是否合理？数据流是否正确？ |
| 技术选型 | 25% | 每个关键组件是否有充分对比？选择理由是否convincing？ |
| 可行性 | 15% | 实施路线是否切实可行？是否考虑了团队能力？ |
| 成本意识 | 10% | 是否有详细的成本测算？ROI是否可量化？ |
| 风险管理 | 5% | 是否识别了关键风险？应对预案是否具体？ |
| 表达呈现 | 5% | PPT是否清晰？演讲是否有逻辑？答疑是否到位？ |

**通过线**: 3位以上评委评分 ≥ 3.5 分

### 评委常见追问（需提前准备）

```
1. "为什么选择StarRocks而不是ClickHouse？如果让你们用ClickHouse，方案会怎么改？"
2. "你们的容量估算是怎么算的？如果数据量是现在的10倍，架构哪里会先撑不住？"
3. "方案中哪一个决策你最有信心？哪一个最没信心？为什么？"
4. "如果给你砍掉50%的预算，你会砍什么？为什么？"
5. "这个方案最大的失败风险是什么？如果失败了你会怎么做？"
6. "如果现在就要上线Phase 1，你觉得团队还需要补什么能力？"
7. "方案中有没有过度设计的地方？如果重新做，会简化什么？"
```

## 考核三：成本优化评估

### 考核方式

```
形式: 方案文档评审 + 量化评估
内容: 基于项目12（FinOps成本优化方案）

量化要求:
  - 优化方案可量化的成本节省 ≥ 30%
  - 每项优化措施有计算过程和数据支撑
  - ROI分析有说服力，考虑了实施成本
```

### 评分标准

| 评分维度 | 权重 | 评估要点 |
|----------|------|----------|
| 审计深度 | 25% | 是否覆盖了计算/存储/Kafka？是否有数据支撑？ |
| 方案可行性 | 30% | 优化方案是否具体可执行？是否有实施优先级？ |
| 量化准确性 | 25% | 成本节省计算是否有依据？假设是否合理？ |
| ROI分析 | 15% | 是否考虑了实施成本？回本周期是否合理？ |
| 持续机制 | 5% | 是否有持续优化机制（Dashboard/Review流程）？ |

**通过线**: 评委评分 ≥ 3.5 分，且量化节省 ≥ 30%

## 考核四：Mentor成果评估

### 考核方式

```
形式: 学员评价 + 导师自述 + 结果验证
评估内容: L4学员作为Mentor指导的初级学员（L2）的学习成果

评估维度:
  1. L2学员结业通过率 ≥ 80%
  2. Mentor的学生满意度评分（匿名问卷）
  3. Mentor的指导记录（代码Review、1-on-1会议纪要）
  4. Mentor的教学案例（至少1个，描述如何帮助学员突破瓶颈）
```

### 评分标准

| 评分维度 | 权重 | 评估要点 |
|----------|------|----------|
| 学生通过率 | 40% | L2结业通过率 ≥ 80% |
| 学生满意度 | 30% | 匿名评分 ≥ 4.0/5.0 |
| 指导质量 | 20% | 代码Review质量、1-on-1频次和深度 |
| 教学案例 | 10% | 是否有1个以上深刻的教导案例 |

**通过线**: 综合评分 ≥ 3.5 分

## 综合评定

### 最终成绩计算

```
最终得分 = 考核一(30%) + 考核二(30%) + 考核三(25%) + 考核四(15%)

等级:
  4.5 - 5.0: 卓越 (Excellent) —— 直接推荐CTO梯队
  3.5 - 4.4: 通过 (Pass) —— 晋升L5
  2.5 - 3.4: 待改进 (Conditional Pass) —— 2周内补考未过项
  1.0 - 2.4: 不通过 (Fail) —— 重新学习4周后再次考核

补考规则:
  - 每项考核最多补考1次
  - 补考在原考核后2周内进行
  - 补考仍不通过，整体评定为不通过
```

### 晋升L5的标准

通过L4结业考核后，学员晋升至L5（CTO视野）阶段。L5阶段无固定结业时间，以持续评定方式进行。

```
L5评定标准:
  - 技术战略: 年度技术规划被采纳，12个月后回顾准确率 > 70%
  - 团队建设: 团队满意度 > 85%，关键人才保留率 > 90%
  - 商业价值: 能证明每项技术投入的商业回报
  - 行业影响: 每年至少1次大会演讲，或技术博客阅读量 > 5万
```

## 附录：考核FAQ

### Q1: 如果考核一中我想不出答案怎么办？
评委不会期望你给出"完美的答案"。系统设计面试考察的是思考过程，而非结论。即使方案有缺陷，只要体现出清晰的需求分析、合理的容量估算、比较性的选型思维和风险意识，就能得高分。

### Q2: 架构方案答辩需要注意什么？
**千万不能说"A比B好所以选A"**。必须说"A在X方面比B好，但在Y方面不如B。因为我们的场景Z更看重X，所以选择A。但如果场景发生变化......"——这才是一个架构师的思维。

### Q3: 成本优化必须真的在公司落地吗？
不需要（这是模拟场景）。但成本数据需要基于真实的云服务定价，计算逻辑需要严谨。评委可以随时追问"这个数据怎么来的"。

### Q4: 如果我带的L2学员都没过怎么办？
考核四的Mentor评估考察的是你的指导能力和努力程度，不完全等同于学生的绝对成绩。如果你有充分证据证明你付出了努力（如Code Review记录、1-on-1纪要、教学方案等），即使学生通过率不达标，也可以根据指导质量给予补考机会。

### Q5: 考核失败最常的原因是什么？
1. **需求澄清不够**：上来就画图，没问清楚约束
2. **只说一个方案**：没有体现比较和权衡思维
3. **成本意识缺失**：设计了一个完美的方案但成本不可接受
4. **PPT过于技术化**：评委中有非技术背景的"CTO"，听不懂技术黑话
5. **不会承认不知道**：面对不会的问题硬答，不如诚实说"这个我不确定，我的初步想法是..."

## 系统设计面试8道完整考题

### 考题1：设计一个支持100万QPS的实时异常检测系统

**题目场景描述**：
某电商平台每日交易流水超过5亿笔，需要构建一个实时异常检测系统，对每一笔交易进行风险评分。系统接入的数据包括：用户历史行为特征、设备指纹信息、IP地理位置、交易金额、商品类目等共约200维特征。检测模型为在线学习模型，每5分钟更新一次。

**需求澄清提示**：
- 功能性需求：交易风控评分、黑名单实时更新、规则引擎可配置
- 非功能性需求：P99延迟 < 50ms、SLA 99.99%、特征更新延迟 < 5分钟
- 数据量：日均5亿笔交易、峰值100万QPS、每笔交易上下文200维特征
- 特殊约束：金融合规要求（审计日志保留5年）、模型可解释性要求

**预期架构要点**：
- 特征缓存层：使用本地LRU缓存 + Redis Cluster，缓存热点特征以减少远程调用
- 规则引擎：自研规则引擎（支持DSL配置），规则热更新无需重启服务
- 模型服务：TensorFlow Serving / Triton Inference Server，GPU推理加速
- 数据流：Kafka → Flink特征计算 → Redis/Aerospike特征存储 → 在线推理服务
- 落地方案：Caffeine本地缓存(500MB, TTL 60s) → Redis Cluster(热点特征) → Aerospike(全量特征)

### 考题2：设计一个大数据平台的元数据管理系统

**题目场景描述**：
某公司有3000+张Hive表、200+个Kafka Topic、500+个Flink/Spark任务、50+个数据产品。目前元数据散落在Confluence文档、Excel表格、各个系统的配置文件中。数据开发找一张表平均需要15分钟。需要构建统一的元数据管理系统。

**需求澄清提示**：
- 功能性需求：表/字段/Topic/任务的注册与发现、数据血缘（字段级别）、数据字典
- 非功能性需求：搜索响应 < 500ms、元数据变更通知延迟 < 1分钟、API可用性 99.9%
- 用户角色：数据开发(贡献者)、数据分析师(消费者)、数据治理(管理员)
- 特殊约束：需要对接已有系统（Hive Metastore、Kafka Schema Registry、Airflow）

**预期架构要点**：
- 元数据采集：通过Hive Hook/Spark Listener采集任务运行时血缘；通过SQL解析器采集静态血缘
- 存储选型：Neo4j(血缘图) + Elasticsearch(全文搜索) + MySQL(结构化元数据)
- 血缘链路：Flink/Spark SQL解析 → Atlas Hook → Kafka → 血缘图构建 → Neo4j
- 搜索实现：Elasticsearch倒排索引，支持表名/字段名/描述/标签多维搜索
- 开放API：RESTful API + GraphQL（支持灵活字段组合查询）

### 考题3：设计支持100亿条/天的日志分析平台

**题目场景描述**：
某互联网公司日均产生100亿条日志（Nginx访问日志、应用日志、系统日志、业务埋点日志），需要构建统一的日志分析平台。日志格式不统一，部分日志含JSON结构，部分为纯文本。分析需求包括：错误率实时监控、用户行为路径分析、API性能分析、安全审计查询。

**需求澄清提示**：
- 功能性需求：实时日志检索（关键词/正则/时间范围）、日志聚合统计、日志上下文查看、告警规则配置
- 非功能性需求：写入峰值200万条/秒、检索P95 < 3秒（7天内数据）、全量日志保留30天
- 查询特点：90%的查询集中在最近24小时，10%需要查询全量30天数据
- 特殊约束：需支持多租户隔离（不同业务线数据不可互查）

**预期架构要点**：
- 采集层：Filebeat → Kafka（topic按业务线分区，实现多租户隔离）
- 实时索引：Flink ETL解析 + Elasticsearch热集群（SSD，保留3天热数据）
- 冷存储：Kafka数据直接写入HDFS/OSS（Parquet格式，按小时分区）
- 冷查询：Presto/Trino联邦查询HDFS冷数据，延迟 < 30秒
- 可视化：Grafana(实时) + Kibana(即席检索) + Superset(离线分析报表)

### 考题4：设计一个跨10个业务线的数据服务网关

**题目场景描述**：
某集团有10个业务线（电商、物流、金融、广告等），每个业务线有独立的数据团队和技术栈。现在需要构建统一的数据服务网关，提供对外统一的SQL查询入口。要求：支持联邦查询（跨业务线的数据表JOIN）、查询鉴权、资源隔离、查询审计、SQL优化建议。

**需求澄清提示**：
- 功能性需求：SQL解析与路由、联邦查询引擎、查询结果缓存、SQL改写（方言适配）
- 非功能性需求：简单查询P95 < 500ms、复杂联邦查询P95 < 5秒、并发1000 QPS
- 鉴权要求：库/表/列/行四级权限控制（基于RBAC+ABAC混合模型）
- 特殊约束：不同业务线使用不同的SQL引擎（Hive/Spark/ClickHouse/StarRocks/Presto）

**预期架构要点**：
- SQL解析：基于Apache Calcite的SQL解析器，支持方言识别和改写
- 查询路由：元数据中心(哪些表在哪个引擎) → 单源查询直发对应引擎 → 跨源查询走Trino联邦
- 结果缓存：Redis缓存热点查询结果（TTL按表更新频率动态调整）
- 资源隔离：使用Trino Resource Group按业务线分配查询资源
- 限流策略：令牌桶算法，默认100 QPS/业务线，突发200 QPS

### 考题5：设计电商大促的实时库存扣减+对账系统

**题目场景描述**：
电商双11大促，峰值订单创建100万笔/分钟。需要设计一个支持高并发的实时库存扣减系统，同时需要进行T+1日终对账（库存系统的库存数据 vs 订单系统的实际销售量），确保库存数据最终一致。

**需求澄清提示**：
- 功能性需求：实时库存扣减（预占库存 → 支付成功扣减 → 取消订单回滚）、库存预警（低于阈值告警）、库存补货建议
- 非功能性需求：库存扣减P99 < 10ms、SLA 99.99%、对账差异率 < 0.01%
- 并发量：峰值100万QPS库存查询、50万QPS库存扣减
- 特殊约束：严禁超卖，允许少卖（宁可少卖也不能超卖）

**预期架构要点**：
- 库存缓存：Redis Cluster存储实时库存（Key: sku:{id}:stock, Value: 剩余库存数）
- 扣减策略：Redis Lua脚本保证原子性（先检查库存 ≥ 扣减量，再DECR）
- 异步持久化：每100ms将Redis库存快照写入MySQL Binlog，用于灾备恢复
- 对账链路：每日凌晨Spark ETL读取订单表(实际销量) vs 库存系统快照(系统库存)
- 兜底方案：Redis故障时降级到MySQL读写（自动降级开关，QPS降低但保证可用）

### 考题6：设计一个支持2000个数据源的数据同步平台

**题目场景描述**：
某数据中台需要从2000+个异构数据源同步数据（MySQL/PostgreSQL/Oracle/MongoDB/HTTP API/FTP文件/SaaS平台API），统一入仓到数据湖。数据源分布在不同的网络环境（内网/VPN/公网）。要求支持全量同步和增量同步（CDC），断点续传，数据校验。

**需求澄清提示**：
- 功能性需求：全量同步 + 增量同步(CDC)、数据格式自动转换、断点续传、数据校验(行数+摘要)、异常重试
- 非功能性需求：单源全量同步 < 4小时、增量同步延迟 < 1分钟、平台SLA 99.9%
- 异构性：关系型数据库15种 + NoSQL 5种 + API 20+种SaaS + 文件协议5种
- 特殊约束：部分数据源仅支持特定时间段访问(如银行日终对账文件仅在凌晨提供)

**预期架构要点**：
- 插件化架构：Reader插件(数据源抽象) + Writer插件(目标端抽象)，插件热加载
- 调度引擎：基于DAG的调度编排，支持依赖触发、时间触发、事件触发
- CDC方案：MySQL用Debezium、PostgreSQL用Pgoutput、Oracle用LogMiner/OGG
- 数据校验：MD5摘要 + 行数对账，同步完成后自动触发校验任务
- 监控告警：同步延迟监控、数据量波动告警、数据源连通性心跳检测

### 考题7：设计一个面向分析师的自然语言查询系统（Text-to-SQL）

**题目场景描述**：
某公司有2000+张数据表，数据分析师常需要编写SQL查询数据，但部分分析师SQL能力有限。需要构建一个自然语言查询系统，分析师用中文输入问题，系统自动生成SQL并返回结果。要求：简单查询准确率 > 95%，复杂查询准确率 > 80%。

**需求澄清提示**：
- 功能性需求：NL2SQL自动生成、SQL解释（生成的SQL用自然语言回译）、查询结果可视化、对话式追问
- 非功能性需求：简单查询 < 3秒(含生成+执行)、复杂查询 < 10秒、并发100 QPS
- 难点：2000+张表的Schema理解、业务术语映射（"大客户" → `ltv > 10000`）、歧义消解
- 特殊约束：不允许将表数据上传到外部LLM API（数据安全）、生成的SQL必须经过安全校验

**预期架构要点**：
- NL2SQL引擎：基于大模型（内部部署的Qwen/ChatGLM）+ Few-Shot Prompt Engineering
- 上下文增强：RAG（检索增强生成）——先检索相关表Schema和业务元数据，再注入Prompt
- Schema索引：将表和字段信息向量化存储（Milvus/Weaviate向量数据库），支持语义检索
- SQL安全校验：禁止DML语句、限制查询行数(默认1000行)、禁止全表扫描(必须带WHERE条件)
- 反馈闭环：用户对SQL结果的点赞/点踩 → 作为Fine-tuning数据迭代模型

### 考题8：设计一个支持1000+表的数据治理平台

**题目场景描述**：
某数据中台管理1000+张Hive表，存在严重的"数据沼泽"问题：30%的表从未被访问、50%的字段缺少描述、大量相似表存在（如`user_info`和`user_info_v2`和`user_info_new`）、数据血缘链路不清晰。需要构建数据治理平台。

**需求澄清提示**：
- 功能性需求：数据资产盘点、数据质量评分、生命周期管理、相似表检测、数据安全定级
- 非功能性需求：全量表扫描 < 4小时完成、评分计算 < 2小时、前端Dashboard加载 < 3秒
- 治理维度：完整性（字段描述覆盖率）、规范性（命名规范检查）、安全性（敏感字段识别）、价值性（访问热度）
- 特殊约束：治理平台本身不能侵入数据生产流程（非侵入式设计）

**预期架构要点**：
- 资产采集：Hive Metastore API全量拉取 + 定期增量同步，每天凌晨执行
- 评分模型：多维度加权评分（完整性30% + 规范性20% + 安全性20% + 价值性30%）
- 相似表检测：基于表名TF-IDF向量化 + 字段Schema的Jaccard相似度 + 数据分布相似度
- 生命周期管理：冷热分层（30天未访问→冷存、90天未访问→归档、180天未访问→建议删除）
- 治理工作流：问题发现 → 自动分配负责人 → 限期整改 → 自动验收 → 关闭循环

## 补充：答辩现场常见情景及应对策略

### 情景1：评委说"你这个方案成本太高了"

**错误应对**："我会优化成本的"（太笼统，没有具体方案）

**正确应对**：
1. 承认："是的，当前方案基于全功能需求设计，在成本方面存在优化空间"
2. 量化："如果砍掉50%预算，我会优先砍掉以下几项...（具体到组件和功能）"
3. 折中："砍掉后，性能下降约X%，但核心SLA仍可满足"
4. 替代："还可以考虑用Y替代X，成本低但运维复杂度上升"

### 情景2：评委问"这个技术你用过吗？"

**如果用过**："我在XX项目中用过，规模是（具体数字），遇到过以下坑（具体问题），我们的解决方案是（具体方案）"

**如果没用过**："我没有在生产环境用过，但我通过（阅读官方文档/看源码/搭建POC/阅读案例）了解到X的特性是...我的理解可能存在偏差，请您指正"

### 情景3：评委指出方案中的缺陷

**错误应对**："不会的，我觉得没问题"（防御性态度）

**正确应对**：
1. 先确认："我理解您的担忧，这里确实是一个潜在风险"
2. 应对："我的应对思路是...您觉得这个方案是否可行？"
3. 求助："这个方面我确实考虑不深，您有什么建议吗？"

### 情景4：不到30分钟评委就说"可以了，你有什么问题问我们？"

**这是测试**——可能意味着：
- A) 你确实表现很好，评委已经满意（可能性20%）
- B) 你的方案太肤浅，评委不想浪费时间（可能性60%）
- C) 评委在测试你的主动性（可能性20%）

**正确应对**：
- "我还想补充X方面的设计细节，因为这是架构的核心"（展示深度）
- "我还准备了一个备选方案B，想请教评委的意见"（展示比较思维）
- "关于X技术选型，我最近关注到Y的新特性，想听听评委的看法"（展示持续学习）
