## 第2部分：优化方案

### 方案1：冷热数据分层存储

```
策略设计:

┌─────────────────────────────────────────────────────┐
│  热数据 (Hot, <7天)                                   │
│  存储: NVMe SSD (MinIO Hot Pool)                     │
│  数据: ODS近7天 + 当前DWD分区 + StarRocks实时数据     │
│  访问延迟: < 5ms                                      │
│  成本: 高                                             │
├─────────────────────────────────────────────────────┤
│  温数据 (Warm, 7-30天)                                │
│  存储: HDD (MinIO Warm Pool)                         │
│  数据: ODS 7-30天 + DWD近30天分区                     │
│  访问延迟: < 50ms                                     │
│  成本: 中                                             │
├─────────────────────────────────────────────────────┤
│  冷数据 (Cold, >30天)                                 │
│  存储: 归档存储 (阿里云OSS 低频/归档)                 │
│  数据: 历史DWS/ADS汇总 + ODS >30天                    │
│  访问延迟: 分钟级 (需先取回)                          │
│  成本: 极低                                           │
└─────────────────────────────────────────────────────┘

实施方式:
  Iceberg Table TTL配置:
    ALTER TABLE ods_user_behavior_log SET TBLPROPERTIES (
      'write.distribution-mode' = 'hash',
      'history.expire.max-snapshot-age-ms' = '2592000000'  -- 30天
    );
  
  自动化迁移脚本 (Airflow DAG, 每日00:30执行):
    # 1. Hot→Warm: dt < 7天前的分区迁移到Warm Pool
    # 2. Warm→Cold: dt < 30天前的分区迁移到OSS归档
    # 3. 清理过期快照 + 孤儿文件

预期效果:
  当前存储分布:  热150TB + 温250TB + 冷300TB + 死150TB
  优化后分布:    热100TB + 温200TB + 冷400TB + 已清理150TB
  
  成本节省:
    Hot SSD: 150→100TB, 节省 50TB × $0.1 = $5,000/月
    Warm HDD: 250→200TB, 节省 50TB × $0.03 = $1,500/月
    Cold归档: 300→400TB, 增加 100TB × $0.01 = $1,000/月
    清理死数据: 150TB 彻底释放
    
    月节省: $5.5K
    年节省: $66K
    预期总存储成本降低: 40%
```

### 方案2：计算资源弹性伸缩

```
策略设计:

弹性伸缩策略:
  ┌──────────────────────────────────────────────────┐
  │ K8s Cluster Autoscaler + Karpenter               │
  │                                                   │
  │ 基线常驻: 300核 (白天)                             │
  │ 凌晨扩容: 1200核 (02:00-06:00 ETL高峰)            │
  │ 白天缩容: 300核 (06:00-22:00 低负载)              │
  │ 自动降配: 周末/节假日进一步降低基线                │
  └──────────────────────────────────────────────────┘

CronHPA配置:
  apiVersion: autoscaling.alibabacloud.com/v1beta1
  kind: CronHorizontalPodAutoscaler
  metadata:
    name: spark-executor-scaler
  spec:
    scaleTargetRef:
      apiVersion: apps/v1
      kind: Deployment
      name: spark-executor
    jobs:
    - name: "night-etl-scale-up"
      schedule: "0 1 * * *"      # 凌晨1点扩容
      targetSize: 40             # 40个Executor Pod
    - name: "morning-scale-down"
      schedule: "0 7 * * *"      # 早上7点缩容
      targetSize: 8              # 8个Executor Pod
    - name: "weekend-scale-down"
      schedule: "0 0 * * 0,6"    # 周末缩容
      targetSize: 4

Spot实例策略:
  离线ETL任务 (可中断): 100%使用Spot实例
  实时Flink任务 (不可中断): 使用On-Demand实例
  ML训练任务: Spot为主 + On-Demand兜底
  
  Spot中断处理:
    - Karpenter自动检测Spot中断通知
    - 优雅下线: Drain节点 → 重新调度 → 新Spot节点接管
    - Flink任务使用Savepoint恢复

预期效果:
  当前: 固定1200核 × 24h = 28,800核时/天
  优化后:
    常驻300核 × 16h = 4,800核时 (白天)
    扩容1200核 × 4h = 4,800核时 (ETL)
    过渡800核 × 4h = 3,200核时 (过渡)
    ──────────────────────────
    总计: 12,800核时/天 (节省55.6%)
    
    Spot实例占80%: 成本再降70%
    
  月计算成本:
    优化前: 28,800 × 0.30 × 30 = 259,200元/月
    优化后: (12,800 × 0.09 × 30) = 34,560元/月  (Spot混合价)
    月节省: ~224,640元
    年节省: ~270万
    计算成本降低: 87%
```

### 方案3：任务合并与去重

```
当前问题:
  3个团队在做几乎相同的用户画像:
    - 数据平台团队: user_profile_tag (2800核时/天)
    - 算法团队: user_profile_v2 (2100核时/天)
    - 运营团队: 用Python脚本也跑一份 (手动)
  
  重复计算: 4900核时/天 × 30天 = 147,000核时/月

解决方案:
  建立"公共数据集市":
    统一用户画像表: dim.dim_user_profile_wide
    维护团队: 数据平台团队 (唯一Owner)
    产出方式: T+1 Spark ETL
    下游消费:
      - 算法团队直接读取dim表
      - 运营团队通过Superset查询dim表
      - 营销团队通过API获取指定用户画像
  
  其他可合并任务:
    - 订单宽表: 2个团队各自维护 → 统一为dwd.dwd_order_detail
    - 流量报表: 3套不同维度的日报 → 统一DWS层
    
预期效果:
  消除重复计算: 4900核时/天 → 2800核时/天 (节省2100核时/天)
  数据一致性提升: "同一个指标两个数字"问题彻底解决
  维护成本降低: 3个团队各自维护 → 1个团队集中维护
  
  月节省: 2100 × 30 × 0.30 = 18,900元/月
  年节省: ~23万
  额外收益: 数据一致性问题的管理成本节省 (无法量化但价值巨大)
```

### 方案4：查询优化

```
审计方法:
  收集所有SQL查询日志 (Trino/StarRocks/Hive)
  分析Top 10最高成本的查询
  评估优化空间

Top 5 高成本查询分析:

查询1: 运营日报 (日执行500次, 每次扫描500GB)
  SELECT * FROM dwd_order_detail WHERE dt >= '...'
  -- 问题: 每次全表扫描30天的数据
  -- 优化: 创建物化视图, 预聚合到日粒度
  CREATE MATERIALIZED VIEW mv_order_daily_summary AS
  SELECT dt, category_id, SUM(amount), COUNT(*)
  FROM dwd_order_detail
  GROUP BY dt, category_id;
  -- 效果: 查询从60s → 0.5s, 扫描量从500GB → 50MB

查询2: 用户留存分析 (日执行100次, 每次扫描200GB)
  -- 问题: 复杂多步CTE, 重复计算
  -- 优化: 预计算留存表到ADS层, BI直接查询ADS表
  -- 效果: 查询从30s → 1s

查询3: 实时GMV大屏查询 (每秒执行1次)
  -- 问题: 每次都扫描全表
  -- 优化: 使用StarRocks物化视图 + 预聚合
  -- 效果: P99从500ms → 20ms

查询4: 用户行为漏斗 (日执行50次)
  -- 问题: 5步CTE + 多次JOIN
  -- 优化: 预计算每日漏斗转化率表
  -- 效果: 查询从120s → 2s

查询5: Ad-hoc探索查询 (无分区过滤)
  -- 问题: 数据分析师忘加分区条件
  -- 优化: Trino配置 query_max_scan_size=100GB, 强制分区过滤
  -- 效果: 阻止全表扫描, 倒逼分析师加上分区条件

预期效果:
  查询优化后扫描量减少80%
  物化视图占用的额外存储: +50GB (可忽略)
  
  查询成本 (Trino按扫描量计费):
    优化前: 日均扫描50TB × 30天 = 1500TB/月
    优化后: 日均扫描10TB × 30天 = 300TB/月
    月节省扫描量: 1200TB
    
  月节省: 1200TB × $0.5/TB扫描(Trino成本) = $600/月
  查询响应时间缩短: 平均 30s → 3s (用户体验大幅提升)
  年节省: ~$7,200
```

```
月度成本Review (每月第3个工作日):
  议程:
  1. 上月成本回顾 (vs 预算, vs 上月)
  2. 异常成本分析 (Top 3异常)
  3. 优化建议讨论
  4. 下月成本预测

季度成本治理 (每季度):
  议程:
  1. 季度成本趋势
  2. 大项优化措施回顾
  3. 下季度优化目标 (量化)
  4. 预算调整建议

成本红线:
  - 部门月度成本超过预算的120% → 自动暂停非核心任务
  - 单任务日消耗 > 500核时 → 需要架构师审批
  - 新表30天无访问 → 自动归档提醒
```

```

标签强制策略:
```
