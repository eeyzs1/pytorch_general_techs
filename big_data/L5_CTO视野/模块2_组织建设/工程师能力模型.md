# 工程师能力模型

> **定位**：定义大数据工程师从L1到L5的完整能力成长路径，为招聘、晋升、培养提供统一的评估标准。
>
> **核心理念**：能力模型不是"熟练使用XXX"的技能清单，而是可观察、可评估的行为标准。每条标准必须能回答一个问题："我怎么知道这个人达到了这个标准？"

---

## 能力模型设计原则

```yaml
原则1 - 可评估性:
  坏的标准: "精通Spark"
  好的标准: "能够独立使用Spark SQL完成包含3表JOIN+窗口函数的数据分析需求，并解释执行计划中的关键优化"
  
  为什么? "精通"无法衡量，面试官可以随意解读。具体的行为描述才能客观评估。

原则2 - 成长导向:
  模型不是为了"给人贴标签"，而是为了"告诉人该怎么成长"
  每个层级的标准应该清晰回答：从L2到L3，我需要多做什么、做到什么程度？

原则3 - 双通道:
  区分I/C（Individual Contributor，独立贡献者）和M（Manager，管理者）两个方向
  L1-L3为纯I/C通道，L4分叉为I/C和M路线，L5为两者合一

原则4 - 宽严相济:
  核心能力标准是"必须项"（Must Have），晋升必须全部达标
  加分项是"锦上添花"（Nice to Have），有则更好但不强制
```

---

## 能力维度定义

```yaml
五个核心维度:

维度1: 技术能力（Technical Competency）
  定义: 掌握和运用技术解决实际问题的能力
  子维度:
    - 技术广度: 掌握的技术栈范围
    - 技术深度: 对某一领域的深入程度（源码级理解）
    - 工程能力: 代码质量、系统设计、性能调优
    - 学习能力: 新技术的学习速度

维度2: 业务理解（Business Acumen）
  定义: 理解业务、将技术翻译为业务价值的能力
  子维度:
    - 数据意识: 能用数据描述和量化业务问题
    - 产品思维: 理解用户需求和产品逻辑
    - 商业嗅觉: 理解公司的商业模式和竞争环境
    - ROI意识: 做技术决策时考虑投入产出比

维度3: 协作与沟通（Collaboration & Communication）
  定义: 与他人高效协作和沟通的能力
  子维度:
    - 文档能力: 撰写清晰的设计文档和技术方案
    - 沟通效率: 能用对方（业务方/上级/下属）的语言沟通
    - 跨团队协作: 推动跨团队的技术方案落地
    - 知识分享: 对内做技术分享、对外输出技术影响力

维度4: 领导力（Leadership）
  定义: 影响和带领他人取得更大成果的能力
  子维度:
    - 技术领导力: 制定技术方案、Review代码、指导初级工程师
    - 项目领导力: 推动项目从0到1交付
    - 人才培养: 辅导和培养团队成员成长
    - 技术文化: 影响和塑造团队的工程文化

维度5: 影响力（Influence）
  定义: 在公司内外产生技术影响力的能力
  子维度:
    - 团队内影响力: 技术决策被团队采纳的比例
    - 跨团队影响力: 推动跨团队的技术标准和最佳实践
    - 公司级影响力: 参与公司级技术战略制定
    - 行业影响力: 开源贡献、大会演讲、技术博客
```

## L1 初级大数据工程师

### 定位

能够在指导下完成任务的大数据初学者。需要明确的指令和及时的反馈。

### 技术能力

```yaml
必达标准:
  
  数据处理:
    - 能用Python编写数据处理脚本（文件读写、CSV/JSON解析、基础清洗）
    - 能写出JOIN+GROUP BY+窗口函数的标准SQL查询
    - 理解HDFS的基本操作（上传/下载/查看/删除）
  
  大数据工具:
    - 能用Hive SQL完成简单的数据查询和聚合分析
    - 能提交Spark任务并查看Spark UI的基本信息
    - 理解MapReduce的Map和Reduce阶段做了什么
  
  工程能力:
    - 代码提交遵循团队规范（commit message格式、分支命名）
    - 能独立解决简单的merge conflict
    - 能写出基本可运行的单元测试

行为锚定示例:
  给定一个"统计过去7天各品类GMV"的需求:
    L1水平: 能写出正确的SQL，但可能不知道为什么要加分区条件
    L2水平: 会主动加上分区裁剪条件并检查执行计划
```

### 业务理解

```yaml
必达标准:
  - 理解公司核心业务指标的含义（GMV、DAU、留存率、转化率）
  - 能正确的取数：理解"业务方要的是什么"，避免取错数据
  - 能在做完数据需求后，附上"数据口径说明"
```

### 协作与沟通

```yaml
必达标准:
  - 能清晰地描述自己遇到的问题（包括：做了什么、期望什么、实际看到什么）
  - 能看懂别人的代码并给出简单的Code Review意见
  - 能在团队内做10分钟的技术分享（如"我这周学会的一个知识点"）
```

### 领导力

```yaml
必达标准:
  - (L1不要求领导力)
  - 加分项: 能主动帮助更新的新人解决环境搭建问题
```

### 影响力

```yaml
必达标准:
  - (L1不要求影响力)
  - 加分项: 有个人技术博客（持续更新），GitHub有持续的贡献记录
```

### L1 → L2 晋升标准

```
晋升条件: 以下2条全部满足
  1. 独立完成至少1个中等复杂度的数据需求（涉及3表以上JOIN + 窗口函数）
  2. 能够独立排查简单的数据问题（如"数据量为什么少了？"）

晋升考察方式:
  - 代码Review表现
  - 项目交付的质量和独立性（是否需要大量指导）
  - 360度评估（同事+导师评价）
```

## L2 中级大数据工程师

### 定位

能够独立完成中等复杂任务的工程师。是团队的主要战斗力。

### 技术能力

```yaml
必达标准:

  数据处理:
    - 能独立完成包含复杂业务逻辑的数据管道开发
    - 熟练使用Spark DataFrame API和Spark SQL解决分析问题
    - 能理解和解释Spark的执行计划（EXPLAIN），识别常见的性能问题
  
  流处理:
    - 理解Kafka的核心概念（Topic/Partition/Consumer Group/Offset）
    - 能编写基本的Flink流处理任务（Source→Transform→Sink）
    - 能解释Kafka的至少一次（At-Least-Once）语义和重复消费的处理方式
  
  数据仓库:
    - 理解数仓分层（ODS/DWD/DWS/ADS）的概念和设计原则
    - 能独立设计中等复杂度的ETL任务（包含多表Join和聚合）
    - 理解分区表和文件的存储格式（Parquet/ORC）对性能的影响
  
  工程能力:
    - 能编写清晰的代码注释和模块文档
    - 能设计合理的错误处理和重试机制
    - 能使用Git进行分支管理和Code Review

行为锚定示例:
  给定一个"构建用户日活跃汇总表"的需求:
    L2水平: 能独立完成全流程（ODS→DWD→DWS），包含分区策略和压缩格式选择
    L3水平: 在L2基础上，能识别和解决数据倾斜，能设计增量更新机制，处理迟到数据
```

### 业务理解

```yaml
必达标准:
  - 能根据业务需求自主设计合理的分析维度（不依赖业务方给出的具体SQL）
  - 能识别数据异常并向业务方主动预警（如："今日GMV异常下降，经查是XX原因"）
  - 能参与业务需求评审，给出技术可行性和工作量的合理估算

行为锚定:
  业务方说: "帮我看看昨天的用户活跃度"
  L1反应: 直接问"你要什么SQL？"或写一条最简单的DAU SQL
  L2反应: 追问"你要看新客活跃还是老客活跃？是要城市维度还是品类维度？"
           → 主动设计一个包含3-4个分析维度的查询
```

### 协作与沟通

```yaml
必达标准:
  - 能写出清晰的设计文档（问题背景→方案设计→技术细节→风险）
  - 能在Code Review中给出有建设性的反馈（不只是"LGTM"）
  - 能主动分享技术经验（如：写一篇技术博客/内部分享）
  - 能与其他团队工程师有效协作（理解彼此的接口和依赖关系）
```

### 领导力

```yaml
必达标准:
  - 能指导L1工程师完成常规任务
  - 能在自己负责的模块内做技术决策
  - 能承担1个中小型项目的技术Owner角色
```

### 影响力

```yaml
必达标准:
  - (L2不强制要求对外影响力)
  - 加分项: 在团队内发起1个改进倡议并推动落地（如：改进CI流程）
```

### L2 → L3 晋升标准

```
晋升条件: 以下4条必须全部满足
  1. 独立交付过1个跨越3个以上数据层的完整数据管道
  2. 至少1次成功排查和解决生产环境故障（有复盘文档）
  3. 有可证明的指导经验（所带L1成员的能力有明显提升）
  4. 完成DDIA前8章的深度阅读并能映射到实际工作

晋升考察方式:
  - 技术方案评审（考察系统设计思路）
  - 故障排查能力评估（给定模拟故障场景）
  - 被指导者的评价
  - DDIA读书笔记答辩
```

## L3 高级大数据工程师

### 定位

某个技术方向的专家，能解决团队中最难的技术问题。是团队的"技术天花板"。

### 技术能力

```yaml
必达标准:

  技术深度（至少满足以下2项）:
    - 阅读过Spark/Flink/Kafka任意一个核心模块的源码（至少3000行），能解释设计意图
    - 能根据业务场景设计和实现自定义的数据处理算子/Connector
    - 能深入分析性能瓶颈：从JVM GC到Shuffle到OS Page Cache，逐层排查
    - 有1个被合入开源项目的PR（或等价的内部框架贡献）
  
  系统设计:
    - 能独立设计包含5+个子系统的数据处理架构
    - 能对新技术做系统化评估（不只是"看起来不错"，而是有POC数据支撑）
    - 能在实时(流) vs 离线(批)之间做合理取舍和架构设计
  
  性能调优:
    - 能诊断和解决数据倾斜问题（至少掌握3种不同的策略）
    - 能通过参数调优将任务性能提升5倍以上（且有数据支撑的优化报告）
    - 理解Spark/Flink内存模型，能根据任务特征调整内存配置
  
  论文阅读:
    - 已精读GFS/MapReduce/BigTable/Dremel/Spark RDD/Kafka等6+篇经典论文
    - 能用自己的话讲清楚每篇论文的核心设计和权衡

行为锚定示例:
  线上Spark任务运行超时:
    L3水平: 
      1. 打开Spark UI，识别数据倾斜的Stage和Task
      2. 定位倾斜Key，选择加盐/Broadcast/AQE方案
      3. 调优后性能提升8倍
      4. 将排查过程和方法论写成文档分享给团队
```

### 业务理解

```yaml
必达标准:
  - 能将业务问题翻译为技术方案（不只是翻译SQL，而是设计合理的计算和存储方案）
  - 能对业务方说"不"——合理评估需求的技术ROI，避免过度工程
  - 能主动提出技术驱动的业务创新建议（如："基于实时数据我们可以做XX"）

行为锚定:
  业务方提出: "能不能把所有的用户行为数据都实时接入？"
  L2反应: "好，我设计一个Flink管道"（埋头干活）
  L3反应: "为什么要实时？哪些场景需要实时？离线T+1能不能满足核心需求？
           如果只需要10%的指标实时，成本可以降低80%"（追问Why）
```

### 协作与沟通

```yaml
必达标准:
  - 能写出一流的技术设计文档（被其他团队作为参考模板）
  - 能主导跨团队技术方案的评审和决策
  - 能在团队内做深度技术分享（不只是"简介"，而是源码级解读）
  - 能有效Push Back不合理的需求（有理有据，保护团队）
```

### 领导力

```yaml
必达标准:
  - 至少成功指导过2名L2成员的成长（他们承认你的帮助是关键的）
  - 能带领3-5人的小团队完成1个季度的项目交付
  - 能建立和推动团队内的某项技术标准（如：代码规范、SQL编写规范）
```

### 影响力

```yaml:
必达标准（满足1项即可）:
  - 在团队内推动了1项有显著效果的技术改进（如：优化了CI流水线，全团队受益）
  - 有1篇技术博客被外部媒体转载或获得1000+阅读量
  - 在公司级技术大会上做过分享
```

### L3 → L4 晋升标准

```
晋升条件: 以下5条必须全部满足
  1. 独立完成过1个企业级大数据平台的架构设计（被评审团认可）
  2. 在性能调优领域有明显的专长（有一个可量化的成功案例：性能提升10倍+）
  3. 至少指导过3名工程师，其中1名晋升
  4. 完成论文精读12篇 + 源码深潜3000行
  5. 有外部技术影响力（开源贡献/大会演讲/技术博客）

晋升考察方式:
  - 架构方案答辩（由L4+架构师评审团评估）
  - 源码深度问答（随机抽取核心代码段，解释设计意图）
  - 带人成果评估（被指导者的晋升记录）
  - 外部影响力评估（博客/演讲/PR的记录）
```

## L4 数据架构师 / 技术经理

### 定位

L4是分叉点：
- **I/C路线（架构师）**：专注于技术深度，解决公司级架构问题
- **M路线（技术经理）**：开始管理团队，负责人才招聘、培养和绩效

### I/C路线：数据架构师

#### 技术能力

```yaml
必达标准:

  架构设计:
    - 能独立完成支持100TB+日处理量的数据平台架构设计
    - 能在技术选型中做有说服力的Trade-Off分析（至少3个维度的对比矩阵）
    - 能预见架构的技术债务（在设计中就标注"2年后需要重构的部分"）
    - 能设计跨多云/混合云的数据架构方案
  
  技术广度:
    - 对大数据生态（Spark/Flink/Kafka/Hive/Iceberg/Trino/ClickHouse等）有系统性的理解
    - 能说出每个组件"最擅长什么、最不擅长什么、什么场景不该用"
    - 对至少2个技术方向有源码级理解（如Spark Catalyst优化器 + Flink Checkpoint机制）
  
  成本与性能:
    - 能从ROI角度评估技术方案（不只是技术最优，而是性价比最优）
    - 能设计FinOps方案（成本归因、冷热分层、弹性伸缩）
  
  技术前瞻:
    - 能对1-2年的技术趋势做有理有据的判断
    - 能发现团队技术栈的"过度工程"问题并推动简化
```

#### 业务理解

```yaml
必达标准:
  - 能与CTO/VP一起参与年度技术规划的制定
  - 能将公司业务战略翻译为技术战略（"我们要做XX业务 → 所以需要YY技术能力"）
  - 能对技术投入的ROI做量化分析（不是凭感觉）
  - 能识别和驱动"技术驱动的业务创新机会"（不是等业务方提需求）
```

#### 领导力

```yaml
必达标准:
  - 能带领5-10人技术团队交付复杂的平台级项目
  - 能通过文档和标准影响50+人的技术决策（跨团队影响力）
  - 能成为团队成员的"技术导师"（不只是管理者，更是技术榜样）
```

#### 影响力

```yaml
必达标准（满足2项以上）:
  - 开源项目Committer或PMC Member
  - 在QCon/ArchSummit/Data+AI Summit等顶级大会上演讲
  - 技术博客持续更新，阅读量累计10万+
  - 在公司内部推动了1个跨部门的技术标准落地
```

### M路线：技术经理

#### 管理能力

```yaml
必达标准:

  人才招聘:
    - 能设计合理的面试流程和评估标准
    - 独立面试超过30人，招聘命中率（通过试用期/总入职数）> 80%
    - 能识别和吸引优秀的候选人
  
  团队管理:
    - 能有效管理8-15人的团队
    - 能合理分配任务，匹配每个人的能力和成长需求
    - 能处理团队冲突和绩效问题
    - 能帮助团队成员设定清晰的成长路径
  
  项目管理:
    - 能同时管理3-5个并行项目
    - 能合理评估项目风险和时间线
    - 能在项目受阻时有效"疏通"（跨团队协调）
  
  沟通与影响力:
    - 能向上管理：与CTO/VP有效沟通团队进展和问题
    - 能跨部门协调：推动依赖方按时交付
    - 能为团队争取合理的资源和认可
```

#### 技术能力（M路线的技术要求）

```yaml
必达标准:
  - 技术能力不低于L3水平（能看懂团队的技术方案，能Review核心代码）
  - 不需要是最强的技术专家，但必须能判断技术决策的质量
  - 能在技术评审中提出关键问题（即使自己不一定能写出最优方案）
  - 对团队使用的核心技术栈保持基本的动手能力
```

## L5 CTO / 技术VP

### 定位

公司的技术领袖。负责技术战略制定、组织建设、商业翻译和外部影响力。

> L5的详细评定标准见 `../L5持续评定标准.md`

### 技术远见

```yaml
必达标准:
  
  战略制定:
    - 能制定3-5年的技术路线图（不只是架构升级，还要匹配业务发展）
    - 能判断"做不做"的决策（新技术的引入时机、旧技术的淘汰节奏）
    - 能平衡"自研 vs 采购 vs 开源"的投入组合
    - 12个月后回顾，技术规划的准确率 > 70%
  
  风险判断:
    - 能识别技术组合中的关键风险点（单点故障/Bus Factor/供应商锁定）
    - 能在技术事故中做出正确的"停止/继续/回滚"决策
    - 能判断"哪些技术债务必须立即还、哪些可以继续欠"

行为锚定:
  面对一个新技术浪潮（如LLM对数据工程的影响）:
    优秀CTO的反应:
      1. 不是立即决定"用或不用"
      2. 而是：判断这是"炒作期"还是"实质变革"
      3. 如果是实质变革：制定"学习→POC→试点→推广"的渐进路线
      4. 同步调整招聘计划和组织架构
```

### 组织建设

```yaml
必达标准:
  
  人才密度:
    - 团队的L3+工程师占比在同行业中处于前25%
    - 关键岗位有明确的继任者（至少1人，最好是2人）
    - 团队成员对自己在"能力模型"中的位置有清晰认知和成长路径
  
  工程文化:
    - 建立了有效的Code Review文化（Review率 > 90%，有实质Review而非"LGTM"）
    - 建立了常态化的技术分享机制（频率≥每月2次，参与率≥70%）
    - 数据驱动的决策习惯在团队中扎根（方案评审有数据支撑）
  
  组织设计:
    - 团队的组织架构能匹配技术战略需求
    - 职责边界清晰，协作流程顺畅
    - 跨团队/跨部门合作中没有明显的"灰色地带"

行为锚定:
  一位高级工程师想离职:
    优秀CTO的反应:
      1. 1对1了解真实原因（不只是"有更好的机会"）
      2. 分析是个人问题还是系统问题（如果是系统问题，需要组织级修复）
      3. 做出留任或祝福的决定（尊重个人选择）
      4. 如果决定放人：确保知识交接，评估Bus Factor变化
```

### 商业翻译

```yaml
必达标准:
  
  向上沟通:
    - 能用1页PPT让CEO理解"为什么需要这笔技术预算"
    - 能用业务数据（而非技术术语）证明技术投入的价值
    - 能在董事会中清晰陈述技术战略
  
  对外沟通:
    - 能在技术事故中有效管理外部感知（客户/投资人/媒体）
    - 能代表公司在行业大会上发表有影响力的演讲
    - 能与行业伙伴/竞品CTO建立良性的交流关系

行为锚定:
  向CEO申请5000万技术预算:
    合理的沟通方式:
      Page 1: 我们现在的技术问题（用业务影响描述）→ "因为技术债务，每个月浪费300小时救火"
      Page 2: 如果什么都不做 → "明年这个数字会变成500小时，可能导致1次以上的重大业务中断"
      Page 3: 我们的方案（只说做什么）→ "投入5000万，未来3年每年节省2000万人力成本"
      Page 4: ROI计算和风险 → "ROI 120%，最大的风险是..."
      Page 5: 需要董事会做的决定 → "批准预算，授权启动XX项目"
```

### 行业影响

```yaml
必达标准:
  
  内部:
    - 公司的技术品牌在行业中具有识别度（候选人说"我因为你们的技术而想加入"）
  
  外部（满足2项以上）:
    - 开源项目PMC Member或核心贡献者
    - 顶级技术大会Keynote/Session演讲者
    - 技术媒体（InfoQ等）的常驻作者或受访者
    - 参与了行业标准（如Apache项目）的制定或推动
```

## 能力模型的使用指南

### 面试评估

```yaml
面试评估方法:

  技术能力评估:
    L1: 现场编程（Python + SQL）
    L2: 系统设计简答 + 故障排查场景
    L3: 源码问答 + 论文讨论 + 性能调优案例分析
    L4: 架构设计全流程（从需求分析到成本预算）
    L5: 战略决策案例分析（给定场景，做技术决策并说服"董事会"）

  行为面试:
    核心方法: STAR（Situation-Task-Action-Result）
    重点关注:
      - 这个成果"你个人"的贡献有多少？（vs 团队的贡献）
      - 你做过最难的决策是什么？为什么难？
      - 你有没有"搞砸"的经历？你从中学到了什么？
```

### 晋升评估

```yaml
晋升评估流程:

  1. 自评（晋升申请）:
     - 候选人对照能力模型逐条自评
     - 提供每项能力的行为证据（项目链接/文档/同事评价）
     - 自评差距和改进计划
  
  2. 360度评估:
     - 收集3-5名合作者的匿名评价
     - 重点关注：协作中的实际表现（不是自评中的描述）
  
  3. 晋升评审会:
     - 评审委员会: L(N+2)成员 + 跨部门高级成员 + HRBP
     - 候选人15分钟自述 + 15分钟问答
     - 投票: 全票或2/3通过
  
  4. 结果与反馈:
     - 无论通过与否，提供详细的反馈报告
     - 如果未通过：指出差距 + 给出6个月提升计划
```

### 持续更新机制

```yaml
能力模型的版本管理:

  更新频率: 每年Review一次（结合年度技术规划）
  
  更新驱动因素:
    - 技术生态变化（如LLM要求新的能力维度）
    - 公司业务阶段变化（创业期 vs 成长期 vs 成熟期对能力要求不同）
    - 行业实践演进（如DataOps/MLOps成为标配）
  
  更新流程:
    1. 收集反馈（CTO + 架构师 + Tech Lead）
    2. 起草更新（不超过原篇幅的20%）
    3. 团队Review（全员可见，征求反馈）
    4. 版本发布（标注版本号和更新日期）
```
