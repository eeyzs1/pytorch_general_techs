# 技术评审流程设计

> **定位**：设计一套分层分级的、能防止"盖章通过"的技术评审机制。一个好的评审流程应该做到：小需求快速通过，大方案深入讨论，风险点一个不漏。
>
> **核心理念**：评审的目的不是"审批"，而是"集合团队智慧，降低犯错概率"。如果一个评审流程让人觉得是"走形式"，那它就是失败的。

---

## 评审分级机制

### 为什么需要分级？

```
不是所有技术决策都需要CTO拍板。

如果所有决定都向上汇报：
  → CTO成为瓶颈（一个人的带宽是有限的）
  → 团队失去主动性和责任感
  → 决策速度极慢

如果所有决定都下沉：
  → 各自为政，技术栈碎片化
  → 高风险决策没有充分论证
  → 技术债务迅速累积
```

### 四级评审体系

```yaml
L0 级别: 自审（Self-Review）
  适用场景:
    - Bug修复（不涉及架构变更）
    - 小功能迭代（不影响现有数据模型）
    - 配置修改（不改变系统行为边界）
    - 常规数据需求（单表查询/聚合）
  
  决策人: 开发者自己
  要求:
    - 所有代码必须经过Code Review
    - 自认为有风险的点，主动升级评审级别
  
  禁止场景（必须升级至少到L1）:
    - 修改了数据模型（表结构变更）
    - 修改了对外API接口
    - 涉及金钱/用户隐私的数据处理

L1 级别: 同行评审（Peer Review）
  适用场景:
    - 新数据模型设计（单表/ETL任务）
    - 新API接口设计
    - 中等复杂度的数据处理任务
    - 第三方库/工具的引入（非平台级）
  
  责任人: Tech Lead + 1名同组工程师
  评审方式:
    - 设计文档（1-3页）+ 代码Review
    - 异步评审为主（通过文档评论），必要时开会
  评审周期: < 2个工作日
  
  产出物:
    - 评审意见记录（设计文档评论）
    - 通过/修改后通过/不通过 的决定

L2 级别: 技术评审（Technical Review）
  适用场景:
    - 新数据管道架构设计（涉及多个系统）
    - 核心数据模型变更（影响多个下游）
    - 新平台组件引入（如引入新的OLAP引擎）
    - 影响在线服务性能的技术变更
    - 数据迁移方案（全量/增量）
  
  责任人: 架构师 + 相关方Tech Lead
  参与人: 架构师、相关团队Tech Lead、SRE代表、安全代表（如涉及PII数据）
  评审方式:
    - 正式的技术设计文档（5-10页）
    - 必须有评审会议（30-60分钟）
    - 会前材料预读（提前48小时发送）
  评审周期: < 5个工作日
  
  产出物:
    - 评审会议纪要
    - 风险评估矩阵
    - 通过/有条件通过/不通过 的决定
    - 如果是有条件通过：列出条件清单和验证方式

L3 级别: 架构评审（Architecture Review）
  适用场景:
    - 公司级平台架构变更（如数据湖 → 湖仓一体迁移）
    - 影响跨部门用户的技术决策（如更换BI工具）
    - 年度技术路线图
    - 重大技术采购决策（超过50万/年的工具/平台）
    - 涉及数据安全合规的技术方案
  
  责任人: CTO + 架构师委员会
  参与人: CTO、架构师、相关VP、安全负责人、法务（如涉及合规）
  评审方式:
    - 正式架构设计方案（10-20页）+ POC验证报告
    - 必须至少2次评审会议
    - 第1次：方案阐述和质疑（90分钟）
    - 第2次：修订方案和最终决定（60分钟）
    - 必须包含成本分析和ROI预估
  评审周期: < 10个工作日
  
  产出物:
    - 架构决策记录（ADR - Architecture Decision Record）
    - 正式审批签字
    - 项目启动授权（含预算和资源承诺）
```

### 分级决策矩阵

```yaml
如何判断一个方案应该走哪个级别的评审？

使用"影响面×风险×不可逆性"三维矩阵:

影响面（多少人/系统会被影响）:
  1: 1个模块
  2: 1个服务/1个团队
  3: 多个服务/多个团队
  4: 全平台/全公司

风险（搞砸了的后果）:
  1: 轻微不便（可快速回滚）
  2: 影响开发效率（需要额外工作时间来修复）
  3: 影响业务指标（如数据错误导致运营决策偏差）
  4: 影响业务连续性（服务中断/数据丢失/安全事故）

不可逆性（做错了之后能反悔吗）:
  1: 完全可逆（换个版本即可）
  2: 基本可逆（需要数据迁移但可行）
  3: 部分不可逆（数据迁移成本高，部分数据可能丢失）
  4: 完全不可逆（架构选型、供应商选择等）

评分计算:
  总分 = 影响面 × 风险 × 不可逆性
  
  1-8分   → L0 自审
  9-18分  → L1 同行评审
  19-36分 → L2 技术评审
  37-64分 → L3 架构评审
```

## 评审Checklist

### L0 自审Checklist

```yaml
开发者自问:
  [ ] 我的代码是否通过了单元测试？
  [ ] 我的代码是否遵循团队编码规范？
  [ ] 我是否考虑了错误处理（异常场景）？
  [ ] 我是否修改了现有接口/数据模型的签名？（如果是→升级到L1）
  [ ] 我是否引入了新的外部依赖？（如果是→升级到L1）
  [ ] 我的变更对下游消费者是透明的吗？（如果不是→升级到L1）
```

### L1 同行评审Checklist

```yaml
功能正确性:
  [ ] 逻辑是否正确覆盖了所有需求场景？
  [ ] 边界条件是否处理？（空数据/超大数据/异常输入）
  [ ] 是否考虑了并发/幂等性问题？
  [ ] 数据计算的准确性是否可验证？

代码质量:
  [ ] 代码是否清晰易读？（变量命名、函数职责单一）
  [ ] 是否有合理的错误处理和日志？
  [ ] 是否有重复代码可以抽取？
  [ ] 是否引入了不必要的复杂度？

数据设计:
  [ ] 表结构设计是否合理？（主键/索引/分区策略）
  [ ] 字段类型和长度是否恰当？
  [ ] 是否有数据保留策略（TTL/归档）？
  [ ] 是否考虑了数据血缘（数据从哪里来、流向哪里）？

性能:
  [ ] 对大数据量的场景，是否使用了合理的分区裁剪？
  [ ] 是否存在全表扫描的风险？
  [ ] JOIN策略是否合理（Broadcast vs SortMerge）？

安全:
  [ ] 是否涉及用户隐私数据（PII）？（特殊处理要求）
  [ ] 是否有SQL注入的风险？
  [ ] 敏感信息（密钥/密码）是否硬编码？
```

### L2 技术评审Checklist

```yaml
架构设计:
  [ ] 架构图是否清晰？（组件、数据流、接口）
  [ ] 是否有2个以上的备选方案对比？
  [ ] 方案选择的理由是否基于数据和事实？
  [ ] 是否考虑了未来12个月的数据量增长？

系统影响:
  [ ] 对现有系统的兼容性影响是否评估？
  [ ] 是否有明确的回滚方案？
  [ ] 上线和回滚的SOP是否完备？
  [ ] 对下游消费者的影响是否通知和协商？

性能与容量:
  [ ] 是否进行了容量估算？（数据量/QPS/存储）
  [ ] 是否有POC验证关键性能假设？
  [ ] 瓶颈在哪里？如果瓶颈被打满，降级方案是什么？
  [ ] 监控指标和告警阈值是否定义？

稳定性:
  [ ] 故障模式是否分析？（如果XX组件挂了会怎样？）
  [ ] 是否有重试/熔断/降级机制？
  [ ] 数据一致性如何保证？（Exactly-Once / At-Least-Once？）
  [ ] 是否有数据质量校验机制？

运维:
  [ ] 部署方案是否清晰？（如何上线、如何灰度）
  [ ] 运维手册是否完备？（监控、告警、常见故障处理）
  [ ] 是否需要新的运维工具或组件？
  [ ] 对现有运维流程的影响？

成本:
  [ ] 新增基础设施成本是否估算？
  [ ] 新增人力维护成本是否估算？
  [ ] 是否有优化成本的措施？（如冷热分层）
  [ ] ROI是否合理？

安全与合规:
  [ ] 数据加密（传输+存储）是否覆盖？
  [ ] 访问控制和审计是否完备？
  [ ] 是否符合GDPR/等保等合规要求？
  [ ] PII数据是否做了脱敏处理？
```

### L3 架构评审Checklist

```yaml
战略对齐:
  [ ] 方案是否与公司技术战略和路线图一致？
  [ ] 如果是技术战略的调整 → 是否经过了充分的论证？
  [ ] 对竞争格局的影响是否评估？

组织影响:
  [ ] 对团队组织架构的改变需求？
  [ ] 团队技能是否匹配？需要招聘/培训哪些人？
  [ ] 对现有工作流程的改变和过渡期管理？

财务影响:
  [ ] 完整TCO（Total Cost of Ownership）3年投影？
  [ ] ROI计算是否保守合理？
  [ ] 对预算的影响和资金来源？
  [ ] 如果不做，机会成本是什么？

风险评估:
  [ ] Top 5风险及应对预案？
  [ ] 最大的不确定性是什么？如何降低不确定性？
  [ ] 如果项目失败，Plan B是什么？
  [ ] 对业务连续性的最大威胁是什么？

实施路线:
  [ ] 分阶段实施计划是否合理？
  [ ] 每个阶段的验收标准是否明确？
  [ ] 是否有"止血点"（如果某个里程碑未达成，停止投入）？

长期视角:
  [ ] 3年后这个方案是否仍然是最优的？
  [ ] 技术演进方向是否与方案兼容？
  [ ] 供应商锁定风险评估？
  [ ] 迁移/退出的成本和路径？
```

## 防止"盖章通过"的机制

### 反模式识别与应对

```yaml
反模式1: "评审就是走形式"
  症状:
    - 评审会议快速结束（<10分钟就通过）
    - 评论全是"LGTM"或"+1"
    - 评审后几乎不需要修改
    - 但上线后频繁出问题
  
  根因:
    - 评审者不够认真（觉得跟自己无关）
    - 评审者能力不足（看不出问题）
    - 时间压力（被要求"快点通过"）
    - 权力不对等（评审者不敢质疑高级工程师）
  
  解决方案:
    1. 设立"魔鬼代言人"角色（每场评审指定1人专门找问题）
    2. 评审时间强制要求（L2评审不少于30分钟，L3不少于60分钟）
    3. 匿名反馈通道（如果评审者感觉压力，可以匿名提交意见）
    4. 事后复盘（如果评审通过的项目出了问题，组织"评审为什么没发现"的复盘）

反模式2: "设计文档没人认真读"
  症状:
    - 评审会议上大家才第一次看文档
    - 提出的问题文档里已经写了
    - 方案做完了才发现评审时有误解
  
  解决方案:
    1. 强制预读（评审会议前48小时必须提交评论）
    2. 没有预读评论 = 不能参加评审会议
    3. 使用文档评论系统（而非口头讨论），留下书面记录

反模式3: "评审只看技术，不看业务"
  症状:
    - 技术方案很完善，但没人问"这解决了什么业务问题？"
    - 评审通过了但业务方表示"这不是我想要的"
  
  解决方案:
    1. L2+评审必须有业务方代表（产品经理/业务负责人）参与
    2. 设计文档第一页必须是"业务问题描述"和"成功标准"
    3. 方案通过后，业务方代表签字确认

反模式4: "CTO一言堂"
  症状:
    - CTO在评审中表达了倾向，其他人不再表达不同意见
    - 所有L3评审的结论都跟CTO的初始观点一致
  
  解决方案:
    1. CTO在评审中最后一个发言
    2. 采用"匿名投票"先行，再公开讨论
    3. 如果连续5次评审全票通过 → 检查是否有"一言堂"问题
```

### 评审质量评估

```yaml
每季度进行一次评审质量回顾:

关键指标:
  - 评审通过率: 目标70-85%（太高=标准低，太低=流程阻碍进展）
  - 评审后修改次数: 中位数应在1-3次
  - "评审后发现问题"率: 目标<5%（上线后发现的重大问题中有多少是评审应该发现的）
  - 评审周期达标率: 各等级评审是否在规定时间内完成

定性评估:
  - 随机抽查5份评审记录，评估评审质量:
    * 评审意见是否深入技术细节？
    * 是否有建设性的替代方案建议？
    * 是否覆盖了所有Checklist维度？
```

## 架构决策记录（ADR）

### 什么是ADR？

ADR（Architecture Decision Record）是一种轻量级的架构决策文档格式，记录每一次重要的技术决策及其上下文。

### ADR模板

```markdown
# ADR-XXXX: [决策标题]

## 状态
[提议中 / 已接受 / 已废弃 / 已替代]

## 背景
描述需要做这个决策的技术背景和业务上下文。
回答：为什么现在需要做这个决定？

## 决策
描述我们做了什么决定。
回答：我们选择了什么方案？

## 后果
描述应用这个决策后会产生的影响——好的、坏的、中性的。
回答：这个决定让什么变得更容易？什么变得更难？

## 备选方案
列出我们考虑过但最终没有选择的方案，以及不选的原因。

### 方案B: [备选方案名称]
- 优点: ...
- 为什么不选: ...

### 方案C: [备选方案名称]
- 优点: ...
- 为什么不选: ...

## 相关
- 相关ADR: ADR-YYYY
- 相关设计文档: [链接]
- 相关JIRA: [链接]
```

### ADR示例

```markdown
# ADR-0025: 选择Apache Iceberg作为湖仓表格式

## 状态
已接受（2025-03-15）

## 背景
当前我们使用Hive Metastore管理所有数据表。随着业务发展遇到以下问题：
1. CDC场景需要Upsert，Hive ACID性能不佳且运维复杂
2. 分区策略需手动维护，每月浪费约30工时
3. 多引擎（Spark/Trino/Flink）查询同一份数据时需要各自注册表，数据不一致

业务方（增长团队）要求用户标签更新从T+1缩短到T+15分钟，这在当前Hive方案下无法实现。

## 决策
我们决定采用Apache Iceberg作为新的湖仓表格式，通过以下路径逐步迁移：
1. Phase 1 (Q2): 新建表默认使用Iceberg
2. Phase 2 (Q3): 将核心表（用户标签/订单宽表）迁移到Iceberg
3. Phase 3 (Q4): 逐步将剩余Hive表迁移

## 后果
正面影响:
- Upsert延迟从T+1小时降低到实时（分钟级）
- 分区管理从手动变为自动（Hidden Partition），预期每月节省30工时
- 多引擎视图一致（Spark/Trino/Flink共用同一份元数据）
- 支持Time Travel（数据回滚能力）

负面影响:
- 增加了一个新组件（Iceberg REST Catalog），需要运维
- 团队需要学习新工具（预估2周学习曲线）
- 迁移期间需要双写（Hive+Iceberg），增加临时复杂度

## 备选方案

### 方案B: Apache Hudi
- 优点: Upsert性能更好，增量查询功能更强
- 为什么不选: 
  * 组件更多（需要额外部署Timeline Server），运维复杂度更高
  * 我们的核心场景是分析查询而非超低延迟写入，Hudi的写入优化不是我们的刚需
  * 社区生态（Trino/Flink集成）不如Iceberg成熟

### 方案C: Delta Lake
- 优点: 与Spark生态最深度集成
- 为什么不选:
  * 对Trino/Flink的集成不如Iceberg（Databricks生态优先级）
  * 开源版的某些功能受限（如Z-Ordering的某些优化）
  * 与Databricks商业版有潜在生态绑定风险

## 相关
- 相关ADR: ADR-0023（Kafka升级到3.x）
- 设计文档: [Iceberg迁移方案设计文档](link)
- POC报告: [Iceberg vs Hudi vs Delta Lake对比测试](link)
```

### 评审会议规范

```yaml
会议前（评审人）:
  - 提前48小时收到设计文档
  - 提前24小时提交书面评论（在设计文档的评论区域）
  - 如果没有提前提交评论，主持人有权拒绝参加

会议中:
  角色分配:
    - 主持人（Facilitator）: 控制时间，确保讨论不跑题
    - 方案陈述人（Presenter）: 15-20分钟方案介绍
    - 魔鬼代言人: 专门挑刺，挑战每一个假设
    - 记录人（Scribe）: 记录决定、行动项、未解决问题
  
  时间分配（以60分钟会议为例）:
    0-5min:  主持人说明目标和规则
    5-20min: 方案陈述（Presenter）
    20-50min: 讨论与质疑（全体）
    50-55min: 总结与决定（主持人）
    55-60min: 确认行动项（记录人）
  
  讨论原则:
    - 先聚焦"做不做对的事"（方向），再讨论"怎么做好"（细节）
    - 鼓励"愚蠢问题"（有时候最基础的假设是错的）
    - 禁止人身攻击（针对方案不针对人）
    - 如果15分钟内无法达成一致 → 回到数据，做POC验证

会议后:
  - 24小时内发出会议纪要
  - 明确行动项和责任人
  - 如果需要修改方案，约定下次评审时间
```

## 特殊场景的评审机制

### 紧急变更评审

```yaml
适用场景:
  - 生产环境P0/P1事故的修复方案
  - 安全漏洞的紧急修复
  - 因外部依赖（如云服务商变更）导致的紧急适配

加急评审流程:
  1. 发起人同步CTO/架构师（即时通讯，不是邮件）
  2. 如果是已知风险 → 按预案执行
  3. 如果是未知风险 → 30分钟紧急评审会
     - 参会人: CTO/架构师 + 事故负责人 + SRE
     - 讨论: "快速修复 + 后续根治"两步方案
     - 决定: Go/No-Go
  4. 事后必须在24小时内补交正式设计文档
  5. 事后必须在72小时内完成正式评审（回顾决策是否正确）
```

### 技术采购评审

```yaml
适用场景:
  - 购买/订阅商业化技术产品（Databricks/Snowflake/Confluent等）
  - 签署超过30万/年的技术服务合同

额外评审维度:
  - [ ] 供应商尽调（公司规模、财务状况、客户口碑）
  - [ ] 许可证审查（法务参与）
  - [ ] 数据安全审查（数据是否出境？谁有权访问？）
  - [ ] SLA与赔偿条款审查
  - [ ] 退出策略（如果终止合作，数据如何迁移？）
  - [ ] 与开源替代方案的TCO对比
```

### 技术债务修复评审

```yaml
特殊考虑:
  技术债务修复的评审标准应适当放低——因为:
    - 风险更低（不改架构，只是改进现有实现）
    - 收益更明确（减少已知的痛苦点）
  
  简化后的L2评审Checklist:
    [ ] 修复是否解决了根本问题？（不只是打补丁）
    [ ] 修复是否会引入新问题？
    [ ] 修复的测试是否充分？（回归测试覆盖）
    [ ] 修复的时间窗口是否合理？（避开业务高峰期）
    [ ] 回滚方案（如果修复后有新问题，能否快速回退？）
```
