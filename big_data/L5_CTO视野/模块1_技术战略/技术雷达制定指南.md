# 技术雷达制定指南

> **定位**：技术雷达是CTO用来管理技术组合、沟通技术方向的核心工具。一份好的技术雷达能让团队知道"我们应该用什么、应该学什么、应该放弃什么"，避免技术选型的混乱和随意。
>
> **核心理念**：技术雷达不是一次性的文档，而是需要持续更新的"活文档"。它的价值不在"写出来"，而在"讨论过程"和"持续更新"。

---

## 技术雷达基础

### 什么是技术雷达？

技术雷达（Technology Radar）是由ThoughtWorks首创的技术管理框架，将技术分为四个象限和四个环，用于可视化团队的技术组合状态和方向。

### 四个象限

```yaml
技术（Techniques）:
  描述: 软件开发的方法论和实践
  数据领域的示例:
    - 数据建模方法论（维度建模 vs Data Vault vs One Big Table）
    - ETL设计模式（ELT vs ETL、CDC vs 批量、Lambda vs Kappa）
    - 测试策略（数据质量测试、管道测试、Data Contract）
    - 部署策略（Blue-Green、Canary、Feature Flag for Data Pipeline）
    - 工程实践（DataOps、GitOps for Data、MLOps）
  
  关键问题: "我们做事的方式对吗？有没有更好的方法？"

工具（Tools）:
  描述: 开发、测试、部署、运维中使用的软件工具
  数据领域的示例:
    - 调度工具：Airflow / DolphinScheduler / Prefect / Dagster
    - 查询引擎：Trino / Presto / Drill / Impala
    - 开发工具：dbt / Dataform / SQLFluff
    - 可视化：Superset / Metabase / Lightdash / Grafana
    - IDE集成：Jupyter / Hex / Deepnote
  
  关键问题: "这个工具帮我们省时间了吗？还是增加了维护负担？"

平台（Platforms）:
  描述: 基础设施和运行平台
  数据领域的示例:
    - 计算平台：Spark / Flink / Ray
    - 存储平台：HDFS / S3 / MinIO / JuiceFS / Alluxio
    - 数据湖仓：Databricks / Snowflake / Starburst / Dremio
    - 编排平台：Kubernetes / YARN / HashiCorp Nomad
    - 云平台：AWS EMR /阿里云 MaxCompute /GCP BigQuery
  
  关键问题: "我们现在依赖的平台是否限制了我们的能力？有没有更好的选择？"

语言与框架（Languages & Frameworks）:
  描述: 编程语言、框架、库
  数据领域的示例:
    - 数据处理：PySpark / Flink SQL / Kafka Streams
    - 数据格式：Parquet / Avro / ORC / Arrow
    - 序列化：Protocol Buffers / FlatBuffers / MessagePack
    - ML框架：PyTorch / TensorFlow / JAX
    - 流处理DSL：KsqlDB / Flink SQL / Spark Structured Streaming
  
  关键问题: "我们是否在用最适合当前场景的语言和框架？"
```

### 四个环

```yaml
内环——采纳（Adopt）:
  含义: 强烈推荐，应该在合适场景中默认使用
  进入标准:
    - 已在公司生产环境验证，表现良好
    - 有明确的最佳实践和内部文档
    - 大部分团队成员已经熟练使用
    - 社区成熟，有商业支持或活跃的开源社区
  数据领域示例:
    - Apache Spark（批处理事实标准）
    - Apache Kafka（消息队列事实标准）
    - Parquet（列式存储格式事实标准）
    - Airbyte/Fivetran（数据集成工具）
  管理动作: 所有新技术项目默认选择，除非有明确理由不选

第二环——试验（Trial）:
  含义: 值得追求，应该在有限范围内试用
  进入标准:
    - 已在行业中有成熟案例（至少5个知名公司使用）
    - 完成内部POC验证
    - 有1-2个内部项目正在使用，数据良好
    - 风险和收益都经过评估
  数据领域示例:
    - Apache Iceberg（湖仓格式，正在从Hive迁移）
    - Trino（交互式查询，正在替代部分Spark SQL场景）
    - dbt（数据转换，正在试点使用的数据工程团队）
  管理动作: 鼓励在非关键路径项目中使用，要求在POC结束后提交评估报告

第三环——评估（Assess）:
  含义: 值得探索，了解它能如何影响我们的实践
  进入标准:
    - 技术有明确的价值主张，与我们当前或未来的痛点匹配
    - 行业内开始有讨论（技术博客/大会演讲/早期采用者分享）
    - 尚未在公司内部使用，或仅在个人Sandbox试验
    - 技术风险尚不明确
  数据领域示例:
    - DuckDB（嵌入式计算引擎）
    - RisingWave（Rust实现的流数据库）
    - OpenTofu（Terraform开源替代）
    - DataHub（数据目录/数据发现平台）
  管理动作: 指定专人跟踪，每季度汇报进展和推荐变化

外环——暂缓（Hold）:
  含义: 谨慎对待，目前不推荐使用
  进入标准（满足任一条即可）:
    - 曾被采纳但出现重大问题（性能/稳定性/维护成本）
    - 技术方向与公司战略不符
    - 技术已经过时，有更好的替代方案
    - 曾经试用但ROI不达预期
    - 社区已不活跃或维护者已放弃
  数据领域示例:
    - MapReduce（已被Spark替代）
    - Storm（已被Flink替代）
    - Pig/Hive on MR（已被Spark SQL替代）
    - 自研工作流引擎（应迁移到Airflow等成熟方案）
    - Hadoop 2.x（EOL，应升级到3.x或迁移到云原生方案）
  管理动作: 在新项目中禁止使用，制定现有系统的迁移计划
```

## 技术雷达制定流程

### 第一步：组建雷达委员会

```yaml
委员会构成:
  
  主席: CTO（最终决策权）
  
  常任委员:
    - 架构师 × 2（提供技术深度视角）
    - Tech Lead × 2（提供一线实践视角）
    - 数据平台负责人（提供平台视角）
  
  轮值委员（每次会议邀请1-2人）:
    - 轮值目的: 防止委员会成为"小圈子"，引入新鲜视角
    - 人选: 最近有新项目上线的工程师、对外参加过大会讲演的人

委员会职责:
  - 每季度评审和更新技术雷达
  - 对新申请进入"采纳"环的技术进行评审
  - 对连续2个季度处于"评估"环未推进的技术进行"淘汰或推进"决策
```

### 第二步：技术清单盘点

```yaml
盘点模板（每季度填写一次）:

技术名称: Apache Iceberg
当前环: 试验（Trial）
所属象限: 平台

上次评审时间: 2025-Q1
上次评审决定: 从评估→试验

评估维度:

1. 业务价值 (1-5分): 4
   - 解决了什么业务问题?
     * 支持实时Upsert，使标签更新从T+1→T+15分钟
     * 时间旅行功能支持数据回滚，降低了数据事故的影响面
   - 不能解决什么?
     * 不解决查询性能问题（这部分由Trino负责）

2. 技术成熟度 (1-5分): 4
   - 社区活跃度: GitHub 7k+ Stars，Netflix/Apple/Airbnb在大规模使用
   - 生产案例: 内部已有1个项目在使用（用户标签宽表），运行3个月0故障
   - 商业支持: Tabular(Dremio)、AWS、Cloudera提供商业支持
   - 版本稳定性: 2.0版本API已稳定，无破坏性变更

3. 组织适配度 (1-5分): 3
   - 团队学习成本: 与Hive概念相似，需要约2周上手
   - 现有技术栈兼容性: 需要引入REST Catalog，增加1个运维组件
   - 团队意愿: 数据平台组积极，业务组相对保守（担心稳定性）

4. 趋势判断 (前进/稳定/衰退): 前进
   - 行业趋势: 湖仓格式之争正在收敛，Iceberg和Delta Lake两家独大
   - 人才供给: 市场上Iceberg技能者逐渐增多，招聘难度中等
   - 供应商锁定风险: 低（开源+开放格式）

综合推荐:
  - [ ] 晋升到采纳（Adopt）
  - [x] 保持试验（Trial）
  - [ ] 降级到评估（Assess）
  - [ ] 移入暂缓（Hold）

决策理由:
  在试验期内表现良好，但使用者仍少（仅1个项目），建议再观察1个季度，
  鼓励更多项目使用，Q3评审时如果使用项目≥3个且无事故→晋升到采纳。

下一步行动:
  - Q2目标: 至少再有2个项目使用Iceberg
  - 需要完成: Iceberg运维手册（监控/备份/恢复/故障处理）
  - 需要完成: 团队内部Iceberg全员培训
```

### 第三步：季度评审会议

```yaml
会议流程（2小时）:

0. 会前准备（提前1周）
  - 各委员提交各自关注的"新技术提案"（使用上述盘点模板）
  - 秘书汇总所有提案，提前发给全体委员预读

1. 回顾环节（15min）
  - 快速过一遍上次会议的决定
  - 跟踪上次设定的"下一步行动"完成情况
  - 做一个"满意/不满意"快速投票

2. 争议项讨论（60min）
  - 重点讨论:
    * 推荐"晋升"或"降级"的技术
    * 连续2个季度在"评估"环未推进的技术（做出推进或放弃决定）
    * 委员会成员有不同意见的技术
  - 讨论规则:
    * 每个讨论项限时15分钟
    * 赞成方先陈述（3分钟）→ 反对方回应（3分钟）→ 自由讨论（7分钟）→ 投票（2分钟）

3. 新提名讨论（30min）
  - 讨论新进入视野的技术
  - 决定放入"评估"环还是直接"暂缓"
  - 指定跟踪负责人

4. 雷达可视化更新（15min）
  - 在雷达图上更新各技术的位置
  - 生成《Qx技术雷达》PDF版本
  - 讨论"雷达发布后，怎么让团队真正读和用？"
```

### 第四步：技术雷达发布与推广

```yaml
发布策略:
  
  内部发布:
    渠道:
      - 公司Wiki/Confluence（技术雷达专区）
      - 团队Slack/钉钉频道（#tech-radar）
      - 全员邮件（每季度1次）
      - 团队周会展示（5分钟快速过一遍变化）
    
    内容:
      - 《Qx技术雷达》PDF（3-5页）
      - 附带"本季度关键变化"的1页Summary
      - 附带"如果你只读1分钟"的TL;DR
      
    互动设计:
      - 设置"挑战雷达"环节：任何人可以提交对某个技术环位置的异议
      - 在雷达PDF上标注"下次评审（YYYY-MM）"，鼓励提前提交意见
      - 每个技术的"联系人"标注在雷达上，方便想了解细节的人咨询

  对外发布（可选，用于技术品牌建设）:
    适用场景:
      - 公司有技术品牌建设需求
      - 行业内有足够的技术影响力（否则可能被批评"不自量力"）
    
    注意事项:
      - 匿名化或脱敏公司特定的上下文
      - 标注"仅代表本团队观点，不代表行业标准"
      - 准备好接受外部的质疑和挑战
```

## 技术评估维度深度指南

### 维度一：业务适配度评估

```yaml
评估框架: 问题-方案匹配矩阵

步骤1: 明确业务问题
  不要从技术出发，从业务痛点出发:
    ✗ "我们想用Iceberg" 
    ✓ "我们需要支持实时Upsert，因为标签更新延迟T+1导致推荐效果下降15%"

步骤2: 技术方案对比
  
  | 评估维度 | 方案A: Iceberg | 方案B: Hudi | 方案C: 保持Hive+手动分区 | 
  |----------|---------------|-------------|-------------------------|
  | 解决核心痛点的程度 | 完美解决 | 完美解决 | 不解决 |
  | 引入的额外复杂度 | 中（新增REST Catalog） | 高（组件更多） | 无 |
  | 学习成本 | 2周 | 3周 | 0 |
  | 运维成本变化 | +1个组件 | +2个组件 | 不变 |
  | 社区健康度 | 很好 | 很好 | 稳定（但无创新） |
  | 3年前景 | 上升 | 上升 | 下降 |
  
  注意: 一定要包含"保持现状"作为对照方案！不做事也是一个选择。

步骤3: 决策矩阵（加权评分）
  
  | 维度 | 权重 | 方案A | 方案B | 方案C |
  |------|------|-------|-------|-------|
  | 解决痛点的程度 | 40% | 5 | 5 | 0 |
  | 实施复杂度 | 20% | 3 | 2 | 5 |
  | 长期可维护性 | 25% | 4 | 4 | 2 |
  | 社区/人才 | 15% | 4 | 4 | 3 |
  | 加权总分 | 100% | 4.3 | 4.15 | 1.75 |
  
  最终决定: 选择方案A（Iceberg）
  
  重要提示: 权重不是固定的！根据当前阶段调整：
  - 创业早期: "快速上线"权重最高
  - 快速增长期: "可扩展性"权重最高
  - 稳定运营期: "可维护性"权重最高
```

### 维度二：技术风险评估

```yaml
技术风险分类:

1. 技术风险（Technology Risk）:
   - 该技术是否足够成熟？
     * 评估方法: 查看GitHub Release版本号、已知的生产事故报告
     * 红牌: 还在0.x版本 / 最近6个月有2次以上数据丢失的Issue
     * 黄牌: 1.x版本但API不稳定 / 缺少大规模生产案例
     * 绿牌: 2.x+版本 / 有多家知名公司背书
  
   - 性能是否达标？
     * 评估方法: POC验证（必须有，不能仅靠文档和博客）
     * 红牌: POC结果比声称的性能差50%以上
     * 黄牌: POC结果比声称的性能差10-30%
     * 绿牌: POC结果符合或超过预期
  
   - 与现有系统的兼容性？
     * 评估方法: 列出所有相互依赖的系统，逐一检查
     * 红牌: 需要升级或修改核心依赖系统的版本
     * 黄牌: 需要添加适配层
     * 绿牌: 即插即用

2. 供应商/社区风险（Vendor/Community Risk）:
   - 该技术有"单点依赖"吗？（只有一个公司维护？）
     * 评估方法: 查看GitHub贡献者分布
     * 红牌: 80%+的代码由单一公司贡献
     * 黄牌: 50-80%的代码由单一公司贡献
     * 绿牌: 多家公司积极参与
  
   - 许可证风险:
     * 红牌: AGPL/SSPL（可能限制商业使用）
     * 黄牌: GPL（需要评估使用方式是否触发传染条款）
     * 绿牌: Apache 2.0/MIT/BSD
  
   - 社区活跃度趋势:
     * 评估方法: GitHub Insights查看Star数趋势、Issue关闭率、PR合入速度
     * 红牌: Star半年未增长 / Issue平均关闭时间>3个月
     * 黄牌: 增速放缓 / 关键维护者最近离职
     * 绿牌: 增速健康 / 有多名核心维护者

3. 组织风险（Organizational Risk）:
   - 团队意愿:
     * 评估方法: 匿名问卷或团队讨论
     * 红牌: 多数人反对且理由充分
     * 黄牌: 有人担心但可以沟通解决
     * 绿牌: 大多数人愿意尝试
  
   - Bus Factor:
     * 定义: 最少需要几个人离职，这个技术在公司内部就会"失传"
     * 红牌: Bus Factor = 1
     * 黄牌: Bus Factor = 2
     * 绿牌: Bus Factor ≥ 3
  
   - 迁移回滚成本:
     * 定义: 如果用了1年后发现不合适，需要多少成本退回原方案
     * 红牌: 数据格式不兼容，回滚需要全量数据迁移（>1个月）
     * 黄牌: 需要部分数据迁移或适配层重构（1-2周）
     * 绿牌: 可以平滑回退（双写方案或在接口层切换）

风险评估总结模板:

| 技术 | 技术风险 | 供应商风险 | 组织风险 | 综合风险等级 | 是否推荐 |
|------|---------|-----------|---------|------------|---------|
| Iceberg | 低 | 低 | 中(团队学习意愿) | 中低 | 推荐引入 |
| Flink 2.0 | 中(新版本) | 低 | 低 | 中 | 等6个月 |
| RisingWave | 中高(太新) | 高(单一公司) | - | 高 | 暂不引入 |
```

### 维度三：ROI预估

```yaml
技术投入的ROI计算方法:

直接成本:
  - 人力成本: 预计投入人天 × 日薪
    * 例: 迁移到Iceberg需要2人 × 60天 = 120人天
    * 日薪按1000元算：人力成本 = 12万元
  
  - 基础设施成本变化:
    * 新增组件：REST Catalog服务器 × 2（高可用）= 2台虚拟机 × 500元/月 = 1000元/月
    * 存储变化：Iceberg元数据额外占用约5%存储 = 对于100TB数据约5TB = 500元/月
    * 合计增加：1500元/月 = 1.8万元/年
  
  - 许可证/订阅成本:
    * 如果使用Tabular商业版: $5000/月 = 约3.6万/月 = 43.2万/年
    * 如果使用开源版: 0

直接收益:
  - 效率提升:
    * 消除手动分区管理: 每月节省30工时 = 360工时/年 ≈ 1.8人月
    * 价值: 1.8人月 × 2万元/人月 = 3.6万元/年
  
  - 业务价值:
    * 标签更新从T+1到T+15分钟，推荐系统效果提升15%
    * 如果推荐系统贡献年GMV 5亿 × 15% = 7500万增量GMV
    * 技术贡献占比: 技术贡献了效果提升的40%（其余是算法优化）= 3000万GMV
    * 按10%利润率算: 300万利润增量
    * 保守估计: 取上述值的20%作为"可量化的技术贡献" = 60万/年

ROI计算:
  第1年: 收益(3.6+60万) - 成本(12+1.8万) = 49.8万
  ROI = 49.8 / 13.8 = 361%

注意: 
  业务价值的量化是CTO最有挑战也最重要的工作。上面的计算方法只是一个框架，
  实际中需要与业务方合作"认领"贡献比例。关键原则:
  1. 宁可低估，不要高估（建立可信度）
  2. 永远拉上业务方一起算（避免"自说自话"）
  3. 保守估计 + 灵敏度分析（给出"乐观/中性/悲观"三档预估）
```

### 维度四：时机判断

```yaml
时机判断框架——"太早、刚好、太晚":

太早（Too Early）的信号:
  - 技术还在0.x版本，API每个版本都在变
  - 缺乏生产环境案例（只有Demo和Blog Post）
  - 关键插件/集成还不存在（比如没有你需要的Connector）
  - 社区讨论集中在"它能做什么"而不是"怎么在生产中用"
  
  行动: 放入"评估"环，指定跟踪人

刚好（Right Time）的信号:
  - 1.x版已发布，API基本稳定
  - 有3-5个知名公司的生产案例分享（含踩坑经验）
  - 关键生态组件可用（Connector/Driver/Monitoring）
  - 社区讨论开始转向"最佳实践"和"性能调优"
  
  行动: 放入"试验"环，启动内部POC

太晚（Too Late）的信号:
  - 行业标准已经形成（如Spark之于批处理）
  - 不引入已经影响招聘（候选人问"你们为什么不用XX？"）
  - 竞品都已采用并展示了显著优势
  - 现有方案已经无法维护（原维护者离职/社区停更）
  
  行动: 尽快纳入"采纳"环，制定追赶计划

"过时"的信号:
  - 官方宣布EOL（End of Life）
  - 社区主要贡献者转投竞品
  - 大企业用户公开宣布迁移计划
  - 不再有新的生产环境部署案例
  
  行动: 放入"暂缓"环，制定迁移计划
```

### 案例1：某电商公司的数据技术雷达（2025 Q2）

```yaml
技术象限:

    - Airbyte（数据集成）: 替代了之前自研的12个定制化连接器

    - Apache Iceberg（湖仓格式）: 3个核心表已迁移，等待更大规模验证

    - DuckDB: 可能用于数据应用端的嵌入分析
    - Dagster（数据编排）: Airflow的潜在替代方案
    - Text-to-SQL（LLM辅助查询）: 评估GPT-4生成生产SQL的准确性
  
  暂缓（Hold）:
    - Hive on MR: 已全面迁移到Spark SQL
    - Azkaban（调度器）: 已迁移到Airflow
    - Python UDF（Spark场景）: 建议用Pandas UDF或PySpark内置函数替代
    - Zookeeper: 逐步迁移到Kafka KRaft模式

平台象限:

  采纳（Adopt）:
    - Kubernetes: 所有新服务默认部署平台
    - Kafka: 消息队列统一标准
    - AWS S3: 数据湖主存储
  
  试验（Trial）:
    - Spark on K8s: 性能与YARN相当，运维复杂度更低
    - Argo Workflows: 正在替代部分Airflow的数据管道编排场景
  
  评估（Assess）:
    - Ray: 用于ML训练管道的统一计算框架
    - RisingWave: 关注其在流处理领域的进展
  
  暂缓（Hold）:
    - YARN: 计划1年内完成到K8s的迁移
    - Hadoop HDFS: 数据主存储已迁移到S3，HDFS仅保留过渡
    - Cloudera CDH: 已迁移到开源自建方案
```

### 案例2：决策失败后的复盘

```yaml
背景: 某公司在2019年决定采用HBase作为实时特征存储。到2022年，发现：
  - HBase运维成本高（需要HDFS + Zookeeper + HBase × 3套集群）
  - 性能不如预期（读写延迟不如Redis + Cassandra组合）
  - 社区活跃度下降（核心贡献者转向其他项目）

复盘——用技术雷达框架分析"我们当初为什么错了":

  2019年决策时的雷达评估回顾（应该是什么样子）:
    技术风险: 中（需要HDFS+ZK基础设施，运维复杂度高）→ 未充分评估
    供应商风险: 低（Apache项目）
    组织风险: 高（团队无人有HBase生产运维经验）→ 忽略了
    ROI: 未量化计算 → 我们当时没有做
    
  根因分析:
    1. 我们被"大数据公司都在用HBase"的叙事影响，而非自身的痛点匹配
    2. 没有做POC验证，仅凭技术博客和大会分享做决策
    3. 忽略了运维成本（这是CTO的失职）
    4. 没有设立"回滚触发条件"——用了2年才发现不合适，太晚了
  
  吸取的教训（更新到雷达制定流程中）:
    1. 任何进入"试验"环的技术，必须有POC报告（不能仅靠外部资料）
    2. 平台级技术（如存储引擎）的评估，运维成本权重提升到30%
    3. 每季度检查一次"暂缓"环，主动寻找需要"扔掉"的技术
    4. 设定"自动降级"机制：连续2季度在"评估"环无进展 → 移入"暂缓"
```

## 技术雷达的反模式

```yaml
反模式1: "雷达只是挂在墙上的装饰" 
  症状: 雷达更新了但没人看，团队继续按自己的喜好选技术
  解决: 把雷达与项目启动流程绑定——每个新项目必须标注"使用了哪些雷达上的技术及理由"
  
反模式2: "CTO一个人定的雷达"
  症状: 雷达体现了CTO的个人偏好，没有团队共识
  解决: 必须有多人参与的评审流程，公开每个人的投票和理由

反模式3: "为了漂亮而塞满"
  症状: 每个象限塞了几十个技术项，眼花缭乱无人细读
  解决: 采纳+试验环合计不超过15个项目，评估环不超过10个

反模式4: "只加不减"
  症状: 每次会议只加新技术进"评估"，从不把技术移入"暂缓"
  解决: 设立"一进一出"规则——每新增1个评估项，必须淘汰1个暂缓项

反模式5: "雷达是技术选型的唯一依据"
  症状: 不看雷达的不审批，看了雷达就不思考
  解决: 雷达是"建议"而非"规则"，任何项目都可以挑战雷达的推荐，但需要写书面理由

反模式6: "雷达就是工具清单"
  症状: 只列技术和工具，忽视方法论（技术象限）的引导
  解决: 技术象限中至少包含Code Review/测试策略/部署流程等方法论条目
```

## CTO使用技术雷达的沟通技巧

```yaml
对工程师团队:
  沟通要点:
    - "这是我们集体智慧的结晶"（不是CTO个人意志）
    - "如果你不同意某个位置，欢迎挑战——需要数据和逻辑"
    - "使用雷达上'采纳'的技术有'免审批权'"（激励机制）
  避免:
    - "CTO定了这个，你们照着做"（扼杀创新和主动性）

对CEO/董事会:
  沟通要点:
    - "我们用这个工具来管理技术投资组合，就像你管理投资项目组合"
    - "每个季度我们都会审视一遍，确保技术在为业务服务"
    - "这是'活文档'，如果业务方向变了，技术方向也会跟着调整"
  避免:
    - 展示雷达图本身（太技术化）→ 展示"因为技术雷达我们省了/赚了多少钱"

对外（招聘/技术品牌）:
  沟通要点:
    - "这是我们团队的技术品味和方向"（吸引志同道合的人）
    - "我们不只是使用技术，我们主动选择和演进技术"
  避免:
    - 对外发布过于激进的观点（如"XX技术已死"）
    - 标明"暂缓"的技术如果说得太绝对，可能得罪社区
```
